`pgmq-migration` `0.4.0.1` vendors PGMQ `1.11.0` as a native `pg-migrate` component. It installs
the schema without requiring the PostgreSQL extension.

## Public component

```haskell
module Pgmq.Migration
  ( DefinitionError
  , MigrationComponent
  , pgmqMigrations
  )

pgmqMigrations :: Either DefinitionError MigrationComponent
```

The component identity is `pgmq`, with no component dependencies. Its manifest contains:

| Migration | Purpose |
| --- | --- |
| `pgmq/0001-install-v1.11.0` | exact vendored PGMQ 1.11 schema baseline |
| `pgmq/0002-schema-management-comment` | non-destructive native-runner handoff canary |

The package exports no standalone runner and no `migrate`, `upgrade`, `validate`, `getMigrations`,
or `version` functions. The application owns the final plan, connection policy, and CLI or
deployment process.

## Predecessor history

Existing databases may have PGMQ history in `public.schema_migrations` from the removed
`hasql-migration` runner. Native commands do not inspect that ledger. The explicit adapter exports:

```haskell
data AlternativeHistoryPolicy
  = DirectFullInstallHistory
  | EquivalentTwoStepUpgradeHistory

data SourceLedgerPolicy
  = RequireExclusiveSourceLedger
  | AllowUnselectedSourceRows

pgmqHasqlMigrationMappings
  :: AlternativeHistoryPolicy
  -> Either HistoryDefinitionError (NonEmpty HistoryMapping)

pgmqHasqlMigrationSourceConfig
  :: ConnectionProvider
  -> AlternativeHistoryPolicy
  -> Either HasqlMigrationDefinitionError HasqlMigrationSourceConfig

pgmqHasqlMigrationSourceConfigWithPolicy
  :: ConnectionProvider
  -> AlternativeHistoryPolicy
  -> SourceLedgerPolicy
  -> Either HasqlMigrationDefinitionError HasqlMigrationSourceConfig
```

`DirectFullInstallHistory` selects `pgmq_v1.11.0`. The adapter reproduces its stored base64 MD5
from the exact vendored payload, then maps the matching SHA-256 payload to the native baseline with
`SamePayload`.

`EquivalentTwoStepUpgradeHistory` selects both `pgmq_v1.10.0_to_v1.10.1` and
`pgmq_v1.10.1_to_v1.11.0`. It requires both exact predecessor checksums plus the packaged, read-only
PGMQ 1.11 schema-state validator. Import also requires the operator to opt in with
`AllowEquivalentHistory`; equivalent history is never implicit.

The default policy requires the selected rows to be the whole predecessor ledger.
`AllowUnselectedSourceRows`, added in `0.4.0.1`, supports a deliberately shared
`public.schema_migrations` table. It relaxes only the unselected-row check; every selected PGMQ row
keeps its exact checksum and state-evidence requirements.

## Import and native handoff

History import acquires no cooperating `hasql-migration` source lock, so quiesce every predecessor
writer. It validates evidence and atomically records target ledger plus audit rows without executing
target DDL. After import, native `up` reports `0001` already applied and applies only the `0002`
canary.

Fresh databases skip import and apply both native migrations. After schema installation,
`pgmq-config` owns queue provisioning; it is not part of component migration.

See [Install or migrate the schema](/docs/pgmq/how-to/install-or-migrate-schema) and
[Import hasql-migration history](/docs/pg-migrate/how-to/import-hasql-migration-history).
