---
title: "Ergonomic emit"
description: "Choose between the field-keyed TermFields record and the positional (t1 *: t2 *: oNil) form of emit, and use emitWith inside an ε-edge that has no enclosing command."
---

`B.emit` accepts two surface forms for an event's payload: a **field-keyed record**
(`<Ctor>TermFields`) and a **positional** `OutFields` chain (`t1 *: t2 *: oNil`). They produce an
identical AST. This guide shows when each reads better, and when you must reach for `emitWith` because
there is no enclosing command to recover the `InCtor` from.

<Callout type="info">
  Assumes you can author a single-event aggregate with the builder. If not, start with
  [Your first aggregate](/docs/keiki/tutorials/your-first-aggregate).
</Callout>

## Goal

Pick the `emit` form that reads clearly for the event at hand, and emit correctly from inside an
ε-edge body (`onEpsilon`) where `emit` cannot find an `InCtor`.

## The two forms

Both forms feed the same `ToOutFields` overload, so the resulting `OutTerm` is byte-identical. The
choice is purely about readability.

<Tabs items={["Field-keyed record", "Positional chain"]}>
<Tab value="Field-keyed record">

The TH-emitted `<Ctor>TermFields rs ci` record (one per event constructor, generated by
`deriveWireCtors` / `deriveAggregate`) has `Term`-typed mirrors of the event's payload fields. Call
sites read top-to-bottom, keyed by name — there is no positional order to keep straight. This is the
form the shipped aggregates use. `Jitsurei.EmailDelivery`'s `EmailSent`:

```haskell
B.emit wireEmailSent EmailSentTermFields
  { recipient = d.recipient
  , subject   = d.subject
  , at        = d.at
  }
```

Prefer the record form for any event with more than one or two fields, or where the field types are
similar enough that positional order is easy to transpose.

</Tab>
<Tab value="Positional chain">

A bare `OutFields` HList built with `(*:)` / `oNil` is accepted through the same `emit` overload (the
passthrough `ToOutFields` instance is `id`). The order matches the event payload's record-field order:

```haskell
B.emit wireEmailSent
  (d.recipient *: d.subject *: d.at *: oNil)
```

This produces exactly the same `OutTerm` as the record form above. Reach for it for a one- or
two-field event where naming each field adds noise, or when you are constructing the chain
programmatically.

</Tab>
</Tabs>

<Callout type="warn">
  The `<Ctor>TermFields` record is part of the builder's `ToOutFields` machinery — it is **not**
  available in the bare AST form. If you hand-author edges against `Keiki.Core`, you must use the
  `OFCons … OFNil` chain (or its `*:` / `oNil` sugar). See
  [Drop down to the AST](/docs/keiki/how-to/drop-down-to-the-ast).
</Callout>

## Emitting from an ε-edge with `emitWith`

`B.emit` recovers the input-side `InCtor` from the enclosing `B.onCmd` — that is what lets it pin the
command constructor into the output term without you naming it. An **ε-edge body** opened with
`B.onEpsilon` pins **no** `InCtor` (it has no command match and no `PayloadProj`), so a plain `emit`
there has nothing to recover. It raises a finalize-time error telling you to switch to `emitWith`:

```haskell
-- Keiki.Builder.emit: no enclosing onCmd pinned an InCtor.
-- Use 'emitWith ic wc fs' inside 'onEpsilon', …
```

`emitWith` is the escape hatch: pass the `InCtor` explicitly as the first argument, then the
`WireCtor` and the payload (record or positional, same overload as `emit`):

```haskell
emitWith
  :: ToOutFields rec rs ci ifs fs
  => InCtor ci ifs        -- the InCtor emit would otherwise recover
  -> WireCtor co fs
  -> rec
  -> EdgeBuilder rs ci co v w w ()
```

```haskell
B.from SomeVertex do
  B.onEpsilon $ B.do
    B.requireGuard someGuard
    B.emitWith inCtorTick wireTicked
      TickedTermFields { at = #lastSeenAt }
    B.goto NextVertex
```

<Callout type="info">
  Inside an `onCmd` body, the `InCtor`-less `emit` is preferred — it reads cleaner and cannot pin the
  wrong constructor. Use `emitWith` only inside `onEpsilon`, or when you deliberately need to override
  the constructor the enclosing `onCmd` bound. Like `emit`, `emitWith` snoc-appends, so multiple calls
  build a multi-event edge.
</Callout>

A body with no `emit`/`emitWith` at all is already an ε-edge (empty output); `B.noEmit` exists only to
state that intent explicitly and is a documentation no-op.

## Verify it worked

Both forms compile to the same edge, so the cross-form equivalence is observable: an aggregate
authored with the record form and one authored with the positional chain produce identical `delta`,
`omega`, and `reconstitute` results. `Jitsurei.EmailDeliveryBuilderSpec` exercises this shape for the
`EmailSent` event.
