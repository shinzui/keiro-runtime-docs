keiki constructs composites with three combinators, all in `Keiki.Composition`:

- `compose` ŌĆö sequential composition (A's output alphabet is B's input alphabet).
- `alternative` ŌĆö disjoint-input dispatch (`Either ci1 ci2` routes to one of two arms).
- `feedback1` ŌĆö a finite aggregate-policy-aggregate two-copy cascade.

Validated sequential pipelines add two public boundary tools: `checkComposeAlignment` diagnoses
constructor/field drift, and `composeChecked` admits the composite only when that warning list is
empty. `compose` remains the unchecked construction primitive. This page explains how the algebra
fits together and where its experimental contract stops. For the derivations themselves,
see [What gets derived](/docs/keiki/explanation/what-gets-derived) and the
[shape reference](/docs/keiki/reference/shape); this page asserts the preservation rather than
re-deriving it.

## The three combinators and their vertex shapes

Every combinator builds its composite vertex out of the same `Composite s1 s2` newtype (a strict pair
with hand-rolled `Bounded`/`Enum`). What differs is *how* the sub-vertices nest and route.

```mermaid
flowchart TD
  subgraph compose["compose t1 t2"]
    C["Composite s1 s2"]
    C --> C1["s1 (t1 vertex)"]
    C --> C2["s2 (t2 vertex)"]
  end
  subgraph alternative["alternative t1 t2"]
    P["Composite s1 s2 (product)"]
    P -->|"Left ci1 -> t1"| P1["advance s1, keep s2"]
    P -->|"Right ci2 -> t2"| P2["advance s2, keep s1"]
  end
  subgraph feedback1["feedback1 t f"]
    F["Composite s1 (Composite s2 s1)"]
    F --> F1["s1 (outer t)"]
    F --> F2["Composite s2 s1 (policy, inner t)"]
  end
```

- **`compose`** pairs the two transducers' vertices: `Composite s1 s2`. The register file is
  `Append rs1 rs2`; a composite edge fires t1's edge and t2's edge in lockstep. Multi-event paths
  symbolically thread downstream writes so later steps see the same state as sequential execution.
- **`alternative`** uses the *same product* vertex `Composite s1 s2`, but the two arms have
  **independent state**: a `Left` input advances only `s1`, a `Right` input advances only `s2`. The
  `PLeftArm`/`PRightArm` and the `Either` input alphabet keep the arms apart, including for
  epsilon/register-only `PTop` guards. (An earlier sum-vertex design was tried and found
  degenerate ŌĆö it could never leave the first arm ŌĆö so the shipped form is the product vertex.)
- **`feedback1`** is literally `compose t (compose f t)`, so its vertex is the nested
  `Composite s1 (Composite s2 s1)` ŌĆö outer `t`, then (policy, inner `t`). Even though the inner and
  outer `t` share a Haskell type, they occupy distinct dimensions of the tuple. The policy command
  advances the inner copy from its own state; this is not shared-state feedback.

<Callout type="warn">
`Keiki.Composition` is experimental. Prefer `composeChecked` for aggregate pipelines. The checker
reports structural alignment facts, but an empty result does not prove arbitrary opaque guard logic.
Likewise, the profunctor/category wrapper rejects mapped boundaries and slot overlap at runtime; it is
not an unqualified lawful category across every public value.
</Callout>

## Why this is the minimal closed set

The design space (catalogued from crem's `StateMachineT`) offered six combinators: `Sequential`,
`Parallel`, `Alternative`, `Feedback`, `Kleisli`, plus the profunctor hierarchy. keiki ships three.
The reasoning for *why those three and no more*:

<Accordions>

<Accordion title="compose subsumes both Sequential and Kleisli">
`Sequential` is plain categorical composition. `Kleisli` is the variant that lifts composition over a
*`Foldable` of inner events* ŌĆö needed only when an edge can emit zero-or-many events. Originally
keiki edges emitted at most one event, so `Kleisli` collapsed to `Sequential`. Once `Edge.output`
became a list (`[OutTerm rs ci co]` ŌĆö `[]` is the ╬Ą-edge, `[o]` a letter edge, `[o1..on]` a
multi-event edge), `compose`'s chain expansion (`PartialPath` + `expandPaths`) walks t2's state
through each of t1's mid-symbols and re-collapses the path into one composite edge. That expansion
*is* the `Kleisli` semantics. So a single `compose` now subsumes **both** crem's `Sequential` and
`Kleisli` ŌĆö there is no capability a separate `Kleisli` would add.
</Accordion>

<Accordion title="alternative is its own idiom, not expressible via compose">
`alternative` dispatches on a disjoint input alphabet (`Either ci1 ci2`) to two arms with independent
state. That is structurally different from sequential threading and cannot be expressed as a
`compose` chain, so it earns its place. Its single-valuedness is automatic: the `Left`/`Right` arms'
guards are conjoined with explicit arm predicates, so no new cross-transducer mutual-exclusion check
is needed.
</Accordion>

<Accordion title="feedback1 is a finite two-copy shape">
A general "iterate until quiescence" `feedback` needs an iteration model, and unbounded iteration can
diverge ŌĆö which keiki's pure formalism deliberately excludes. The non-trivial *pure* form is
exactly one finite cascade, `feedback1 t f`, which is itself just two stacked `compose`s. It does not
feed the policy command into the outer copy's updated state, and there is no `feedback1Checked` that
changes that fact. Multi-round patterns nest `feedback1`s; unbounded loops are pushed to the runtime
boundary. (Note: the
`Disjoint (Names rs1) (Names (Append rs2 rs1))` constraint forces `rs1 = '[]`, so `feedback1`'s
aggregate must currently be stateless ŌĆö a documented consequence of the two-stacked-`compose`
reduction.)
</Accordion>

<Accordion title="parallel and the profunctor hierarchy are out of scope">
`parallel` runs two machines on a strict `(ci1, ci2)` tuple in lock-step, but keiki's queue-driven
runtime delivers one command per tick ŌĆö bounded-context inputs are *sum*-shaped (handled by
`alternative`), not paired. The `Profunctor`/`Strong`/`Choice` hierarchy is a separate concern that
builds *on top of* these combinators rather than competing with them.
</Accordion>

</Accordions>

The set is **closed** in the sense that every orchestration pattern keiki targets ŌĆö choreography,
process managers, sagas, policies, feedback loops (see
[Process managers, sagas, and choreography as transducers](/docs/keiki/explanation/process-managers-sagas-choreography-as-transducers))
ŌĆö reduces to a chain of these three, and each combinator's output is itself a `SymTransducer` that
the others accept as input.

## Guarantees and admission checks

A single keiki transducer earns trust from mechanical analyses. On honestly aligned structural
boundaries, the core transformations preserve the three original derivations. `composeChecked` is
the admission check that detects name drift, positional-field mismatch, and poisoned mapped names
before construction. The analyses themselves are documented in
[What gets derived](/docs/keiki/explanation/what-gets-derived).

In short: on a checked, honest structural boundary, `solveOutput` still inverts events back to
commands, `checkHiddenInputs` still flags fields that fail to reach the output, and the symbolic
diagnostic remains conservative about overlap. Mapped profunctor boundaries are not honest structural
boundaries: provenance marks them as poisoned and categorical composition rejects them.
