<Callout type="warn">
The `Decider` is a **legacy compatibility façade**, not keiki's (継起) recommended surface. Author and
drive aggregates with the transducer itself — `step` / `stepEither` / `delta` / `omega` /
`reconstitute`. Reach for `toDecider` only when you are migrating a codebase already written against a
hand-written naive-decider shape and want to keep that code running unchanged while you move it over.
This page explains the façade so you can recognise it and its limits, not to recommend it.
</Callout>

## The Chassaing shape

Jérémie Chassaing's *Functional Event Sourcing Decider* popularised a small record:

```haskell
data Decider c e s = Decider
  { decide       :: c -> s -> [e]
  , evolve       :: s -> e -> s
  , initialState :: s
  , isTerminal   :: s -> Bool
  }
```

`decide` turns a command and the current state into events; `evolve` folds an event back into the
state. It is a clean shape, and a lot of code is written against it. keiki's `Decider` mirrors it,
with two extra fields for streaming replay:

```haskell
data Decider c e s s_streaming = Decider
  { decide          :: c -> s -> [e]
  , evolve          :: s -> e -> s
  , evolveStreaming :: s_streaming -> e -> Maybe s_streaming
  , initialState    :: s
  , isTerminal      :: s -> Bool
  }
```

## Façade vs transducer-direct

`toDecider` projects a `SymTransducer` onto that record. The state carriers are pairs — keiki edge
guards depend on the register file as well as the control vertex, so the letter-replay state is
`(s, RegFile rs)` and the streaming-replay state is `(InFlight s co, RegFile rs)`:

```haskell
toDecider
  :: (BoolAlg phi (RegFile rs, ci), Eq co)
  => SymTransducer phi rs s ci co
  -> Decider ci co (s, RegFile rs) (InFlight s co, RegFile rs)
```

Field by field, the façade is a thin re-expression of transducer verbs:

- `decide d cmd (s, regs)` is `omega t s regs cmd` — the forward step. After the EP-19 output
  widening, `omega` returns the **full** event list a command produces, so a single command yielding
  two or more events comes back directly, with no state-refinement scaffolding.
- `evolve d (s, regs) ev` is the **letter-only** inverse, `applyEvent`. It handles edges whose output
  is length 0 or 1.
- `evolveStreaming d ws ev` is `applyEventStreaming`, threading the `InFlight` wrapper across a
  length-2+ edge so replay can stay mid-chain between the events of one transition.
- `initialState d` is `(initial t, initialRegs t)`; `isTerminal d (s, _)` is `isFinal t s`.

Because the façade is *exactly* these projections, it can express nothing the transducer cannot — and
one thing the transducer can that it deliberately does not.

### `evolve` is defensive

`applyEvent` returns `Nothing` when an event cannot be replayed letter-by-letter from `(s, regs)`. To
keep the Chassaing signature non-`Maybe`, the façade's `evolve` returns the **input state unchanged**
on failure. That is convenient for the Chassaing shape but it hides replay failures. Callers that
want strict replay use `evolveStreaming` (whose `Maybe` is explicit) or `Keiki.Core.applyEvents`
(which returns `Nothing` on the first failure across a chunk).

## The deliberate ε-edge gap

The one semantic gap is **ε-edges** — edges whose output is `[]`. The transducer transitions state
without emitting an event: in the user-registration aggregate, a GDPR request before confirmation
moves `RequiresConfirmation → Deleted` silently.

The façade no-ops on these. `decide` returns `[]` for the ε-driven command, and since there is no
event, `evolve []` leaves the state untouched — from the Decider record's perspective, identical to
"nothing happened". The state never reaches `Deleted` through the façade.

`DeciderSpec` (`test/Keiki/DeciderSpec.hs`) pins both sides of this. The façade path:

```haskell
-- ε-edge limitation: GDPR from RequiresConfirmation yields [] from decide,
-- so folding evolve leaves the state at RequiresConfirmation.
fst afterGdpr `shouldBe` RequiresConfirmation
```

and the cross-check that the transducer itself *does* make the move:

```haskell
-- delta drives the same ε-edge input straight to Deleted.
```

That is the crux of the trade-off. The Decider's `decide`/`evolve` are an **event-driven** view: the
event is the only carrier of a transition, so a transition that emits no event is invisible. The
transducer is a **transition-driven** view: `delta` and `step` advance on the edge regardless of how
much output it carries. When ε-driven transitions matter — and in any aggregate with silent
state changes they do — use `delta` / `step` directly.

<Callout type="info">
Letter-only vs streaming is the other axis. `evolve` collapses a length-2+ edge: it agrees with
`evolveStreaming` on length-0 and length-1 commands and diverges only on multi-event edges, where the
streaming path stays `InFlight` between the events while `evolve` does not. If you replay events one
at a time across multi-event transitions, you need `evolveStreaming`, not `evolve`.
</Callout>

## When to use it

Use the façade when, and only when, you are porting code already shaped around `decide`/`evolve` and
the cost of rewriting it against the transducer up front is not worth paying yet. It lets that code
drive a keiki aggregate today. As you migrate, move call sites onto `step` / `stepEither` / `delta` /
`omega` / `reconstitute`, which have none of the ε-edge blindness, none of the defensive-`evolve`
silence, and the full multi-event behaviour first-class. For the field-level type signatures and the
shipped record, see the `/docs/keiki/reference/decider` reference page.
