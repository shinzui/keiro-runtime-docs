---
title: "Inspect a per-vertex machine-state view"
description: "Use deriveView for a typed, version-coupled view of an already-hydrated RegFile."
---

A cookbook recipe is a short, copy-paste solution to one problem. Here: inspect current machine
state while exposing only the registers live at one control vertex.

## Problem

Machine-local code handling a `Confirmed` user should not be able to ask for `cConfirmedAt` on a
`PotentialCustomer`, where that slot is meaningless. You want a typed presentation of the current
`RegFile` where the type system blocks asking a vertex for a slot it does not have.

## Solution

`deriveView` generates a singleton, a view sum, and an accessor function from the shared
register file. List, per vertex, the "live" slots that vertex exposes.

```haskell
-- Jitsurei.UserRegistration
$(deriveView ''Vertex ''UserRegRegs
    "SUserVertex" "UserView" "userView"
    [ ("PotentialCustomer",    [])
    , ("RequiresConfirmation", ["email", "confirmCode"])
    , ("Confirmed",            ["email", "confirmedAt"])
    , ("Deleted",              ["email", "deletedAt"])
    ])

-- Pattern-matching on the machine-state view yields only the live slots:
--   userView SConfirmed regs  ==>  ConfirmedV { cEmail = …, cConfirmedAt = … }
-- Asking SPotentialCustomer for cConfirmedAt does not type-check.
```

<Callout type="warn">
  This recipe does **not** build a projection or read model. `userView` requires the current Keiki
  `RegFile`; it cannot fold or replay an event log. Do not use it from `InlineProjection`,
  `AsyncProjection`, a read-model query, or a projection rebuild.
</Callout>

## How it works

The splice emits one `UserView` constructor per vertex carrying record selectors for that
vertex's live slots only, plus a singleton (`SUserVertex`) that ties the term-level vertex to
the type-level view, and `userView :: SUserVertex v -> RegFile UserRegRegs -> UserView`. The
accessor is opt-in and downstream: adding a view does not change `delta` or `omega`.

The reverse dependency remains intentional and strong. `deriveView` reifies the transducer's slot
list, so removing a named slot fails the splice, changing a slot type changes the generated field
type, and changing the vertex schema can change the generated GADTs. That coupling is acceptable for
machine-local presentation and unacceptable for a durable projection contract.

The variance is the point: `LoanApplication` exposes `appApplicantId` at `Intake`, then switches
focus to decision inputs (`appCreditScore`, `appEmploymentVerified`) at `UnderReview`, then to a
terminal summary at `Approved`. See the
[per-vertex view how-to](/docs/keiki/how-to/read-a-per-vertex-view) for consuming a view and
[B-presentation views](/docs/keiki/explanation/b-presentation-views) for the design rationale.
