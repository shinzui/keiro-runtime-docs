The types and functions on this page live in `Keiki.Acceptor`. They name, as a first-class data
type, the two acceptor projections that every keiki (継起) `SymTransducer` carries: the **input
projection** over the command language and the **output projection** over the event language.

In [`Keiki.Core`](/docs/keiki/reference/core) these projections are *implicit* — π₁ is `delta` and π₂
is `applyEventStreaming`. This module *names* them so downstream code (UI, validation, generated
documentation) can pattern-match on a known shape instead of plumbing the step functions by hand. For
the conceptual model of a transducer's two languages, read
[The SymTransducer](/docs/keiki/explanation/the-symtransducer).

## `Acceptor`

A minimal acceptor over alphabet `a` with state carrier `s`.

<TypeTable
  type={{
    aStep: { type: "s -> a -> Maybe s", description: "Single-step transition; Just on success, Nothing to reject (absence of a transition is rejection)" },
    aInitial: { type: "s", description: "The start state" },
    aIsFinal: { type: "s -> Bool", description: "Final-state predicate; a run accepts iff it ends in a state satisfying it" },
  }}
/>

```haskell
data Acceptor a s = Acceptor
  { aStep    :: s -> a -> Maybe s
  , aInitial :: s
  , aIsFinal :: s -> Bool
  }
```

<Callout type="info">
An `Acceptor` holds closures and a `RegFile`, so it has **no `Show` or `Eq` instance**. Assert on the
results of `runAcceptor` or `accepts` instead of on the record itself.
</Callout>

## `inputAcceptor` — the π₁ projection

The acceptor over the **command** alphabet, whose `aStep` is `delta`. Its language is the set of
command sequences the aggregate accepts.

```haskell
inputAcceptor
  :: BoolAlg phi (RegFile rs, ci)
  => SymTransducer phi rs s ci co
  -> Acceptor ci (s, RegFile rs)
```

The state carrier is `(s, RegFile rs)` because edge guards depend on the register file as well as the
control vertex; `aIsFinal` ignores the register file and consults `isFinal t`. `accepts
(inputAcceptor t) cmds` is `True` iff successively applying `delta` to each command reaches a final
control vertex, and is rejected as soon as any step finds zero or multiple satisfied outgoing edges.

## `outputAcceptor` — the π₂ projection

The acceptor over the **event** alphabet, whose `aStep` is `applyEventStreaming`. Its language is the set of
event sequences the aggregate could have produced — the set of **replayable logs**.

```haskell
outputAcceptor
  :: (BoolAlg phi (RegFile rs, ci), Eq co)
  => SymTransducer phi rs s ci co
  -> Acceptor co (InFlight s co, RegFile rs)
```

The carrier is `(InFlight s co, RegFile rs)` because replay may be between the events emitted by one
edge. It starts at `Settled (initial t)`. A tail event advances the pending queue without applying
the edge update twice; a complete queue returns to `Settled`. `aIsFinal` accepts only `Settled s`
where `isFinal t s` is true, so a truncated multi-event log finishes with an inspectable `InFlight`
carrier but is not accepted.

## Folding helpers

`runAcceptor` runs an acceptor over a sequence, returning `Just` the terminal state if every step
succeeds and `Nothing` on the first step that rejects.

```haskell
runAcceptor :: Acceptor a s -> [a] -> Maybe s
```

`accepts` decides membership: `True` iff the input runs to completion *and* the terminal state is
final.

```haskell
accepts :: Acceptor a s -> [a] -> Bool
```

```haskell
-- accepts is runAcceptor plus the final-state check:
accepts a xs = case runAcceptor a xs of
  Just s  -> aIsFinal a s
  Nothing -> False
```

## Two projections of one transducer

Both acceptors hide the register file inside their carrier and use `isFinal` for settled control
vertices. The output side additionally preserves a pending multi-event tail:

| Projection | Function | Alphabet | `aStep` | Language |
| --- | --- | --- | --- | --- |
| π₁ input | `inputAcceptor` | commands `ci` | `delta` | command sequences the aggregate accepts |
| π₂ output | `outputAcceptor` | events `co` | `applyEventStreaming` | replayable event logs, including multi-event chains |

The output acceptor agrees with strict replay on acceptance — the **round-trip property**:

```haskell
accepts (outputAcceptor t) log
  == maybe False (isFinal t . fst) (reconstitute t log)
```

## Tested behaviour

`test/Keiki/AcceptorSpec.hs` anchors both projections:

- **`inputAcceptor`** accepts the canonical `userReg` command sequence and rejects `ConfirmAccount`
  issued from `PotentialCustomer` (no satisfied edge).
- **`outputAcceptor`** accepts complete letter and multi-event logs and rejects a foreign first
  event because `applyEventStreaming` returns `Nothing` on that step.
- **Truncation is visible:** running only the first event of the user-registration entrance returns
  `Just (InFlight RequiresConfirmation [ConfirmationEmailSent ...], regs)` but `accepts` is `False`.
- **The round-trip property** holds for complete, truncated, and foreign logs: output acceptance
  agrees with whether `reconstitute` reaches a final settled state.
- **`aIsFinal`** matches `isFinal` under `fst` (`Deleted` final; `Confirmed` and `PotentialCustomer`
  not).
