Short answers to the questions people ask most about keiki. Each links to the page that goes deeper.

<Accordions>
  <Accordion title="Can I store a Map or a list in a register?">
    Yes ŌĆö if it **arrives whole on the command**, a register can hold any value you like. What breaks
    is *guarding on its contents*: the symbolic analyses see a collection lookup as an opaque `TApp`
    term, so the solver cannot reason about it and any branch on it is unverified. The fix is to
    **project to a scalar** the guard can read ŌĆö a count, a sum, a boolean flag ŌĆö and keep the
    collection itself for the carried payload. See [Model a
    collection](/docs/keiki/how-to/model-a-collection).
  </Accordion>

  <Accordion title="My guard branches on a collection and the build is green ŌĆö is it verified?">
    Probably not. A clean build does **not** mean a guard is solver-visible; the opaque-guard audit
    is opt-in. Set `warnOpaqueGuards = True` and re-run the well-formedness check ŌĆö it flags every
    edge whose guard branches on an opaque `TApp` term the symbolic analyses cannot see. Once you
    know which guard is invisible, project it to a scalar. See [Why
    SMT?](/docs/keiki/explanation/why-smt).
  </Accordion>

  <Accordion title="Why does my replay return Nothing?">
    `applyEvents` and `reconstitute` are compatibility wrappers that collapse replay defects to
    `Nothing`. Use `replayEvents`, `applyEventsEither`, or `reconstituteEither` to learn the failing
    event index, wrapper state, and exact `ReplayFailureReason` ŌĆö ambiguous inversion, queue mismatch,
    inversion failure, or a truncated multi-event chain. A hidden input is one possible design cause,
    and the validation warnings identify it before replay. See [Replay
    failures](/docs/keiki/reference/step-failure#replay-failures-are-not-command-rejections).
  </Accordion>

  <Accordion title="step returned Nothing ŌĆö was the command rejected or was it ambiguous?">
    `step` collapses every non-result to `Nothing`. Use `stepEither` to recover the reason: it
    returns `NoOutgoingEdges` (the source vertex has no edges at all), `NoMatchingEdge` (edges exist
    but none guarded true for this command ŌĆö a legitimate domain rejection), or `AmbiguousEdges`
    (two or more guards matched at once). `AmbiguousEdges` is a **defect**, not a rejection: it means
    overlapping guards, which the well-formedness check would have caught. See [Step
    failure](/docs/keiki/reference/step-failure) and [Diagnose a rejected
    command](/docs/keiki/how-to/diagnose-a-rejected-command).
  </Accordion>

  <Accordion title="How do I evolve an event or register schema?">
    Pin deployed constructor kinds. For an additive field, keep the event version and give an absent
    key a `Maybe`/`fcOnMissing` default. For a structural rewrite, bump `currentVersion` and provide
    every whole-envelope upcaster rung. A register-shape change invalidates only the snapshot cache;
    replay rebuilds it from the event log. Keiki 0.2 also causes one intentional miss for every
    non-empty 0.1 shape because built-in names are now pinned. See
    [Shape](/docs/keiki/reference/shape) and [Evolve a schema
    safely](/docs/keiki/cookbook/evolve-a-schema-safely).
  </Accordion>

  <Accordion title="Should I call delta and omega, or step?">
    Use **`step`** ŌĆö and `stepEither` when you want the rejection reason. `delta` (the next-state
    half) and `omega` (the output half) are the lower-level pieces `step` is built from; reach for
    them only when you genuinely need one half in isolation. For everyday command handling, `step`
    is the right surface.
  </Accordion>

  <Accordion title="Should this become its own aggregate or stay in registers?">
    Ask whether it has **its own identity and lifecycle**. If it is created, changes state, and is
    retired on its own timeline, promote it to its **own aggregate** and coordinate the two with a
    Process. If all you need are **set-level facts** about it ŌĆö how many, the total, whether any are
    open ŌĆö keep it in registers as a **scalar tally**. See [Model a
    collection](/docs/keiki/how-to/model-a-collection).
  </Accordion>

  <Accordion title="The (.>) operator will not compile ŌĆö it clashes with lens.">
    `(.>)` collides with lens's composition operator. Inside a `B.do` block, use the verb form ŌĆö
    `B.requireGt` and its siblings ŌĆö which reads cleanly and never clashes. Outside a builder block,
    bring the operators in under a qualified `Keiki.Operators` import so the symbol is namespaced.
    See [Operators](/docs/keiki/reference/operators).
  </Accordion>

  <Accordion title="Is compose how the runtime fires cross-context flows?">
    No. `compose` is **lockstep** ŌĆö it fuses two transducers that advance together, in the same step,
    as one machine. It does not model an asynchronous, cross-context flow. The runtime models that
    with durable routing and explicit dispatch. `lmapMaybeCi` is a useful standalone forward router,
    but composing across it is deliberately poison-stamped because the partial map cannot support
    replay-safe inversion. See [The composition
    algebra](/docs/keiki/explanation/the-composition-algebra).
  </Accordion>
</Accordions>
