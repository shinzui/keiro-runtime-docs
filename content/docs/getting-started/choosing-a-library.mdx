---
title: Choosing a library
description: Pick the smallest useful keiro runtime package or integration layer for your problem.
---

Choose the highest layer that owns your problem. Its documentation will point
downward when you need lower-level control; you do not need to begin at every
foundation package.

## Match the problem to the surface

| Need | Start with | Why |
| --- | --- | --- |
| A full event-sourced service with commands, projections, orchestration, or workflows | [keiro](/docs/keiro) | It is the application framework and already integrates the pure and persistence layers. |
| An append-only PostgreSQL event store without the Keiro framework | [Kiroku](/docs/kiroku) | It exposes streams, expected versions, transactions, subscriptions, and operations directly. |
| Pure state-machine/transducer modeling, validation, or codec derivation without IO | [keiki](/docs/keiki) | It has no persistence or worker responsibility. |
| Supervised consumption with backpressure, batching, explicit acknowledgements, and metrics | [Shibuya](/docs/shibuya) | Choose a [source adapter](/docs/integrations/shibuya-adapters) separately. |
| A PostgreSQL-native queue, topics, FIFO groups, or queue topology | [pgmq-hs](/docs/pgmq) | It owns queue mechanics; add the Shibuya adapter when you want supervised handlers. |
| Typed background jobs inside a Keiro application | [keiro-pgmq](/docs/integrations/keiro-with-pgmq) | It wraps PGMQ and Shibuya behind `Job` and `JobOutcome`. |
| Manifest-backed PostgreSQL migrations, component composition, deployment, or predecessor cutover | [pg-migrate](/docs/pg-migrate) | Libraries export components; the application owns the final plan and migration executable. |
| A typed `.keiro` service specification, scaffolding, conformance harness, or evolution diff | [keiro-dsl](/docs/keiro/explanation/the-keiro-dsl-toolchain) | It is the authoring toolchain, not a runtime service dependency. |

## Common combinations

- **Keiro service:** `keiro` + Kiroku, with keiki semantics through Keiro's
  validated event-stream boundary.
- **Keiro service with jobs:** add `keiro-pgmq`, the pgmq 0.4 family, and
  `shibuya-pgmq-adapter 0.12`.
- **Standalone event consumer:** Kiroku + `shibuya-kiroku-adapter`, or Shibuya
  with the adapter for an existing Kafka/PGMQ/Message DB source.
- **PostgreSQL deployment:** add the migration components for every selected
  library to one application-owned pg-migrate plan.

<Callout type="info">
  If you are building a complete event-sourced application, begin with Keiro.
  Begin with Kiroku only when you specifically want the store without Keiro's
  domain/runtime abstractions, and with keiki only for pure modeling.
</Callout>
