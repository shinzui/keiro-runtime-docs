title: Choosing a library

You usually do not pick just one — the libraries compose — but each owns a distinct
problem. Use this page to find the right starting point.

## Match the problem to the library

- **"I need to durably store an ordered log of events."** → **kiroku** (記録, an
  append-only PostgreSQL event store). Start at [kiroku](/docs/kiroku).
- **"I need to handle commands and run workflows over those events."** → **keiro**
  (経路, the event-sourcing framework). Start at [keiro](/docs/keiro).
- **"I need pure, law-abiding stream/transducer composition with no IO."** →
  **keiki** (継起, the pure core). Start at [keiki](/docs/keiki).
- **"I need to reliably consume a message queue with backpressure and batching."**
  → **shibuya** (queue processing). Start at [shibuya](/docs/shibuya).
- **"I need a durable work queue to enqueue and lease jobs."** → **pgmq** (a
  PostgreSQL message queue via `pgmq-hs`), consumed by shibuya through the
  [shibuya ⇄ pgmq adapter](/docs/integrations/shibuya-pgmq-adapter). Start at
  [pgmq](/docs/pgmq).
- **"I want a typed background-job queue for my keiro app — declare a job, write a
  handler."** → **keiro-pgmq** (`Keiro.PGMQ`), keiro's typed `Job` layer over pgmq +
  shibuya. Start at [Background jobs with
  PGMQ](/docs/keiro/explanation/background-jobs-with-pgmq).

<Callout type="info">
  New to event sourcing? Begin with kiroku — it is the foundation the rest of the family builds on.
</Callout>
