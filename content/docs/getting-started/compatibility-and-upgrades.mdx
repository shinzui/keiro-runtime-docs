This matrix records the exact committed releases reviewed by this documentation
pass. It is an evidence boundary, not a promise that arbitrary newer versions
are compatible. Pin package versions, read upstream release notes, and rerun your
own build, migration, and integration tests before upgrading.

## Breaking source upgrades

Treat every pre-1.0 minor line as potentially breaking.

- **keiki 0.2:** the `Keiki.Decider` facade is removed; builders require
  explicit output intent; replay failures are structured; pinned shape names
  intentionally cause one cache miss and full replay for non-empty 0.1
  snapshots. See [persist a register
  snapshot](/docs/keiki/how-to/persist-a-register-snapshot#upgrading-from-keiki-01).
- **Keiro 0.3 line:** command, replay, worker, read-model, and workflow outcomes
  are typed more precisely; framework tables live in `keiro`; public event
  streams require replay validation. For `.keiro` specs, follow [migrate a spec
  to 0.2](/docs/keiro/how-to/migrate-a-keiro-spec-to-0-2).
- **Kiroku 0.3:** update cursor, validation, store-error, subscription-source,
  CLI, and adapter handling using the [0.3 upgrade
  matrix](/docs/kiroku/how-to/upgrade-to-0-3).
- **Shibuya 0.8:** handlers receive `Message`, application runners take
  `AppConfig`, batching is first-class, and adapter finalizers must be
  idempotent. Start with the [Shibuya landing page](/docs/shibuya).
- **pgmq-hs 0.4:** `pgmq-migration` exports a component instead of its removed
  standalone runner. Compose it through pg-migrate and use the explicit
  predecessor-history path for existing databases.

## Database upgrade order

For an existing PostgreSQL deployment, do not deploy the new runtime first and
hope startup repairs history:

1. Build the complete application migration plan and inspect `plan`, `status`,
   and `verify` output against a backup or rehearsal database.
2. Quiesce old migration writers and back up the production database.
3. If upgrading Keiro 0.1, follow the [framework-table relocation
   procedure](/docs/keiro/how-to/upgrade-a-0-1-database).
4. Import verified Kiroku/Keiro Codd history and PGMQ
   `public.schema_migrations` history where those predecessor ledgers exist.
   Imports adopt evidence without replaying schema DDL.
5. Run the complete native Kiroku → Keiro → PGMQ → application plan, then run
   strict verification.
6. Deploy runtime processes, reconcile additive queue topology, and monitor the
   typed failure and durable-worker outcomes described by each package.

Fresh databases skip the predecessor imports. Use the [runtime composition
recipe](/docs/pg-migrate/cookbook/compose-runtime-components) and [deployment
runbook](/docs/pg-migrate/how-to/deploy-a-migration-plan) for the executable and
operator gates.

## Stability and pending work

- Keiki, Keiro, Kiroku, Shibuya, PGMQ client packages, and adapters are pre-1.0.
  This documentation does not promise API stability beyond each package's
  release notes and bounds.
- pg-migrate is reviewed at 1.1.0.0, but its compatibility page remains the
  authority for compiler, PostgreSQL, Hasql, and ledger-version support.
- The Message DB adapter is not on the Shibuya 0.8 compatibility line, as noted
  above.
- `keiro-runtime-jitsurei` and `content/docs/example-app/` remain pending
  modernization. They are an older architecture tour, not proof of the current
  package matrix.
