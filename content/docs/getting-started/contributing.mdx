This guide is the authoring and style reference for the keiro runtime
documentation. It explains where each kind of page lives, how to write it, and how
to add it to the navigation. The goal: any contributor can add a correct page
without inventing structure.

<Callout type="info">
  These docs follow the **Diátaxis** framework — every page serves exactly one of
  four needs (Tutorial, How-To Guide, Reference, Explanation), plus the extras
  Cookbook, Code Walkthrough, FAQ, and Theory.
</Callout>

## Voice and tense by document type

Match the voice to the document's purpose:

- **Tutorials** — second person, present/imperative, encouraging ("you will
  build…", "now run…"). Hold the reader's hand; never skip a step.
- **How-To Guides** — imperative, terse, goal-first ("To do X, …"). Assume the
  reader knows the basics.
- **Reference** — neutral, factual, present tense. No second person, no narrative,
  no opinions. Just the facts the reader looks up.
- **Explanation** — reflective; may use "we"/"you". Discusses trade-offs and *why*.
- **Cookbook** — like a how-to but shorter and fully self-contained.
- **Code Walkthrough** — narrative; reads the real source and explains intent.
- **Theory** — precise; defines all notation. Lives inside `explanation/` (a theory
  essay *is* an explanation).

The house voice is **problem-first**: open with the problem the reader has, then the
mechanism. Em-dashes are the house punctuation style.

## Naming conventions

- Product names are always **lowercase**: kiroku, keiro, keiki, shibuya. On the
  first mention of a product on any page, gloss it once — e.g. "kiroku (記録, an
  append-only PostgreSQL event store)".
- Use **"jitsurei"** (実例) for runnable worked examples / example collections.
- The task-oriented quadrant is always labelled **"How-To Guides"** in the UI.

## File naming

- Lowercase, hyphen-separated, `.mdx` (e.g. `appending-events.mdx`).
- Walkthrough pages take a two-digit numeric prefix to force order
  (`00-start-here.mdx`, `01-…`).
- A folder's landing page is always `index.mdx`.

## Where each document type lives

| Document type            | Location                          |
| ------------------------ | --------------------------------- |
| Tutorial                 | `<product>/tutorials/`            |
| How-To Guide             | `<product>/how-to/`               |
| Reference                | `<product>/reference/`            |
| Explanation and Theory   | `<product>/explanation/`          |
| Cookbook recipe          | `<product>/cookbook/`             |
| Code Walkthrough         | `<product>/walkthrough/`          |
| FAQ                      | `<product>/faq.mdx`               |
| Cross-product integration| `integrations/`                   |

## Code-block conventions

- Always tag the fence with a language. Never use a bare fence.
- Haskell samples use the `haskell` tag and **must reflect the real API** — no
  invented functions. PragmataPro ligatures and Haskell-aware highlighting are
  applied automatically; you only choose the tag.
- Other tags in use: `bash`, `sql`, `json`, `nix`, `text`.

## Diagram conventions

- Use a `mermaid` fence for diagrams; keep them small and labelled. They render as
  interactive, zoomable diagrams automatically.

## Shared authoring components

These components are registered globally — use them directly, with **no per-page
imports**:

```mdx
<Callout type="warn">kiroku is in active development; APIs may change.</Callout>

<Steps>
  <Step>### First do this</Step>
  <Step>### Then do that</Step>
</Steps>

<Tabs items={['hasql', 'raw SQL']}>
  <Tab value="hasql">The hasql way.</Tab>
  <Tab value="raw SQL">The raw SQL way.</Tab>
</Tabs>

<Cards>
  <Card title="Tutorials" href="/docs/kiroku/tutorials" />
</Cards>

<Accordions>
  <Accordion title="A question?">An answer.</Accordion>
</Accordions>

<TypeTable type={{ poolSize: { description: 'Connection pool size.', type: 'Int', default: '10' } }} />
```

## How to add a new page

1. Pick the doc type and **copy the matching template** from `_templates/` (e.g.
   `_templates/tutorial.mdx`).
2. Save it under the correct `<product>/<section>/` folder with a
   lowercase-hyphenated name.
3. Fill in the frontmatter `title` and `description`.
4. Write the body following the **voice for that type** (see above).
5. Add the file name (without `.mdx`) to that folder's `meta.json` `pages` array,
   in the position you want it to appear. If you added a whole new subfolder, also
   add the folder name to the *parent* folder's `meta.json` `pages`.
6. Run the build (`pnpm build`) and confirm the page appears in the sidebar and
   compiles without errors.

<Callout type="info">
  Templates live in `content/docs/_templates/` and are deliberately hidden from the
  sidebar (they are not listed in any `meta.json` `pages`). They are still validated
  by the build, so they always stay correct.
</Callout>
