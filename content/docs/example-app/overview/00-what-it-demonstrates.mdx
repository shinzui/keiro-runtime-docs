The reason to read `keiro-runtime-jitsurei` is that it uses **almost every keiro runtime feature
in anger** — with real domain code, across a real service boundary, under one trace. If you have
learned a keiro subsystem abstractly elsewhere on this site, this application is where you can see
it wired into a working program.

The application's own feature-selection guide
(`docs/walkthroughs/runtime-feature-use.md`) frames each feature as "reach for X when…"; the table
below turns that into a "demonstrated by" index. Every row links the keiro concept page on the left
to the app source on the right.

## Feature → demonstrated by

that lives in just one service is the **PGMQ background job** — Hospital Capacity's reservation
work queue.

## How the three scenarios exercise the features

The application packages three runnable scenarios. Each lights up a different slice of the runtime
(from `docs/scenarios/README.md`):

| Scenario | Primary boundary | Runtime features it exercises |
|---|---|---|
| `mass-casualty-transfer` | Incident Command → Hospital Capacity | TypeID, Keiki, Kiroku, command processor, process manager, router, workflow, outbox, Kafka, inbox, PGMQ, Shibuya, OpenTelemetry |
| `hospital-divert-reroute` | Hospital Capacity local aggregate + read model | Keiki guard rejection, Kiroku append, capacity read model, reroute-ready state |
| `supply-shortage-escalation` | Hospital Capacity local escalation | Keiki supply register, process manager, timer worker, capacity read model |

`mass-casualty-transfer` is the full cross-boundary demo — it is the one that touches every box in
the architecture diagram, and it is the narrative spine of the
[cross-service](/docs/example-app/cross-service) chapters. The other two are tighter, single-service
stories that isolate guard rejection and local escalation.
