This is an **ordered source tour** of the `hospital-capacity` service — the receiving side of
`keiro-runtime-jitsurei`. It reads the real Haskell at the pinned commit (`04420ed`), one chapter
per concern, and links each feature back to the canonical keiro reference page.

## What the service is

`hospital-capacity` (Haskell package `hospital-capacity`, modules `HospitalCapacity.*`, source under
`services/hospital-capacity/src/`) absorbs the patients. It owns three aggregates — **Hospital**,
**Capacity**, and **Reservation** — a **Surge** process manager, a durable **reservation workflow**,
and its signature feature: a **`keiro-pgmq` background-job queue** that drains reservation work. When
[Incident Command](/docs/example-app/incident-command) publishes a transfer need, this service
consumes it, routes it to reservation work, holds a bed, and publishes its response back over Kafka.

The service ships two executables:

```haskell
-- services/hospital-capacity/app/HospitalCapacityCli.hs
run tracer args =
  case args of
    ["smoke"] -> putStrLn smokeMessage
    ["scenario", "regional-surge"] -> putStr regionalSurgeScenario
    ["scenario", "regional-surge", "--durable"] -> durableRegionalSurgeScenarioWithTelemetry tracer >>= putStr
    ["diagrams", "keiki"] -> do { … }
```

- **`hospital-capacity-cli`** drives the synchronous write path and the packaged `regional-surge`
  scenario (with `--durable` for the surge PM + reservation workflow), plus `diagrams keiki`.
- **`hospital-capacity-worker`** drives the asynchronous side: the surge timer, the inbox consumer,
  the outbox publishers, and the **reservation-work queue** (`setup` / `enqueue-latest` /
  `consume once` / `enqueue-malformed` / `confirm-latest`). Its full command set is in
  [chapter 06](/docs/example-app/hospital-capacity/06-wiring-and-the-cli).
