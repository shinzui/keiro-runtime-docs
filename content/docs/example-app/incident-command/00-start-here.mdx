This is an **ordered source tour** of the `incident-command` service — the command side of
`keiro-runtime-jitsurei`. It reads the real Haskell at the pinned commit (`04420ed`), one chapter
per concern, and links each feature back to the canonical keiro reference page. Read the chapters
in order.

## What the service is

`incident-command` (Haskell package `incident-command`, modules `IncidentCommand.*`, source under
`services/incident-command/src/`) runs the emergency scene. It owns three aggregates — **Incident**,
**FieldResource**, and **Triage** — plus an **Escalation** process manager and a durable
**evacuation workflow**. When a triage marks a critical (red-tag) patient it emits a *transfer
need*; that need leaves the service through the outbox and Kafka and arrives at
[Hospital Capacity](/docs/example-app/hospital-capacity).

The service ships two executables:

```haskell
-- services/incident-command/app/IncidentCommandCli.hs
run :: Telemetry.ServiceTracer -> [String] -> IO ()
run tracer args =
  case args of
    ["smoke"] -> putStrLn smokeMessage
    ["scenario", "mass-casualty"] -> putStr massCasualtyScenario
    ["scenario", "mass-casualty", "--durable"] -> durableMassCasualtyScenarioWithTelemetry tracer >>= putStr
    "command" : commandArgs -> runCommandCliWithTelemetry tracer commandArgs >>= putStrLn
    ["diagrams", "keiki"] -> do
      writeKeikiDiagrams defaultDiagramFile
      putStrLn ("wrote Keiki diagrams to " <> defaultDiagramFile)
    _ -> putStrLn usage
```

- **`incident-command-cli`** (`app/IncidentCommandCli.hs`) drives the synchronous write path:
  individual commands (`command …`), the packaged `mass-casualty` scenario (with `--durable` to run
  the escalation PM + evacuation workflow), and `diagrams keiki` to regenerate the Keiki state
  diagrams.
- **`incident-command-worker`** (`app/IncidentCommandWorker.hs`) drives the asynchronous side:
  projections, the escalation timer, the inbox consumer, and the outbox publishers. Its subcommands
  are listed in [chapter 07](/docs/example-app/incident-command/07-wiring-and-the-cli).

Both executables wrap everything in a top-level OpenTelemetry span (`withApplicationSpan`) so a
single CLI invocation is one trace; tracing is off unless `SEIHOU_TRACING_ENABLED` is set.
