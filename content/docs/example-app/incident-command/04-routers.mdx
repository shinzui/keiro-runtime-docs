title: "04 — Routers"
description: "Content-based fan-out: the resource-dispatch and capacity-change routers resolve targets from read-model state and emit commands."
---

A **router** is keiro's stateless content-based dispatch: given one accepted event, it queries
read-model state, resolves a set of target aggregates, and emits a command to each. Unlike a process
manager it has **no memory** — it is a pure function from event to commands. Read the keiro
[routers &amp; content-based dispatch](/docs/keiro/explanation/routers-and-content-based-dispatch)
explanation, the [router](/docs/keiro/reference/router) reference, and the
[event fan-out cookbook](/docs/keiro/cookbook/event-fan-out-with-routers) alongside this. Incident
Command has two routers.

## The resource-dispatch router

When a `ResourceDispatched` event lands, this router decides whether the dispatched resource should
be assigned to the incident, by querying the `field_resource_assignments` read model and emitting an
`AssignResource` command to the FieldResource aggregate:

```haskell
-- services/incident-command/src/IncidentCommand/ResourceRouter.hs
resourceDispatchRouter
  :: (IOE :> es, Store :> es)
  => Router ResourceDispatchedData (HsPred FieldResourceRegs FieldResourceCommand)
       FieldResourceRegs FieldResourceVertex FieldResourceCommand FieldResourceEvent es
resourceDispatchRouter =
  Router
    { name = "incident-command-resource-dispatch"
    , key = \dispatched -> showText dispatched.incidentId <> ":" <> showText dispatched.resourceId
    , resolve = \dispatched -> do
        result <- runQuery Nothing fieldResourceAssignmentReadModel (FieldResourceAssignmentQuery dispatched.resourceId)
        pure $
          case result of
            Right (Just assignment)
              | assignmentTargetsIncident dispatched assignment ->
                  [ PMCommand
                      { target = fieldResourceCommandStream dispatched.resourceId
                      , command = AssignResource AssignResourceData { … }
                      }
                  ]
            _ -> []
    , targetEventStream = fieldResourceEventStream
    , targetProjections = const [fieldResourceAssignmentProjection]
    }
```

The four fields are the whole router contract:

- **`key`** — a dedupe/idempotency key for this routing decision (here `incidentId:resourceId`), so
  re-running the router on the same source event is safe.
- **`resolve`** — the content-based part: query a read model and return a list of `PMCommand`s, each
  naming a `target` stream and a `command`. Returning `[]` means "nothing to do."
- **`targetEventStream`** / **`targetProjections`** — the EventStream the emitted commands run
  against, and the projections to run inline when they append.

`runRouterOnce` (wrapped here as `runResourceDispatchRouterOnce`) replays the source event and
applies each resolved command through the normal command cycle.

## The capacity-change router

The second router closes the cross-service loop. When Incident Command's inbox accepts a
**hospital capacity snapshot** from the other service, this router finds the local incidents in the
snapshot's region that still have pending transfer needs and tells each to **reconsider its transfer
plan**:

```haskell
-- services/incident-command/src/IncidentCommand/CapacityChangeRouter.hs
capacityChangeRouter
  :: (IOE :> es, Store :> es)
  => Router AcceptedHospitalCapacitySnapshot (HsPred IncidentRegs IncidentCommand)
       IncidentRegs IncidentVertex IncidentCommand IncidentEvent es
capacityChangeRouter =
  Router
    { name = "incident-command-capacity-change"
    , key = \snapshot -> snapshot.messageId
    , resolve = \snapshot -> do
        result <- resolveCapacityIncidentCandidates snapshot
        case result of
          Left _ -> pure []
          Right candidates -> pure (rights (capacityChangeCommandForCandidate snapshot <$> candidates))
    , targetEventStream = incidentEventStream
    , targetProjections = const [incidentDashboardProjection]
    }
```

Its `resolve` has a content gate before it even queries — a snapshot only routes if the destination
is **not in total divert** and has **free ICU beds**:

```haskell
-- services/incident-command/src/IncidentCommand/CapacityChangeRouter.hs
capacitySnapshotCanRoute :: AcceptedHospitalCapacitySnapshot -> Bool
capacitySnapshotCanRoute snapshot =
  snapshot.divertStatus /= "total-divert" && snapshot.availableIcuBeds > 0
```

When it does route, it queries `pending_transfer_needs` grouped by region (a SQL `GROUP BY
incident_id`) and emits one `ReconsiderTransferPlan` command per candidate incident. This is the
mirror image of the resource router: same `Router` shape, different read model and target
aggregate. Both demonstrate the keiro rule — *routers query read models to resolve targets; they
never hold state of their own.*

This router is driven from the inbox path; the [cross-service tour](/docs/example-app/cross-service)
shows where the accepted snapshot comes from.
