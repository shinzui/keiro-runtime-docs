Short answers to the questions people ask most about kiroku (記録, an append-only
PostgreSQL event store). Each answer links out for depth.

<Accordions>
  <Accordion title="Does kiroku run my database migrations?">
    No. kiroku expects its schema to exist before you use it — migrate first, then point kiroku at
    the database. Use the native component and operator gate in
    [Run against PostgreSQL](/docs/kiroku/how-to/run-against-postgres).
  </Accordion>
  <Accordion title="What changes when I upgrade to Kiroku 0.3?">
    Backward cursors now paginate correctly; invalid stream names and empty batches fail before
    database work; link and transaction failures preserve typed evidence; subscription live sources
    depend on target/group shape; checkpoint-load and zero-buffer failures are loud; and the CLI is
    a remote HTTP client. Follow [Upgrade to Kiroku 0.3](/docs/kiroku/how-to/upgrade-to-0-3) for the
    compatibility and operator matrices.
  </Accordion>
  <Accordion title="Is kiroku ready for production use?">
    kiroku is in active development and its APIs may still change. Pin a version and read the
    maturity banner on the kiroku landing page before depending on it.
  </Accordion>
  <Accordion title="How does kiroku relate to keiro, keiki, and shibuya?">
    kiroku is the persistence foundation the rest of the family writes through. See [The keiro
    family](/docs/getting-started/the-keiro-family).
  </Accordion>
</Accordions>
