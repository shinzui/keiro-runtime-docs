The types on this page live in `Kiroku.Store.Types` (and `Kiroku.Store.Error`). They are the
vocabulary every operation speaks. Records use `DuplicateRecordFields` + `OverloadedLabels` +
`generic-lens`, so application code reads fields with labels — `evt ^. #globalPosition`.

## Newtype identifiers

```haskell
newtype StreamName     = StreamName Text       -- "orders-1"; prefix before first '-' is the category
newtype StreamId       = StreamId Int64        -- DB surrogate id (rarely used by app code)
newtype EventId        = EventId UUID          -- UUIDv7 by default
newtype EventType      = EventType Text        -- free-form discriminator, e.g. "OrderPlaced"
newtype StreamVersion  = StreamVersion Int64   -- per-stream; starts at 0, +1 per event
newtype GlobalPosition = GlobalPosition Int64  -- gap-free $all counter; the subscription cursor
newtype CategoryName   = CategoryName Text     -- a StreamName's prefix before the first '-'
```

## Result and info types

```haskell
data AppendResult = AppendResult
  { streamId :: StreamId, streamVersion :: StreamVersion, globalPosition :: GlobalPosition }
  -- describes the LAST event in the appended batch

data LinkResult = LinkResult { streamId :: StreamId, streamVersion :: StreamVersion }
  -- no globalPosition: linking does not advance $all

data StreamInfo = StreamInfo
  { id :: StreamId, name :: StreamName, version :: StreamVersion
  , createdAt :: UTCTime, deletedAt :: Maybe UTCTime }

data EventFilter
  = FilterCorrelation !UUID
  | FilterCausationDescendants !EventId
  | FilterCausationAncestors !EventId
```

## `ExpectedVersion` (the concurrency guard)

```haskell
data ExpectedVersion
  = NoStream                    -- must NOT exist; creates it (else StreamAlreadyExists)
  | StreamExists                -- must exist, version irrelevant (else StreamNotFound)
  | ExactVersion !StreamVersion -- current version must match exactly (else WrongExpectedVersion)
  | AnyVersion                  -- upsert / create-or-append
```

See [Optimistic concurrency](/docs/kiroku/explanation/optimistic-concurrency) for the full
mapping of constructor → failure.

## `StoreError`

The error type interpreters fail with (`runStoreIO` returns `Either StoreError`).

| Constructor | Meaning |
|---|---|
| `WrongExpectedVersion !StreamName !ExpectedVersion !StreamVersion` | An `ExactVersion` expectation did not match; carries the actual version. |
| `StreamNotFound !StreamName` | The stream does not exist (or is soft-deleted). |
| `ReservedStreamName !StreamName` | Target is store-internal (`$all`) and cannot be appended to. |
| `StreamAlreadyExists !StreamName` | `NoStream` expectation against an existing stream (or a conflicting link target). |
| `DuplicateEvent !(Maybe EventId)` | A caller-supplied `eventId` collided with a committed event. |
| `PoolAcquisitionTimeout` | The pool timed out acquiring a connection; usually retryable. |
| `ConnectionLost !Text` | A network/session error tore down the connection mid-operation; usually retryable. |
| `UnexpectedServerError !Text !Text` | A PostgreSQL `SQLSTATE` outside the recognised set (code, message); investigate. |
| `ConnectionError !Text` | Backward-compatible catch-all; prefer the specific constructors. |

<Callout type="warn">
  `WrongExpectedVersion` and `DuplicateEvent` are *expected* control-flow signals, not bugs —
  retry the decide loop on the former, treat the latter as success for an idempotent append.
  `PoolAcquisitionTimeout` / `ConnectionLost` are generally retryable after a small backoff.
</Callout>
