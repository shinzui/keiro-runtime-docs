---
title: "Run against PostgreSQL"
description: "Apply Kiroku's native migration plan and configure ConnectionSettings for the dedicated kiroku schema."
---

Apply Kiroku's schema as a deployment step, then open the store with matching connection settings.
Opening a `Store` never runs DDL.

<Callout type="info">
  Assumes PostgreSQL is running and `kiroku-store-migrations` is available. If you are starting
  from zero, follow the [getting-started tutorial](/docs/kiroku/tutorials/getting-started).
</Callout>

## 1. Inspect and apply the native plan

```console
export DATABASE_URL='postgresql://service_owner@localhost/app'

cabal run kiroku-store-migrate -- plan --json
cabal run kiroku-store-migrate -- status --json
cabal run kiroku-store-migrate -- verify --json
cabal run kiroku-store-migrate -- up --lock-timeout 30000 --json
cabal run kiroku-store-migrate -- verify --json
```

On a fresh database, preflight verification reports pending migrations. Stop on changed applied
bytes, unknown rows, gaps, or unresolved state. Post-apply verification must report no issues.
Preserve the JSON with the deployment record.

The component creates all event-store objects in `kiroku`. Its manifest contains seven historical
migrations plus the native `0008-schema-management-comment` canary. It does not use `public` for
Kiroku tables.

For an existing database whose history is recorded by Codd, do not run `up` first. Use the exact
checked-in Codd mapping to [import predecessor history](#existing-codd-database) without replaying
the seven historical SQL payloads.

## 2. Configure the store connection

```haskell
let settings = (defaultConnectionSettings "postgresql://app@localhost/app")
      { poolSize = 20
      , schema = "kiroku"
      , extraSearchPath = ["public"]
      }

withStore settings \store -> do
  -- The store is open against the already-migrated database.
  pure ()
```

`schema` controls both `search_path` and the `LISTEN/NOTIFY` channel, so it must match the migrated
schema. `extraSearchPath` makes application projection tables resolvable. Tune
`idleInTransactionTimeout` and `statementTimeout` for the workload.

## Existing Codd database

Quiesce writers and back up the database. Build `kirokuMigrationPlan`, then configure the supported
source with `kirokuCoddSourceConfig`. The checked-in mapping covers every one of the seven historical
rows:

```haskell
plan <- either (fail . show) pure kirokuMigrationPlan
config <-
  either (fail . show) pure
    (kirokuCoddSourceConfig sourceProvider True reason Confirmed)

report <-
  importCoddHistory
    defaultImportOptions
    config
    targetProvider
    plan
    kirokuCoddHistoryMappings
```

Strict import checks the source ledger, exact supplied payloads, and checked-in manifest before it
atomically writes native ledger and audit rows. It executes no target DDL. After import, run normal
`verify`, `up`, and `verify`; only the native canary should be newly applied. Native commands never
inspect the Codd ledger implicitly.

## Verify runtime readiness

```console
psql "$DATABASE_URL" -c '\dt kiroku.*'
```

The schema includes `streams`, `events`, `stream_events`, `subscriptions`, and `dead_letters`.
After that check, a `withStore` against the same database should open and accept an append.

See [Schema migrations](/docs/kiroku/reference/schema-migrations) for the API and CLI contract, and
[Deploy a migration plan](/docs/pg-migrate/how-to/deploy-a-migration-plan) for the production gate.
