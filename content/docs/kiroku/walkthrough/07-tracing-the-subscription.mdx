## What this part covers

`kiroku-otel/src/Kiroku/Otel/Subscription.hs` — the handler that turns a subscription worker's
lifecycle into OpenTelemetry spans. It is the natural sequel to the subscription machinery: the
[worker](/docs/kiroku/walkthrough/02-the-worker-driver) emits a structured `KirokuEvent` on every
state transition, and this module is a **pure read-side consumer** of that stream. It adds no
behaviour to the worker, changes no core type, and adds no `hs-opentelemetry` dependency to
`kiroku-store`. If you are contributing here, this is the file to understand.

For the API surface (span names, attribute keys) see the
[reference](/docs/kiroku/reference/opentelemetry); for _why_ the span shapes are what they are, the
[explanation](/docs/kiroku/explanation/tracing-subscriptions). This part is about the **code**.

## The seam it plugs into

Everything starts from one field on the connection settings:

```haskell
-- kiroku-store/src/Kiroku/Store/Connection.hs
eventHandler :: !(Maybe (KirokuEvent -> IO ()))
```

The worker calls this callback **synchronously on its own thread** for every transition (see
[`Observability.hs`](/docs/kiroku/explanation/subscriptions-and-consumer-groups) for the full
`KirokuEvent` sum). `subscriptionTraceHandler` is a factory that produces exactly such a callback:

```haskell
-- kiroku-otel/src/Kiroku/Otel/Subscription.hs
subscriptionTraceHandler :: Tracer -> IO (KirokuEvent -> IO ())
subscriptionTraceHandler tracer = do
  cells <- newIORef Map.empty
  pure (onEvent tracer cells)
```

Two things to notice already. It is `IO (KirokuEvent -> IO ())`, not a pure function — because it
allocates `cells`, the registry that holds all open-span state. And the returned closure captures
both the `Tracer` and that registry, so each installed handler owns its own bookkeeping.

## Why a mutable registry at all

The handler is called from more than one thread. A **consumer group runs one worker per member**,
each on its own thread ([part 06](/docs/kiroku/walkthrough/06-consumer-groups-and-policy)), and they
all share the one handler closure. Span A's "open" and span A's "close" arrive as two separate calls
that must find the same in-flight `Span` object — so the handler needs mutable state that survives
across calls and is safe under concurrency.

The state is a `Map` from a per-worker key to that worker's open spans, but the value is held behind
a per-key `IORef` rather than inline — that **striping** is what keeps workers off a shared lock:

```haskell
-- A worker is identified by its subscription name plus group member (if any).
type SpanKey = (Text, Maybe Int32)

-- The striped registry: an outer IORef of per-key cells, each its own single-writer IORef.
type SpanCells = IORef (Map SpanKey (IORef OpenState))

data OpenState = OpenState
  { catchup   :: !(Maybe Span)      -- catch-up episode (opened on Started)
  , reconnect :: !(Maybe Span)      -- reconnect episode (opened on first Reconnecting)
  , pause     :: !(Maybe Span)      -- backpressure pause episode (opened on Paused)
  , retries   :: !(Map Int64 Span)  -- per-poison-event retries, keyed by global position
  }
  deriving stock (Generic)
```

The registry maps each `SpanKey` to its **own** `IORef OpenState`, so two members never collide:
each `(name, member)` has its own cell. A non-grouped worker keys on `(name, Nothing)`. Within one
worker the four fields track the four kinds of episode that can be open _at the same time_ — a
worker can be retrying a poison event while also inside a reconnect episode, so they cannot share one
slot.

<Callout type="info">
  The outer `IORef` is **read-mostly**: it is mutated (via `atomicModifyIORef'`) only when a key is
  first seen (`Started`) or removed (`Stopped`). Each key's inner `IORef OpenState` is
  **single-writer** — only its own worker thread touches it — so the per-batch hot path reads the
  outer map lock-free and writes only its own cell with a plain `writeIORef`. There is no shared
  lock at all. See [the hot path holds no shared lock](#the-hot-path-holds-no-shared-lock) below for
  why that is both faster and still correct.
</Callout>

## The state-cell combinators

Every branch of the dispatch goes through one of these helpers, so the registry mechanics live in
one place. The hot-path helper, `withKey`, never touches a shared lock — it resolves the key's own
cell, reads it, runs the span `IO`, and writes the cell back:

```haskell
-- Resolve a key's own cell, creating an empty one on first touch (its Started event).
-- The outer map is read lock-free; it is written (atomicModifyIORef') only to insert a new key.
cellFor :: SpanCells -> SpanKey -> IO (IORef OpenState)
cellFor cells key = do
  m <- readIORef cells
  case Map.lookup key m of
    Just ref -> pure ref
    Nothing  -> do
      fresh <- newIORef emptyOpenState
      atomicModifyIORef' cells $ \m' ->
        case Map.lookup key m' of
          Just ref -> (m', ref)                      -- another thread inserted meanwhile; reuse
          Nothing  -> (Map.insert key fresh m', fresh)

-- Run a state-update against a key's own cell; span IO in `f` holds no shared lock.
withKey :: SpanCells -> SpanKey -> (OpenState -> IO OpenState) -> IO ()
withKey cells key f = do
  ref <- cellFor cells key       -- outer map: lock-free read (rare insert on a new key)
  st  <- readIORef ref           -- this key's cell only
  st' <- f st                    -- span IO: no shared lock
  writeIORef ref st'             -- this key's cell only

-- Run a finalizer against a key's OpenState, then drop the key from the outer registry.
dropKey :: SpanCells -> SpanKey -> (OpenState -> IO ()) -> IO ()
dropKey cells key f = do
  mref <- atomicModifyIORef' cells $ \m -> (Map.delete key m, Map.lookup key m)
  st   <- maybe (pure emptyOpenState) readIORef mref
  f st                           -- unlocked: end spans
```

`withKey` is the workhorse: resolve the worker's own cell (creating it on the first event), read it,
run the `IO` closure that opens/closes spans, then write the cell back — all without serializing on
anything another worker touches. `dropKey` is the terminal variant — it removes the key from the
outer map in one `atomicModifyIORef'`, then runs the finalizer unlocked. It is used only for
`KirokuEventSubscriptionStopped`, so a stopped worker leaves no entry behind. The next section
explains why this lock-free scheme is safe.

## The dispatch, event by event

`onEvent` is one big `\case` over the `KirokuEvent` constructors. The pattern is always the same:
pick the key with `keyOf`, then `withKey` to mutate that worker's open spans. A few representative
branches:

```haskell
onEvent :: Tracer -> SpanCells -> KirokuEvent -> IO ()
onEvent tracer cell = \case
  KirokuEventSubscriptionStarted name pos grp ->
    withKey cell (keyOf name grp) $ \st -> do
      mapM_ closeSpan (catchup st)                -- defensively close a leftover catch-up span
      sp <- openSpan tracer spanCatchup $
              baseAttrs name grp
                ++ [(attrState, toAttribute ("catchup" :: Text)), (attrCheckpoint, posAttr pos)]
      pure st{catchup = Just sp}

  KirokuEventSubscriptionCaughtUp name pos grp ->
    withKey cell (keyOf name grp) $ \st -> do
      let endAttrs = [(attrCheckpoint, posAttr pos)]
      mapM_ (`closeSpanWith` endAttrs) (catchup st)                       -- close catch-up...
      mapM_ (\sp -> setStatus sp Ok >> closeSpanWith sp endAttrs) (reconnect st)  -- ...or reconnect
      mapM_ (\sp -> setStatus sp Ok >> closeSpan sp) (Map.elems (retries st))     -- retries succeeded
      pure st{catchup = Nothing, reconnect = Nothing, retries = Map.empty}
```

`Started` **opens** the catch-up episode and parks the live `Span` in `catchup`. `CaughtUp` is the
matching **close**: a single caught-up signal resolves whichever of catch-up / reconnect was open,
and — because the worker has demonstrably moved past every event — marks any open retry spans `Ok`
and clears them. Note the asymmetry: opens write one field; closes often resolve several, because
reaching `Live` is good news for everything that was pending.

The per-batch **deliver** span is the one transition that fires constantly, on every target and in
both phases — it is keyed on the new `KirokuEventSubscriptionDelivered` event and opens-and-ends a
span at once:

```haskell
  KirokuEventSubscriptionDelivered name count phase grp ->
    withKey cell (keyOf name grp) $ \st -> do
      let stateText = case phase of
            DeliveredCatchUp -> "catchup" :: Text
            DeliveredLive    -> "live"
      sp <- openSpan tracer spanDeliver $
              baseAttrs name grp
                ++ [(attrState, toAttribute stateText), (attrBatchRows, intAttr count)]
      closeSpan sp
      case phase of
        DeliveredLive -> do                         -- a live batch means the worker advanced...
          mapM_ (\rsp -> setStatus rsp Ok >> closeSpan rsp) (Map.elems (retries st))  -- ...past any retry
          pure st{retries = Map.empty}
        DeliveredCatchUp -> pure st                 -- catch-up delivery resolves nothing
```

This is the span that closed the old blind spot. Previously the per-batch span keyed on
`KirokuEventSubscriptionFetched`, which only the **DB-driven live loops** emit — so an `$all`
subscription's live phase (pushed from the in-memory queue, never "fetched") produced no per-batch
span and went dark on the timeline once it caught up. `Delivered` is emitted from the single
delivery primitive on **every** path and **both** phases, carrying `kiroku.subscription.state =
catchup | live` and `kiroku.batch.rows`, so every batch is now traced. A live delivery also doubles
as "the worker moved on," so it closes any open retry spans `Ok` — the same resolution `CaughtUp`
does; a catch-up delivery is just progress and resolves nothing.

<Callout type="info">
  `KirokuEventSubscriptionFetched` is now a **no-op** in the tracer — matched explicitly (`{} -> pure
  ()`) so the incomplete-patterns net still fires, but not traced. The DB-driven live loops emit
  _both_ `Fetched` (per fetch, the live-fetch-rate signal) and `Delivered` (per delivered batch);
  keying the span on `Delivered` is what stops the category/consumer-group live path from emitting
  two spans per batch.
</Callout>

The retry branch is the one place a `Map` rather than a `Maybe` is needed, because a worker can have
several poison events in flight, each its own episode keyed by global position:

```haskell
  KirokuEventSubscriptionRetrying name pos attempt grp ->
    withKey cell (keyOf name grp) $ \st -> do
      let GlobalPosition p = pos
      case Map.lookup p (retries st) of
        Nothing -> do                                   -- first attempt: open a per-position span
          sp <- openSpan tracer spanRetrying $
                  baseAttrs name grp
                    ++ [ (attrState, toAttribute ("retrying" :: Text))
                       , (attrEventPos, toAttribute p), (attrAttempt, intAttr attempt) ]
          pure st{retries = Map.insert p sp (retries st)}
        Just sp -> do                                   -- later attempt: annotate the open span
          spanEvent sp "retry.attempt" [(attrAttempt, intAttr attempt)]
          setAttrs sp [(attrAttempt, intAttr attempt)]
          pure st
```

This "first occurrence opens a span, subsequent occurrences add a span _event_" idiom recurs:
`Reconnecting` does the same with `reconnect` and a `reconnect.attempt` event. It keeps one span
per logical episode instead of one per retry tick.

### Episode precedence and DB errors

`KirokuEventSubscriptionDbError` does not always open a span. If an episode is already open it
annotates that instead, choosing the most relevant one:

```haskell
primaryEpisode :: OpenState -> Maybe Span
primaryEpisode st = reconnect st <|> catchup st <|> pause st
```

The `<|>` on `Maybe` gives a fixed precedence — reconnect, then catch-up, then pause — so a DB error
during a reconnect attaches to the reconnect span (the thing actually going wrong) rather than
spawning a competing `db_error` span. Only when _no_ episode is open does it emit a standalone
`spanDbError`. The same `primaryEpisode` is where `Stopped` records its stop reason before tearing
everything down.

### Stop: the teardown branch

```haskell
  KirokuEventSubscriptionStopped name pos reason grp ->
    dropKey cell (keyOf name grp) $ \st -> do
      let stopAttrs = [ (attrStopReason, toAttribute (T.pack (show reason)))
                      , (attrCheckpoint, posAttr pos) ]
      -- Always emit a short standalone terminal span, even with no open episode.
      term <- openSpan tracer spanStopped (baseAttrs name grp ++ stopAttrs)
      closeSpan term
      mapM_ (`setAttrs` stopAttrs) (primaryEpisode st)   -- also annotate the most relevant open episode
      mapM_ closeSpan (catchup st)                       -- then end EVERY open span so none leaks
      mapM_ closeSpan (reconnect st)
      mapM_ closeSpan (pause st)
      mapM_ closeSpan (Map.elems (retries st))
```

This is the only `dropKey` branch. It **always** emits a standalone `spanStopped` carrying the stop
reason and checkpoint — so a healthy worker that catches up, runs `Live`, and stops with no open
episode still leaves a terminal marker on the timeline. It then records the stop reason on the
primary episode _if one is open_, closes **every** span the worker had open, and removes its cell
from the registry. A worker that crashes mid-episode therefore still flushes its spans (with the
stop reason recorded) rather than leaking them — directly serving the export-on-end constraint from
the [explanation](/docs/kiroku/explanation/tracing-subscriptions).

## The span helpers, and one easy-to-miss subtlety

The low-level helpers are thin wrappers over `OpenTelemetry.Trace.Core`. All spans are **root**
spans (`Context.empty`):

```haskell
openSpan :: Tracer -> Text -> [(Text, Attribute)] -> IO Span
openSpan tracer name attrs = do
  sp <- createSpan tracer Context.empty name defaultSpanArguments
  setAttrs sp attrs
  pure sp

closeSpan :: Span -> IO ()
closeSpan sp = endSpan sp Nothing       -- end at current time
```

The subtlety is in `setAttrs`, and it is the kind of thing a future contributor will trip over:

```haskell
-- Set attributes on a span, OVERRIDING any existing value for the same key.
setAttrs :: Span -> [(Text, Attribute)] -> IO ()
setAttrs sp = mapM_ (\(k, v) -> addAttribute sp k v)
```

It uses the singular `addAttribute` (an insert/override) in a loop, **not** the bulk
`addAttributes`. The bulk version does a left-biased union that _keeps_ a key's existing value, so
refreshing a checkpoint position or attempt counter would silently no-op. If you batch these for
"efficiency," you reintroduce that bug.

## The hot path holds no shared lock

The `eventHandler` runs synchronously on the worker's emit-site thread, and a consumer group drives
the shared handler from one thread per member. An earlier version held the whole `Map SpanKey
OpenState` behind a single `MVar`; once the per-batch deliver span was added — emitted on **every**
batch on **every** target — that shared lock would serialize **every** worker across span syscalls
(`createSpan` / `endSpan` are each a `clock_gettime` plus an allocation). The striped registry
removes the shared structure from the hot path entirely: the per-batch path reads the outer map
lock-free, then reads and writes only its **own** key's `IORef`. Workers never queue behind each
other.

This is safe because each `SpanKey` is **single-writer**. Every `(subscription name, member)` is
emitted by exactly one worker thread, and the `eventHandler` callback is synchronous on that thread,
so a key's `OpenState` cannot change between the `readIORef` and the `writeIORef` of its cell — there
is no second writer to race with. The only cross-thread structure is the **outer** registry, and it
is mutated only on the rare `Started` (insert a key's cell) and `Stopped` (delete it), each via
`atomicModifyIORef'`, so a concurrent insert of _another_ key never clobbers this one. `Stopped` is a
key's last event, so nothing touches its cell afterward.

Even with the lock out of the way, span **export** must still stay off the worker thread:

<Callout type="warn">
  Configure the `Tracer`'s `TracerProvider` with a **batch** span processor (the SDK default).
  Export does not happen in `endSpan` — it happens later on a background thread. With a
  simple/synchronous processor, `endSpan` would block on network I/O **on the worker's own thread**,
  coupling event-processing throughput to your tracing backend's latency.
</Callout>

## Contributing: adding a newly-traced event

The `KirokuEvent` constructor set is **additive** — new events are added, existing ones are never
changed. `onEvent` is a total `\case`, so a new constructor surfaces as a `-Wincomplete-patterns`
warning at compile time rather than a silently untraced transition. To trace a new event:

1. Decide its span shape (episode / per-batch / standalone — see the
   [explanation](/docs/kiroku/explanation/tracing-subscriptions)).
2. If it is an episode, add a field to `OpenState` (and `emptyOpenState`) to hold the open `Span`;
   if it pairs open/close like retries, key it in a `Map`.
3. Add the `\case` branch: `withKey` (or `dropKey` for a terminal event), open or close the span,
   set attributes via `setAttrs`.
4. Export any new span-name / attribute-key constant at the bottom of the module and document it in
   the [reference](/docs/kiroku/reference/opentelemetry).

The non-subscription operational events (`KirokuEventNotifierReconnecting`, `...Reconnected`,
`KirokuEventPublisherPoolError`, `KirokuEventHardDeleteIssued`) are intentionally matched and
ignored here — they are not subscription-FSM transitions. Leave them explicit so the
incomplete-patterns safety net keeps working.

## A note on the append side

This part covered the subscription handler. The other module, `Kiroku.Otel.TraceContext`, is the
producer/consumer trace-propagation seam (`injectTraceContext` / `extractTraceContext`). It is
worth one contributor caveat: `injectTraceContext` is marked `{-# NOINLINE #-}` and uses
`unsafePerformIO` to call the propagator's `IO`-typed `encodeSpanContext` from a pure interface. That
is sound because the encode is observably pure on the frozen span returned by `wrapSpanContext` (no
shared mutable state, no exceptions) — but it is exactly the sort of thing not to "tidy up" without
understanding why it is there.

## The handler, in one sentence

`subscriptionTraceHandler` is a pure read-side consumer of the worker's `KirokuEvent` stream that
holds per-worker open-span state in a striped, lock-free registry of per-key `IORef`s, opening short
spans on state entry, a `deliver` span on every batch, and a `stopped` span on every exit — ending
them promptly on the matching exit, so the trace timeline shows real episode durations and nothing
leaks when a worker stops.
