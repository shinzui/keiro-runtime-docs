## What this part covers

`kiroku-store/src/Kiroku/Store/SQL.hs` — the four `append*SQL` templates. This is the most important
file on the write path, because **the append's correctness is the statement's correctness.** There is
no Haskell-level transaction around a single-stream append; atomicity, the optimistic version check,
and the global-position assignment are all properties of this one CTE. We read `appendExpectedVersion`
in full, then see how the other three differ.

## The whole statement

Here is `appendExpectedVersionSQL`, the `ExactVersion` variant — the one that implements the classic
optimistic-concurrency check:

```sql
-- src/Kiroku/Store/SQL.hs — appendExpectedVersionSQL
WITH
  new_events AS (
    SELECT * FROM unnest($1::uuid[], $2::text[], $3::uuid[], $4::uuid[],
                         $5::jsonb[], $6::jsonb[], $7::timestamptz[])
      WITH ORDINALITY AS t(event_id, event_type, causation_id, correlation_id,
                           data, metadata, created_at, idx)
  ),
  stream_update AS (                              -- (1) the optimistic check
    UPDATE streams
    SET stream_version = stream_version + (SELECT count(*) FROM new_events)
    WHERE stream_name = $8
      AND stream_version = $9                     -- <-- only proceeds if the version still matches
      AND deleted_at IS NULL
    RETURNING stream_id, stream_version - (SELECT count(*) FROM new_events) AS initial_version
  ),
  inserted_events AS (                            -- (2) write the event bodies
    INSERT INTO events (event_id, event_type, causation_id, correlation_id, data, metadata, created_at)
    SELECT event_id, event_type, causation_id, correlation_id, data, metadata, created_at
    FROM new_events
    WHERE EXISTS (SELECT 1 FROM stream_update)    -- <-- nothing inserts unless the check passed
    ORDER BY idx
  ),
  source_links AS (                               -- (3) link events into their own stream
    INSERT INTO stream_events (event_id, stream_id, stream_version, original_stream_id, original_stream_version)
    SELECT ne.event_id, su.stream_id, su.initial_version + ne.idx, su.stream_id, su.initial_version + ne.idx
    FROM new_events ne CROSS JOIN stream_update su
  ),
  all_update AS (                                 -- (4) bump the global $all counter
    UPDATE streams
    SET stream_version = stream_version + (SELECT count(*) FROM new_events)
    WHERE stream_id = 0
      AND EXISTS (SELECT 1 FROM stream_update)
    RETURNING stream_version - (SELECT count(*) FROM new_events) AS initial_global_version
  ),
  all_links AS (                                  -- (5) link events into $all in batch order
    INSERT INTO stream_events (event_id, stream_id, stream_version, original_stream_id, original_stream_version)
    SELECT ne.event_id, 0, au.initial_global_version + ne.idx, su.stream_id, su.initial_version + ne.idx
    FROM new_events ne CROSS JOIN all_update au CROSS JOIN stream_update su
  )
SELECT su.stream_id,                              -- (6) report the final positions
       su.initial_version       + (SELECT count(*) FROM new_events),
       au.initial_global_version + (SELECT count(*) FROM new_events)
FROM stream_update su CROSS JOIN all_update au
```

It looks dense, but it is six steps in a fixed order. Read it once top to bottom with the annotations,
then we will pull out the four properties that matter.

## Property 1 — it is atomic because it is one statement

The entire append is a **single SQL statement**. PostgreSQL executes one statement atomically (in
autocommit, it is its own implicit transaction), so either every CTE term applies or none does. There
is no window in which the events exist but the stream version has not moved, or the source link is
written but the `$all` link is not. The interpreter does not need a `BEGIN`/`COMMIT` for a
single-stream append precisely because the statement already gives all-or-nothing.

## Property 2 — the optimistic check is `WHERE stream_version = $9`

This one clause (1) is the whole optimistic-concurrency mechanism. The `UPDATE` only matches a row if
the stream's current version still equals the caller's expected version `$9`. Then every downstream
term is guarded by `WHERE EXISTS (SELECT 1 FROM stream_update)`. So:

- **Version matches** → `stream_update` returns one row → events insert, links write, `$all` bumps,
  and the final `SELECT` returns one row (the new positions).
- **Version does not match** (a concurrent writer advanced it, or the stream was soft-deleted) →
  `stream_update` matches **zero** rows → every `EXISTS` is false → **nothing is written** → the final
  `SELECT` (`FROM stream_update …`) returns **zero rows**.

That empty result is how a conflict travels back to Haskell: no exception, no partial write, just
`Right Nothing`, which [part 03](/docs/kiroku/write-path/03-outcomes-and-errors) turns into
`WrongExpectedVersion`. The same shape gives each precondition its semantics — `NoStream` against an
existing stream, `StreamExists` against a missing one — all surface as the same clean "zero rows."

## Property 3 — `$all` is dual-written, gap-free, in batch order

kiroku keeps a single global sequence (`$all`, the seeded `stream_id = 0` row) so subscriptions and
`readAllForward` have one totally-ordered spine ([The $all stream and global
order](/docs/kiroku/explanation/all-stream-and-global-order)). Steps (4) and (5) maintain it _in the
same statement_ as the per-stream write:

- (4) bumps the `$all` counter by the batch size, returning `initial_global_version` (the value
  _before_ the bump).
- (5) inserts one `$all` junction row per event at `initial_global_version + idx`, where `idx` is the
  1-based `WITH ORDINALITY` index — so positions are **contiguous** (`+1, +2, …`) and assigned in
  batch order.

Because `all_update` carries the same `EXISTS (SELECT 1 FROM stream_update)` guard, a **rejected
append never touches `$all`**. That is why the global position is **gap-free**: positions are computed
as `initial + idx` inside a statement that either fully commits or writes nothing, so a failed append
consumes no positions (unlike a sequence/`nextval`, which would leave holes). The per-stream version is
assigned the same way (`initial_version + idx`), one-indexed within the stream.

## Property 4 — order is fixed by `ORDER BY idx`

`WITH ORDINALITY` stamps each unnested event with its position `idx` in the input arrays, and the body
insert uses `ORDER BY idx`. So the batch is written — and positioned, on both the source stream and
`$all` — in exactly the order the caller passed it. The columnar parameters from [part
01](/docs/kiroku/write-path/01-the-store-effect-and-append-api#parameter-packing-is-columnar-for-one-unnest)
are what `unnest` expands here.

## The other three variants differ only in step (1)

The four `append*SQL` templates share steps (2)–(6) verbatim; they differ only in how the first CTE
establishes the target stream and its `initial_version`:

| `ExpectedVersion` | First CTE                                                                                         | Returns 0 rows when…                       |
| ----------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------ |
| `ExactVersion v`  | `UPDATE … WHERE stream_version = v AND deleted_at IS NULL`                                        | version mismatch or soft-deleted           |
| `StreamExists`    | `UPDATE … WHERE stream_name = $8 AND deleted_at IS NULL` (no version clause)                      | stream missing or soft-deleted             |
| `NoStream`        | `INSERT INTO streams … ON CONFLICT (stream_name) DO NOTHING`                                      | stream already exists                      |
| `AnyVersion`      | `INSERT … ON CONFLICT (stream_name) DO UPDATE SET version = version + n WHERE deleted_at IS NULL` | only when the existing row is soft-deleted |

`AnyVersion` is the interesting one: a plain `INSERT` then a separate `UPDATE` in the same CTE would
not work, because data-modifying CTE terms cannot see each other's changes. A single
`INSERT … ON CONFLICT DO UPDATE` (an upsert) both creates the stream and bumps its version atomically,
which is exactly the "create-or-append" semantics. In every variant, the rest of the statement is
guarded on "did the first CTE produce a row," so the conflict-as-empty-result contract is uniform.

## Why it is correct under concurrency

The optimistic check would be worthless if two writers could both pass it. They cannot. The
`UPDATE streams … WHERE stream_version = $9` takes a **row lock** on the stream row for the duration of
the statement. Under the default `ReadCommitted` isolation, a second concurrent `ExactVersion` append
to the same stream **blocks** on that lock until the first commits, then re-evaluates its `WHERE`
against the now-committed version — which no longer equals `$9`, so it matches zero rows and is
rejected. There is no lost update and no need for `SERIALIZABLE`: serialization happens on the single
`streams` row, and the loser sees a clean empty result. (Multi-stream appends, which touch several
stream rows, need an explicit lock _order_ to avoid deadlocks — that is [part
04](/docs/kiroku/write-path/04-multi-stream-and-transactions).)
