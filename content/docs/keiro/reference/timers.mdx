The types and functions on this page live in `Keiro.Timer` (the wire types are defined in
`Keiro.Timer.Types` and the storage in `Keiro.Timer.Schema`, both re-exported from `Keiro.Timer`).
They schedule, claim, and fire durable timers backed by the `keiro_timers` table. For the
narrative, see [Understanding durable timers](/docs/keiro/explanation/durable-timers).

## `TimerId`

A timer's stable identifier. A caller-chosen id makes scheduling idempotent (rescheduling the same
id updates rather than duplicates).

```haskell
newtype TimerId = TimerId UUID
```

## `TimerRequest`

A request to schedule a timer.

```haskell
data TimerRequest = TimerRequest
  { timerId            :: !TimerId
  , processManagerName :: !Text
  , correlationId      :: !Text
  , fireAt             :: !UTCTime
  , payload            :: !Value
  }
```

    processManagerName: { type: "Text", description: "Identifies the saga instance the timer belongs to, so the firing can be routed back to it." },
    correlationId: { type: "Text", description: "The correlation key of the saga instance (e.g. the incident id)." },

## `TimerStatus`

A timer's lifecycle state.

```haskell
data TimerStatus = Scheduled | Firing | Fired | Cancelled | Dead
```

<TypeTable
  type={{
    Scheduled: { type: "—", description: "Waiting for its fireAt; claimable." },
    Firing: { type: "—", description: "Claimed by a worker and being processed. It becomes claimable again only after requeueStuckTimers or requeueStuckTimer moves it back to Scheduled." },
    Fired: { type: "—", description: "Successfully fired; terminal." },
    Cancelled: { type: "—", description: "Withdrawn before firing (via cancelTimer); terminal. ALSO the decode fallback for an unrecognized stored value." },
    Dead: { type: "—", description: "Abandoned after exceeding the attempt ceiling (via deadLetterTimer or the maxAttempts auto-dead-letter); terminal; carries an explanatory last_error." },
  }}
/>

<Callout type="info">
`statusFromText` maps any **unknown** stored status string to `Cancelled`. The stored strings are
`scheduled | firing | fired | cancelled | dead`. Cancellation and dead-lettering ship as the
[recovery operations](#recovery-operations) below.
</Callout>

## `TimerRow`

A timer row as stored: the original `TimerRequest` fields plus the live status, the attempt count
(incremented on each claim), and the fired event id once it fires.

```haskell
data TimerRow = TimerRow
  { timerId            :: !TimerId
  , processManagerName :: !Text
  , correlationId      :: !Text
  , fireAt             :: !UTCTime
  , payload            :: !Value
  , status             :: !TimerStatus
  , attempts           :: !Int
  , firedEventId       :: !(Maybe EventId)
  }
```

<TypeTable
  type={{
    timerId: { type: "TimerId", description: "The timer's stable id." },
    processManagerName: { type: "Text", description: "The owning process manager's name." },
    correlationId: { type: "Text", description: "The saga instance's correlation key." },
    fireAt: { type: "UTCTime", description: "When the timer became due." },
    payload: { type: "Value", description: "The opaque JSON payload." },

## `scheduleTimerTx`

Schedule a timer inside the caller's transaction (typically a process manager's append). Upserts on
`timerId`: a conflicting row is re-armed **only while it is still `Scheduled`**, so a timer that has
already fired or been cancelled is not resurrected.

```haskell
scheduleTimerTx :: TimerRequest -> Tx.Transaction ()
```

The re-arm guard is the `ON CONFLICT … WHERE keiro_timers.status = 'scheduled'` clause on the
underlying `INSERT`.

## `claimDueTimer`

Atomically claim the single earliest timer due at `now`, moving it to `Firing` and bumping its
attempt count. Uses `FOR UPDATE SKIP LOCKED` so concurrent workers each get a distinct timer.
Returns `Nothing` when none is due.

```haskell
claimDueTimer :: (Store :> es) => UTCTime -> Eff es (Maybe TimerRow)
```

## `markTimerFired`

Mark a claimed timer `Fired`, recording the id of the event its firing produced.

```haskell
markTimerFired :: (Store :> es) => TimerId -> EventId -> Eff es ()
```

## `countDueTimers`

Count timers that are `scheduled` and already due at `now` — the timer backlog. Read-only; mirrors
`claimDueTimer`'s `WHERE` clause but counts rather than locking. The worker records this as the
[`keiro.timer.backlog`](/docs/keiro/reference/telemetry#metric-catalogue) gauge.

```haskell
countDueTimers :: (Store :> es) => UTCTime -> Eff es Int
```

## `runTimerWorker`

Claim and fire at most one timer due at `now`, using `defaultTimerWorkerOptions`: no attempt ceiling,
and automatic requeue of `Firing` rows stale for five minutes. Each pass performs that stale-row sweep,
atomically claims the earliest due timer (marking it `Firing`), runs `fire` on it, and — **if** `fire`
returns the id of the event it produced — marks the timer `Fired`. Returns the claimed `TimerRow`, or
`Nothing` when nothing is due.

```haskell
runTimerWorker ::
  (IOE :> es, Store :> es) =>
  Maybe KeiroMetrics ->
  UTCTime ->
  (TimerRow -> Eff es (Maybe EventId)) ->
  Eff es (Maybe TimerRow)
```

A `fire` that returns `Nothing` (or a crash) leaves the timer `Firing`. With the default options, a
pass at least five minutes later requeues it to `Scheduled`, after which it can be claimed again —
this is what gives **at-least-once** firing. A `fire` that runs longer than that timeout can overlap a
reclaim, so handlers must be idempotent. The caller drives this on a schedule (e.g. once per tick) and
supplies `now`. The first argument is the opt-in
[`KeiroMetrics`](/docs/keiro/reference/telemetry#metrics-surface) handle (`Nothing` disables all
timer instruments); pass `Nothing` if you have not wired a `Meter`. See [Drive the timer
worker](/docs/keiro/how-to/drive-the-timer-worker).

<Callout type="info">
`runTimerWorker` gained an `IOE` constraint and the leading `Maybe KeiroMetrics` argument when the
metrics surface landed (it was `(Store :> es) => UTCTime -> …` in 0.1.0.0). Existing callers pass
`Nothing` for the new argument.
</Callout>

## `TimerWorkerOptions`

Options controlling `runTimerWorkerWith`.

```haskell
data TimerWorkerOptions = TimerWorkerOptions
  { maxAttempts      :: Maybe Int
  , requeueStuckAfter :: Maybe NominalDiffTime }

defaultTimerWorkerOptions :: TimerWorkerOptions
-- TimerWorkerOptions { maxAttempts = Nothing, requeueStuckAfter = Just 300 }

data TimerWorkerConfigError
  = InvalidTimerMaxAttempts Int
  | InvalidTimerRequeueStuckAfter NominalDiffTime

mkTimerWorkerOptions :: TimerWorkerOptions -> Either TimerWorkerConfigError TimerWorkerOptions
```

<TypeTable
  type={{
    maxAttempts: { type: "Maybe Int", default: "Nothing", description: "When Just n, a claimed timer whose post-claim attempts exceeds n is moved to Dead (auto-dead-lettered) instead of fired, rather than ping-ponging forever on a timer that never completes. Nothing never auto-dead-letters." },
    requeueStuckAfter: { type: "Maybe NominalDiffTime", default: "Just 300", description: "Before each pass, move Firing rows at least this old back to Scheduled. Nothing disables automatic recovery. A timeout shorter than the longest valid fire action can cause concurrent duplicate firing." },
  }}
/>

Validate options once before starting the loop. `mkTimerWorkerOptions` rejects a negative attempt
ceiling and a non-positive requeue timeout.

## `runTimerWorkerWith`

Like `runTimerWorker` but applies the given `TimerWorkerOptions`. `runTimerWorker` is
`runTimerWorkerWith defaultTimerWorkerOptions`.

```haskell
runTimerWorkerWith ::
  (IOE :> es, Store :> es) =>
  Maybe KeiroMetrics ->
  TimerWorkerOptions ->
  UTCTime ->
  (TimerRow -> Eff es (Maybe EventId)) ->
  Eff es (Maybe TimerRow)
```

`claimDueTimer` increments `attempts` *before* the ceiling check, so the comparison sees the
post-claim count: with `maxAttempts = Just 0` the very first claim dead-letters; with `Just 2` the
third claim does. A dead-lettered timer is moved to `Dead` with an explanatory `last_error` (e.g.
`"timer exceeded attempt ceiling of 2"`) and the `fire` action is **not** run.

Before that claim, `requeueStuckAfter = Just ttl` runs `requeueStuckTimers ttl now` and records the
count in `keiro.timer.requeued`. Requeued timers retain their original due time and are immediately
eligible on a later claim; there is no retry backoff. Choose `ttl` above the longest legitimate fire
duration, because a still-running action that crosses the timeout can be fired again.

## Recovery operations

A timer left in `Firing` by a crash is not directly claimable. The default worker first moves rows
stale for five minutes back to `Scheduled`; callers that set `requeueStuckAfter = Nothing` own that
recovery themselves. When a timer cannot make progress — its `fire` action keeps failing, or the row
is genuinely stranded — these operations let an operator inspect and resolve it. They live in
`Keiro.Timer.Schema` (re-exported from
`Keiro.Timer`). For the runbook, see [Understanding durable timers](/docs/keiro/explanation/durable-timers).

### `StuckTimerFilter`

Criteria selecting timers stranded in `Firing`. A row is "stuck" when its `status` is `firing` and it
matches every set bound.

```haskell
data StuckTimerFilter = StuckTimerFilter
  { minAge      :: !(Maybe NominalDiffTime)
  , minAttempts :: !(Maybe Int)
  }

anyStuckTimer :: StuckTimerFilter   -- StuckTimerFilter Nothing Nothing
```

<TypeTable
  type={{
    minAge: { type: "Maybe NominalDiffTime", default: "Nothing", description: "Row qualifies when it has been firing at least this long — its updated_at is at or before now - minAge." },
    minAttempts: { type: "Maybe Int", default: "Nothing", description: "Row qualifies when it has been claimed at least this many times." },
  }}
/>

### `findStuckTimers`

List timers stranded in `Firing` matching the filter, ordered oldest-first by `updated_at`.

```haskell
findStuckTimers :: (Store :> es) => UTCTime -> StuckTimerFilter -> Eff es [TimerRow]
```

### `countStuckTimers`

Count timers stranded in `Firing` matching the filter. The worker records `countStuckTimers now
anyStuckTimer` as the [`keiro.timer.stuck`](/docs/keiro/reference/telemetry#metric-catalogue) gauge.

```haskell
countStuckTimers :: (Store :> es) => UTCTime -> StuckTimerFilter -> Eff es Int
```

### `requeueStuckTimers` and `requeueStuckTimer`

The bulk form moves every `Firing` timer at least `olderThan` old back to `Scheduled` and returns the
number changed. `runTimerWorkerWith` invokes this before each claim when `requeueStuckAfter` is set.
Because `fire_at` is unchanged, a due timer can be claimed in that same worker pass.

```haskell
requeueStuckTimers :: (Store :> es) => NominalDiffTime -> UTCTime -> Eff es Int
```

Move a timer from `Firing` back to `Scheduled` so the ordinary claim loop re-fires it. Leaves
`fire_at` unchanged, so a due timer becomes immediately re-claimable. Idempotent (only `firing` rows
match). Returns `True` when a row changed.

```haskell
requeueStuckTimer :: (Store :> es) => TimerId -> Eff es Bool
```

### `cancelTimer`

Move a timer from `Scheduled` or `Firing` to the terminal `Cancelled` state so it never fires.
Terminal rows (`fired`, `cancelled`, `dead`) are left untouched. Idempotent. Returns `True` when a
row changed.

```haskell
cancelTimer :: (Store :> es) => TimerId -> Eff es Bool
```

### `deadLetterTimer`

Move a timer from `Scheduled` or `Firing` to the terminal `Dead` state, recording `reason` in
`last_error` so an operator can see why it was abandoned (`SELECT * FROM keiro_timers WHERE status =
'dead'`). Terminal rows are left untouched. Idempotent. Returns `True` when a row changed.

```haskell
deadLetterTimer :: (Store :> es) => TimerId -> Text -> Eff es Bool
```

## The `keiro_timers` table

The bootstrap migration creates the table and its due-timer index; the
`2026-05-17-03-00-00-keiro-timer-recovery` migration adds the `last_error` column and narrows the
index to active rows:

```sql
CREATE TABLE IF NOT EXISTS keiro_timers (
  timer_id UUID PRIMARY KEY,
  process_manager_name TEXT NOT NULL,
  correlation_id TEXT NOT NULL,
  fire_at TIMESTAMPTZ NOT NULL,
  payload JSONB NOT NULL,
  status TEXT NOT NULL DEFAULT 'scheduled',
  attempts BIGINT NOT NULL DEFAULT 0,
  fired_event_id UUID,
  last_error TEXT,                     -- added by the timer-recovery migration
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- timer-recovery migration: the index now also covers 'firing' rows (for the
-- recovery queries) and is partial on active statuses.
CREATE INDEX IF NOT EXISTS keiro_timers_due_idx
  ON keiro_timers (status, fire_at, process_manager_name)
  WHERE status IN ('scheduled', 'firing');
```

The status column stores the lowercase strings `scheduled | firing | fired | cancelled | dead`.
`last_error` is populated only for dead-lettered (`dead`) rows; it is not surfaced on the `TimerRow`
record, so operators read it directly from the table.
