The types and functions on this page live in `Keiro.ProcessManager`. A process manager reacts to
an incoming event by advancing its own event-sourced state, dispatching commands to target
aggregates, and scheduling durable timers. For the narrative, see [Understanding process managers
and sagas](/docs/keiro/explanation/process-managers-and-sagas).

<Callout type="warn">
The manager-state append and its timers commit in **one** transaction; **each** dispatched
`PMCommand` commits in its **own** transaction. This is **not** one atomic multi-stream commit.
Replay-safety comes entirely from `deterministicCommandId` + `eventAlreadyIn` + the store's
uniqueness constraint.
</Callout>

## `ProcessManager`

The full wiring: the manager's own state machine and the target aggregate it drives. It is
parameterised over the manager aggregate (`phi rs s ci co`), the target aggregate (`targetPhi
targetRs targetState targetCi targetCo`), and the incoming event type (`input`).

```haskell
data ProcessManager input phi rs s ci co targetPhi targetRs targetState targetCi targetCo = ProcessManager
  { name              :: !Text
  , correlate         :: !(input -> Text)
  , eventStream       :: !(ValidatedEventStream phi rs s ci co)
  , streamFor         :: !(Text -> Stream (EventStream phi rs s ci co))
  , targetEventStream :: !(ValidatedEventStream targetPhi targetRs targetState targetCi targetCo)
  , targetProjections :: !(Stream targetCi -> [InlineProjection targetCo])
  , handle            :: !(input -> ProcessManagerAction ci targetCi)
  }
```

## `PMCommandResult`

The outcome of one dispatched target command.

```haskell
data PMCommandResult target
  = PMCommandAppended  !(CommandResult target)  -- the dispatch appended events
  | PMCommandDuplicate !EventId                 -- idempotent replay; the id already existed
  | PMCommandFailed    !CommandError            -- worker classifies transient retry vs deterministic halt
```

<Callout type="warn">
`PMCommandFailed` is not automatically fatal anymore. The worker retries transient store/conflict
failures with `transientRetryDelay`, and halts deterministic command errors. Model benign target
rejections as **total** transitions so they never surface as deterministic `PMCommandFailed`. See
[Keep target commands total](/docs/keiro/how-to/keep-target-commands-total).
</Callout>

## `PMStateResult`

The outcome of the manager's own state append. Unlike `PMCommandResult` there is no failure case —
a manager-state append that genuinely errors aborts the whole reaction via an outer `Left
CommandError`.

```haskell
data PMStateResult target
  = PMStateAppended  !(CommandResult target)
  | PMStateDuplicate !EventId
```

## `ProcessManagerResult`

The complete result of reacting to one event.

```haskell
data ProcessManagerResult managerTarget commandTarget = ProcessManagerResult
  { managerResult   :: !(PMStateResult managerTarget)
  , commandResults  :: ![PMCommandResult commandTarget]
  , timersScheduled :: !Int
  }
```

## `runProcessManagerOnce`

React to a single source event: advance the manager's state, dispatch its target commands, and
schedule its timers — each under a deterministic, idempotent write id.

```haskell
runProcessManagerOnce ::
  ( HasCallStack, IOE :> es, Store :> es, Error StoreError :> es
  , BoolAlg phi (RegFile rs, ci)
  , BoolAlg targetPhi (RegFile targetRs, targetCi)
  , Eq co, Eq targetCo ) =>
  RunCommandOptions ->
  ProcessManager input phi rs s ci co targetPhi targetRs targetState targetCi targetCo ->
  RecordedEvent ->
  input ->
  Eff es (Either CommandError (ProcessManagerResult (EventStream phi rs s ci co) (EventStream targetPhi targetRs targetState targetCi targetCo)))
```

Returns `Left` **only** when the manager-state append fails for a non-duplicate reason; per-command
failures are reported inside `commandResults`. A duplicate manager append short-circuits to
`PMStateDuplicate` but **still re-runs the dispatch loop**, so a crash between the state append and
a command dispatch is recovered on replay.

<Callout type="info">
**Timers survive a no-op manager command.** Timers are scheduled in the manager-append transaction.
When that append is a duplicate (an idempotent replay, so it commits nothing), it cannot schedule
them inline — so `runProcessManagerOnce` schedules any pending `timers` in a **follow-up
transaction** instead. A replayed source event therefore never silently drops a timer just because
the manager command it carried was a no-op.
</Callout>

## `runProcessManagerWorker`

Run a process manager as a live subscription draining a shibuya `Adapter`.

```haskell
runProcessManagerWorker ::
  ( HasCallStack, IOE :> es, Store :> es, Error StoreError :> es
  , BoolAlg phi (RegFile rs, ci)
  , BoolAlg targetPhi (RegFile targetRs, targetCi)
  , Eq co, Eq targetCo ) =>
  RunCommandOptions ->
  ProcessManager input phi rs s ci co targetPhi targetRs targetState targetCi targetCo ->
  Adapter es msg ->
  (msg -> Maybe (RecordedEvent, input)) ->
  Eff es ()
```

Each message is decoded with the caller's `msg -> Maybe (RecordedEvent, input)`. A message that
fails to decode follows the worker's `PoisonPolicy` (default `PoisonHalt`). Successful appends and
duplicate deterministic ids finalize `AckOk`; transient store and command failures finalize
`AckRetry transientRetryDelay`; deterministic command errors finalize `AckHalt (HaltFatal …)`. The
worker invokes the message's ack handle exactly once. See [Run a process manager as a
subscription](/docs/keiro/how-to/run-a-process-manager-as-a-subscription).

```haskell
data PoisonPolicy es msg
  = PoisonHalt | PoisonSkip (Envelope msg -> Eff es ()) | PoisonDeadLetter (Envelope msg -> Eff es ())

data WorkerOptions es msg = WorkerOptions
  { poisonPolicy :: PoisonPolicy es msg
  , transientRetryDelay :: RetryDelay
  , metrics :: Maybe KeiroMetrics
  }

defaultWorkerOptions :: WorkerOptions es msg
```

`defaultWorkerOptions` halts on poison messages, retries transient dispatch failures after
`RetryDelay 5`, and records no metrics. The same options type is used by router workers.

## `deterministicCommandId`

Derive a stable, collision-resistant `EventId` for a manager write from `(manager name,
correlation id, source event id, emit index)` via a v5 UUID.

```haskell
deterministicCommandId :: Text -> Text -> EventId -> Int -> EventId
```

The same inputs always yield the same id, so a replayed source event produces the same write ids
and the store's uniqueness constraint collapses the duplicate. The manager-state append uses an
emit index of `-1`; dispatched commands are numbered `0, 1, 2, …`.

## `eventAlreadyIn`

Check whether an event with the given id is already present in a stream, by scanning it forward.
Used as the pre-dispatch idempotency guard.

```haskell
eventAlreadyIn :: (Store :> es) => RunCommandOptions -> StreamName -> EventId -> Eff es Bool
```
