---
title: "Keiro.ProcessManager"
description: "Process-manager definitions, target-bearing results, rejection policy, deterministic dispatch, and durable rejected-dispatch records."
---

`Keiro.ProcessManager` implements a stateful coordinator over a Shibuya adapter. A reaction advances
the manager's own stream, schedules timers, and dispatches target commands. Its crash-safety is
deterministic and at-least-once; it is not one multi-stream transaction.

<Callout type="warn">
  The manager-state append and its timers commit together. A timer-only no-op uses a separate timer
  transaction. Each target command and its inline projections then commit in their own transaction.
  A crash or rejected target can therefore leave manager history without every target write.
</Callout>

## Definition and action

```haskell
data ProcessManager input phi rs s ci co targetPhi targetRs targetState targetCi targetCo =
  ProcessManager
    { name              :: Text
    , correlate         :: input -> Text
    , eventStream       :: ValidatedEventStream phi rs s ci co
    , streamFor         :: Text -> Stream (EventStream phi rs s ci co)
    , targetEventStream :: ValidatedEventStream targetPhi targetRs targetState targetCi targetCo
    , targetProjections :: Stream targetCi -> [InlineProjection targetCo]
    , handle            :: input -> ProcessManagerAction ci targetCi
    }

data ProcessManagerAction ci targetCi = ProcessManagerAction
  { command  :: ci
  , commands :: [PMCommand targetCi]
  , timers   :: [TimerRequest]
  }

data PMCommand targetCi = PMCommand
  { target  :: Stream targetCi
  , command :: targetCi
  }
```

`handle` is pure, so its positional target-command order is stable for the same input. That is why
process-manager target IDs may include `emitIndex`. `correlate` may join events from different source
streams, but those streams have no relative business-order guarantee; the manager state machine must
accept either arrival order.

## Results retain target identity

```haskell
data PMCommandResult target
  = PMCommandAppended (CommandResult target)
  | PMCommandDuplicate EventId
  | PMCommandFailed StreamName CommandError

data PMStateResult target
  = PMStateAppended (CommandResult target)
  | PMStateDuplicate EventId

data ProcessManagerResult managerTarget commandTarget = ProcessManagerResult
  { managerResult   :: PMStateResult managerTarget
  , commandResults  :: [PMCommandResult commandTarget]
  , timersScheduled :: Int
  }
```

The `StreamName` in `PMCommandFailed` identifies the target that actually failed. A manager-state
failure still appears as the outer `Left CommandError`; worker policy represents it as a
`DispatchFailure` with `emitIndex = -1` and the manager stream as target.

## Worker policy

```haskell
data RejectedCommandPolicy
  = RejectedHalt
  | RejectedDeadLetter
  | RejectedSkip

data DispatchFailure = DispatchFailure
  { emitIndex        :: Int
  , targetStreamName :: StreamName
  , commandError     :: CommandError
  }

data WorkerOptions es msg = WorkerOptions
  { poisonPolicy          :: PoisonPolicy es msg
  , rejectedCommandPolicy :: RejectedCommandPolicy
  , transientRetryDelay    :: RetryDelay
  , metrics                :: Maybe KeiroMetrics
  }
```

`defaultWorkerOptions` uses `PoisonHalt`, `RejectedHalt`, `RetryDelay 5`, and no metrics. The worker
classifies a whole source-event reaction in this order:

1. any systemic deterministic error (decode/replay/gap/encode, or an unconfirmed duplicate) =>
   `AckHalt`;
2. otherwise, any transient store/conflict error => `AckRetry transientRetryDelay`;
3. otherwise, an empty failure list => `AckOk`;
4. otherwise all failures are `CommandRejected` or `CommandAmbiguous` and follow
   `rejectedCommandPolicy`.

| Rejection policy | Durable record | Source acknowledgement |
| --- | --- | --- |
| `RejectedHalt` (default) | None | `AckHalt`; checkpoint does not advance. |
| `RejectedDeadLetter` | One idempotent `keiro.keiro_dead_letters` row per failed dispatch | `AckOk`; source may advance. |
| `RejectedSkip` | None | `AckOk`; only the handled/dead-letter metric records the choice. |

`CommandAmbiguous` is included in the rejection-class set so policy is exhaustive, but it remains an
aggregate-definition defect. Keep the default halt unless an explicit operator process owns the
durable witness and code repair.

<Callout type="warn">
  A process-manager dispatch dead letter splits history: manager state may already record the
  reaction while the target command never applied. If the saga must remember that failure, model a
  domain `DispatchFailed` command/event and drive it from an operator workflow. Keiro cannot append
  an unknown generic correction to the manager stream.
</Callout>

## Durable dispatch dead letters

`RejectedDeadLetter` writes application-dispatch evidence to `keiro.keiro_dead_letters` before
returning `AckOk`:

```haskell
data DispatchDeadLetterRecord = DispatchDeadLetterRecord
  { deadLetterId         :: Int64
  , dispatcherKind      :: DispatcherKind
  , dispatcherName      :: Text
  , correlationId       :: Text
  , sourceEventId       :: EventId
  , sourceGlobalPosition :: GlobalPosition
  , emitIndex           :: Int
  , targetStreamName    :: StreamName
  , errorClass          :: Text
  , errorDetail         :: Text
  , attemptCount        :: Int
  , createdAt           :: UTCTime
  }

listDispatchDeadLetters :: Text -> Eff es [DispatchDeadLetterRecord]
```

Rows are idempotent on `(dispatcher_name, source_event_id, emit_index)`, error detail is capped at
1024 characters, and listing is newest first. These are not Kiroku subscription dead letters:

- `keiro.keiro_dead_letters` means a target dispatch was rejected and the source event was handled;
- `kiroku.dead_letters` means source delivery exhausted its retry policy and the subscription
  checkpoint advanced atomically with parking that source event.

## Single-event and worker runners

```haskell
runProcessManagerOnce ::
  ( HasCallStack, IOE :> es, Store :> es, Error StoreError :> es
  , KirokuStoreResource :> es, BoolAlg phi (RegFile rs, ci)
  , BoolAlg targetPhi (RegFile targetRs, targetCi), Eq co, Eq targetCo ) =>
  RunCommandOptions -> ProcessManager ... -> RecordedEvent -> input ->
  Eff es (Either CommandError (ProcessManagerResult ...))

runProcessManagerWorkerWith ::
  WorkerOptions es msg -> RunCommandOptions -> ProcessManager ... -> Adapter es msg ->
  (msg -> Maybe (RecordedEvent, input)) -> Eff es ()
```

`runProcessManagerWorker` is the `defaultWorkerOptions` specialization. Each ingested message's ack
handle is finalized exactly once. A duplicate manager-state append still reruns target dispatch, so
a crash after manager commit but before a target commit is filled in on redelivery.

## Deterministic IDs and duplicate confirmation

```haskell
deterministicCommandId :: Text -> Text -> EventId -> Int -> EventId
eventAlreadyIn         :: RunCommandOptions -> StreamName -> EventId -> Eff es Bool
confirmBenignDuplicate :: StreamName -> EventId -> CommandError -> Eff es Bool
```

The deterministic ID is a UUIDv5 over manager name, correlation, source event ID, and emit index;
manager state uses `-1`, target commands use `0..`. `eventAlreadyIn` uses Kiroku's point lookup in
the intended stream.

Kiroku event IDs are globally unique, so a `DuplicateEvent` with the expected ID is not sufficient
proof of success: that ID might exist in another stream. `confirmBenignDuplicate` accepts a matching
or ID-less duplicate report only when a point lookup finds the attempted ID in the intended target.
A mismatched ID, cross-stream collision, or non-duplicate error remains the original failure and
cannot be acknowledged as `PMCommandDuplicate`.

See [Run a process manager as a subscription](/docs/keiro/how-to/run-a-process-manager-as-a-subscription)
and [Inspect a rejected dispatch](/docs/keiro/how-to/inspect-a-rejected-dispatch).
