The types and functions on this page live in the **`Keiro.Subscription.Shard`** module family. They
add **consumer-group sharding** for a high-volume [category subscription](/docs/keiro/reference/read-model):
a pool of identical worker processes cooperatively partitions the category among themselves so every
event is processed by exactly one worker, none twice, none skipped — with automatic, **coordinator-free**
failover when a worker dies.

For the concept, read [Scaling the workers](/docs/keiro/explanation/scaling-the-workers); for the task,
[Run sharded subscriptions](/docs/keiro/how-to/run-sharded-subscriptions).

```haskell
import Keiro.Subscription.Shard         -- WorkerId, ShardLease, the ownership operations
import Keiro.Subscription.Shard.Worker  -- ShardedWorkerOptions, runShardedSubscriptionGroup
-- Keiro.Subscription.Shard.Schema       -- the Transaction-level lease statements (rarely imported directly)
```

## What kiroku provides, and what this adds

kiroku already partitions a category into `N` **buckets** (consumer-group member indices in `[0, N)`)
by a stable hash of the originating stream id, and keeps a per-member checkpoint — so `N` disjoint
readers can drain a busy category in parallel. What kiroku leaves to the operator is *membership*:
"exactly one live process must own each bucket at a time" is otherwise a manual `[0..N-1]` wiring.

This module supplies the missing operability layer — a **lease** over each bucket so a pool of
identical workers agree, with no external coordinator (no etcd/ZooKeeper/Consul), on who owns which
bucket right now, and re-divide the buckets automatically when a worker joins, leaves, or dies.
Coordination lives entirely in the `keiro_subscription_shards` lease table.

<Callout type="info">
The lease — a renewable `lease_expires_at` timestamp — is the ownership mechanism, **not** a held lock.
A transaction-scoped advisory lock auto-releases at transaction end (so it cannot span a worker's
multi-transaction lifetime) and a session-scoped lock has no connection affinity through kiroku's
pooled `Store` — the same finding the [resume worker](/docs/keiro/reference/durable-workflows#the-resume-worker)
records. A renewable lease gives lifetime ownership and automatic failover without depending on
connection affinity; disjointness rests on the `FOR UPDATE SKIP LOCKED` claim, not on the liveness
estimate being exact.
</Callout>

## The lease descriptor and ownership operations

`Keiro.Subscription.Shard` is the typed `Eff`-level surface over kiroku's `Store`.

```haskell
newtype WorkerId = WorkerId UUID

data ShardLease = ShardLease
  { subscriptionName :: SubscriptionName
  , workerId         :: WorkerId
  , shardCount       :: Int              -- N; fixed per subscription name (every worker must agree)
  , leaseTtl         :: NominalDiffTime  -- how long a claim/renew keeps a bucket before it expires
  }

freshWorkerId      :: (IOE :> es) => Eff es WorkerId
ensureShards       :: (Store :> es) => ShardLease -> Eff es ()
acquireOwnedBuckets :: (IOE :> es, Store :> es) => ShardLease -> Int -> Eff es (Set Int)
renewOwnedBuckets  :: (IOE :> es, Store :> es) => ShardLease -> Eff es (Set Int)
relinquish         :: (Store :> es) => ShardLease -> Set Int -> Eff es ()
ownershipSnapshot  :: (Store :> es) => ShardLease -> Eff es [(Int, Maybe WorkerId, Maybe UTCTime)]
fairShareTarget    :: Int -> Int -> Int   -- ceil(N / liveWorkers)
```

<TypeTable
  type={{
    freshWorkerId: { type: "Eff es WorkerId", description: "Mint a fresh per-process owner id (a random UUID). Two restarts of the same binary get different ids, so a restarted process never inherits the dead process's leases — it claims afresh once they expire." },
    ensureShards: { type: "ShardLease -> Eff es ()", description: "Idempotently populate the N shard rows (ON CONFLICT DO NOTHING). Safe to call on every worker startup." },
    acquireOwnedBuckets: { type: "ShardLease -> Int -> Eff es (Set Int)", description: "One reconcile pass, in a single transaction: renew the leases this worker holds, then — if below its fair share — claim ONE more bucket (unowned or expired). Returns the buckets owned after the pass. The Int is the caller's estimate of live workers." },
    renewOwnedBuckets: { type: "ShardLease -> Eff es (Set Int)", description: "Heartbeat only — write a fresh expiry for every bucket this worker still holds, and return them." },
    relinquish: { type: "ShardLease -> Set Int -> Eff es ()", description: "Graceful release of the given buckets (clean shutdown) so they become claimable immediately instead of after lease expiry." },
    ownershipSnapshot: { type: "ShardLease -> Eff es [(Int, Maybe WorkerId, Maybe UTCTime)]", description: "Read every bucket's (bucket, owner, lease_expires_at). The worker uses it to estimate liveWorkers (distinct non-expired owners) before a pass." },
    fairShareTarget: { type: "Int -> Int -> Int", description: "ceil(N / liveWorkers): when k workers are live they collectively claim all N buckets and no single worker hogs them. Non-positive liveWorkers is treated as one." },
  }}
/>

<Callout type="info">
**Claiming one bucket per pass is deliberate.** If a cold worker grabbed its whole fair share at once
it could, racing alone before its peers' first pass, monopolise every bucket and then never see the
idle peers (they own nothing, so they are invisible in the lease table). Taking one bucket per pass
means concurrently-starting workers each grab one, become visible after the first pass, and climb to an
even share together. Ownership spreads over up to `N` reconcile intervals — a deliberate trade of
spin-up latency for coordinator-free fairness. A stale `liveWorkers` estimate only changes how
aggressively a worker claims, never whether ownership stays disjoint.
</Callout>

## The sharded worker

`Keiro.Subscription.Shard.Worker` turns leased bucket ownership into running kiroku consumer-group
readers. Start the **same** worker binary `N` times — on `N` hosts, in `N` containers, in an
autoscaling group — each calling `runShardedSubscriptionGroup` with the same `SubscriptionName` and
options, and the workers cooperatively partition the category.

```haskell
data ShardedWorkerOptions = ShardedWorkerOptions
  { shardCount    :: Int              -- N buckets, fixed per subscription name
  , leaseTtl      :: NominalDiffTime  -- claim/renew validity (default 30 s)
  , renewInterval :: NominalDiffTime  -- gap between reconcile passes (default 10 s)
  , target        :: SubscriptionTarget  -- the category (or AllStreams) to shard
  , batchSize     :: Int32            -- events per DB fetch per bucket reader (default 100)
  , bufferSize    :: Natural          -- per-reader bounded-queue capacity (default 256)
  }

defaultShardedWorkerOptions :: SubscriptionTarget -> Int -> ShardedWorkerOptions

reconcileShardsOnce :: KirokuStore -> ShardLease -> ShardedWorkerOptions
                    -> IORef (Map Int RunningReader) -> (RecordedEvent -> IO ()) -> IO (Set Int)
runShardedSubscriptionGroup :: KirokuStore -> SubscriptionName -> ShardedWorkerOptions
                            -> (RecordedEvent -> IO ()) -> IO ()
```

<TypeTable
  type={{
    leaseTtl: { type: "NominalDiffTime", default: "30 s", description: "How long a claim/renew keeps a bucket before it expires. A dead worker loses every bucket within one TTL." },
    renewInterval: { type: "NominalDiffTime", default: "10 s", description: "Gap between reconcile passes; well under leaseTtl so a live worker renews several times per TTL (~3 renews per 30 s). A single missed renewal does not lose ownership." },
    batchSize: { type: "Int32", default: "100", description: "Events per database fetch, per bucket reader." },
    bufferSize: { type: "Natural", default: "256", description: "Per-reader bounded-queue capacity — backpressure." },
  }}
/>

`reconcileShardsOnce` is the single testable unit (the analogue of `resumeWorkflowsOnce`). One pass:

1. **Estimate** how many workers are live (distinct non-expired owners in the lease table, plus self).
2. **`acquireOwnedBuckets`** — renew held leases and claim up to a fair share more (taking over expired
   leases).
3. **Shed** any buckets held beyond the fair share when more than one worker is live, so a worker that
   over-claimed on a cold start gives the excess back and the pool converges to an even split. A lone
   worker never sheds — it must own every bucket.
4. **Reconcile readers** — open a kiroku consumer-group reader for each newly-owned bucket, stop the
   reader for each newly-lost bucket. A bucket that moves owners resumes at its own kiroku per-member
   checkpoint, so no event is dropped.

`runShardedSubscriptionGroup` is the loop driver: mint a `WorkerId`, `ensureShards`, then
`reconcileShardsOnce` every `renewInterval` forever. On shutdown (the loop thread is killed) a `finally`
stops every reader this worker holds; its leases then expire and a surviving worker re-claims its
buckets (failover).

<Callout type="warn">
**A sharded subscription delivers at-least-once.** A brief overlap is possible while a bucket changes
owners, so the `RecordedEvent -> IO ()` handler **must be idempotent** — keyed on `eventId` — exactly
as keiro's [async-projection guidance](/docs/keiro/how-to/make-an-async-projection-idempotent) already
requires. The lease table governs *assignment only*; event positions stay in kiroku's per-member
checkpoints (`(subscription_name, consumer_group_member)`), so a re-homed bucket resumes where its
previous owner left off.
</Callout>

## The lease table

One table in the **`kiroku` schema**, added by the `keiro-subscription-shards` migration.

```sql
CREATE TABLE keiro_subscription_shards (
  subscription_name  TEXT        NOT NULL,
  bucket             INT         NOT NULL,   -- kiroku consumer-group member index
  shard_count        INT         NOT NULL,   -- N; fixed per subscription_name
  owner_worker_id    UUID,                   -- NULL = unowned / claimable
  lease_expires_at   TIMESTAMPTZ,            -- NULL when unowned
  heartbeat_at       TIMESTAMPTZ,            -- last renewal (observability)
  updated_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (subscription_name, bucket),
  CONSTRAINT keiro_subscription_shards_bucket_range_chk CHECK (bucket >= 0 AND bucket < shard_count),
  CONSTRAINT keiro_subscription_shards_count_chk        CHECK (shard_count >= 1)
);
```

One row per `(subscription_name, bucket)` records *who owns each bucket right now*, as a renewable
lease. A live worker renews `lease_expires_at` on a heartbeat; a dead worker stops renewing, its lease
expires, and another worker re-claims the bucket. The table does **not** re-hash anything and does
**not** store event positions. The `Schema` module's `claimShardsTx` uses `FOR UPDATE SKIP LOCKED` over
the claimable rows (exactly as the [timer worker's claim](/docs/keiro/reference/timers)), so two workers
racing the same bucket can never both win. See [Migrations and schema](/docs/keiro/reference/migrations-and-schema)
for the full table map.
