title: "Subscription sharding"
description: "Reference for leased consumer-group ownership, acknowledgment-coupled delivery, bounded retry, and source-event dead letters."
---

`Keiro.Subscription.Shard` and `Keiro.Subscription.Shard.Worker` let identical worker processes
share a Kiroku consumer group. Keiro leases bucket assignment; Kiroku owns event delivery,
per-member checkpoints, bounded retry, and `kiroku.dead_letters`.

The guarantee is **at-least-once**, not exactly-once. A lease handoff can briefly overlap readers,
and a crash can replay successful work whose batch-tail checkpoint was not saved. Make every handler
idempotent on the source `eventId`.

```haskell
import Keiro.Subscription.Shard
import Keiro.Subscription.Shard.Worker
```

For deployment steps, see [Run sharded subscriptions](/docs/keiro/how-to/run-sharded-subscriptions).
For the design, see [Scaling the workers](/docs/keiro/explanation/scaling-the-workers).

## Ownership API

Kiroku partitions a category into `N` consumer-group members by a stable hash of the originating
stream id. It stores one checkpoint per `(subscription_name, member)`. Keiro adds membership: one
renewable lease row for every member index in `[0, N)`.

```haskell
newtype WorkerId = WorkerId UUID

data ShardLease = ShardLease
  { subscriptionName :: SubscriptionName
  , workerId         :: WorkerId
  , shardCount       :: Int
  , leaseTtl         :: NominalDiffTime
  }

freshWorkerId       :: (IOE :> es) => Eff es WorkerId
ensureShards        :: (Store :> es) => ShardLease -> Eff es ()
acquireOwnedBuckets :: (IOE :> es, Store :> es) => ShardLease -> Int -> Eff es (Set Int)
renewOwnedBuckets   :: (IOE :> es, Store :> es) => ShardLease -> Eff es (Set Int)
relinquish          :: (Store :> es) => ShardLease -> Set Int -> Eff es ()
ownershipSnapshot   :: (Store :> es) => ShardLease -> Eff es [(Int, Maybe WorkerId, Maybe UTCTime)]
fairShareTarget     :: Int -> Int -> Int
```

| Operation | Contract |
| --- | --- |
| `ensureShards` | Idempotently creates the `N` rows. It raises `ShardCountMismatch` when the same subscription name already has another configured count. |
| `acquireOwnedBuckets` | In one transaction, renews buckets still held and claims at most one unowned or expired bucket when below `ceil(N / liveWorkers)`. |
| `renewOwnedBuckets` | Renews held leases without claiming. A bucket taken after expiry is absent from the result. |
| `relinquish` | Clears leases still owned by this worker so another worker can claim immediately. |
| `ownershipSnapshot` | Lists bucket, owner, and expiry for health and liveness estimation. |

Claiming one bucket per pass prevents a cold-starting worker from monopolizing all buckets before its
peers become visible. The liveness count tunes only the fair-share target. `FOR UPDATE SKIP LOCKED`
over claimable lease rows is the actual exclusion mechanism.

<Callout type="warn">
  Exclusion applies to valid lease claims, not to a stale process after its lease expires. A paused
  or partitioned worker can keep its old reader alive briefly while a successor claims the bucket.
  Both may deliver the same event; idempotency makes that overlap safe.
</Callout>

## Delivery API

Use the acknowledgment-aware entry point when the handler can fail.

```haskell
data ShardDelivery = ShardDelivery
  { event   :: RecordedEvent
  , attempt :: Word
  , bucket  :: Int
  }

data ShardAck
  = ShardAckOk
  | ShardAckRetry RetryDelay
  | ShardAckDeadLetter DeadLetterReason

type ShardEventHandler = ShardDelivery -> IO ShardAck

runShardedSubscriptionGroupAck
  :: KirokuStore
  -> SubscriptionName
  -> ShardedWorkerOptions
  -> ShardEventHandler
  -> IO ()

runShardedSubscriptionGroup
  :: KirokuStore
  -> SubscriptionName
  -> ShardedWorkerOptions
  -> (RecordedEvent -> IO ())
  -> IO ()
```

`attempt` is zero-based: `0` is the first delivery and `1` the first redelivery. `bucket` is the
Kiroku consumer-group member. The acknowledgment maps directly to Kiroku:

| Keiro result | Kiroku result | Durable consequence |
| --- | --- | --- |
| `ShardAckOk` | `Continue` | The worker may continue. Ordinary successful events are checkpointed at the batch tail, so a crash before then can replay them. |
| `ShardAckRetry delay` | `Retry delay` | The same event is redelivered after `delay`; its checkpoint has not advanced. |
| `ShardAckDeadLetter reason` | `DeadLetter reason` | Kiroku atomically inserts `kiroku.dead_letters` and advances this member's checkpoint past the event. |

The worker uses Kiroku's `subscriptionAckStream`. That bridge blocks until the consumer supplies one
of these results. If a reader is shed or shut down while the handler is in flight, the asynchronous
exception supplies no acknowledgment; the next owner reads again from the unadvanced checkpoint.

`runShardedSubscriptionGroup` is a compatibility wrapper. It passes only the event to the handler
and returns `ShardAckOk` when that handler completes. A synchronous exception is still caught by the
acknowledgment-aware layer and converted to a bounded retry. New handlers that need to distinguish a
transient failure from poison data should use `runShardedSubscriptionGroupAck` directly.

## Retry and exception policy

```haskell
data ShardedWorkerOptions = ShardedWorkerOptions
  { shardCount        :: Int
  , leaseTtl          :: NominalDiffTime
  , renewInterval     :: NominalDiffTime
  , target            :: SubscriptionTarget
  , batchSize         :: Int32
  , bufferSize        :: Natural
  , handlerRetryDelay :: RetryDelay
  , retryPolicy       :: RetryPolicy
  , onShardError      :: Maybe (ShardWorkerError -> IO ())
  }
```

| Field | Default | Meaning |
| --- | --- | --- |
| `leaseTtl` | 30 seconds | How long a claim remains valid without renewal. |
| `renewInterval` | 10 seconds | Delay between reconcile passes. Keep it below the TTL. |
| `batchSize` | 100 | Events fetched per bucket reader. |
| `bufferSize` | 256 | Capacity of each acknowledgment bridge; one event can also be in flight. |
| `handlerRetryDelay` | 1 second | Delay used when a synchronous handler exception is converted to retry. |
| `retryPolicy` | `RetryPolicy 5` | Maximum **total deliveries**, including the first. After the last retry result, Kiroku dead-letters with `DeadLetterMaxAttempts`. |
| `onShardError` | `Nothing` | Reports snapshot, acquisition, ensure, and reader-stream failures. Ordinary synchronous handler failures use retry and do not become `ShardReaderDied`. |

`mkShardedWorkerOptions` validates all positive bounds, requires `leaseTtl > renewInterval`, rejects a
negative handler retry delay, and requires at least one total delivery. Call it during application
startup; the loop entry points accept the options record directly.

A synchronous `SomeException` from the handler becomes `ShardAckRetry handlerRetryDelay`.
`SomeAsyncException` is rethrown so shedding and shutdown do not invent a handler decision. An
explicit `ShardAckDeadLetter` bypasses remaining retries. Repeated `ShardAckRetry` results exhaust at
`retryMaxAttempts`; Kiroku then records `DeadLetterMaxAttempts attempts` and continues with the next
event.

## Reconciliation and lifecycle

```haskell
reconcileShardsOnce
  :: KirokuStore
  -> ShardLease
  -> ShardedWorkerOptions
  -> IORef (Map Int RunningReader)
  -> ShardEventHandler
  -> IO (Set Int)
```

One pass:

1. estimates live workers from non-expired owners plus self;
2. renews held buckets and claims toward the fair share;
3. sheds excess buckets when peers are visible; and
4. starts readers for newly owned buckets and cancels readers for lost buckets.

An acquisition failure preserves the prior local ownership set and reports `ShardAcquireFailed`.
Each dead reader removes itself from the map, reports `ShardReaderDied`, and can be restarted by a
later pass if the lease is still owned.

`runShardedSubscriptionGroupAck` mints a fresh worker ID, ensures rows, and reconciles forever. Its
`finally` block stops current readers and relinquishes their leases. A graceful stop therefore makes
them immediately claimable; a process crash relies on lease expiry.

Kiroku's member checkpoint upsert is monotonic. A late acknowledgment from a stale reader cannot
move a checkpoint backward, though it can duplicate work already performed by a successor.

## The lease table

`keiro.keiro_subscription_shards` stores assignment only: subscription name, bucket, fixed shard
count, owner, lease expiry, heartbeat, and update time. It never stores event positions.

### Terminal source-event evidence

`kiroku.dead_letters` stores source deliveries parked explicitly or after retry exhaustion. Its
insert and member-checkpoint advance are one database statement: if the insert fails, the checkpoint
does not advance and the reader fails, so the source event reappears after restart.

Use [Replay subscription dead letters](/docs/keiro/how-to/replay-subscription-dead-letters) for the
operator path. Replay retains these rows; they are durable evidence, not a work queue whose items are
deleted on success.
