This page owns every non-aggregate runtime node. References to aggregates, projections, read models,
contracts, commands, queues, workflows, and fields are resolved by `check`.

## Process and timer

```keiro-dsl
process HospitalSurge
  name "hospital-surge"
  input SurgeInput { hospitalId availableIcuBeds:Int observedAt:Time }
  correlate input.hospitalId via idText
  saga Surge category "hospitalSurge"
  target Hospital
  projections [ hospitalReadiness ]
  on SurgeInput
    advance NoteSurgeThreshold { hospitalId timerId=timer.id }
    dispatch Hospital@input.hospitalId ActivateSurge { hospitalId }
      on-appended AckOk ; on-duplicate AckOk ; on-failed Retry
    schedule surgeFollowUp
  dispatch-id strategy=uuidv5 from=(name, correlationId, sourceEventId, emitIndex)
  rejected => halt
  poison => halt
  timer surgeFollowUp
    id uuidv5 "hospital-surge-timer:" <> correlationId
    fireAt input.observedAt + 5m
    payload { kind="hospital-surge-follow-up" hospitalId }
    fire dispatch Surge@correlationId MarkSurgeTimerFired { hospitalId timerId }
      fired-event-id uuidv5 "hospital-surge-fired:" <> correlationId
      on-ok Fired ; on-reject Fired ; on-ambiguous Retry ; on-error Retry ; not-mine Retry
    decode unknown-status => Cancelled
    max-attempts 5 dead-letter "surge timer exceeded ceiling"
```

Durations are decimal digits plus `s`, `m`, or `h`. The fire time must use a declared `:Time` input;
dispatch IDs are runtime-owned. Saga categories are non-empty camelCase without `-`, `:`, whitespace,
control characters, or `$all`; generated category constants are used with `entityStream`.

Dispatch outcomes are `AckOk`, `Retry`, or `deadLetter "reason"`. Node-level `rejected` and `poison`
are each `halt`, `deadLetter`, or `skip`; rejection-class failures include `CommandAmbiguous` for an
ordinary dispatch. Timer outcomes are `Fired` or `Retry`, but `on-ambiguous Fired` is invalid.
`max-attempts` must be at least 1.

Generated: `Process`, `ProcessHarness`, and aggregate modules. Hand-owned: `ProcessHoles` for handle,
deadline/fire behavior, SQL, and runtime assembly. Pass generated worker options to
`runProcessManagerWorkerWith`. A manual duplicate path must call `confirmBenignDuplicate` against
the target stream.

## Router

```keiro-dsl
router PagingRouter
  name "jitsurei-paging"
  input IncidentRaised { incidentId service }
  key input.incidentId via idText
  resolve stable via read-model service_oncall row { responderId }
  target Page
  projections [ ]
  dispatch-each SendPage { incidentId=input.incidentId responderId=resolved.responderId }
    on-appended AckOk ; on-duplicate AckOk ; on-failed Retry
  dispatch-id strategy=uuidv5 from=(name, key, sourceEventId, targetStreamName, occurrence)
  rejected => deadLetter
  poison => halt
```

The resolver source is `read-model <name>` or `hole`. `stable` is mandatory: retries dedupe repeated
targets but a drifting resolver accumulates the union of attempt outputs. Bindings read declared
`input.*`, `resolved.*`, or quoted literals. Name, key, target, and target-based dispatch identity
are replay-sensitive.

Generated: `Router`, `RouterHarness`. Hand-owned: `RouterHoles` plus the effectful resolver and
assembled runtime value. Pass generated options to `runRouterWorkerWith`. Target/command/projection,
key/row/binding, read-model, and policy consistency are checked.

## Contract, intake, emit, and publisher

### Contract

```keiro-dsl
contract emergency {
  schemaVersion 1
  discriminator messageType
  topic incidentEvents "emergency.incident.events"
  event IncidentTransferNeedDeclared on incidentEvents {
    incidentId: typeid "inc"
    region: text
    redCount: int
  }
}
```

Contract field types are `typeid "prefix"`, `text`, or `int`. Topic aliases map to actual Kafka
topics; events declare their topic affinity. Generated `Contract` modules own payload ADTs/codecs.

### Intake

```keiro-dsl
intake incidentInbox {
  contract emergency
  topic incidentEvents
  accept IncidentTransferNeedDeclared
  bind messageId from header "keiro-message-id" required cross-check body
  bind key from kafka-key
  dedupe key messageId policy PreferIntegrationMessageId
  persist = dedupe-only
  decode { envelope strict-required lenient-optional ; body strict schemaVersion == 1 }
  disposition {
    processed => ackOk
    duplicate => ackOk
    inProgress => retry 5s
    previouslyFailed => deadLetter "prior failure"
    decodeFailed => deadLetter
    dedupeFailed => deadLetter
    storeFailed => retry 5s
  }
}
```

Binding sources are `header "name"`, `body`, `kafka-key`, or `kafka-cursor`; `required` and
`cross-check body` are optional modifiers. `persist` is `full-envelope` (default) or `dedupe-only`.
The latter drops successful payload bytes but failed rows always retain the full envelope.

The directly scaffoldable dedupe policies are `PreferIntegrationMessageId`,
`PreferSourceEventIdentity`, and `KafkaDeliveryIdentity`. A custom runtime key remains hand-owned.

All seven disposition rows are mandatory and unique. `duplicate` acknowledges, prior failure and
decode poison dead-letter, and transient store failure retries. The generated `inboxPersistence`
value is passed to `runInboxTransactionWith`. Consumer group/sharding settings remain deployment
configuration, outside the notation.

### Emit and publisher

```keiro-dsl
emit reservationResponse {
  contract emergency ; topic hospitalEvents ; source "hospital-capacity" ; key incidentId
  map status { "confirmed" => TransferReservationAccepted _ => skip }
  messageId derive "msg" hole ; idempotencyKey derive hole
}

publisher hospitalPublisher {
  emit reservationResponse ; ordering PerKeyHeadOfLine ; maxAttempts 10
  backoff exponential 2s max=60s multiplier=2.0
  outboxId stable from messageId
}
```

The emit map requires `_ => skip`; all mapped events must belong to the chosen contract topic.
Publisher backoff is `constant <duration>` or
`exponential <initial> max=<duration> multiplier=<decimal>`. Exponential form requires both options.
Ordering is one of `PerKeyHeadOfLine`, `PerSourceStream`, `StopTheLine`, or `BestEffort`. The
publisher references an emit and captures the stable outbox identity field. Mapping, derivation,
ordering, and stable identity are evolution surfaces.

## Workqueue and dispatch

```keiro-dsl
workqueue reservation_work {
  queue logical = "hospital_capacity.reservation_work"
  derive physical = "hospital_capacity_reservation_work"
         dlq = "hospital_capacity_reservation_work_dlq"
         table = "pgmq.q_hospital_capacity_reservation_work"
  ordering fifo-throughput
  group key from reservationId via raw
  provision standard
  payload ReservationWorkItem { reservationId -> "reservation_id" text required }
  retry maxRetries = 3 delay = 5s dlq = on
  disposition {
    storeFailure -> retry 5s
    decodeFailure -> deadLetter
    commandRejected -> deadLetter
    onCodecReject -> deadLetter
  }
}
```

Ordering is `unordered` (default), `fifo-throughput`, or `fifo-roundrobin`. FIFO requires exactly one
group key; unordered forbids it. `via raw` requires a `text` payload field. An opaque `via <name>`
requires `fixture "input => output"`.

Provisioning is `standard` (default), `unlogged`, or
`partitioned(interval="daily", retention="7 days")`. Unlogged queues can be truncated to empty after
a database crash and produce `WqUnloggedDurability`. Partition settings are non-empty, require
`pg_partman`, and apply only when creating a queue; reconciliation does not migrate an existing
queue. FIFO adds its GIN index. A DLQ remains standard.

Payload field types include the queue codec surface (`text`, `int`, `bool`) with explicit wire name
and optional `required`. The captured physical/DLQ/table trio is re-derived from the logical name.
All four disposition rows are mandatory; store failure retries, decode failure dead-letters, and
`dlq = on` requires a positive retry ceiling.

```keiro-dsl
dispatch reservation_work_dispatch {
  source readModel = accepted_transfer_needs key = reservationId
  fanout body = resolveTransferCandidates
  dedup key = reservationId
        seenIn readModel = transfer_decisions field = reservation_id
        seenIn queue = reservation_work field = reservation_id
  enqueue to = reservation_work
}
```

Source/dedupe read models, the declared read-model column, queue, queue wire field, and enqueue target
must resolve. Fan-out and raw-SQL queue dedupe are hand-owned. Generated queue output is `Queue`,
`QueuePolicy`, and a `QueueCodec` module; headers, batch/visibility/poll/concurrency tuning, metrics,
and handler behavior stay in runtime configuration.

### The generated `QueueCodec`

Scaffolded workqueues ship a `QueueCodec` module holding a **versioned** `keiroJobCodec` envelope
(`{v,t,data}`) starting at schema version 1, rather than a bare `aesonJobCodec` payload. New queues
therefore begin inside an envelope that makes rolling-deploy skew self-describing: an old worker
reading a future envelope returns `JobPayloadFromFuture` and the job is retried.

Payload-evolution *syntax* in the DSL remains future work — `diff` still classifies workqueue payload
changes with its normal rules — but the runtime envelope is in place from day one.

<Callout type="warn">
A **fresh** queue needs no migration. An **existing** queue carrying bare-payload messages must be
drained before you swap its codec, or use a transitional decoder that accepts both shapes; otherwise
in-flight bodies fail as `JobPayloadMalformed` and dead-letter. See
[deploy-ordering rule 3](/docs/keiro/reference/deploy-ordering#3-upgrade-versioned-job-workers-before-producers).
</Callout>

## Workflow and operation

```keiro-dsl
workflow HospitalTransferReservation
  name "hospital-transfer-reservation"
  in ReservationWorkflowInput { reservationId:Id hospitalId:Id coolingOffDelay:Duration }
  out ReservationWorkflowSummary
  id from input.reservationId via idText
  body
    step create-transfer-hold -> ReservationHold
    patch fraud-check-v2 {
      step fraud-check -> FraudCheckResult
    }
    await reservation-confirmation -> ReservationConfirmation
    sleep cooling-off after coolingOffDelay
    child ship-order id input via shipChildId -> Text
    continueAsNew RolloverSeed
```

Body item forms are `step`, `await`, injected-delay `sleep`, child workflow, nested `patch`, and
top-level terminal `continueAsNew`. Labels are unique and replay matches labels. Patch IDs are
opaque, globally unique within the workflow, colon-free, and never reused. Use a patch for a
cross-cutting change to several journaled steps; rename one changed step instead of patching it.

`continueAsNew <Seed>` must be the final top-level item and cannot appear in a patch. The hand-owned
body calls `restoreSeed` at the beginning and `continueAsNew` at the tail. Generated output is
`WorkflowFacts` and `WorkflowRuntime`; behavior stays in a hand-written body module.

The four operation shapes are:

```keiro-dsl
operation ConfirmReservation
  command on Reservation
    stream from reservationId via reservationStream
    project [ transferDecision ]

operation QueryTransferDecision
  query transferDecision
    input TransferReservationId
    result Maybe TransferDecision
    consistency Strong

operation SignalReservationConfirmation
  signal reservation-confirmation of HospitalTransferReservation
    key from reservationId via reservationWorkflowId
    value ReservationConfirmation

operation RunReservationWorkflow
  run HospitalTransferReservation
    input ReservationWorkflowInput
    outcome -> ReservationWorkflowRun
```

Aggregate/read-model/workflow references resolve. A signal label and value type must match an
`await`; workflow ID/sleep fields must exist. Operations validate entry points but own no separate
persisted decode or identity surface in `diff`.
