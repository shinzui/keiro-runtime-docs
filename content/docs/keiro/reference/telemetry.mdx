The types and functions on this page live in `Keiro.Telemetry`. This is the **single** module where
keiro touches `hs-opentelemetry-api` and `hs-opentelemetry-semantic-conventions`. It carries two
independent, opt-in surfaces: **tracing** (the span helpers below) and **metrics** (the
[`KeiroMetrics`](#metrics-surface) instrument set). Tracing is opt-in
through `Maybe Tracer`: with `Nothing` every
helper is a pass-through that runs the body and emits **zero** spans; with `Just tracer` it opens a
span. The application makes the `Tracer` (typically `hs-opentelemetry-sdk`'s `makeTracer`) and
threads it in via the option records owned by the command and outbox surfaces
([`RunCommandOptions.tracer`](/docs/keiro/reference/command),
[`OutboxPublishOptions.tracer`](/docs/keiro/reference/outbox)). For the wiring narrative, see
[Enable OpenTelemetry](/docs/keiro/how-to/enable-opentelemetry).

## Span helpers

All three are `MonadUnliftIO m` with `HasCallStack`. Each takes a `Maybe Tracer`; under `Nothing` it
calls `body Nothing` and returns — no span. The body receives a `Maybe Span` so it can add
attributes or record an error on the live span.

### `withProducerSpan`

```haskell
withProducerSpan
  :: (MonadUnliftIO m, HasCallStack)
  => Maybe Tracer -> IntegrationEvent -> KafkaProducerRecord -> (Maybe Span -> m a) -> m a
```

Opens a `Producer`-kind span named `"send " <> destination` and sets the messaging attributes for an
outbox publish (`messaging.system = kafka`, `messaging.operation.type = publish`,
`messaging.destination.name`, `messaging.message.id`, and the message key when present). Used by the
outbox publisher.

### `withConsumerSpan`

```haskell
withConsumerSpan
  :: (MonadUnliftIO m, HasCallStack)
  => Maybe Tracer -> Maybe Text -> KafkaInboundRecord -> Maybe IntegrationEvent
  -> (Maybe Span -> m a) -> m a
```

Opens a `Consumer`-kind span named `"process " <> topic`. The optional `Text` is the consumer group
name (recorded as `messaging.consumer.group.name`); the `Maybe IntegrationEvent` is the decoded
envelope (`Nothing` on a decode-failure path, in which case only broker-known attributes are set).
**It extracts the upstream parent** from the inbound record's W3C `traceparent`/`tracestate` headers,
so this span becomes a **child** of the producer span in the upstream process — joining the two
traces by trace id.

### `withCommandSpan`

```haskell
withCommandSpan
  :: (MonadUnliftIO m, HasCallStack)
  => Maybe Tracer -> Text -> Maybe Int64 -> (Maybe Span -> m a) -> m a
```

Opens an `Internal`-kind span named after the resolved stream. Sets `keiro.stream.name` and, when the
`Maybe Int64` retry attempt is given, `keiro.retry.attempt`. The caller adds `keiro.events.appended`
on the returned span after a successful append.

### `withWorkflowSpan`

```haskell
withWorkflowSpan
  :: (MonadUnliftIO m, HasCallStack)
  => Maybe Tracer -> WorkflowName -> WorkflowId -> Maybe StepName -> (Maybe Span -> m a) -> m a
```

Opens an `Internal`-kind span named `"workflow " <> name` around a [durable-workflow](/docs/keiro/reference/durable-workflows)
run. Sets `keiro.workflow.name`, `keiro.workflow.id`, and — when a `StepName` is supplied —
`keiro.workflow.step`. `runWorkflowWith` calls it once per run with the tracer from
`WorkflowRunOptions.tracer`, so a `Nothing` tracer makes the whole run untraced.

## The W3C TraceContext bridge

W3C TraceContext is the web standard for propagating a trace across processes via the
`traceparent`/`tracestate` headers. These helpers convert between an in-memory span and the keiro
`TraceContext` record carried on `IntegrationEvent` envelopes and outbox rows. The application is
expected to have installed the W3C propagator on its `TracerProvider`; these are keiro's side of the
bridge.

```haskell
traceContextFromCurrentSpan :: (MonadIO m) => m (Maybe TraceContext)
traceContextFromHeaders     :: [(Text, Text)] -> Maybe TraceContext
injectTraceContext          :: (MonadIO m) => [(Text, Text)] -> m [(Text, Text)]
```

- `traceContextFromCurrentSpan` reads the current thread-local span and formats it as a
  `TraceContext`; `Nothing` when no span is active.
- `traceContextFromHeaders` lifts a `TraceContext` out of a flat header list (it does not validate
  the `traceparent` format — the propagator's parser does that at consume time).
- `injectTraceContext` appends the current span's `traceparent`/`tracestate` headers to a header
  list, returning it unchanged when no span is active. Used by the outbox publisher.

## Attributes keiro sets

`messaging.*` follow the OpenTelemetry messaging semantic conventions; the `keiro.*` keys are
bespoke to keiro.

| Attribute | Set on | Value |
|---|---|---|
| `messaging.system` | producer, consumer | `"kafka"` |
| `messaging.operation.type` | producer / consumer | `"publish"` / `"process"` |
| `messaging.operation.name` | producer / consumer | `"send"` / `"process"` |
| `messaging.destination.name` | producer, consumer | destination / topic |
| `messaging.message.id` | producer, consumer | the envelope `messageId` |
| `messaging.destination.partition.id` | consumer | inbound record partition |
| `messaging.kafka.offset` | consumer | inbound record offset |
| `messaging.consumer.group.name` | consumer | the consumer group, when given |
| `messaging.kafka.message.key` | producer, consumer | the message key, when present |
| `keiro.stream.name` | command | the resolved stream name |
| `keiro.retry.attempt` | command | the 1-based retry attempt, when given |
| `keiro.events.appended` | command | events appended (added by the caller after append) |
| `keiro.workflow.name` | workflow | the workflow definition name |
| `keiro.workflow.id` | workflow | the workflow instance id |
| `keiro.workflow.step` | workflow | the step name, when a per-step span is opened |

## Attribute keys

keiro links `hs-opentelemetry-semantic-conventions 1.40.0.0` (generated from spec **v1.40**)
directly. Every `messaging.*` / `db.*` typed `AttributeKey` keiro sets is **imported** from
`OpenTelemetry.SemanticConventions` and **re-exported** from `Keiro.Telemetry`, so this module stays
the one-stop telemetry import while every convention name is anchored to the spec-generated module
rather than a hand-typed string. The exported semantic-convention keys (binding name → wire name):

```haskell
messaging_operation_type           -- "messaging.operation.type"
messaging_operation_name           -- "messaging.operation.name"
messaging_destination_partition_id -- "messaging.destination.partition.id"
messaging_consumer_group_name      -- "messaging.consumer.group.name"
messaging_client_id                -- "messaging.client.id"
messaging_kafka_offset             -- "messaging.kafka.offset"
db_system_name                     -- "db.system.name"
db_namespace                       -- "db.namespace"
db_collection_name                 -- "db.collection.name"
db_operation_name                  -- "db.operation.name"
```

Only the bespoke `keiro.*` keys have no upstream equivalent, so they are **defined locally** in
`Keiro.Telemetry`:

```haskell
keiro_stream_name                  -- "keiro.stream.name"
keiro_retry_attempt                -- "keiro.retry.attempt"
keiro_events_appended              -- "keiro.events.appended"
keiro_workflow_name                -- "keiro.workflow.name"
keiro_workflow_id                  -- "keiro.workflow.id"
keiro_workflow_step                -- "keiro.workflow.step"
```

<Callout type="warn">
**Span coverage is partial; metric coverage is broad.** Four sites emit **spans** today:
**outbox publish**, **inbox consume**, **command run**, and a **durable-workflow run**
(`withWorkflowSpan`). Hydration and snapshot reads are still **not traced**. The four background
workers — outbox, inbox, timer, and projection — instead carry the **metric** instruments documented
in [Metrics surface](#metrics-surface) below, so their backlog, lag, and outcome counts are observable
even though they emit no spans. The durable-workflow runtime carries both a span and the
[`keiro.workflow.*` instruments](#workflow-instruments). Both surfaces are tracked in the keiro repo's
`docs/research/opentelemetry-semconv-audit.md`.
</Callout>

## Metrics surface

Alongside the span helpers, `Keiro.Telemetry` exposes an OpenTelemetry **metrics** instrument set for
the four background workers. Like tracing, it is opt-in: workers accept a `Maybe KeiroMetrics`, and
every `record*` helper is a no-op under `Nothing`, so an application that has not wired a `Meter` is
unaffected.

### `Meter` and the instrumentation library

`Meter` is re-exported from `hs-opentelemetry-api`. The application builds it once from its
`MeterProvider` against keiro's instrumentation library:

```haskell
keiroInstrumentationLibrary :: InstrumentationLibrary
keiroInstrumentationLibrary =
  InstrumentationLibrary { libraryName = "keiro", libraryVersion = "", ... }
```

Typically `meter <- getMeter mp keiroInstrumentationLibrary` after constructing the SDK
`MeterProvider`.

### `KeiroMetrics`

The full instrument set. Build it once at startup with `newKeiroMetrics` and thread the
`Just metrics` value into each worker; share **one** value across all workers.

```haskell
data KeiroMetrics = KeiroMetrics
  { outboxBacklog          :: Gauge Int64
  , outboxPublished        :: Counter Int64
  , outboxRetried          :: Counter Int64
  , outboxDeadlettered     :: Counter Int64
  , outboxReclaimed        :: Counter Int64
  , inboxProcessed         :: Counter Int64
  , inboxDuplicates        :: Counter Int64
  , inboxFailed            :: Counter Int64
  , inboxPoisoned          :: Counter Int64
  , inboxBacklog           :: Gauge Int64
  , timerBacklog           :: Gauge Int64
  , timerFireLag           :: Histogram
  , timerAttempts          :: Histogram
  , timerStuck             :: Gauge Int64
  , timerRequeued          :: Counter Int64
  , projectionLag          :: Gauge Int64
  , projectionWaitTimeouts :: Counter Int64
  , dispatchFailed         :: Counter Int64
  , dispatchDuplicates     :: Counter Int64
  , dispatchPoison         :: Counter Int64
  , workflowStepsExecuted     :: Counter Int64
  , workflowStepsReplayed     :: Counter Int64
  , workflowResumed           :: Counter Int64
  , workflowActive            :: Gauge Int64
  , workflowJournalLength     :: Histogram
  , workflowAwakeablesPending :: Gauge Int64
  }

newKeiroMetrics :: (MonadIO m) => Meter -> m KeiroMetrics
```

`newKeiroMetrics` creates every instrument from the `Meter`. Under a no-op meter the instruments are
themselves no-ops, so it is safe to call unconditionally.

### Metric catalogue

These are the wire names keiro registers. The `record*` helpers (and the `*Name` `Text` bindings for
each wire name) are exported from `Keiro.Telemetry`; gauges and counters take `Int64`, histograms
take `Double`. All instruments are recorded with empty attributes (no per-call labels).

| Wire name | Kind | Unit | Record helper | Emitted by |
|---|---|---|---|---|
| `keiro.outbox.backlog` | Gauge | `{event}` | `recordOutboxBacklog` | `outboxMaintenancePass` / `sampleOutboxBacklog` — rows in `pending`/`failed` awaiting publish. |
| `keiro.outbox.published` | Counter | `{event}` | `recordOutboxPublished` | Outbox publisher, per-pass summary — rows successfully published. |
| `keiro.outbox.retried` | Counter | `{event}` | `recordOutboxRetried` | Outbox publisher — publish attempts that failed and will retry. |
| `keiro.outbox.deadlettered` | Counter | `{event}` | `recordOutboxDeadlettered` | Outbox publisher — rows parked in `dead` after exhausting retries. |
| `keiro.outbox.reclaimed` | Counter | `{event}` | `recordOutboxReclaimed` | `outboxMaintenancePass` — stale `publishing` rows reclaimed from a crashed or stalled publisher. |
| `keiro.inbox.processed` | Counter | `{message}` | `recordInboxProcessed` | Inbox transaction — messages processed to completion. |
| `keiro.inbox.duplicates` | Counter | `{message}` | `recordInboxDuplicates` | Inbox transaction — deliveries skipped as duplicates by `(source, dedupe_key)`. |
| `keiro.inbox.failed` | Counter | `{message}` | `recordInboxFailed` | Inbox transaction — messages whose handler raised. |
| `keiro.inbox.poisoned` | Counter | `{message}` | `recordInboxPoisoned` | Retrying inbox wrapper — handler failures that reached the configured attempt ceiling. |
| `keiro.inbox.backlog` | Gauge | `{message}` | `recordInboxBacklog` | `sampleInboxBacklog` — rows in `processing`/`failed` (non-terminal). |
| `keiro.timer.backlog` | Gauge | `{timer}` | `recordTimerBacklog` | Timer worker, once per pass — `scheduled` timers already due. |
| `keiro.timer.fire.lag` | Histogram | `ms` | `recordTimerFireLag` | Timer worker, per claimed timer — milliseconds between `fire_at` and the actual fire. |
| `keiro.timer.attempts` | Histogram | `{attempt}` | `recordTimerAttempts` | Timer worker, per claimed timer — the post-claim attempt count. |
| `keiro.timer.stuck` | Gauge | `{timer}` | `recordTimerStuck` | Timer worker, once per pass — timers stranded in `firing`. |
| `keiro.timer.requeued` | Counter | `{timer}` | `recordTimerRequeued` | Timer worker — stale `firing` rows moved back to `scheduled`. |
| `keiro.projection.lag` | Gauge | `{event}` | `recordProjectionLag` | `recordProjectionLag` (application-driven) — events between the log head and a projection's checkpoint. |
| `keiro.projection.wait.timeouts` | Counter | `{timeout}` | `recordProjectionWaitTimeouts` | Read-model `waitFor` — calls that timed out before the projection caught up. |
| `keiro.dispatch.failed` | Counter | `{command}` | `recordDispatchFailed` | Process-manager/router workers — dispatched commands or source reactions that failed. |
| `keiro.dispatch.duplicates` | Counter | `{command}` | `recordDispatchDuplicate` | Process-manager/router workers — deterministic command ids skipped as duplicates. |
| `keiro.dispatch.poison` | Counter | `{message}` | `recordDispatchPoison` | Process-manager/router workers — messages classified as poison by the decode seam. |

The worker entry points thread the handle for you: the [timer worker](/docs/keiro/reference/timers#runtimerworker)
records the backlog, stuck, fire-lag, and attempts instruments inside each pass; the
[outbox](/docs/keiro/reference/outbox) publisher records publish outcome counters while
`outboxMaintenancePass`/`sampleOutboxBacklog` record reclaim and backlog gauges; the
[inbox](/docs/keiro/reference/inbox) helpers record per-message result counters while
`sampleInboxBacklog` records the backlog gauge; and the projection path exposes
[`recordProjectionLag`](/docs/keiro/reference/projection#recordprojectionlag) for the application to
call on its drain loop. See [Enable OpenTelemetry](/docs/keiro/how-to/enable-opentelemetry) for the
end-to-end wiring.

### Workflow instruments

The nine `keiro.workflow.*` instruments cover the [durable-workflow runtime](/docs/keiro/reference/durable-workflows).
They are recorded only when a `KeiroMetrics` handle is threaded through `WorkflowRunOptions.metrics`
(directly, or via `WorkflowResumeOptions.runOptions` for a resumed run); the two per-pass gauges below
are sampled by the resume worker. The `*Name` `Text` bindings and `record*` helpers are exported from
`Keiro.Telemetry`.

| Wire name | Kind | Unit | Record helper | Emitted by |
|---|---|---|---|---|
| `keiro.workflow.steps.executed` | Counter | `{step}` | `recordWorkflowStepExecuted` | A `step`/`awaitStep` **miss** — the action ran and was journaled. |
| `keiro.workflow.steps.replayed` | Counter | `{step}` | `recordWorkflowStepReplayed` | A `step`/`awaitStep` **hit** — the recorded result was returned without running the action. |
| `keiro.workflow.resumed` | Counter | `{workflow}` | `recordWorkflowResumed` | The resume worker — one increment per re-invocation of a registered workflow. |
| `keiro.workflow.failed` | Counter | `{workflow}` | `recordWorkflowFailed` | The resume worker — workflows marked terminally failed after exhausting retry attempts. |
| `keiro.workflow.resume.errors` | Counter | `{error}` | `recordWorkflowResumeErrors` | The resume worker — transient store errors while advancing individual workflow instances. |
| `keiro.workflow.lease.skipped` | Counter | `{lease}` | `recordWorkflowLeaseSkipped` | The resume worker — candidates skipped because another worker holds a live instance lease. |
| `keiro.workflow.active` | Gauge | `{workflow}` | `recordWorkflowActive` | The runtime — workflow runs currently in progress in this process (sampled on run entry and exit). |
| `keiro.workflow.journal.length` | Histogram | `{event}` | `recordWorkflowJournalLength` | The runtime — a completing run's journal event count (the `Completed` path only). |
| `keiro.workflow.awakeables.pending` | Gauge | `{awakeable}` | `recordWorkflowAwakeablesPending` | The resume worker, once per pass — `pending` rows in `keiro_awakeables`. |

The `steps.executed` vs `steps.replayed` split is the live-execution vs journal-replay signal: a
healthy resume after a crash is mostly replays plus a small executed tail.
