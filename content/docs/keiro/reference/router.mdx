A `Router` is the stateless sibling of a process manager. For each source event, it resolves zero or
more target commands effectfully and dispatches each in its own transaction.

## Definition

```haskell
data Router input targetPhi targetRs targetState targetCi targetCo es = Router
  { name              :: Text
  , key               :: input -> Text
  , resolve           :: input -> Eff es [PMCommand targetCi]
  , targetEventStream :: ValidatedEventStream targetPhi targetRs targetState targetCi targetCo
  , targetProjections :: Stream targetCi -> [InlineProjection targetCo]
  }

newtype RouterResult target = RouterResult
  { commandResults :: [PMCommandResult target] }
```

`resolve` computes the target set; it does not dispatch. Keep it read-only in effectâ€”typically an
explicitly registered read-model queryâ€”because it may run again for the same source event. Returning
`[]` is an intentional no-op. Each target's inline projections share that target command's append
transaction; the entire fan-out is not atomic.

`RouterResult` contains one target-bearing `PMCommandResult` per resolved command, in resolution
order. There is no manager-state append and therefore no outer `Either CommandError`.

## Target-derived IDs

```haskell
deterministicRouterCommandId ::
  Text -> Text -> EventId -> StreamName -> Int -> EventId
```

The UUIDv5 name includes router name, correlation key, source event ID, resolved target stream name,
and the command's occurrence among commands addressing that same stream. Text fields use
length-prefixed UTF-8, avoiding delimiter ambiguity and non-ASCII truncation.

Target identityâ€”not list positionâ€”is the critical 0.2 contract. If a redelivery returns targets in
a different order, the same target and same-stream occurrence still derive the same ID. If `resolve`
returns a newly added target, that target receives a new dispatch. Across attempts, the durable
recipient set is therefore the union of resolved sets; keep `resolve` stable when one exact snapshot
of recipients matters.

Multiple commands to one target in the same result use occurrences `0`, `1`, and so on, preserving
distinct IDs.

## Upgrade probe for positional IDs

Before dispatch, the router checks both the current target-derived ID and a legacy
`deterministicCommandId ... legacyIndex`. The legacy check is scoped to the same target stream, so it
cannot mistake another target's old positional event for success.

<Callout type="warn">
  The legacy positional lookup is a transition bridge, not a permanent exactly-once guarantee
  across simultaneous code and resolver-output changes. If target ordering/membership changes while
  old positional writes are still being redelivered, an already handled logical target may receive
  one new target-derived ID. Keep resolution stable through the compatibility window and remove the
  probe in a later coordinated release.
</Callout>

## Dispatch and duplicate proof

```haskell
runRouterOnce ::
  ( HasCallStack, IOE :> es, Store :> es, Error StoreError :> es
  , KirokuStoreResource :> es
  , BoolAlg targetPhi (RegFile targetRs, targetCi), Eq targetCo ) =>
  RunCommandOptions -> Router ... -> RecordedEvent -> input -> Eff es (RouterResult ...)
```

For each command, the runner:

1. derives the target stream name and same-stream occurrence;
2. checks the current ID in that target;
3. checks the legacy positional ID in that target during the upgrade window;
4. otherwise dispatches with `runCommandWithProjections`; and
5. folds `DuplicateEvent` to `PMCommandDuplicate` only when `confirmBenignDuplicate` proves the
   current attempted ID is present in the intended target.

A globally colliding ID that exists only in another stream remains
`PMCommandFailed targetStreamName error`. It is a systemic deterministic failure and halts rather
than being silently acknowledged.

## Worker policy

```haskell
runRouterWorkerWith ::
  WorkerOptions es msg -> RunCommandOptions -> Router ... -> Adapter es msg ->
  (msg -> Maybe (RecordedEvent, input)) -> Eff es ()
```

`runRouterWorker` supplies `defaultWorkerOptions`. Poison input follows `PoisonPolicy`; transient
failures retry; systemic deterministic failures halt; and an all-rejection group follows
`RejectedCommandPolicy` (`RejectedHalt`, `RejectedDeadLetter`, or `RejectedSkip`). The default is
`RejectedHalt`. Each ingested ack handle is finalized once.

`RejectedDeadLetter` writes `keiro.keiro_dead_letters` rows with
`DispatcherRouter`, the target stream, source identity, error class/detail, emit index, and attempt
count before acknowledging the source. It is separate from Kiroku's retry-exhaustion table.

## Changing what a router dispatches

A router holds no state of its own, so it has no replay surface â€” but its **target-derived ids** make
a dispatch change deploy-sensitive in exactly the way a process manager's do.

<Callout type="warn">
Change `resolve` or `targetProjections` while old source events are still in the redelivery window and
a redelivered event dispatches the *new* command under the *same* deterministic id. The store confirms
it as a benign duplicate of the old dispatch, so overlapping targets deduplicate while only the newly
selected targets actually run â€” a silently mixed half-old, half-new fan-out.

Drain the subscription, resolve or explicitly discard its dead letters, deploy, then resume:
[deploy-ordering rule 4](/docs/keiro/reference/deploy-ordering#4-drain-process-manager-and-router-decide-changes).
</Callout>

`keiro-dsl diff` raises the non-breaking `RouterDecideSurfaceChanged` advisory for spec-visible
resolve/dispatch edits. Hole-only changes remain invisible, so the drain rule stands whenever
hand-owned dispatch logic changes. A changed router *stable name*, key, or target is BREAKING instead
â€” it re-keys durable identity.

[Inspect a rejected dispatch](/docs/keiro/how-to/inspect-a-rejected-dispatch).
