The types and functions on this page live in `Keiro.Router` (re-exported from `Keiro`). A `Router`
is a **stateless, content-based router** (in the Enterprise Integration Patterns sense): for each
incoming event it resolves a data-dependent set of target streams *effectfully* and dispatches one
command to each. See [Route events to commands](/docs/keiro/how-to/route-events-to-commands) for the
task recipe.

## `Router`

<TypeTable
  type={{
    name: { type: "Text", description: "Part of every dispatched command's deterministic id" },
    key: { type: "input -> Text", description: "Correlation string for the source event" },
    resolve: { type: "input -> Eff es [PMCommand targetCi]", description: "Effectful target set (e.g. a runQuery against a read model)" },
    targetEventStream: { type: "ValidatedEventStream targetPhi targetRs targetState targetCi targetCo", description: "The validated event stream each dispatched command runs against" },
    targetProjections: { type: "Stream targetCi -> [InlineProjection targetCo]", description: "Inline projections to run in the dispatch transaction" },
  }}
/>

```haskell
data Router input targetPhi targetRs targetState targetCi targetCo es = Router
  { name              :: !Text
  , key               :: !(input -> Text)
  , resolve           :: !(input -> Eff es [PMCommand targetCi])
  , targetEventStream :: !(ValidatedEventStream targetPhi targetRs targetState targetCi targetCo)
  , targetProjections :: !(Stream targetCi -> [InlineProjection targetCo])
  }
  deriving stock (Generic)
```

`PMCommand`, `PMCommandResult`, `deterministicCommandId`, and `eventAlreadyIn` come from
`Keiro.ProcessManager` (see the [process-manager reference](/docs/keiro/reference/process-manager)),
and `InlineProjection` / `runCommandWithProjections` from `Keiro.Projection` (see the [projection
reference](/docs/keiro/reference/projection)).

### The `resolve` seam

`resolve :: input -> Eff es [PMCommand targetCi]` is the one effectful field, and the source of most
confusion: it **computes the target set, it does not dispatch**. Given the decoded source event it
returns a list of `PMCommand`s (each a target `Stream` paired with the command to run on it); keiro
performs the idempotent dispatch *after* `resolve` returns.

- It is a **query, not a write** — read a read model (`runQuery`) or config and return; never append
  events from inside `resolve`. keiro owns the writes.
- The list may be **empty, one, or many** — that is the dynamic-recipient-list behaviour. `[]` means
  "routes nowhere" (an intentional no-op, not an error).
- It can run **more than once** for the same source event (the worker retries on failure), which is
  safe precisely because it only reads; the dispatch it feeds is idempotent by deterministic id.

The `key :: input -> Text` field is the correlation string (an incident id, a transaction id) folded
into each `deterministicCommandId`, so re-derived ids are stable per source event. For the EIP
lineage and the stateless mental model, see [Routers and content-based
dispatch](/docs/keiro/explanation/routers-and-content-based-dispatch).

### `targetProjections`

`targetProjections :: Stream targetCi -> [InlineProjection targetCo]` runs read-model updates in the
**same transaction** as each dispatched command's append. Use `const []` for append-only dispatch
(the common case); return projections when a dispatched event must update a read model atomically
with its write.

## `RouterResult`

```haskell
newtype RouterResult target = RouterResult { commandResults :: [PMCommandResult target] }
  deriving stock (Generic, Eq, Show)
```

A failed dispatch surfaces as a `PMCommandFailed` element of the list, **not** as an outer `Either`.

## `runRouterOnce`

Dispatch the commands for one source event.

```haskell
runRouterOnce ::
  forall input targetPhi targetRs targetState targetCi targetCo es.
  (HasCallStack, IOE :> es, Store :> es, Error StoreError :> es,
   BoolAlg targetPhi (RegFile targetRs, targetCi), Eq targetCo) =>
  RunCommandOptions ->
  Router input targetPhi targetRs targetState targetCi targetCo es ->
  RecordedEvent ->
  input ->
  Eff es (RouterResult (EventStream targetPhi targetRs targetState targetCi targetCo))
```

<Callout type="info">
Dispatch is **exactly-once-per-target by construction**. For each resolved target, the router derives
a `deterministicCommandId name correlationId sourceEventId emitIndex`, pre-checks it with
`eventAlreadyIn` (skip → `PMCommandDuplicate`), otherwise runs the command and folds a store-level
`DuplicateEvent` rejection into a benign `PMCommandDuplicate`. Re-processing the same source event
writes nothing the second time.
</Callout>

## `runRouterWorker`

Run a router as a live subscription, draining a shibuya `Adapter`.

```haskell
runRouterWorker ::
  forall msg input targetPhi targetRs targetState targetCi targetCo es.
  (HasCallStack, IOE :> es, Store :> es, Error StoreError :> es,
   BoolAlg targetPhi (RegFile targetRs, targetCi), Eq targetCo) =>
  RunCommandOptions ->
  Router input targetPhi targetRs targetState targetCi targetCo es ->
  Adapter es msg ->
  (msg -> Maybe (RecordedEvent, input)) ->
  Eff es ()
```

The worker decodes each message to `(RecordedEvent, input)` and finalizes the ingested message's
`AckHandle`:

- an **undecodable** message → the configured `PoisonPolicy` from `WorkerOptions` (default
  `AckHalt (HaltFatal …)`);
- every dispatch `PMCommandAppended` / `PMCommandDuplicate` → `AckOk`;
- transient `PMCommandFailed` values → `AckRetry transientRetryDelay`;
- deterministic `PMCommandFailed` values → `AckHalt (HaltFatal …)`.

Use `runRouterWorkerWith` to override the poison policy, transient retry delay, or dispatch metrics.
The worker invokes the adapter ack handle exactly once per ingested message.

The worked example is `jitsurei` `pagingRouter` (`jitsurei/src/Jitsurei/Paging.hs`): for each
`IncidentRaisedData` it `runQuery`s the on-call read model and dispatches one `SendPage` per
responder, with `targetProjections = const []`. Run it with `just jitsurei-paging`.
