The types and functions on this page live in the **`Keiro.Workflow`** module family. A *durable
workflow* is an ordinary `effectful` computation whose side effects are recorded ("journaled") at
**named** checkpoints, so it can pause тАФ across a crash, a redeploy, or an idle wait тАФ and resume by
re-invoking it from the top while short-circuiting the steps it already completed. For the concept
and the design rationale, read [Understanding durable execution](/docs/keiro/explanation/durable-execution);
for the source tour, read [the durable-execution walkthrough](/docs/keiro/walkthrough/durable-execution/00-start-here).

The umbrella `Keiro` module does **not** re-export the workflow surface. Import `Keiro.Workflow*`
directly, the same way you import `Keiro.Timer` or `Keiro.Outbox`:

```haskell
import Keiro.Workflow           -- the effect, step, runWorkflow/runWorkflowWith, the journal codec
import Keiro.Workflow.Sleep     -- sleepNamed / sleep, runWorkflowTimerWorker
import Keiro.Workflow.Awakeable -- awakeableNamed / awakeable, signalAwakeable, cancelAwakeable
import Keiro.Workflow.Child     -- spawnChild, awaitChild, cancelChild
import Keiro.Workflow.Resume    -- resumeWorkflowsOnce, the registry, the fixed-poll and push-aware workers
import Keiro.Workflow.Snapshot  -- workflowStateCodec (the snapshot discriminant)
import Keiro.Workflow.Gc        -- gcWorkflowsOnce / runWorkflowGcWorker
```

`continueAsNew`, `restoreSeed`, and `patch` ([versioning and rotation](#versioning-and-rotation)) are
re-exported from `Keiro.Workflow`. The [push-aware resume worker](#the-resume-worker) additionally
rides `Keiro.Wake` тАФ see the [push-delivery reference](/docs/keiro/reference/push-delivery).

## The effect and the authoring surface

`Keiro.Workflow` defines one effect, `Workflow`, interpreted by `runWorkflowWith`. You rarely send
its operations directly; the four primitives below are built on them. The two primitive operations
are `step` (run-once-and-journal) and `awaitStep` (the suspension primitive every wake source builds
on).

```haskell
data Workflow :: Effect

step          :: (Workflow :> es, ToJSON a, FromJSON a) => StepName -> Eff es a -> Eff es a
awaitStep     :: (Workflow :> es, FromJSON a)           => StepName -> Eff es () -> Eff es a
currentWorkflow :: (Workflow :> es) => Eff es (WorkflowName, WorkflowId)
freshOrdinal  :: (Workflow :> es) => Text -> Eff es Int
```

- **`step name action`** runs `action` once and journals its JSON-encoded result. On a replay where
  `name` is already journaled, the recorded result is returned and `action` is **not** run again.
  If the process crashes after `action` runs but before the journal append commits, a later resume has
  no record of the step and may run it again, so step side effects are **at-least-once across
  crashes**. Identity is the **name**, not the source position, so inserting or reordering code
  elsewhere in the workflow never corrupts an in-flight instance.
- **`awaitStep name arm`** returns `name`'s journaled result if a wake source has recorded it;
  otherwise it runs the idempotent `arm` action once and **suspends** the run (`runWorkflow` returns
  `Suspended`). `arm` **must be idempotent** тАФ a resumed run re-enters `awaitStep` from the top on
  every resume until the result is journaled, so it re-runs `arm` each time. The sleep, awakeable, and
  child primitives are all thin wrappers over `awaitStep`.

  On a miss in the in-memory replay map, `awaitStep` first consults the generation-scoped
  `keiro_workflow_steps` index тАФ *before* checking cancellation, running `arm`, or suspending. An
  index hit is inserted into the map and returned exactly like an ordinary replay hit. This is what
  makes snapshots safe to combine with wake sources: see
  [the step-index fallback](#the-step-index-fallback-under-snapshots). The fallback applies **only**
  to `awaitStep`; a `step` or `patch` miss attempts an append whose transaction rechecks the same
  index under the step lock, so those already converge.
- **`currentWorkflow`** returns the running instance's `(WorkflowName, WorkflowId)` тАФ used to key wake
  sources deterministically.
- **`freshOrdinal namespace`** is a per-run, per-namespace counter (from 0) backing the ordinal
  convenience forms (`sleep`, `awakeable`); ordinal names are only stable if call order does not
  change across deploys.

## Identity and the journal

`Keiro.Workflow.Types` owns the identity newtypes, the stream naming, and the journal codec
(`Keiro.Workflow` re-exports all of it).

```haskell
newtype WorkflowName = WorkflowName { unWorkflowName :: Text }   -- the definition (e.g. "orderFulfillment")
newtype WorkflowId   = WorkflowId   { unWorkflowId   :: Text }   -- one instance
newtype StepName     = StepName     { unStepName     :: Text }   -- a checkpoint label
newtype PatchId      = PatchId      { unPatchId      :: Text }   -- a versioning patch (see "Versioning and rotation")

data WorkflowIdentityError
  = WorkflowNameEmpty | WorkflowNameInvalidChar Char Text
  | WorkflowIdEmpty   | WorkflowIdInvalidChar Char Text

mkWorkflowName :: Text -> Either WorkflowIdentityError WorkflowName
mkWorkflowId   :: Text -> Either WorkflowIdentityError WorkflowId

workflowStreamName           :: WorkflowName -> WorkflowId -> StreamName          -- "wf:<name>-<id>"
workflowGenerationStreamName :: WorkflowName -> WorkflowId -> Int -> StreamName    -- the physical stream for one generation
```

Every instance owns one kiroku stream named **`wf:<name>-<id>`**. `:` and `-` are structural тАФ names
and ids must not make the boundary ambiguous (the same caveat as the process-manager
`pm:<name>-<correlationId>` convention), and `#` is reserved for generation suffixes. Prefer the
validated constructors: `mkWorkflowName` rejects empty names plus `:`, `-`, and `#`; `mkWorkflowId`
rejects empty ids plus `:` and `#`, while allowing UUID-style hyphens. The raw constructors remain
available for compatibility but can create ambiguous journal stream names. `continueAsNew` (below)
rotates an instance's *physical* journal onto a fresh **generation**: generation 0 keeps the bare
`wf:<name>-<id>` name (so never-rotated workflows are byte-for-byte unchanged), and generation `g > 0`
appends a `#<g>` suffix тАФ `workflowGenerationStreamName name id g`. The *logical* identity
`(WorkflowName, WorkflowId)` you and the resume registry see is stable across rotations.

```haskell
data WorkflowJournalEvent
  = StepRecorded     { stepName :: Text, result :: Value, recordedAt :: UTCTime }
  | WorkflowCompleted { recordedAt :: UTCTime }
  | WorkflowCancelled { recordedAt :: UTCTime }
  | WorkflowFailed    { reason :: Text, recordedAt :: UTCTime }
  | WorkflowContinuedAsNew { generation :: Int, recordedAt :: UTCTime }   -- the EP-48 rotation marker

workflowJournalCodec :: Codec WorkflowJournalEvent   -- schemaVersion 1, no upcasters
```

Each `step` appends a `StepRecorded`; the terminal `WorkflowCompleted` closes the journal. The
suspension primitives journal their completions as the **same** `StepRecorded` event under reserved
step-name prefixes тАФ never as new event types тАФ so the replay loop stays uniform. `continueAsNew` adds
the one new terminal-for-this-generation marker, `WorkflowContinuedAsNew` (its `generation` is the
*next* generation the rotation opens); it is additive within `schemaVersion = 1`, so old journals never
carry the tag and no upcaster is needed.

| Reserved binding | Value | Used by |
|---|---|---|
| `sleepStepPrefix` | `"sleep:"` | a durable sleep's completion |
| `awakeableStepPrefix` | `"awk:"` | an awakeable's completion |
| `childStepPrefix` | `"child:"` | a child spawn record and its `:result` completion |
| `patchStepPrefix` | `"patch:"` | a `patch` branch decision (`patch:<patchId>`) |
| `completedStepName` | `"__workflow_completed__"` | the terminal completion marker (index row) |
| `cancelledStepName` | `"__workflow_cancelled__"` | the terminal cancellation marker (index row) |
| `failedStepName` | `"__workflow_failed__"` | the terminal failure marker (index row) |
| `continuedAsNewStepName` | `"__workflow_continued_as_new__"` | the rotation marker (terminal *for its generation*) |
| `continueSeedStepName` | `"__workflow_seed__"` | the carried `continueAsNew` seed, on the next generation |

The helper `patchStepName :: PatchId -> Text` is `patchStepPrefix <> unPatchId`.

```haskell
type WorkflowState = Map Text Value   -- the accumulated step-name -> result map a run folds
```

## Running a workflow

```haskell
data WorkflowOutcome a = Completed a | Suspended | Cancelled | Failed | ContinuedAsNew

runWorkflow     :: (IOE :> es, Store :> es, Error StoreError :> es) => WorkflowName -> WorkflowId -> Eff (Workflow : es) a -> Eff es (WorkflowOutcome a)
runWorkflowWith :: (IOE :> es, Store :> es, Error StoreError :> es) => WorkflowRunOptions -> WorkflowName -> WorkflowId -> Eff (Workflow : es) a -> Eff es (WorkflowOutcome a)
```

`runWorkflowWith` is the **single canonical entry point**; `runWorkflow` is it with
`defaultWorkflowRunOptions`. Each run first resolves the instance's *current* (highest) generation and
operates only on it; a never-rotated workflow stays at generation 0, so naming, load, and append are
byte-for-byte as before. The resume worker re-invokes through `runWorkflowWith`, so a resumed run
honours the same options. The typed `Error StoreError` channel is required even when the workflow body
does not use it: journal failures remain caller-visible, while the runner catches only post-commit
snapshot-store failures so they cannot reverse an already-durable transition. The outcome is:

<TypeTable
  type={{
    "Completed a": { type: "тАФ", description: "The computation ran to its end; a WorkflowCompleted marker is journaled." },
    Suspended: { type: "тАФ", description: "The run parked at an unresolved awaitStep (a sleep, awakeable, or child wait); a wake source was armed and a later resume continues." },
    Cancelled: { type: "тАФ", description: "The journal already carries a WorkflowCancelled marker (a child cancelled by its parent); the run short-circuited and executed nothing." },
    Failed: { type: "тАФ", description: "The journal already carries a WorkflowFailed marker; the run short-circuited and executed nothing." },
    ContinuedAsNew: { type: "тАФ", description: "The run rotated onto a fresh journal generation via continueAsNew. Still unfinished (distinct from Completed): the new generation has no terminal marker, so the resume worker re-invokes it on its current generation." },
  }}
/>

Running the same `(name, id)` again replays the journal and short-circuits completed steps. If the
journal already carries a `WorkflowCancelled` marker the run short-circuits to `Cancelled` without
executing any step; if it carries `WorkflowFailed`, it short-circuits to `Failed`.

### Lease renewal and `WorkflowLeaseLost`

The resume worker populates `leaseHeartbeat` automatically from the claim it just made; direct
`runWorkflow` calls leave it `Nothing`.

When it is present, the runtime **renews the lease for another full TTL immediately before each fresh
boundary** тАФ a step action that is about to run for the first time, and an await arm that is about to
be armed. Replay hits neither write nor renew. If another worker owns the row at such a boundary, the
runtime throws:

```haskell
data WorkflowLeaseLost = WorkflowLeaseLost   -- an Exception
```

before performing any further side effects. The resume worker catches it, counts a `leaseSkipped`,
and **consumes no crash attempt** тАФ losing a lease is not a workflow bug.

<Callout type="info">
Size `WorkflowResumeOptions.leaseTtl` above the longest **individual** step action or await arm,
including its normal timeout тАФ not above the whole multi-step advance. A healthy multi-step run no
longer loses ownership merely because its original claim aged out. `leaseTtl` still bounds
dead-worker takeover delay, so do not inflate it beyond what the slowest single action needs.
</Callout>

<Callout type="warn">
Set option fields with the **generic-lens label**, not a bare record update:

```haskell
defaultWorkflowRunOptions & #snapshotPolicy .~ Every 2
                          & #metrics .~ Just m
```

A bare record update on `snapshotPolicy` is ambiguous тАФ the name collides with keiki's `EventStream`
field of the same name.
</Callout>

### Errors

```haskell
data WorkflowError
  = WorkflowStepDecodeError   Text Text   -- a journaled step result didn't decode into the replaying type (step name, message)
  | WorkflowJournalDecodeError Text        -- a journal event didn't decode during pre-load
  | WorkflowJournalEncodeError Text        -- a journal event didn't encode for append
  | WorkflowJournalAppendError Text        -- a journal append failed for a non-conflict reason
```

All four surface through `throwIO`, so they propagate through the surrounding store/IO error channel.
A `WorkflowStepDecodeError` is a programmer error: a step's result type changed incompatibly тАФ see
[Failure, retries, and resurrection](#failure-retries-and-resurrection) for the recovery path, and
[deploy-ordering rule 8](/docs/keiro/reference/deploy-ordering#8-treat-workflow-step-results-as-permanent)
for how to avoid it.

`WorkflowLeaseLost` is a separate exception (not a `WorkflowError` constructor) because it is not a
failure of the workflow: the resume worker classifies it as a lease skip.

## Versioning and rotation

These three functions on the authoring surface (re-exported from `Keiro.Workflow`) let a workflow
keep an unbounded history bounded, and let you evolve a workflow's logic while instances are in flight.

```haskell
continueAsNew :: (Workflow :> es, ToJSON s)               => s -> Eff es a
restoreSeed   :: (Workflow :> es, ToJSON s, FromJSON s)   => s -> Eff es s
patch         :: (Workflow :> es)                         => PatchId -> Eff es Bool
```

### `continueAsNew` / `restoreSeed` тАФ journal rotation

`continueAsNew seed` rotates a long-running workflow onto a fresh journal **generation**: it snapshots
the carried `seed`, journals a terminal `WorkflowContinuedAsNew` marker on the current generation, and
unwinds *this* run (the result type is fully polymorphic because control never returns to the caller тАФ
the rotated continuation runs on the next run/resume). This is how a workflow that runs an *unbounded*
number of steps тАФ a poller, a per-day rolling process тАФ keeps its per-generation journal bounded so
replay and hydration stay fast forever.

`restoreSeed def` reads the carried seed back at the **top of the workflow body**: it is an ordinary
journaled `step` under `continueSeedStepName`, so on a rotated-into generation it hits the seed the
rotation wrote (returning the carried value without re-running), and on the first generation it misses
and records `def`.

```haskell
pollerWorkflow :: (Workflow :> es, Store :> es, IOE :> es) => Eff es a
pollerWorkflow = do
  cursor <- restoreSeed (Cursor 0)                 -- carried value, or the default on generation 0
  batch  <- step (StepName "poll") (pollFrom cursor)
  sleepNamed (StepName "interval") 60
  continueAsNew (advance cursor batch)             -- rotate; never returns
```

A rotating run returns the [`ContinuedAsNew`](#running-a-workflow) outcome; the resume worker drives
it forward on its current generation. The rotation is crash-safe: the next generation's seed step is
appended **first** (advancing `currentGeneration`), so even a crash mid-rotation converges to
"continue from the seed" and never re-runs the closed generation. The one-entry seed map is offered
to the snapshot store unconditionally on rotation (regardless of `snapshotPolicy`). A successful
write makes the next generation hydrate in O(1); a failed write is counted and the next generation
still converges by replaying its already-committed seed entry.

The rotation also records the deployment's **active patch set atomically with the seed**, in the same
transaction. Without that, an asynchronous append racing the rotation тАФ a duplicate awakeable signal,
a child completion тАФ could land on the new generation before its patch set existed, silently forcing
every later `patch` call in that instance to decide `False`. Generation 0 has no rotation moment, so
it still records its set on the first run.

### `patch` тАФ versioning in-flight workflows

`patch pid` returns a **stable, journaled** branch decision: `True` for an instance that is fresh at
this point (run the new logic) and `False` for one already in flight when the patch shipped (keep the
old logic). The decision is journaled once under `patch:<pid>` and replayed verbatim forever.

<Callout type="warn">
`patch` is an **escape hatch** for a change that *cross-cuts several steps*. For the common case тАФ one
step changed тАФ do **not** use `patch`: **rename the step** instead. A renamed `StepName` has no
journaled history under its new name, so its action runs fresh on the next replay, which is exactly
right for a single-step change. Reach for `patch` only when an in-flight instance would be left
incoherent by the new code (the change adds, removes, or reorders steps, or changes the meaning of an
already-journaled result). **Never reuse a retired `PatchId`.**
</Callout>

See [Write a durable workflow тЖТ Versioning](/docs/keiro/how-to/write-a-durable-workflow#versioning-a-running-workflow)
for the rename-vs-patch decision worked through an example.

## The four primitives

`step` is the first primitive (above). The other three are wrappers over `awaitStep`, each in its own
module.

### Durable sleep

In `Keiro.Workflow.Sleep`:

```haskell
sleepNamed :: (Workflow :> es, Store :> es, IOE :> es) => StepName -> NominalDiffTime -> Eff es ()
sleep      :: (Workflow :> es, Store :> es, IOE :> es) => NominalDiffTime -> Eff es ()
```

`sleepNamed name delta` arms a deterministic row in the existing `keiro_timers` table and suspends.
When a timer worker fires the row it appends a `sleep:<name>` completion to the journal, so a later
run continues past the sleep. The wait survives a restart with **no external scheduler and no
in-memory timer thread** тАФ its only durable state is one `keiro_timers` row. `sleep delta` is the
ordinal convenience form (`sleep:0`, `sleep:1`, тАж); prefer `sleepNamed` for anything that must survive
a code edit mid-flight.

A workflow-sleep timer row carries
`{"kind":"keiro.workflow.sleep","step":"sleep:<suffix>","gen":<n>}` in its `payload`, which is how one
timer worker distinguishes a workflow sleep from an ordinary process-manager timer **and** pins the
fire to the generation that armed it. **A worker must drain the timers** for a sleep ever to fire:

```haskell
workflowSleepFireAction :: (Store :> es, IOE :> es) => TimerRow -> Eff es (Maybe EventId)
runWorkflowTimerWorker  :: (IOE :> es, Store :> es) => Maybe KeiroMetrics -> UTCTime -> (TimerRow -> Eff es (Maybe EventId)) -> Eff es (Maybe TimerRow)
```

`runWorkflowTimerWorker` is one `runTimerWorker` pass that handles **both** workflow sleeps and
ordinary timers: for each claimed timer it tries `workflowSleepFireAction`, and falls back to the
supplied process-manager fire action when the row is not a sleep. A workflow-only deployment passes
`\_ -> pure Nothing` as the fallback. Firing is idempotent (the journal append uses a deterministic id
and pre-checks the step), so at-least-once timer firing yields exactly-once journaling. Workflow sleeps
inherit the timer subsystem's [stuck-row recovery API](/docs/keiro/reference/timers#recovery-operations)
for free. Helpers: `sleepTimerId`, `sleepStepName`, `sleepTimerPayload`, `parseSleepPayload`,
`matchSleepTimerGeneration`, `workflowSleepKind`.

#### Sleep timers are generation-owned lifecycle state

Three rules make a sleep behave correctly across rotation, re-arms, and collection.

**The arming generation owns the completion.** `workflowSleepFireAction` appends directly to the
generation recorded in the payload, through `prepareJournalAppend`. A legacy payload with no `gen` key
recovers its generation by matching its deterministic timer id against candidate generations
(`matchSleepTimerGeneration`). So a timer left in `firing` and requeued after `continueAsNew` becomes
an idempotent append check on its *original* generation, and can never resolve a same-named sleep on
the next one.

**Only the arming insert owns the wake hint.** `scheduleTimerOnceTx` now returns `Bool` тАФ `True` when
its first-arm-wins insert created the row. Only that successful insert writes the instance's
`wake_after`. A replay arm therefore leaves both `fire_at` and `wake_after` untouched, so re-entering
a due or already-fired sleep cannot postpone discovery. A successful fire clears `wake_after` in the
**same transaction** as its journal append, so a fired sleeper is rediscovered promptly.

<Callout type="warn">
`scheduleTimerOnceTx` changed from `Tx.Transaction ()` to `Tx.Transaction Bool`. Callers that do not
need insertion ownership can discard it with `void`.
</Callout>

**A terminal instance refuses its own timers.** Before appending, a claimed sleep checks the instance
row: if it exists with `completed`, `cancelled`, or `failed` status, the fire cancels the timer and
appends nothing. A *missing* instance row is deliberately **not** treated as terminal тАФ a workflow can
crash after arming a first-operation sleep but before recording its suspended instance, and that fire
is allowed to append, recreating the discoverable running instance.

Together with [garbage collection](#garbage-collection) deleting sleep timers in *every* lifecycle
state, this closes the hole where a surviving scheduled timer could resurrect and re-execute a
workflow whose journal and instance data had already been collected.

### Awakeables

In `Keiro.Workflow.Awakeable` тАФ a durable promise an external system resolves: a webhook, a human
approver, an LLM tool call.

```haskell
newtype AwakeableId = AwakeableId UUID   -- deriving ToJSON/FromJSON over the inner UUID

awakeableNamed :: (Workflow :> es, Store :> es, IOE :> es, FromJSON a) => StepName -> Eff es (AwakeableId, Eff es a)
awakeable      :: (Workflow :> es, Store :> es, IOE :> es, FromJSON a) => Eff es (AwakeableId, Eff es a)

signalAwakeable     :: (IOE :> es, Store :> es, ToJSON r) => AwakeableId -> r -> Eff es Bool
signalAwakeableFrom :: (IOE :> es, Store :> es, ToJSON r) => AwakeableRow -> r -> Eff es Bool
cancelAwakeable     :: (Store :> es) => AwakeableId -> Eff es Bool

deterministicAwakeableId :: WorkflowName -> WorkflowId -> Text -> AwakeableId
```

`awakeableNamed label` returns a journaled opaque `AwakeableId` (hand it to the external system) and an
`await` action; the workflow suspends on `await` until `signalAwakeable aid payload` records the
payload in the journal. New allocations are random v4 UUIDs recorded under `awkid:<label>`, so replay
reads the same id the workflow already handed out without making it predictable from public
coordinates. `deterministicAwakeableId` is retained only as a legacy generation-0 adoption helper: if
a pre-change workflow already registered a deterministic row, the first post-change allocation adopts
that row. `awakeable` is the ordinal convenience form (label `ord:N`); prefer `awakeableNamed`.

The `pending` row is **registered inside the journaled allocation step's action**, before the id can
be appended, returned, or handed off. An immediate external signal can therefore never be mistaken for
an unknown id. The await arm keeps its own idempotent registration as a repair path for historical
allocations whose journal entry exists without a row. A crash between row registration and allocation
journaling can leave an unreachable pending row; garbage collection removes awakeables by owner
coordinates, and the id was never exposed.

- **`signalAwakeable`** is idempotent and crash-safe. It returns `True` only on the
  `pending тЖТ completed` transition; a second signal returns `False` and leaves the stored payload
  unchanged. It **re-appends** the journal entry whenever the row is `completed` (the value just
  written, or the stored payload), so a crash between the row update and the journal append heals on a
  later signal. That external re-append is itself **idempotent**: it pre-checks the journal for the
  entry's deterministic id and re-checks after a racing insert, so a concurrent or repeated signal
  coalesces on the existing entry instead of duplicating it. A `False` return therefore does not mean
  "nothing happened". Returns `False` for an unknown id.

  It also **re-reads status inside its own transaction** when the guarded `pending тЖТ completed`
  update loses a race. It appends when it performed the transition, when it repaired an
  already-completed row from the stored payload, or when the in-transaction read shows another signal
  completed the row. If **cancellation** won, it appends nothing and returns `False` тАФ so
  cancellation and completion stay mutually exclusive, and a cancellation winner cannot both trigger
  compensation and leak a value into the journal.

- **`signalAwakeableFrom`** exposes that transaction-decision core so a race test can interpose
  deterministically between the initial row read and the guarded completion. Ordinary callers should
  use `signalAwakeable`.
- **`cancelAwakeable`** abandons a still-`pending` promise (returns `True` on the transition). A
  workflow that later re-enters the awakeable's `await` then throws `WorkflowAwakeableCancelled`, which
  the author can `catch` for compensation and the resume worker records as a failure.

```haskell
newtype WorkflowAwakeableCancelled = WorkflowAwakeableCancelled AwakeableId
```

Helpers: `awakeableIdToUuid`, `awakeableIdText`.

### Child workflows

In `Keiro.Workflow.Child` тАФ fan-out / fan-in: a parent spawns a child (a second workflow with its own
journal), waits for its result, and may cancel it.

```haskell
data ChildHandle a = ChildHandle { childName :: WorkflowName, childWfId :: WorkflowId }

spawnChild  :: (Workflow :> es, Store :> es) => WorkflowName -> WorkflowId -> Eff (Workflow : es) a -> Eff es (ChildHandle a)
awaitChild  :: (Workflow :> es, Store :> es, FromJSON a) => ChildHandle a -> Eff es a
cancelChild :: (IOE :> es, Store :> es) => ChildHandle a -> Eff es Bool
```

- **`spawnChild name id def`** records a `child:<id>` `StepRecorded` in the **parent's** journal (so a
  replay never re-spawns) and inserts a `running` row in `keiro_workflow_children`. It does **not** run
  the child inline тАФ the resume worker drives the child to completion from the registry, so the
  child's `WorkflowName` **must be registered there**. The `def` argument is taken for authoring
  ergonomics; the registry is the source of truth for the child body.
- **`awaitChild handle`** suspends the parent (an `awaitStep` on `child:<id>:result`) until the child
  completes, then decodes and returns its result. Parent-journal values are tagged envelopes:
  `{"ok": result}` returns, `{"cancelled": true}` throws `WorkflowChildCancelled`, and
  `{"failed": reason}` throws `WorkflowChildFailed`; legacy raw success values still decode.

  On a journal **miss**, `awaitChild` consults the durable child link row: a `ChildFailed` row raises
  `WorkflowChildFailed` directly, on **any** parent generation. That is why the row persists its
  terminal `failureReason` (migration `0020-keiro-workflow-children-failure-reason.sql`) тАФ a parent
  that has rotated past the generation carrying the original failure sentinel can still observe the
  failure without importing a prior generation's journal. Rows written before the column fall back to
  the child instance's `last_error`, then to a stable generic message.
- **`cancelChild handle`** flips the child row to `cancelled`, writes a `WorkflowCancelled` marker to
  the **child** journal (so the child's next resume short-circuits), and writes a `{"cancelled": true}`
  sentinel as the parent's await-step result (so a suspended parent's `awaitChild` throws). Retrying a
  historical row-only cancel repairs both markers even when the return value is `False`.

The resume worker drives any workflow that is some parent's child through `runChildWorkflow` (rather
than bare `runWorkflowWith`) so a finished child wakes its waiting parent:

```haskell
runChildWorkflow    :: (IOE :> es, Store :> es, Error StoreError :> es, ToJSON a) => WorkflowRunOptions -> WorkflowName -> WorkflowId -> Eff (Workflow : es) a -> Eff es (WorkflowOutcome a)
childCompletionHook :: (IOE :> es, Store :> es) => WorkflowName -> WorkflowId -> Value -> Eff es ()

data WorkflowChildCancelled = WorkflowChildCancelled WorkflowName WorkflowId
data WorkflowChildFailed    = WorkflowChildFailed WorkflowName WorkflowId Text
```

`runChildWorkflow` runs `runWorkflowWith` and, on `Completed`, calls `childCompletionHook`, which
flips the child row to `completed` and appends the `child:<id>:result` `StepRecorded` to the parent's
journal in one transaction. An already-completed row re-appends from the stored result to repair an
old wedge; cancelled and failed rows do not fabricate success. Helpers: `childSpawnStepName`,
`childResultStepName`.

## The resume worker

In `Keiro.Workflow.Resume` тАФ a background worker that, each pass, discovers workflows whose journal
lacks a terminal marker and
re-invokes each so it proceeds (past a now-resolved wait, or after a crash).

```haskell
data WorkflowDef es = forall a. (ToJSON a) => WorkflowDef { runDef :: WorkflowId -> Eff (Workflow : es) a }
type WorkflowRegistry es = Map WorkflowName (WorkflowDef es)

resumeWorkflowsOnce         :: (IOE :> es, Store :> es, Error StoreError :> es) => WorkflowResumeOptions -> WorkflowRegistry es -> Eff es ResumeSummary
runWorkflowResumeWorker     :: (IOE :> es, Store :> es, Error StoreError :> es) => WorkflowRegistry es -> Eff es ()
runWorkflowResumeWorkerWith :: (IOE :> es, Store :> es, Error StoreError :> es) => WorkflowResumeOptions -> WorkflowRegistry es -> Eff es ()
```

A workflow's body is application code; only its journal lives in the database. So the application
supplies a **`WorkflowRegistry`** mapping each `WorkflowName` to a `WorkflowDef` that rebuilds the
body from an id alone тАФ the resume-worker analogue of the `fire` action `runTimerWorker` takes.

`resumeWorkflowsOnce` is the single, testable pass; `runWorkflowResumeWorker(With)` loops it on a poll
interval. Discovery unions `findUnfinishedWorkflowIds now` (non-terminal `keiro_workflows` rows whose
`wake_after` has expired) with `findRunningChildIds` (freshly-spawned, zero-step children), so a child
with no steps yet is still driven. **No kiroku `wf:` prefix subscription is used**. Each candidate is
claimed through an expiry-based row lease in `keiro_workflows` before it advances; a live foreign lease
skips only that instance and increments `leaseSkipped`. A discovered workflow whose name is absent from
the registry is surfaced as `unknownName`, never silently dropped.

```haskell
data ResumeSummary = ResumeSummary
  { discovered :: Int, resumed :: Int, completed :: Int, stillSuspended :: Int
  , unknownName :: Int, failed :: Int, transientErrors :: Int, leaseSkipped :: Int
  }

data WorkflowResumeOptions = WorkflowResumeOptions
  { runOptions :: WorkflowRunOptions, pollInterval :: Int, maxAttempts :: Int
  , leaseTtl :: NominalDiffTime, logEvent :: ResumeLogEvent -> IO ()
  }

defaultWorkflowResumeOptions :: WorkflowResumeOptions   -- default run options, 1s poll, 5 attempts, 60s lease
emptyResumeSummary          :: ResumeSummary
```

<TypeTable
  type={{
    runOptions: { type: "WorkflowRunOptions", default: "defaultWorkflowRunOptions", description: "Threaded into runWorkflowWith / runChildWorkflow, so a resumed run honours the same snapshot and telemetry options as its first run. The metrics handle here also backs the per-pass keiro.workflow.awakeables.pending and keiro.workflow.resumed instruments." },
    pollInterval: { type: "Int", default: "1_000_000", description: "Microseconds the loop driver sleeps between passes. Under the push-aware worker its meaning shifts to the fallback timeout: the maximum gap when no NOTIFY arrives." },
    maxAttempts: { type: "Int", default: "5", description: "Synchronous workflow exceptions before the resume worker appends a terminal WorkflowFailed marker." },
    leaseTtl: { type: "NominalDiffTime", default: "60", description: "How long a claimed workflow instance stays leased without reaching another fresh workflow boundary. It bounds dead-worker recovery time and must exceed the longest single step action or await arm." },
    logEvent: { type: "ResumeLogEvent -> IO ()", default: "stderr renderer", description: "Hook for unknown-name, transient-store-error, crash, marked-failed, and pass-failed events." },
  }}
/>

A rotated run (`ContinuedAsNew`) counts toward `resumed`: its new generation has no terminal marker, so
`findUnfinishedWorkflowIds` still reports it and the next pass drives the rotated generation forward.

Each claimed instance is driven with `leaseHeartbeat` set from the claim, so the runtime renews the
lease at every fresh boundary. A `WorkflowLeaseLost` throw is counted as `leaseSkipped` and **does
not** consume a crash attempt.

### Failure, retries, and resurrection

A synchronous exception escaping a workflow body consumes one workflow attempt. The worker records
it, stores `last_error`, sets `next_attempt_at` with exponential backoff тАФ 2, 4, 8, 16, 32, then 64
seconds (capped) тАФ and releases the lease without resetting attempts. Store errors are classified
separately as **transient**: they count toward `transientErrors` and consume no attempt.

With the default `maxAttempts = 5`, the fifth failure is terminal: the runtime appends
`WorkflowFailed`, sets instance status to `failed`, and drops the instance from ordinary resume
discovery. A direct `runWorkflow` then returns `Failed`. There is roughly 30 seconds of scheduled
backoff before that point, plus polling time and each attempt's own runtime тАФ choose `maxAttempts` for
the longest application outage you want the worker to ride through.

For a child, the worker also records the child failure (preserving the reason on the link row) and
wakes the parent with `{"failed": reason}`.

Once the problem is repaired, an operator returns the instance to the runnable pool through the
supported API in `Keiro.Workflow.Instance`:

```haskell
data ResurrectOutcome = WorkflowResurrected | WorkflowNotFailed | WorkflowNotFound

resurrectFailedWorkflow :: (Store :> es) => WorkflowName -> WorkflowId -> Eff es ResurrectOutcome
```

```haskell
resurrectFailedWorkflow (WorkflowName "order-fulfillment") (WorkflowId "order-123") >>= \case
  WorkflowResurrected -> pure ()
  WorkflowNotFailed   -> putStrLn "instance is not terminally failed"
  WorkflowNotFound    -> putStrLn "instance does not exist"
```

In **one transaction** it resets status, attempts, error, retry time, lease, and completion metadata;
deletes the current generation's derived `__workflow_failed__` step-index row; and revives a failed
child link when the instance is a child (an otherwise-excluded failed child row would short-circuit
its own runs).

<Callout type="warn">
**Failure history is immutable; only its derived runnable-state projections are revivable.**
`resurrectFailedWorkflow` appends no audit event and never deletes the historical `WorkflowFailed`
event тАФ deleting journal history would violate the append-only contract and erase the evidence of
repeated failure. Because `WorkflowFailed` events use store-generated UUIDv7 ids (rather than the
deterministic id every other journal entry uses), a later failure on the same generation appends a
distinct event rather than colliding with the first. Concurrent failure writers stay deduplicated by
the workflow-step advisory lock and the in-transaction index check.

A failure sentinel already delivered to a **parent** journal is likewise immutable. Resurrect the
parent separately if its own terminal failure should be retried. Never repair a failed instance with
manual SQL.
</Callout>

A second call after a successful revival is a guarded no-op reported as `WorkflowNotFailed`; an
unknown instance reports `WorkflowNotFound`. After resurrection the next resume **replays** every
completed step rather than repeating its side effects.

Failure audit identity тАФ who resurrected what, and why тАФ remains the caller's responsibility; the
runtime journal records failures, not administrative commands.

### Push-aware variant

`Keiro.Workflow.Resume` also ships a **push-aware** worker that wakes within sub-second of any append
instead of waiting out a fixed poll, by riding kiroku's existing per-store `LISTEN/NOTIFY` notifier.

```haskell
runWorkflowResumeWorkerPush :: KirokuStore -> WorkflowResumeOptions -> WorkflowRegistry '[Store, Error StoreError, IOE] -> IO ()
runPollLoopWith            :: WakeSignal -> Int -> IO () -> IO ()   -- generic: one pass, then wait on the signal
```

`runWorkflowResumeWorkerPush` runs `resumeWorkflowsOnce` each pass and, between passes, waits on the
store's notifier тАФ falling back to `pollInterval` (now read as the *fallback* timeout) so a dropped
notification still drains the backlog. It opens **no new database connection**. This is the push-aware
sibling of `runWorkflowResumeWorker(With)`, which remain the durable fixed-poll baseline. See the
[push-delivery reference](/docs/keiro/reference/push-delivery) and
[Enable push delivery](/docs/keiro/how-to/enable-push-delivery).

## Snapshots

In `Keiro.Workflow.Snapshot`. For long journals, set a `snapshotPolicy` so a resume does not replay
every step.

```haskell
workflowStateCodec        :: StateCodec WorkflowState
workflowStateShapeHash    :: Text   -- the fixed sentinel "keiro.workflow.stepmap.v1"
workflowStateCodecVersion :: Int    -- 1
lookupWorkflowSnapshot    :: (Store :> es) => StreamName -> Eff es (Either SnapshotMissReason (WorkflowState, StreamVersion))
loadWorkflowSnapshot      :: (Store :> es) => StreamName -> Eff es (Maybe (WorkflowState, StreamVersion))
writeWorkflowSnapshot     :: (Store :> es) => StreamId -> StreamVersion -> WorkflowState -> Eff es ()
```

The accumulated step-result map snapshots via `workflowStateCodec`. Its shape hash is a **fixed
sentinel** тАФ distinct from the `regFileShapeHash` used by aggregate snapshots, because workflow step
names are dynamic runtime strings with no static slot list. A snapshot is advisory: a missing,
mismatched, or undecodable snapshot collapses to a full version-0 replay, so a stale snapshot degrades
performance at worst, never correctness. Per-step result-type evolution stays each step's own
`ToJSON`/`FromJSON` concern, surfaced at the `step` decode.

### The step-index fallback under snapshots

A snapshot seeds the replay map at a recorded stream version, after which hydration reads only later
journal events. That is sound for steps the snapshotting run performed itself тАФ but **not** for a
wake source that appended concurrently.

A signalled awakeable, a completing child, or a firing sleep can journal a result while the owning run
is still executing. If that run then writes a snapshot at a *later* stream version, its in-memory map
does not contain the concurrent result even though the result's journal event is at or below the
snapshot version. Seed plus exclusive tail read then under-approximates the journal тАФ permanently.
The old `awaitStep` map-miss path armed the wake source and suspended without appending anything, so
it had no way to converge.

The fix: the generation-scoped `keiro_workflow_steps` index is the **authoritative point lookup** for
whether a step has been journaled тАФ every journal append maintains it in the same transaction, so it
is complete whenever the append is visible. On a map miss, `awaitStep` queries that index *before*
checking cancellation, arming, or suspending. An index hit is inserted into the map, recorded as a
replay, decoded, and returned exactly like an ordinary replay hit.

<Callout type="info">
The fallback is **generation-scoped**: a prior generation's result never resolves an await in a later
generation. It costs one indexed point read, confined to the miss path тАФ normal step execution and
replay hits are untouched. A recorded wake result has settled-history semantics: it is delivered
*before* a pending cancellation check, matching the ordinary in-memory replay path.

Correctness now explicitly depends on every journal append path maintaining `keiro_workflow_steps` in
the same transaction. New append paths must preserve that invariant.
</Callout>

The upshot: snapshots stay advisory, compact replay seeds тАФ no coordination with concurrent wake
writers, no invalidation тАФ and awakeables, child completions, and sleeps cannot be hidden permanently
by a snapshot or a stale same-run map.

`lookupWorkflowSnapshot` retains `SnapshotNoStream`, `SnapshotNotFound`, or
`SnapshotDecodeFailed`; `loadWorkflowSnapshot` is the convenience form that collapses every miss to
`Nothing`. `writeWorkflowSnapshot` is the direct store primitive and can fail. The runtime calls it
only **after** the corresponding step, completion, or continue-as-new seed journal append commits,
through an advisory wrapper: a `StoreError` increments `keiro.snapshot.write.failures` and leaves the
workflow outcome successful. The journal is durable truth; a snapshot is replaceable acceleration.

## The step index and discovery

`Keiro.Workflow` re-exports the `keiro_workflow_steps` index helpers from `Keiro.Workflow.Schema`. The
journal stream is the source of truth for replay; this table is a derived fast-lookup view kept in
sync inside the same transaction as each journal append.

```haskell
data WorkflowStepRow = WorkflowStepRow
  { workflowId :: Text, workflowName :: Text, generation :: Int, stepName :: Text, result :: Value, recordedAt :: UTCTime }

recordStepTx            :: WorkflowStepRow -> Tx.Transaction ()        -- INSERT ... ON CONFLICT DO NOTHING
loadStepIndex           :: (Store :> es) => WorkflowName -> WorkflowId -> Int -> Eff es (Map Text Value)
stepExists              :: (Store :> es) => WorkflowName -> WorkflowId -> Int -> Text -> Eff es Bool
currentGeneration       :: (Store :> es) => WorkflowName -> WorkflowId -> Eff es Int   -- MAX(generation), or 0
setWorkflowWakeAfterTx      :: WorkflowName -> WorkflowId -> UTCTime -> Tx.Transaction ()
clearWorkflowWakeAfterTx    :: WorkflowName -> WorkflowId -> Tx.Transaction ()
findUnfinishedWorkflowIds   :: (Store :> es) => UTCTime -> Eff es [(Text, Text)]    -- non-terminal workflow rows due for resume
```

EP-48 threads the **generation** through every lookup. `currentGeneration` returns `MAX(generation)`
for the logical id (0 when the workflow has no rows). The instance table is now the resume discovery
surface: `findUnfinishedWorkflowIds now` returns rows in non-terminal status whose `wake_after` is
`NULL` or due. `continuedAsNewStepName` is not a terminal workflow status, so a rotated, still-running
workflow remains discoverable on its current generation.

The `wake_after` hint has one writer and one clearer. Only the sleep arm whose
`scheduleTimerOnceTx` insert **won** sets it, to that timer's `fire_at`; a successful fire clears it
atomically with the journal append. Not-yet-due sleepers therefore drop out of resume discovery until
the timestamp expires, while a re-arm cannot push the hint forward and a fired sleeper is
rediscovered immediately.

This index is also the authority behind the
[`awaitStep` step-index fallback](#the-step-index-fallback-under-snapshots), so every journal append
path must keep maintaining it inside the same transaction as the append.

## Journal append helpers

Used by wake-source code (a fired timer, `signalAwakeable`, a finishing child) to record an awaited
step's resolution. The append uses a deterministic event id over
`("keiro":"workflow":name:id:generation:stepName)`, so concurrent or retried writes collapse to one
row тАФ and the same step name on two generations yields *different* ids. These helpers resolve the
current generation internally (a wake source always targets the generation the suspended run is parked
on, the current one), so callers do not pass it.

```haskell
appendJournalEntry            :: (IOE :> es, Store :> es) => WorkflowName -> WorkflowId -> WorkflowJournalEvent -> Eff es ()
appendJournalEntryReturningId :: (IOE :> es, Store :> es) => WorkflowName -> WorkflowId -> WorkflowJournalEvent -> Eff es EventId
deterministicJournalId        :: WorkflowName -> WorkflowId -> Int -> Text -> EventId
```

`prepareJournalAppend` is the generation-explicit primitive underneath them, used where the caller
must pin a *specific* generation rather than the current one тАФ notably a sleep fire, which appends to
the generation that armed the timer. `deterministicJournalId` exposes the id derivation so such a
caller can report the appended id without re-reading it.

<Callout type="info">
`WorkflowFailed` is the **one exception** to deterministic ids: it requests none, so the store assigns
a UUIDv7. That is what lets a resurrected instance fail again on the same generation and append a
distinct historical event instead of colliding with the first. Deduplication of concurrent failure
writers falls to the workflow-step advisory lock and the in-transaction index check.
</Callout>

## Tables

Four tables in the **`keiro` schema**, added by the workflow migrations. The journal streams
(`wf:<name>-<id>`) remain the source of truth for replay; these tables are derived indexes plus
operator-visible state.

| Table | Migration | Purpose |
|---|---|---|
| `keiro_workflow_steps` | `keiro-workflow-steps` (+ `keiro-workflow-generation`) | Step-lookup index backing replay and unfinished-workflow discovery; PK `(workflow_id, workflow_name, generation, step_name)`. |
| `keiro_workflows` | `keiro-workflows-instances` (+ `wake-after`, `gc-index`) | One lifecycle row per logical workflow instance: current generation, status, attempts, last error, next attempt, lease owner/expiry, timestamps, and wake hint. |
| `keiro_awakeables` | `keiro-awakeables` | Pending external completions (`pending`/`completed`/`cancelled`), keyed by `awakeable_id`. |
| `keiro_workflow_children` | `keiro-workflow-children` (+ `children-failure-reason`) | Durable parentтЖТchild links (`running`/`completed`/`cancelled`/`failed`) with the terminal `failure_reason`, PK `(child_id, child_name)`. |

The `keiro-workflow-generation` migration (EP-48) adds the `generation` column (default 0) to
`keiro_workflow_steps` and widens its primary key to include `workflow_name` and `generation`, so two
generations of the same logical workflow never collide on a reserved step name.

`0020-keiro-workflow-children-failure-reason.sql` adds the nullable `failure_reason` column. It
backfills nothing and rewrites no rows: pre-migration failed links fall back to the child instance's
`last_error`, then to a stable generic message. See
[Migrations and schema](/docs/keiro/reference/migrations-and-schema) for the full table map.

## Garbage collection

`Keiro.Workflow.Gc` provides an operator-scheduled cleanup worker for terminal workflow instances.

```haskell
data WorkflowGcPolicy = WorkflowGcPolicy { retention :: NominalDiffTime, batchSize :: Int }
data WorkflowGcSummary = WorkflowGcSummary { scanned :: Int, deleted :: Int }

gcWorkflowsOnce     :: (Store :> es) => UTCTime -> WorkflowGcPolicy -> Eff es WorkflowGcSummary
runWorkflowGcWorker :: (IOE :> es, Store :> es) => WorkflowGcPolicy -> Int -> Eff es ()
```

Eligibility comes from `keiro_workflows`: terminal `completed`/`cancelled`/`failed` rows older than
the retention cutoff are scanned oldest-first. Completed children are not deleted while their parent
is still non-terminal, because the parent may still attach to the stored child result. Deletion removes
every generation's journal stream, workflow snapshots, step-index rows, owned awakeables, child links,
**every owned workflow-sleep timer regardless of status** (scheduled, firing, fired, cancelled, and
dead), and finally the instance row.

<Callout type="warn">
Collecting only *terminal* timer rows left a hole: a still-`scheduled` timer could survive the
deletion of its workflow's journal and instance, fire later, and recreate the workflow from incomplete
history. GC now removes them all, and тАФ as defense in depth for the partial-GC recovery window тАФ a
claimed sleep whose instance row still exists in `completed`, `cancelled`, or `failed` status cancels
itself and appends nothing. A *missing* instance row is not treated as terminal, because that is the
legitimate crash-before-suspend case.
</Callout>
