The types and functions on this page live in `Keiro.Projection`. A projection folds events into an
application-owned read-model table; see
[Projections, read models, and snapshots](/docs/keiro/explanation/projections-read-models-and-snapshots)
for the conceptual distinction.

## `InlineProjection`

`runCommandWithProjections` delegates to `runCommandWithSqlEvents`; the append and every inline
`apply` share one transaction. A projection failure aborts the append, so events and read-model rows
commit together or not at all. Like the other SQL command runners, this function requires
`KirokuStoreResource` so Kiroku enrichment runs before the callback sees recorded events.

```haskell
runCommandWithProjections ::
  ( HasCallStack, IOE :> es, Store :> es, Error StoreError :> es
  , KirokuStoreResource :> es, BoolAlg phi (RegFile rs, ci), Eq co ) =>
  RunCommandOptions ->
  ValidatedEventStream phi rs s ci co ->
  Stream (EventStream phi rs s ci co) ->
  ci ->
  [InlineProjection co] ->
  Eff es (Either CommandError (CommandResult (EventStream phi rs s ci co)))
```

Keiro supplies the per-event transaction, deduplication, and registry fence. It does not supply the
subscription polling loop or checkpoint acknowledgement policy.

## Live application outcomes

```haskell
data AsyncApplyOutcome
  = AsyncApplied
  | AsyncDuplicate
  | AsyncFenced

applyAsyncProjection ::
  AsyncProjection -> RecordedEvent -> Tx.Transaction AsyncApplyOutcome
```

`applyAsyncProjection` reads `readModelName` from `keiro.keiro_read_models` with `FOR SHARE` in the
same transaction as the dedup insert and application write.

| Outcome | Database effect | Worker disposition |
| --- | --- | --- |
| `AsyncApplied` | Dedup key and read-model update commit together. | Acknowledge/checkpoint when the enclosing delivery transaction succeeds. |
| `AsyncDuplicate` | Existing retained dedup key suppresses `applyRecorded`. | Treat as idempotent success and acknowledge/checkpoint. |
| `AsyncFenced` | Registry row is missing or not `Live`; no dedup or model write occurs. | Do **not** acknowledge or checkpoint past the event. Fail/park it and retry after promotion. |

That last rule is essential: checkpointing `AsyncFenced` would permanently skip an event while a
rebuild owns the model.

## Rebuild-only application

```haskell
applyAsyncProjectionUnfenced ::
  AsyncProjection -> RecordedEvent -> Tx.Transaction AsyncApplyOutcome
```

This bypasses the registry status check so the designated rebuilder can write while the model is
`Rebuilding`. It still inserts the projection/event dedup key, so its outcomes are `AsyncApplied` or
`AsyncDuplicate`. Live workers must use `applyAsyncProjection`; never use the unfenced function to
avoid handling `AsyncFenced`.

## Dedup retention

```haskell
pruneAsyncProjectionDedupBefore :: UTCTime -> Eff es Int64
```

The dedup key and `applyRecorded` run in the same transaction. Prune only before the oldest event the
subscription system can redeliver; pruning intentionally reopens older events for application. It
is not a rebuild reset. `startRebuild` clears only the named projections' keys atomically with the
writer fence, table truncation, and checkpoint reset.

## Projection lag

```haskell
recordProjectionLag ::
  Maybe KeiroMetrics -> AsyncProjection -> Eff es ()
```

Call this from the application-owned drain loop, normally after applying a batch. It records
`keiro.projection.lag` as the non-negative `$all` head minus the projection subscription checkpoint;
`Nothing` makes it a no-op.

See [Make an async projection idempotent](/docs/keiro/how-to/make-an-async-projection-idempotent)
for a complete outcome handler and
[Rebuild a read model](/docs/keiro/how-to/rebuild-a-read-model) for fenced replay.
