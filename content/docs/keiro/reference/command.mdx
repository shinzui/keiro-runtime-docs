The types and functions on this page live in `Keiro.Command` (re-exported from `Keiro`). They run a
command through the Hydrate → Transduce → Append cycle; for the narrative, see
[The command cycle](/docs/keiro/explanation/the-command-cycle).

<Callout type="warn">
The command API now requires `ValidatedEventStream`, not a bare `EventStream`. Build the stream with
`mkEventStream`/`mkEventStreamWith`, or use `mkEventStreamOrThrow` only for generated code and
fixtures with a sibling validation proof. This is the compile-time guard that keeps unchecked
streams out of hydration and replay.
</Callout>

## `CommandResult`

The outcome of a successfully handled command. `eventsAppended` is `0` for a no-op command (one the
transducer accepted but that emitted no events).

<TypeTable
  type={{
    target: { type: "Stream target", description: "The stream the command ran against" },
    streamVersion: { type: "StreamVersion", description: "The stream's version after the append" },
    globalPosition: { type: "Maybe GlobalPosition", description: "$all position of the last appended event; Nothing on a no-op" },
    eventsAppended: { type: "Int", description: "How many events were appended; 0 for a no-op" },
  }}
/>

## `CommandError`

Why a command did not complete.

```haskell
data CommandError
  = HydrationDecodeFailed !CodecError       -- a stored event could not be decoded while rehydrating
  | HydrationReplayFailed !StreamVersion    -- replay stalled: the machine rejected a committed event
  | CommandRejected                         -- the transducer rejected the command in the hydrated state
  | EncodeFailed !CodecError                -- an emitted event could not be encoded for append
  | StoreFailed !StoreError                 -- the store rejected the append (non-retryable)
  | RetryExhausted !Int !StoreError         -- optimistic-concurrency retries ran out (limit, last error)
  | ConflictFixpoint !StreamVersion !StoreError -- retry observed no stream-version progress
  deriving stock (Generic, Eq, Show)
```

<Callout type="warn">
Tell three outcomes apart: **rejected** (`Left CommandRejected` — no edge for this command in the
hydrated state, nothing written), **no-op** (`Right CommandResult { eventsAppended = 0 }` — accepted
but emitted nothing, nothing written, *not* a rejection), and **store failure**
(`Left (StoreFailed …)` / `Left (RetryExhausted …)`). They flow through the same
`Either CommandError CommandResult` and are easy to conflate.
</Callout>

## `RunCommandOptions` and `defaultRunCommandOptions`

Knobs for a single command invocation.

<TypeTable
  type={{
    retryLimit: { type: "Int", description: "Rehydrate-and-replay attempts after a conflict", default: "3" },
    pageSize: { type: "Int32", description: "Read-batch size during hydration", default: "256" },
    eventIds: { type: "[EventId]", description: "Caller ids assigned to emitted events, in order", default: "[]" },
    beforeAppend: { type: "IO ()", description: "Hook run before each append attempt (test seam)", default: "pure ()" },
    retryBackoffMicros: { type: "Int", description: "Base OCC retry backoff; capped at 100ms and jittered; 0 disables", default: "5000" },
    metrics: { type: "Maybe KeiroMetrics", description: "Optional command/snapshot metrics handle", default: "Nothing" },
    tracer: { type: "Maybe Tracer", description: "Optional OpenTelemetry tracer", default: "Nothing" },
    metadata: { type: "Maybe Value", description: "JSON merged into every event's metadata", default: "Nothing" },
  }}
/>

```haskell
-- Defaults: 3 retries, 256-event read pages, no caller ids, no-op hook,
-- 5ms base retry backoff, no metrics, no tracer, no metadata.
defaultRunCommandOptions :: RunCommandOptions
```

Retry behavior:

- Optimistic-concurrency conflicts are retried by rehydrating, re-planning, and appending again.
  Each retry records conflict/retry metrics when a metrics handle is installed.
- `StreamAlreadyExists` can be retryable, but if the next hydration observes the same stream version
  as the previous attempt, the command stops with `ConflictFixpoint`. This catches soft-deleted or
  otherwise unreadable streams that keep colliding without making visible progress.
- `DuplicateEvent` is not retried; it is surfaced as `StoreFailed` and counted with
  `keiro.command.duplicates` when metrics are enabled.
- Snapshot writes happen after a successful append. If the snapshot write fails, the command result
  still succeeds and `keiro.snapshot.write.failures` is incremented.

## `runCommand` — append only

```haskell
runCommand ::
  forall phi rs s ci co es.
  (HasCallStack, IOE :> es, Store :> es, Error StoreError :> es, BoolAlg phi (RegFile rs, ci), Eq co) =>
  RunCommandOptions ->
  ValidatedEventStream phi rs s ci co ->
  Stream (EventStream phi rs s ci co) ->
  ci ->
  Eff es (Either CommandError (CommandResult (EventStream phi rs s ci co)))
```

## `runCommandWithSql` — append plus an in-transaction callback

Runs `afterAppend` in the **same** Hasql transaction as the append (e.g. to update an inline read
model atomically). The callback sees the `AppendResult`; its return is surfaced as `Maybe a`
(`Nothing` on a no-op, when no append happened).

```haskell
runCommandWithSql ::
  forall phi rs s ci co a es.
  (HasCallStack, IOE :> es, Store :> es, Error StoreError :> es, BoolAlg phi (RegFile rs, ci), Eq co) =>
  RunCommandOptions ->
  ValidatedEventStream phi rs s ci co ->
  Stream (EventStream phi rs s ci co) ->
  ci ->
  (AppendResult -> Tx.Transaction a) ->
  Eff es (Either CommandError (CommandResult (EventStream phi rs s ci co), Maybe a))
```

## `runCommandWithSqlEvents` — the general primitive

The most general runner: the in-transaction callback also receives each emitted event paired with its
persisted `RecordedEvent`. This is the primitive that
[projections](/docs/keiro/reference/projection) (`Keiro.Projection`), process managers, and the
[router](/docs/keiro/reference/router) build on.

```haskell
runCommandWithSqlEvents ::
  forall phi rs s ci co a es.
  (HasCallStack, IOE :> es, Store :> es, Error StoreError :> es, BoolAlg phi (RegFile rs, ci), Eq co) =>
  RunCommandOptions ->
  ValidatedEventStream phi rs s ci co ->
  Stream (EventStream phi rs s ci co) ->
  ci ->
  ([(co, RecordedEvent)] -> AppendResult -> Tx.Transaction a) ->
  Eff es (Either CommandError (CommandResult (EventStream phi rs s ci co), Maybe a))
```

See [Run a command in a transaction](/docs/keiro/how-to/run-a-command-in-a-transaction) for the
transactional pattern and [Configure concurrency retries](/docs/keiro/how-to/configure-concurrency-retries)
for tuning `retryLimit`/`pageSize`.
