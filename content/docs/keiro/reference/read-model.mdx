The types and functions on this page live in `Keiro.ReadModel`, `Keiro.ReadModel.Schema`,
`Keiro.ReadModel.Rebuild`, and `Keiro.Connection`. A read model is application-owned derived data;
Keiro keeps only its registry metadata in the framework-owned `keiro` schema.

## `ReadModel`

`schema` belongs to the application data table, not to Keiro's framework metadata. Qualify every
application table reference:

```haskell
qualifiedTableName :: ReadModel q r -> Text
qualifyTable       :: Text -> Text -> Text
```

Both produce a double-quoted reference such as `"app_reads"."order_summary"`. Build the
projection's create/read/write statements from that qualified name. Do not rely on an unqualified
`CREATE TABLE`: on a default Kiroku connection it can land in `kiroku`, which is not the model's
declared application schema.

`withProjectionSchema` adds an application schema to Kiroku's `extraSearchPath` without changing
Kiroku's own `schema = "kiroku"`. `keiroConnectionSettings connectionString projectionSchema` is the
corresponding convenience constructor. `ensureProjectionSchema` is an opt-in development/test
helper; Keiro never creates an application schema automatically.

## Registration is explicit

Register each model once during projection/application startup, after its application table exists
and before accepting projection work or queries:

```haskell
registerReadModel
  (readModel ^. #name)
  (readModel ^. #version)
  (readModel ^. #shapeHash)
```

`registerReadModel` is idempotent, but an existing row is returned unchanged: it does not overwrite
stored `version`, `shapeHash`, or status. `runQuery` and `runQueryWith` never create the row. A missing
row returns `ReadModelUnregistered name`, which exposes omitted startup wiring instead of turning a
read into a registry write.

## Consistency and `StrongScope`

```haskell
data ConsistencyMode
  = Strong
  | Eventual
  | PositionWait PositionWaitOptions

data StrongScope
  = EntireLog
  | CategoryHead Text
```

- `Eventual` queries immediately.
- `PositionWait options` waits for `options.target`; `Nothing` skips waiting.
- `Strong` captures a target at query start, selected by the model's `strongScope`, and waits for the
  model's subscription cursor to reach it.

`EntireLog` uses the `$all` head and is correct only when the subscription observes the whole log.
`CategoryHead category` uses the latest global position originating in that Kiroku category, so
unrelated category traffic cannot hold a category-fed model behind forever. A model fed by multiple
categories should use an explicit `PositionWait`, or `EntireLog` with an all-stream subscription.

```haskell
data PositionWaitOptions = PositionWaitOptions
  { target        :: Maybe GlobalPosition
  , timeoutMicros :: Int
  , pollMicros    :: Int
  }

defaultStrongWaitOptions = PositionWaitOptions
  { target = Nothing, timeoutMicros = 5_000_000, pollMicros = 10_000 }
```

The wait reads the minimum `subscriptions.last_seen` for `subscriptionName`. Kiroku currently does
not advance category checkpoints on empty fetches; that is why `CategoryHead` avoids choosing an
unreachable target created only by another category.

## Query functions and errors

```haskell
runQuery     :: Maybe KeiroMetrics -> ReadModel q r -> q
             -> Eff es (Either ReadModelError r)
runQueryWith :: Maybe KeiroMetrics -> ConsistencyMode -> ReadModel q r -> q
             -> Eff es (Either ReadModelError r)
waitFor      :: Maybe KeiroMetrics -> PositionWaitOptions -> ReadModel q r
             -> GlobalPosition -> Eff es (Either ReadModelError ())

data ReadModelError
  = ReadModelUnregistered Text
  | ReadModelStaleSchema Text Int Int Text Text
  | ReadModelWaitTimeout Text GlobalPosition GlobalPosition
  | ReadModelNotLive Text ReadModelStatus
```

`runQueryWith` checks that the registry row already exists, validates its version, hash, and `Live`
status, performs the selected wait, then runs `query`. It has no registration side effect. A wait
deadline records `keiro.projection.wait.timeouts` when a metrics handle is supplied.

The related position helpers are:

```haskell
readSubscriptionPosition :: Text -> Eff es (Maybe GlobalPosition)
storeHeadPosition        :: Eff es GlobalPosition
categoryHeadPosition     :: Text -> Eff es GlobalPosition
```

## Registry lifecycle

```haskell
data ReadModelStatus = Live | Rebuilding | Paused | Abandoned | UnknownStatus Text

registerReadModel :: Text -> Int -> Text -> Eff es ReadModelMetadata
lookupReadModel   :: Text -> Eff es (Maybe ReadModelMetadata)
markRebuilding    :: Text -> Int -> Text -> Eff es ReadModelMetadata
markLive          :: Text -> Int -> Text -> Eff es ReadModelMetadata
markAbandoned     :: Text -> Int -> Text -> Eff es ReadModelMetadata
```

Unknown stored status text becomes `UnknownStatus raw`, retaining the database value for
diagnostics. Registry rows live in `keiro.keiro_read_models`; the application `schema` is deliberately
not part of that persisted identity.

## Supported rebuild lifecycle

```haskell
data RebuildError
  = RebuildProducedNoApplies Text GlobalPosition

startRebuild  :: ReadModel q r -> [Text] -> GlobalPosition
              -> Eff es ReadModelMetadata
finishRebuild :: ReadModel q r -> [Text] -> GlobalPosition
              -> Eff es (Either RebuildError ReadModelMetadata)
abandonRebuild :: ReadModel q r -> Eff es ReadModelMetadata
```

`startRebuild model projectionNames replayFrom` performs one transaction: transition the registry
row to `Rebuilding`, truncate the qualified application table, delete dedup keys for only the named
projections, and reset every matching subscription member to `replayFrom`. The registry transition
happens first and locks the row used to fence live async writers.

Replay through `applyAsyncProjectionUnfenced`, which is reserved for the designated rebuilder and
retains ordinary dedup semantics. Then call `finishRebuild` with the same projection names and replay
position. It checks the store head and newly created dedup rows in the promotion transaction. If the
log extends beyond `replayFrom` but a non-empty projection list produced no applications, it returns
`RebuildProducedNoApplies` and leaves the model `Rebuilding`. An empty projection-name list denotes
an inline-only model and skips that dedup guard.

<Callout type="warn">
  `rebuild` and `promote` still exist as low-level status-only operations. `promote` bypasses the
  zero-apply guard. Use `startRebuild` and `finishRebuild` for the supported destructive in-place
  workflow. There is no shadow-table or online cutover; queries return `ReadModelNotLive` during the
  rebuild.
</Callout>

See [Rebuild a read model](/docs/keiro/how-to/rebuild-a-read-model) for the operational sequence and
[Projection](/docs/keiro/reference/projection) for live-writer outcomes.
