A `.keiro` file is a typed service specification. It starts with one context, may choose module
placement, declares shared types/rules, and then declares top-level nodes. The current grammar has
twelve top-level node constructors; nested projections, snapshots, timers, and patches are
documented with their owners below.

## File structure

```keiro-dsl
context hospital-capacity
module Acme.Services
layout collocated

id TransferReservationId prefix=rsv
enum DivertStatus { Open=open TotalDivert=total-divert }
rule lifeCriticalOverride : DivertStatus -> Bool
  ex Open => true ; TotalDivert => false

# top-level nodes follow
```

`#` begins a line comment. Whitespace and newlines separate tokens but indentation is not structural;
keywords and braces define the grammar. Quoted strings support `\"`, `\\`, `\n`, `\t`, and `\r`;
raw line breaks inside a quoted string are rejected.

Identifiers use ASCII letters, digits, and underscores. Node/type/constructor names are
`PascalCase` where the grammar expects generated constructors. Fields and registers begin with a
lowercase letter or `_`. Haskell keywords, duplicate names, unsafe constructor forms, and generated
vertex-constructor collisions are rejected.

## Module placement

Both placement clauses are optional:

| Clause | Values/default | Effect |
|---|---|---|
| `module <Dotted.Prefix>` | no prefix | Prepends a PascalCase namespace to generated and hole modules |
| `layout <style>` | `prefixed` | `prefixed` uses `Generated.<Ctx>…`; `collocated` uses `<Ctx>.<Node>.Generated…` |

CLI `--module-root` and `--collocate` flags override the corresponding spec clauses. Precedence is
CLI flag, then spec clause, then built-in default.

## Shared declarations

- `id Name prefix=text` generates a `Text` newtype and records its durable prefix.
- `enum Name { Constructor=wire-spelling … }` defines a closed enum and explicit wire spellings.
- `rule name : Enum -> Bool` followed by `ex Constructor => expression` rows defines a total rule for
  use in aggregate expressions.

Rules must resolve their enum domain, cover every constructor exactly, and use known constructors.
Changing an ID prefix or enum wire spelling is breaking evolution.

## Complete node inventory

| Grammar family | Nested clauses | Owning reference |
|---|---|---|
| `aggregate` | registers, states, commands, events, transitions, `wire`, `projection`, `snapshot` | [Domain and read-side nodes](/docs/keiro/reference/keiro-dsl-domain-nodes) |
| `readmodel` | table/schema/columns, captured shape, consistency/scope/feed/subscription | [Domain and read-side nodes](/docs/keiro/reference/keiro-dsl-domain-nodes) |
| `process` | handle/dispatch policy and one nested `timer` | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `router` | stable resolver, target bindings, dispatch identity and worker policy | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `contract` | topics and integration event schemas | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `intake` | bindings, persistence, decode and seven-outcome disposition | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `emit` | contract/topic map and deterministic ID holes | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `publisher` | ordering, attempt ceiling, backoff and stable outbox ID | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `workqueue` | captured queue names, ordering/group, provisioning, payload/retry/disposition | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `dispatch` | read-model source/fan-out, read-model/queue dedupe and enqueue target | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `workflow` | ordered steps/awaits/sleeps/children, `patch`, terminal `continueAsNew` | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |
| `operation` | command, query, signal, or workflow-run entry point | [Runtime and integration nodes](/docs/keiro/reference/keiro-dsl-runtime-nodes) |

`projection`, `snapshot`, and `timer` are not independent top-level nodes. A patch and
`continueAsNew` are workflow body items. An operation is top-level but owns no persisted decode or
identity surface of its own; it validates references to other nodes.

## Ownership and generated output

`HOLE` means behavior remains hand-owned. Scaffold creates hole modules only when absent and
overwrites only files marked `-- @generated`. Typical output is:

| Node | Generated | Hand-owned |
|---|---|---|
| aggregate | `Domain`, `Codec`, `EventStream`, `Projection`, `Harness` | `Holes` |
| read model | `ReadModelTable`, `ReadModel`, `ReadModelHarness` | `ReadModelHoles` |
| process | `Process`, `ProcessHarness` | `ProcessHoles` |
| router | `Router`, `RouterHarness` | `RouterHoles` and runtime value/fill |
| contract/intake/publisher | `Contract`, `Inbox`, `Publisher` | integration/transaction fills |
| workqueue | `Queue`, `QueuePolicy` | fan-out, handler, and raw-SQL dedupe fills |
| workflow | `WorkflowFacts`, `WorkflowRuntime` | workflow body module |

Operations primarily validate and connect references; they do not produce a separate generated
module. Always use the emitted manifest as the exact module/dependency inventory for a concrete spec.

## CLI and lifecycle

The command surface, exit codes, scaffold protections, manifests, stale records, and diff output are
in the [keiro-dsl CLI reference](/docs/keiro/reference/keiro-dsl-cli). The practical sequence is
`new/write → parse/check → scaffold → fill → harness → diff`.
