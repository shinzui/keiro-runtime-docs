`EventStream` (and its `SnapshotPolicy`/`StateCodec`) live in `Keiro.EventStream`; replay-safety
validation lives in `Keiro.EventStream.Validate`; the `Stream` handle lives in `Keiro.Stream`. All
three surfaces are re-exported from `Keiro`.

## `EventStream`

The complete description of one persistent event stream: a keiki transducer plus everything keiro
needs to run it against a durable store. The type parameters thread through from the transducer —
`phi` = guard alphabet, `rs` = register set, `s` = control state, `ci` = command input, `co` = event
output.

<TypeTable
  type={{

    eventCodec: { type: "Codec co", description: "Serializes/deserializes/migrates this stream's events" },

    snapshotPolicy: { type: "SnapshotPolicy (s, RegFile rs)", description: "When to write a snapshot of the folded state" },
    stateCodec: { type: "Maybe (StateCodec (s, RegFile rs))", description: "Nothing disables snapshotting regardless of policy" },
  }}
/>

<Callout type="info">
`stateCodec = Nothing` disables snapshotting at runtime. `mkEventStream` rejects a stream when
`snapshotPolicy` is not `Never` but `stateCodec` is `Nothing`, because that configuration asks for
snapshots that can never be written. The `jitsurei` order stream shows both ends: `orderEventStream`
uses `snapshotPolicy = Never`, `stateCodec = Nothing`, while `snapshotOrderEventStream` uses
`snapshotPolicy = Every 2`, `stateCodec = Just (defaultStateCodec @OrderRegs @OrderState 1)`.
</Callout>

## `ValidatedEventStream` and replay-safety validation

An `EventStream` is only as safe to replay as the keiki transducer inside it.
`Keiro.EventStream.Validate` lifts keiki's pure umbrella check (`validateTransducer`) to the
`EventStream` boundary, so a service can reject an unsafe durable stream **before** it is ever
hydrated — with no SMT solver. The check covers hidden input, first-event recoverability, ambiguous
event inversion, unguarded input reads, state-changing output-free edges, structural determinism,
and dead edges. Keiro also validates snapshot configuration and forces the initial snapshot encode.

This is now a breaking API boundary: `runCommand`, `runCommandWithSql`,
`runCommandWithSqlEvents`, `runCommandWithProjections`, routers, and process managers accept a
`ValidatedEventStream`, not a bare `EventStream`. The raw record remains available for constructing
and validating definitions; the constructor for `ValidatedEventStream` is intentionally hidden.

```haskell
data EventStreamWarning = EventStreamWarning
  { eswStreamLabel :: !Text   -- the caller-supplied stream label
  , eswReason      :: !Text   -- rendered from the keiki warning
  }
  deriving stock (Eq, Show)

validateEventStream ::
  (Bounded s, Enum s, Ord s, Show s) =>
  Text -> EventStream (HsPred rs ci) rs s ci co -> [EventStreamWarning]

validateEventStreamWith ::
  (Bounded s, Enum s, Ord s, Show s) =>
  ValidationOptions -> Text -> EventStream (HsPred rs ci) rs s ci co -> [EventStreamWarning]

mkEventStream ::
  (Bounded s, Enum s, Ord s, Show s) =>
  Text -> EventStream (HsPred rs ci) rs s ci co ->
  Either [EventStreamWarning] (ValidatedEventStream (HsPred rs ci) rs s ci co)

mkEventStreamWith ::
  (Bounded s, Enum s, Ord s, Show s) =>
  ValidationOptions -> Text -> EventStream (HsPred rs ci) rs s ci co ->
  Either [EventStreamWarning] (ValidatedEventStream (HsPred rs ci) rs s ci co)

mkEventStreamOrThrow ::
  (HasCallStack, Bounded s, Enum s, Ord s, Show s) =>
  Text -> EventStream (HsPred rs ci) rs s ci co ->
  ValidatedEventStream (HsPred rs ci) rs s ci co

unvalidated :: ValidatedEventStream phi rs s ci co -> EventStream phi rs s ci co

mkEventStreamUnchecked ::
  EventStream phi rs s ci co -> ValidatedEventStream phi rs s ci co
```

- `validateEventStream label es` runs the check with `defaultValidationOptions` and returns the
  labelled warnings; an **empty list** means the stream passed every enabled check.
- `validateEventStreamWith opts label es` is the same with caller-chosen `ValidationOptions` (e.g.
  to narrow the checks for a stream with a known-benign warning).
- `mkEventStream label es` is the fail-fast smart constructor: `Left warnings` for an unsafe stream,
  `Right validated` for a sound one.
- `mkEventStreamOrThrow label es` is the partial constructor used by generated modules and fixtures
  that have a sibling validation proof. Hand-authored service boot code should usually prefer
  `mkEventStream` and handle `Left` explicitly.
- `mkEventStreamUnchecked es` skips **every** keiki and keiro check. It exists for tests and
  emergency forensics only; it is not a production escape hatch.

The bare `EventStream { … }` record literal stays available for construction, validation, and
low-level internals. It is not enough to call the command runners.

### Lifecycle of a validated stream

1. **Author the raw definition.** Write `fooEventStreamDef :: EventStream …` with the transducer,
   initial state/registers, event codec, stream-name resolver, snapshot policy, and optional
   `StateCodec`.
2. **Validate at the boundary.** Call `mkEventStream "foo" fooEventStreamDef` at application boot, in
   a registry module, or from generated code. Hand-authored services should handle `Left warnings`
   and refuse to start; generated code and fixtures may use `mkEventStreamOrThrow` when the adjacent
   harness proves validation stays green.
3. **Export the wrapper.** Export `fooEventStream :: ValidatedEventStream …` and pass that value to
   `runCommand`, `runCommandWithSql`, `runCommandWithProjections`, `ProcessManager`, or `Router`.
4. **Let internals unwrap.** Runtime modules call `unvalidated` internally after the type boundary has
   done its job. The raw record is then used for hydration, `Keiki.step`, event encoding,
   stream-name resolution, and snapshot handling.

That shape is why migration is compiler-driven: any old call that passes `fooEventStreamDef` to a
runner fails to typecheck until the stream has a validated wrapper.

### What validation proves

With `defaultValidationOptions`, `mkEventStream` accepts only streams whose transducer passes these
enabled checks:

- **Hidden input.** Every command field read while producing an event chain must be recoverable from
  persisted output.
- **Head recoverability.** The first event in a chain must recover every command field needed to
  choose and replay its edge. Later events cannot repair an ambiguous first event.
- **Inversion ambiguity.** Outgoing edges whose first events could identify more than one original
  transition are rejected conservatively.
- **Guarded input reads.** A command-field read must be protected by an earlier matching-constructor
  guard.
- **State-changing output-free edges.** An edge that emits nothing may not change durable state or
  write registers, because there would be no event from which to recover that change.
- **Structural determinism and reachability.** Proven command-side overlaps and statically dead
  edges are warnings, and therefore construction failures at this durable boundary.
- **Snapshot coherence and initial encodability.** A non-`Never` policy requires a `StateCodec`.
  When a codec is present, validation forces it over `(initialState, initialRegisters)` so an
  `uninit: <slot>` thunk installed by `emptyRegFile` fails startup instead of the first snapshot.
- **Event-codec construction invariants.** Validation runs `mkCodec` over `eventCodec`, so an invalid
  schema version, duplicate event tags, duplicate or out-of-range upcaster sources, and an incomplete
  chain to the current version all fail *validated construction* — before the service can touch a
  stored stream. The warning renders as `event codec misconfigured: <reason>`.

`mkEventStreamWith` still accepts custom `ValidationOptions`, but keiro force-enables
`checkHeadRecoverability` and `checkStateChangingEpsilon` after applying the caller's options.
Persisted events are the only durable state, so those two checks cannot be disabled here. Every
remaining enabled warning is also a `Left`; only narrow another check for a documented benign case.

### What validation does not prove

Validation is intentionally cheap and local. It does **not** prove that the codec round-trips every
domain value, that a stream name is the one you meant (`StreamCategory` helps there), or that
external effects in handlers are idempotent. It also does not make stale snapshots authoritative:
snapshots remain advisory and miss safely back to full replay.

<Callout type="warn">
Most importantly, validation checks the **new machine in isolation** — never the new machine against
your real old log. It cannot tell you that tightening a guard just orphaned ten thousand stored
events, because it never reads the store. That question belongs to the
[replay audit](/docs/keiro/reference/replay-audit); the whole layering is laid out in
[Evolution and replayability](/docs/keiro/explanation/evolution-and-replayability).
</Callout>

### Replay-only edges pass the boundary

A keiki `Edge` carries an `EdgeMode` (`Live` or `ReplayOnly`). A `ReplayOnly` edge is never taken by
forward stepping; it exists so events emitted under a retired rule keep an inverting edge. Two
validator scopings make the pattern usable at this boundary:

- forward-determinism checks consider only `Live`/`Live` pairs; and
- inversion-ambiguity flags only **same-mode** pairs.

So a live edge and its replay-only twin — which emit the same event by construction — are not
refused by `mkEventStream`, even though the ambiguity check is deliberately guard-blind. Every other
check (hidden input, head recoverability, guarded input reads, state-changing epsilon, both dead-edge
analyses) applies to replay-only edges unchanged, and structural reachability still traverses them,
so a vertex reachable only through a replay-only edge stays live for replay continuation.

Build one with `Keiki.Builder.replayOnly` (or `mode = ReplayOnly` on a raw `Edge`); `keiro-dsl`
lowers the `replay-only` transition prefix to it. See
[Tighten a guard with a replay-only twin](/docs/keiro/how-to/tighten-a-guard-with-a-replay-only-twin).

## How keiro makes streams replayable and safe

Replayability is not one check; keiro layers several independent guards:

- **Pure transition semantics.** `EventStream.transducer` is a keiki `SymTransducer`. Hydration folds
  stored events through `Keiki.applyEventStreaming`, and command handling calls `Keiki.step` with the
  hydrated `(state, registers)`. There is no effectful decide/evolve callback hidden in the stream.
- **Compile-time runner boundary.** Public command and projection runners require
  `ValidatedEventStream`, so a bare record cannot accidentally reach the write path.
- **Build-time stream validation.** `validateEventStream` runs keiki's full default replay contract,
  plus snapshot coherence and initial-encode checks. Every warning rejects construction.
- **Snapshot coherence.** `mkEventStream` rejects snapshotting policies without a `stateCodec`, and
  snapshot reuse is gated by the three-component discriminator (`stateCodecVersion`, `shapeHash`,
  `stateShapeHash`). A stale or undecodable snapshot is a cache miss, so hydration falls back to full
  event replay.
- **Codec invariants.** `mkCodec` validates schema versions, duplicate event types, duplicate
  upcaster sources, out-of-range upcasters, and missing migration rungs — and validated stream
  construction runs it for you, so a misconfigured codec cannot reach hydration. On read, unknown
  event types and malformed schema-version stamps are fatal instead of being skipped.
- **Typed stream names.** `Stream a` and `StreamCategory a` keep target streams and category reads
  aligned. `category` rejects ambiguous category prefixes before they can fan events into the wrong
  subscription or read model.
- **Idempotent append inputs.** `RunCommandOptions.eventIds` lets routers, process managers, and
  replayed handlers supply deterministic event ids, so repeated fan-out attempts coalesce at the
  store boundary.

The `label` is echoed into every `EventStreamWarning` so a multi-aggregate service can tell *which*
stream is at fault; `eswReason` renders the underlying keiki warning (`hidden-input @…`,
`head-unrecoverable @…`, `inversion-ambiguity @…`, `unguarded-input-read @…`,
`state-changing-epsilon @…`, `nondeterministic @…`, `possibly-dead @…`, or `opaque-guard @…`, each
tagged with its source). The check requires the control state `s` to satisfy
`(Bounded s, Enum s, Ord s, Show s)` —
it walks `[minBound .. maxBound]` exactly as the renderers and `defaultStateCodec` shape hash do —
and the transducer must use keiki's standard `HsPred` guard carrier.

The enabled checks come from keiki's `ValidationOptions` (carried through `validateTransducer`):

## `SnapshotPolicy`

When keiro writes a snapshot of the folded `(state, registers)`.

```haskell
data SnapshotPolicy state
  = Never                                  -- never snapshot; always replay the full log
  | Every !Int                             -- snapshot when version is a positive multiple of n
  | OnTerminal                             -- snapshot only at a final state
  | Custom !(Terminality -> state -> StreamVersion -> Bool)
  deriving stock (Generic)
```

Command handling evaluates policies over the half-open stream-version span `(preVersion,
postVersion]` for an append. `Every n` fires when any positive multiple of `n` lies inside that span,
so a multi-event append that jumps from version `2` to `4` still writes a snapshot for `Every 3`.
`OnTerminal` and `Custom` use the post-append state and terminality.

## `StateCodec`

How a snapshot is encoded. It is **defined** here, but the runtime snapshot read/write mechanics —
`lookupSnapshotSeed`, `encodeSnapshotStrict`, `writeSnapshotEncoded`, `defaultStateCodec`, policy evaluation, and the
`keiro_snapshots` table — are documented on the read side's [snapshot
reference](/docs/keiro/reference/snapshot).

## `Stream` — the phantom-typed handle

A `StreamName` tagged with a phantom type `a` identifying which aggregate it names, so a name for one
aggregate cannot be passed where another is expected.

```haskell
newtype Stream a = Stream { name :: StreamName }
  deriving stock (Generic, Eq, Ord, Show)

stream        :: Text -> Stream a                                   -- build a handle from a raw stream-name Text
streamName    :: Stream a -> StreamName                              -- recover the underlying StreamName
mapStreamName :: (StreamName -> StreamName) -> Stream a -> Stream a  -- transform the name, keep the tag
```

<Callout type="info">
The design notes' old name for this type was `AggregateId a`; the shipped type is `Stream a`. Trust
the source.
</Callout>

## Safe, category-based construction (`StreamCategory`)

`stream` takes a raw `Text`, so the physical name `"order-" <> orderIdText orderId` is hand-assembled
at every call site. That leaks an invariant the **store**, not the caller, owns: in kiroku a stream
name's *category* is the substring before its first `-`, and category-scoped reads
([`Kiroku.Store.Read.readCategory`](/docs/kiroku/reference/core-types)) and category subscriptions
fan in every stream that shares that prefix. So the literal `"order-"` is not cosmetic — it is the
**join key** deciding which streams a projection or subscription sees. Hand-writing it per call site
has three hazards: no single source of truth (a typo spawns an orphan category), silent
mis-parsing (a "category" that itself contains a `-`, e.g. a saga prefix `"hospital-surge-"`, parses
as the category `hospital`, fanning the saga's stream into the wrong aggregate), and no typed bridge
back to the `CategoryName` a `readCategory`/subscription needs.

`Keiro.Stream` closes this with a validated, phantom-typed **category**: declare it once and derive
both per-entity streams and the category name from it.

```haskell
newtype StreamCategory a = StreamCategory { categoryTextOf :: Text }
  deriving stock (Generic, Eq, Ord, Show)

data CategoryError
  = CategoryEmpty                    -- the empty string
  | CategoryContainsSeparator !Text  -- contains the reserved `-` category/id boundary
  | CategoryReserved !Text           -- equals a store-reserved name ($all)
  | CategoryContainsIllegalChar !Char !Text -- whitespace or control character
  deriving stock (Eq, Show, Generic)

category       :: Text -> Either CategoryError (StreamCategory a)  -- validating smart constructor
categoryUnsafe :: Text -> StreamCategory a                         -- partial; `error`s on an invalid literal
categoryName   :: StreamCategory a -> CategoryName                 -- the matching CategoryName for category reads/subs
entityStream   :: StreamCategory a -> Text -> Stream a             -- render "<category>-<id>"
entityStreamId :: StreamIdSegment i => StreamCategory a -> i -> Stream a

class StreamIdSegment i where
  renderIdSegment :: i -> Text
-- instances for Text and String ship; add one per typed id:
-- instance StreamIdSegment OrderId where renderIdSegment = orderIdText
```

- `category t` **rejects** the empty string, any text containing `-` (kiroku's category/id boundary —
  this is hazard #2 turned into a fail-stop), the reserved name `$all`, and whitespace/control
  characters. Returns `Left CategoryError` otherwise.
- `categoryUnsafe t` is for static, known-good literals at a definition site
  (`orderCategory = categoryUnsafe "order"`); it `error`s on an invalid category, so **never** pass
  user input to it.
- `entityStream`/`entityStreamId` delegate the name mechanics to kiroku's `streamNameInCategory`, so
  the category rule stays single-sourced in the store. The phantom type carries from the
  `StreamCategory`, so the result is correctly tagged.
- `categoryName cat` recovers the `CategoryName` for category-scoped reads and subscription targets
  **from the same definition** — and by construction `categoryName cat` always equals
  `Kiroku.Store.Types.categoryName (streamName (entityStream cat id))`, so the per-entity stream and
  the category read can never disagree.

```haskell
orderCategory :: StreamCategory a
orderCategory = Stream.categoryUnsafe "order"          -- validated once at the definition site

orderStream :: OrderId -> Stream OrderEventStream
orderStream = Stream.entityStream orderCategory . orderIdText   -- renders "order-<id>"

orderCommandStream :: OrderId -> Stream OrderCommand
orderCommandStream = Stream.entityStream orderCategory . orderIdText
```

This is exactly how the `jitsurei` order stream now builds its handles — the two `stream ("order-" <>
…)` literals collapse to one declared category.

<Callout type="warn">
The type is named `StreamCategory`, **not** `Category`, to avoid clashing with the
`Kiroku.Store.Subscription.Types.Category` subscription-target constructor that consumer code often
imports alongside this module. Write a **compound** category in camelCase (e.g. `"hospitalSurge"` for
a saga over hospital surges) — never with `-`, which `category` rejects. The `:` separator is reserved
for the workflow stream family (`wf:<name>`).
</Callout>

For why an `EventStream` carries a `SymTransducer` directly rather than a decide/evolve pair, see
[Why SymTransducer, not Decider](/docs/keiro/explanation/why-symtransducer-not-decider).
