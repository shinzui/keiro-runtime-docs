keiro's "workflow engine" comes in two layers, both **shipped**. The **v1** layer treats aggregates,
process managers, and sagas as the *same* mathematical object (a keiki `SymTransducer`) persisted to
*one* substrate (your Postgres). The **v2** layer — **durable execution** — adds journaled named-step
workflows *on top of* v1, for processes that read more naturally as a sequence than as a pile of
per-event reactions. This page separates the two so you can pick the right tool.

<Callout type="info">
An earlier version of this page described v2 as a *roadmap idea, not shipped*. That is now out of
date: the durable-execution runtime (`Keiro.Workflow`) has landed with real, compilable signatures.
See the [durable workflows reference](/docs/keiro/reference/durable-workflows) and
[Understanding durable execution](/docs/keiro/explanation/durable-execution).
</Callout>

## v1 — the process-manager + timer substrate

Everything you need to coordinate multi-step, event-driven work:

- **Process managers** (`Keiro.ProcessManager`) — stateful coordinators that fold the events they
  have seen into their own event-sourced state, dispatch commands to target aggregates, and schedule
  timers. See [Understanding process managers and sagas](/docs/keiro/explanation/process-managers-and-sagas).
- **Sagas** — the same primitive used for compensation; there is no separate type. See [Write a saga
  with compensation](/docs/keiro/how-to/write-a-saga-with-compensation).
- **Durable timers** (`Keiro.Timer`) — `keiro_timers` rows fired by a bare, caller-driven worker with
  at-least-once semantics and `FOR UPDATE SKIP LOCKED` claiming, plus a stuck-row **recovery API**
  (`findStuckTimers` → `requeueStuckTimer` / `cancelTimer` / `deadLetterTimer`) and an optional
  `maxAttempts` auto-dead-letter ceiling. See [Understanding durable timers](/docs/keiro/explanation/durable-timers).
- **Worker observability** (`Keiro.Telemetry`) — an opt-in `KeiroMetrics` instrument set covering the
  outbox, inbox, timer, and projection workers, alongside the OpenTelemetry command/producer/consumer
  spans. See the [Telemetry reference](/docs/keiro/reference/telemetry#metrics-surface).

This expresses timeouts, escalations, retries, and compensating transactions — the jitsurei
order-fulfillment and incident-escalation domains are built entirely on it. The model's strength is
its honesty: every effect is an ordinary command-cycle write, keyed by a deterministic id, replay-safe
under at-least-once delivery. There is no hidden runtime to operate.

## v2 — durable execution (named-step workflows)

The v1 model asks you to express a workflow as a *reaction per event*: each incoming event is handled
independently and the manager's progress lives in its state stream. That is powerful, but some
workflows read more naturally as a *sequence* — "do this, then sleep for an hour, then wait for an
external signal, then do that". Expressing such a sequence as a pile of per-event reactions can
obscure the control flow.

**Durable execution** (`Keiro.Workflow`) lets you write that workflow as ordinary sequential code
whose checkpoints are individually durable:

| Feature | Shipped v2 shape |
|---|---|
| Authoring model | an ordinary `Eff (Workflow : es) a` function |
| Durable side effects | `step "name" action` journals a result under an explicit name; a replay returns it without re-running |
| Sleep | `sleepNamed` / `sleep`, backed by the v1 `keiro_timers` table |
| External completion | `awakeableNamed` + `signalAwakeable` / `cancelAwakeable` — durable promises (`keiro_awakeables`) |
| Child workflows | `spawnChild` / `awaitChild` / `cancelChild` (`keiro_workflow_children`) |
| Journal storage | one kiroku stream per instance, `wf:<name>-<id>` |
| Crash recovery | `resumeWorkflowsOnce` discovers and re-invokes unfinished workflows from a registry |
| Snapshots | `runWorkflowWith` + `snapshotPolicy` compact long journals |
| Rotation | `continueAsNew` / `restoreSeed` rotate onto a fresh journal generation so unbounded histories stay bounded |
| Versioning | `patch` gives a stable, journaled branch decision for cross-cutting logic changes (prefer renaming a step for single-step changes) |
| Observability | the `keiro.workflow.*` instruments and a `workflow <name>` span |

The key design decision is **named steps, not positional history**: step identity is an explicit
string, stable across source reordering, which avoids the Temporal-style replay nondeterminism that
ordinary code movement can otherwise cause. See
[Understanding durable execution](/docs/keiro/explanation/durable-execution) for why this matters.

### Phase-2 additions (MasterPlan 6)

The phase-2 work closes the two pieces that were once deferred and adds two scaling primitives shared
with the read side:

- **Continue-as-new journal rotation** (`continueAsNew` / `restoreSeed`) keeps an unbounded-length
  history bounded by rotating onto a fresh journal generation.
- **The versioning / patch API** (`patch`) gives a stable, journaled branch decision for a change that
  cross-cuts several steps; a single-step change is still better expressed by renaming the step.
- **LISTEN/NOTIFY push delivery** (`Keiro.Wake`, `runWorkflowResumeWorkerPush`) wakes the resume worker
  (and any poll-loop worker) within sub-second of an append, riding kiroku's existing per-store
  notifier with no new connections and a durable poll fallback.
- **Consumer-group sharding** (`Keiro.Subscription.Shard`, `runShardedSubscriptionGroup`) lets a pool
  of identical workers drain a high-volume category subscription disjointly, with coordinator-free
  failover.

The last two are about *scaling the workers* rather than the workflow model itself — see
[Scaling the workers](/docs/keiro/explanation/scaling-the-workers).

Still deferred (rejected or demand-driven): multi-region / global ordering, server-side scripted
projections, a schema registry, and field-level encryption.

## Which layer to build on

They are complementary. Reach for a **process manager** when the work reads as *"whenever X happens,
react"*; reach for a **durable workflow** when it reads as *one long-running function with in-line
waits*. A process manager is effectively a workflow whose steps are explicit event-driven reactions;
v2 reuses v1's command cycle, timers, snapshots, codecs, and migration story. The
[when-to-use comparison](/docs/keiro/explanation/durable-execution#when-to-use-a-workflow-vs-a-process-manager)
has the full table.
