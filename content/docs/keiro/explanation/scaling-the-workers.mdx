---
title: "Scaling the workers"
description: "How push-aware polling reduces latency and leased, acknowledgment-aware consumer groups increase throughput without weakening durability."
---

Keiro has two separate scaling tools:

- [`Keiro.Wake`](/docs/keiro/reference/push-delivery) reduces the idle latency of durable poll loops.
- [`Keiro.Subscription.Shard`](/docs/keiro/reference/subscription-sharding) spreads a busy category
  across a leased Kiroku consumer group.

Both are coordination mechanisms. Neither replaces durable state or idempotent work.

## Push changes when a pass runs

A fixed poll is correct but turns its interval into worst-case reaction time. A push-aware loop waits
on Kiroku's in-process broadcast of PostgreSQL `NOTIFY` events or on a fallback timeout, whichever
happens first. It reuses the store's listener connection.

`NOTIFY` is best-effort, so a wake never carries correctness. The durable poll remains the unit of
work, and the fallback guarantees another pass even if every notification is missed. Push changes
latency, not truth.

## Sharding changes how many readers run

Kiroku assigns every originating stream to one of `N` consumer-group members and keeps a checkpoint
for each member. Keiro adds dynamic membership: identical processes lease those member indices from
`keiro.keiro_subscription_shards`, then open or close Kiroku readers to match their current leases.

The lease table solves assignment, not delivery. A reconcile pass:

1. estimates the current live worker count;
2. renews existing leases and claims at most one bucket toward a fair share;
3. sheds excess buckets after peers become visible; and
4. reconciles live readers with that ownership set.

`FOR UPDATE SKIP LOCKED` ensures two valid claims cannot win the same lease row. Claiming only one
bucket per pass lets concurrently starting workers become visible and converge instead of allowing
the first process to monopolize the pool.

## Why overlap still exists

Lease exclusion does not stop code in a stale process. A worker can pause past `leaseTtl` while its
reader remains alive; a successor then claims the expired bucket and starts from the durable member
checkpoint. Until the stale reader stops, both can deliver the same event.

This is why the real guarantee is at-least-once:

- make handler effects idempotent on `eventId`;
- keep checkpoint writes monotonic; and
- couple each event's handler outcome to Kiroku before allowing progress.

Kiroku's checkpoint upsert cannot move backward, so a late stale acknowledgment does not regress
progress. It can still repeat an external effect unless the handler deduplicates it.

## Acknowledgment is the safety boundary

The sharded worker uses Kiroku's `subscriptionAckStream`, not the plain pull bridge. The reader emits
one `ShardDelivery` and blocks for a `ShardAck`:

| Handler outcome | Delivery behavior | Checkpoint behavior |
| --- | --- | --- |
| `ShardAckOk` | Continue to later events. | Successful events are covered when the batch-tail checkpoint is saved. |
| `ShardAckRetry delay` | Redeliver the same event after the delay. | Remains before the event. |
| `ShardAckDeadLetter reason` | Park the source event and continue. | Dead-letter insert and member-checkpoint advance happen atomically. |
| Shedding, shutdown, or another asynchronous interruption | No handler decision is written. | Remains before any unacknowledged event, which the next owner receives again. |

This separates a **lease handoff** from a **delivery decision**. Losing assignment cancels the reader;
it does not invent success. The next owner resumes from Kiroku's durable position.

Ordinary `Continue` events use batch-tail checkpointing. If a crash happens after their idempotent
effects but before the tail is saved, they replay. That is another normal source of at-least-once
delivery, independent of lease overlap.

## Bounded failure isolation

`runShardedSubscriptionGroupAck` lets the handler choose retry or dead letter. A synchronous handler
exception is converted to retry using `handlerRetryDelay`; an asynchronous exception is rethrown so
shutdown cannot be mistaken for failure policy. `retryPolicy` bounds total deliveries (five by
default). Exhaustion records `DeadLetterMaxAttempts` in `kiroku.dead_letters` and advances atomically,
allowing later events in the member to drain.

The compatibility `runShardedSubscriptionGroup` handler returns only `IO ()`. Completion maps to
success, while synchronous exceptions use the same bounded retry ladder. Use the acknowledgment-aware
entry point when the application knows that invalid input should be parked immediately.

Dead letters are durable operator evidence. `replaySubscriptionDeadLetters` re-runs an idempotent,
domain-specific handler and reports fresh, duplicate, failed, or missing outcomes. It deliberately
does not delete or mark the source rows.

## How the two tools compose

Push and sharding are orthogonal today. The shard worker's ownership reconcile loop uses a fixed
`renewInterval`; `Keiro.Wake` signals event appends, not lease changes. A future dedicated rebalance
notification could reduce handoff latency, but lease expiry and polling would remain the correctness
path.

Choose push for idle reaction time. Choose sharding for category throughput. In both cases, verify
the durable proof: workflow journal or worker tables for completed work, Kiroku member checkpoints
for progress, and `kiroku.dead_letters` for terminal source-event failure.
