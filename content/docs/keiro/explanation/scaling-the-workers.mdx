title: "Scaling the workers"
description: "Two ways keiro scales its background workers without changing the durable model: LISTEN/NOTIFY push delivery cuts latency (react in sub-second instead of polling), and consumer-group sharding scales throughput (a pool of workers drains a busy category disjointly). Both keep correctness in the durable layer, not in the optimization."
---

keiro's background workers — the [resume worker](/docs/keiro/explanation/durable-execution#who-turns-the-crank-the-resume-worker),
the [timer worker](/docs/keiro/explanation/durable-timers), the [outbox publisher](/docs/keiro/explanation/the-outbox-pattern),
and [async-projection subscriptions](/docs/keiro/explanation/projections-read-models-and-snapshots) —
all share one shape: poll the database, do a batch of idempotent work, commit, repeat. That shape is
correct and operable, but at scale two pressures show up: **latency** (a fixed poll interval is also
the worst-case reaction time) and **throughput** (one worker draining a busy category is a bottleneck).

keiro answers each with an additive primitive that leaves the durable model untouched. The throughline:
**correctness stays in the durable poll/lease layer; the optimization only changes *when* and *how
many* workers run.**

## Latency: push delivery

A poll loop sleeps a fixed interval between passes — the resume worker's default is a full second — so
a workflow whose timer just fired can wait up to that interval before anything notices. You could
shorten the interval, but that trades latency for wasted database round-trips on an idle system.

**Push delivery** ([`Keiro.Wake`](/docs/keiro/reference/push-delivery)) lets a worker instead *wait to
be woken*. kiroku already fires a Postgres `NOTIFY` on every append and already runs **one** dedicated
listener connection per store, fanning notifications to an in-process channel. A keiro worker
subscribes to that channel (an in-memory `dupTChan`, **not** a new database connection) and blocks
until either a notification arrives or a fallback timeout elapses — whichever first. So the system
reacts within sub-second of real work, yet costs nothing on an idle one.

The discipline that makes this safe: **push is an optimization over a durable poll, never a
replacement.** A `NOTIFY` is best-effort — drop the listener for a moment and the notification is gone.
So correctness never depends on a notification arriving. A missed wake only delays the next pass to the
fallback interval, exactly as the old fixed poll did; the durable, idempotent pass is still the unit of
work. In the worst case (every notification dropped) a push-aware worker behaves identically to the
fixed-poll baseline. That is why the resume worker's `pollInterval` is simply *repurposed* as the
fallback timeout: strictly better for latency, identical in the worst case.

## Throughput: consumer-group sharding

A single subscription worker reads a category's events in order, one batch at a time. For a busy
category that is a throughput ceiling — and you cannot just run two copies of the same worker, because
both would process every event.

kiroku already solves the *partitioning* half: it can split a category into `N` **buckets**
(consumer-group member indices) by a stable hash of the originating stream id, and keeps a per-bucket
checkpoint, so `N` readers can each drain a disjoint slice. What it leaves to the operator is
*membership* — "exactly one live process owns each bucket right now" — which by hand is fragile
`[0..N-1]` wiring that breaks the moment a worker dies.

**Consumer-group sharding** ([`Keiro.Subscription.Shard`](/docs/keiro/reference/subscription-sharding))
supplies that membership layer as a **lease**. You start the *same* worker binary `N` times; each mints
a worker id and, every few seconds, runs one reconcile pass against a small `keiro_subscription_shards`
table: renew the buckets it holds, claim up to its fair share of unowned-or-expired ones, shed any
excess, and open/close kiroku readers to match. The result is a pool that partitions the category among
itself — every event processed by exactly one worker — with **no external coordinator** (no
etcd/ZooKeeper/Consul): coordination lives entirely in the lease table.

Two design choices carry the weight:

- **A lease, not a held lock.** A renewable `lease_expires_at` timestamp gives lifetime ownership that
  survives across the many transactions a worker's life spans — which neither a transaction-scoped
  advisory lock (auto-released at commit) nor a session-scoped one (no connection affinity through a
  pool) can do. When a worker dies it simply stops renewing; its leases expire and a survivor re-claims
  the buckets. **Failover is automatic and coordinator-free.**
- **Claim one bucket per pass.** Disjointness rests on a `FOR UPDATE SKIP LOCKED` claim, so two workers
  racing the same bucket can never both win regardless of how stale their "how many of us are live"
  estimate is. Claiming a single bucket per pass (rather than a whole fair share at once) keeps a
  cold-starting worker from monopolising every bucket before its peers become visible; the pool climbs
  to an even split together over a few intervals.

Because a bucket can briefly overlap owners while it changes hands, a sharded subscription is
**at-least-once** — so its handler must be idempotent (keyed on `eventId`), exactly as keiro's
[async-projection guidance](/docs/keiro/how-to/make-an-async-projection-idempotent) already requires.
The lease governs *assignment only*; event positions stay in kiroku's per-bucket checkpoints, so a
re-homed bucket resumes exactly where its previous owner left off.

## They compose

Push and sharding are orthogonal and stack cleanly. Each sharded reader is still a poll/lease loop, so
the same `Keiro.Wake` channel can shorten *its* idle gap; the rebalance interval is itself a documented
seam where a dedicated rebalance `NOTIFY` could replace the fixed wait. Neither primitive changes what
is durable — the journal, the checkpoints, the leases — so you can adopt either, both, or neither, and
the worst case is always the plain fixed-poll, single-worker behaviour you started with.
