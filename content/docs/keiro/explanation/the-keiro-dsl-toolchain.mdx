A keiro **service** — a bounded context using event sourcing, process managers, durable timers,
durable workflows, and Kafka/PGMQ integration — is today hand-written as a large amount of highly
regular Haskell. `keiro-dsl` is a separate package (and CLI) that makes that regularity explicit: a
`.keiro` file becomes the **typed specification** of the service, the permanent machine-checkable
source of truth, and the toolchain turns it into compiling Haskell.

<Callout type="warn">
keiro-dsl is **new** and in active development alongside keiro `0.1.0.0`. It is an *authoring* tool,
not a runtime dependency: the code it scaffolds compiles against the ordinary keiro/keiki libraries
and has no dependency on keiro-dsl. The spec extension is `.keiro`.
</Callout>

## The observation it is built on

An empirical audit of two real services found that **67%–96% of the code per feature is a
deterministic template** — domain ADTs, ids, codecs, `EventStream`/projection wiring, register lists,
the Template-Haskell splices, the process/timer/contract plumbing. The non-derivable remainder
collapses into a small, closed set of **hole-kinds**: the transducer body, a projection's `apply` SQL,
an upcaster body, a process `handle`, a fan-out body, and so on.

Two things follow. First, the deterministic 67–96% should never be hand-written — typos and drift
there create silent bugs (an orphan stream category, a codec that no longer round-trips). Second, the
genuinely dangerous decisions — inverted error semantics, opaque id derivations, fields whose names
lie about their semantics — are *exactly* the ones a human or a coding agent gets wrong when left
implicit. The spec's job is to **force each dangerous decision to be stated explicitly** and reject
the spec before any Haskell is written if one is missing or stated the unsafe way.

## What you do with a `.keiro` file

A developer (or a coding agent planning a feature) runs a five-step loop:

1. **Write** a service spec in terse notation covering keiro's full node surface: `aggregate`,
   `process` + `timer`, `contract`/`intake`/`emit`/`publisher`, `workqueue`/`dispatch`, `workflow`,
   and `operation`.
2. **`keiro-dsl check`** the spec — rejecting any missing required decision *before any Haskell
   exists*.
3. **`keiro-dsl scaffold`** the spec — emitting the symbol-free deterministic layer into
   `-- @generated` modules, precisely-typed **holes** in hand-owned modules, and a verification
   **harness**.
4. **Fill** the transducer body and the holes (by hand or with a coding agent) against the generated
   signatures, then run the harness to confirm behavior matches the spec.
5. **`keiro-dsl diff --since <ref>`** to classify each later spec change as safe (additive) or
   breaking (needs an upcaster or a deprecation).

The [notation reference](/docs/keiro/reference/keiro-dsl-notation) is the complete grammar and CLI;
the [tutorial](/docs/keiro/tutorials/author-a-service-with-keiro-dsl) walks the whole loop end to end
on one aggregate.

## The load-bearing scope decision: the firewall

keiro-dsl is deliberately **not** a full code generator. It does **not** emit the symbolic transducer
logic — the part that lowers guards and register writes to keiki's `./=` / `lit` / `B.slot` /
`B.requireGuard` operators against Template-Haskell-produced identifiers. That is the most brittle,
most framework-coupled code in a service, and it is exactly what a coding agent writes well from a
spec plus examples. So it is left as a **hole** and pinned by the harness.

This is enforced as a tested invariant — the **firewall**:

<Callout type="info">
**No `-- @generated` line ever contains a keiki symbolic operator.** The symbolic operators (`./=`,
`.==`, `.||`, `lit`, `B.slot`, `B.requireGuard`) appear *only* in the hand-owned hole modules you
write. If one shows up in a generated module, that is a scaffolder bug — not something to "fix" in
place.
</Callout>

The firewall is what makes the whole design tractable: the generated layer is pure boilerplate that is
safe to overwrite on every scaffold, and the dangerous, framework-coupled logic lives in one place
you own and the harness verifies. It also draws a clean line for the two writers — re-running
`scaffold` overwrites every `-- @generated` module unconditionally but **never** clobbers a filled
hole module (those are create-if-absent).

Also out of scope by design: read-model table migrations (delegated to `codd`) and arbitrary in-body
computation — branching, raw SQL, string surgery, inter-step data flow. Those are agent-written,
harness-checked holes, not things the spec expresses.

## Where the value concentrates: the dangerous-decision checks

The bulk of the payoff is `check` rejecting an unsafe spec before any Haskell exists. The checker
forces each dangerous decision to be present *and* flags the inversion if it is stated the
unsafe-but-plausible way:

- **Time is injected, never sampled.** A deadline or sleep is computed from a timestamp carried in the
  input (e.g. `observedAt`), never from a wall-clock read — otherwise replay would compute a different
  time. The checker rejects a sampled clock (`ClockSampled`, `ProcessFireAtNotInjected`).
- **The inbox disposition inversions.** An intake's disposition table must be **complete**, and two
  outcomes are dangerously counter-intuitive: `duplicate => ackOk` (a replay is *success*, not a
  retry) and `previouslyFailed => deadLetter` (a prior failure is *not* retried). A third:
  `decodeFailed => deadLetter` (poison, not unbounded retry). State them correctly; the checker flags
  the wrong way.
- **Dangerous timer defaults.** A timer with no `max-attempts` ping-pongs forever; an unknown timer
  status silently cancels; dispatch-ids are runtime-owned and must be un-overridable. The spec forces
  an explicit ceiling and forbids a user-supplied dispatch-id (`ProcessDispatchIdSupplied`).
- **The pgmq inversions.** A workqueue's `storeFailure => retry` (transient) must retry while
  `decodeFailure => deadLetter` (poison) must dead-letter; and the physical queue/table names are
  re-derived and checked against the captured fixture so a hand-typed name cannot drift
  (`WqPhysicalDivergence`).
- **Await ↔ signal coupling.** Every `signal <label> of <workflow>` must match an `await <label>` of
  that workflow — otherwise the deterministic awakeable id never matches and the workflow waits
  forever (`AwaitSignalMismatch`).
- **Codec evolution.** A versioned event (`vN`, N > 1) needs a contiguous upcaster hole; a status-map
  projection must be total. `diff --since` then classifies a later field-add-without-bump as
  **BREAKING**.

## The harness — not the scaffold — pins behaviour

Two agents can fill the holes differently but correctly and both pass; one wrong guard, mapping, or
disposition fails a *specific* named harness test. The scaffolder emits, alongside the generated layer,
a harness that asserts `validateTransducer == []` (the [replay-safety
check](/docs/keiro/reference/event-stream-and-stream#replay-safety-validation-keiroeventstreamvalidate)),
codec round-trips, the time-injection / disposition / id decisions, and a behavioural accept. So the
guarantee that your fill matches the spec is *behavioral and tested*, not textual. This is also why the
scaffold can safely omit the symbolic logic: the harness, not the generator, is what proves the filled
service is correct.

## When to reach for it

Use keiro-dsl when you are building or evolving a keiro service and want the spec to be the source of
truth — especially when a coding agent is doing the filling, because the spec is what keeps it from
quietly choosing the unsafe inversion. For a one-off aggregate with no registers, guards, process
managers, or integration, hand-writing is fine; the DSL earns its keep on the regular, dangerous,
multi-node services it was audited against.
