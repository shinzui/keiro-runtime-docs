---
title: "The keiro-dsl service toolchain"
description: "How a typed service specification, validation gate, transactional scaffolder, hand-owned holes, harnesses, and exhaustive diff divide service authoring safely."
---

`keiro-dsl` makes a `.keiro` file the machine-checkable source of truth for a Keiro service. The
toolchain derives regular Haskell wiring, forces persistence and delivery decisions into reviewable
notation, and leaves behavior-bearing code in typed hand-owned seams.

It is an authoring tool, not an application runtime dependency. Generated modules compile against
Keiro and Keiki; an application does not import `keiro-dsl`.

## One source, six gates

The lifecycle has six distinct jobs:

1. `new <kind>` prints a minimal valid starter for one supported node family.
2. `parse` proves the notation parses and prints the canonical representation.
3. `check` resolves references and rejects unsafe, incomplete, or contradictory decisions.
4. `scaffold` plans and verifies the entire output set before writing generated and create-once files.
5. The generated harness compiles with hand-filled behavior and proves facts, codecs, replay safety,
   and selected runtime contracts.
6. `diff --since <ref>` classifies later persistence and identity changes as `ADDITIVE`, `WARNING`,
   or `BREAKING`; only breaking changes block the command.

That separation matters. Parsing does not imply semantic validity, scaffolding does not prove a
hand-written guard or resolver, and a green current spec does not prove that an upgrade preserves
durable identity.

## What the specification can own

The top-level node inventory is closed and source-checked:

- domain and read side: `aggregate`, aggregate `projection`, first-class `readmodel`, and `snapshot`;
- orchestration: `process` with one `timer`, and `router`;
- integration: `contract`, `intake`, `emit`, and `publisher`;
- background work: `workqueue` and `dispatch`;
- durable execution: `workflow` and `operation`.

Shared `id`, `enum`, and `rule` declarations supply generated types and total domain rules. Optional
`module` and `layout` clauses control placement. The spec captures identities, schemas, policies,
bindings, and typed topology. Arbitrary SQL, effectful fan-out, workflow bodies, symbolic aggregate
transitions, and deployment tuning remain hand-owned.

## Why validation is the first semantic boundary

Many plausible-looking defaults are wrong for durable systems, so `check` makes them explicit:

- time comes from typed input fields, never a sampled wall clock;
- a process saga names a validated category, and process/router rejection and poison policies are
  mandatory;
- a timer has a separate `on-ambiguous` outcome, which cannot be `Fired`;
- inbox duplicate delivery is success, prior failure and decode poison dead-letter, and transient
  store failure retries;
- workqueue store failure retries while decode failure dead-letters;
- a router acknowledges resolver stability and uses target-based deterministic dispatch identity;
- read-model table shape, consistency, feed, scope, and subscription identity agree;
- event versions have contiguous upcasters and workflow signals match awaits;
- Haskell-facing identifiers are ASCII, namespace-safe, non-keyword, and collision-free.

Diagnostics are located and coded. Warnings remain visible but do not fail `check`; errors do.

## The ownership firewall

Scaffolding produces two kinds of files:

| Ownership | Lifecycle | Typical contents |
|---|---|---|
| Generated, marked `-- @generated` | overwritten on re-scaffold | ADTs, codecs, event streams, tables, policies, harnesses |
| Hand-owned holes | created only when absent | symbolic transitions, SQL, resolvers, fan-out, worker/body assembly |

Generated aggregate event streams expose a raw definition and a public validated value built with
`mkEventStreamOrThrow`. The generated harness checks `validateTransducer`; application and worker
wiring consumes the validated value.

The firewall is mechanical: generated modules may not contain Keiki symbolic operators such as
`./=`, `.==`, `lit`, `B.slot`, or `B.requireGuard`. Those operators belong in hand-owned aggregate
holes, where the generated types constrain them and the harness tests them.

Before the first write, scaffold validates the spec, detects exact and case-folded module collisions,
checks faithful lowering and the firewall, and protects any existing generated target without its
banner. `--force-generated-overwrite` bypasses only that last banner protection. It is an explicit
adoption tool, not a general "ignore safety" switch.

## Manifests and stale output

A successful scaffold emits a manifest with Cabal-ready module/dependency blocks and a per-context
scaffold record. On the next run, paths that are no longer planned appear under `stale:`. Nothing is
deleted automatically: generated entries are reviewable deletion candidates; hole entries are
hand-owned code and require inspection.

The pair prevents two common upgrade failures: forgetting a newly generated module in Cabal and
silently leaving an obsolete generated module in the build.

## Why the harness is the behavioral proof

Two implementations can satisfy the same typed hole in different ways. Text generation cannot prove
which implementation matches the service decision. Harnesses therefore assert observable facts:
codec round-trips, replay-safety validation, aggregate accepts/rejects, process/timer dispositions,
router identity and policy, read-model declarations, queue policy, and workflow facts.

A scaffold report proves the generated half is coherent. A green harness proves the generated and
hand-owned halves agree.

## Why `diff` is a separate upgrade gate

`check` validates one snapshot of a service. `diff` compares two snapshots over an exhaustive
node-family registry and asks whether stored bytes or durable identities survive the change.

- `ADDITIVE` is safe to introduce.
- `WARNING` changes forward behavior or policy and exits zero, but demands review.
- `BREAKING` can strand persisted data, cursors, jobs, or retry identity and exits nonzero.

The output is deliberately regular:

```text
<tier>: <node> <facet> <subject>: <detail> [optional-code]
```

Versioning an event and supplying a contiguous upcaster can turn an otherwise breaking payload
change into an additive one. Renaming a stable workflow, router, process category, queue, read-model
table/subscription, or identifier prefix remains breaking because code cannot migrate the old
identity implicitly.

## What `diff` computes beyond a tier

Three of `diff`'s outputs are not classifications at all — they are *work products* for the deploy.

**A replay-impact verdict.** `--replay-impact-out FILE` answers the narrower question "can the
candidate binary interpret an already-stored log differently?" as machine-readable JSON: either
`replay-neutral` (no data access needed) or `affected` with a conservative per-aggregate event-type
set. New aggregates, events, transitions, and syntactically proven guard loosenings are neutral,
because no old log depends on them.

**A paste-ready replay-only twin.** When a live transition's guard changes with no twin present, the
`AggGuardTightened` advisory computes the removed region — `old ∧ ¬new`, with negation eliminated
inside the guard grammar — and prints a `replay-only` transition carrying it, complete with the old
writes/emits/goto. It re-parses as-is. It is *printed, never auto-applied*: whether history should
stay replayable or be truncated is a business decision.

**Old-payload goldens.** `--emit-goldens DIR` captures the previous wire shape of every bumped event
at the one moment both shapes are knowable. `scaffold --goldens DIR` then embeds them so the harness
exercises `decodeRaw` against a genuine shape. Emission never overwrites a file, so a payload copied
out of production always wins.

## The gate `diff` deliberately is not

`diff` reads two specs. It never reads your database, so it cannot tell you whether real streams
still exercise a retired rule, still carry a deprecated event, or still hold a snapshot seed produced
by the old fold. Nor can it see inside a hole.

That is why the toolchain's last gate lives outside it: the
[real-log replay audit](/docs/keiro/reference/replay-audit), assembled from the scaffolded
`Generated.<Context>.ReplayAudit.auditTargets`. Gates are placed at the earliest boundary with enough
evidence, and only the database has this evidence. See
[Evolution and replayability](/docs/keiro/explanation/evolution-and-replayability) for how all six
layers divide the work.
