Short answers to the questions people ask most about keiro. Each links to the page that goes deeper.

<Accordions>
  <Accordion title="Is keiro a server or a library?">
    A **library you import**, not a server you operate. You add keiro to a Haskell service and call
    its functions; there is no keiro daemon, broker, or control plane. The only infrastructure keiro
    needs is your PostgreSQL database. See [What is keiro?](/docs/keiro/explanation/what-is-keiro)
    and [Get started with keiro](/docs/keiro/tutorials/getting-started).
  </Accordion>

  <Accordion title="How does it relate to kiroku, keiki, and shibuya?">
    keiro **composes three lower libraries**: **kiroku** (the append-only PostgreSQL event store),
    **keiki** (the pure symbolic-register finite-state transducer that is the decision core), and
    **shibuya** (the supervised subscription/worker substrate). keiro's thesis is that aggregates,
    process managers, and sagas are all the *same* keiki transducer persisted to *one* substrate —
    your Postgres. See [The keiro stack](/docs/keiro/explanation/the-keiro-stack).
  </Accordion>

  <Accordion title="Does keiro ship durable workflows (Temporal-style)?">
    Yes — `Keiro.Workflow` is the v2 **durable-execution runtime**: write a process as one function
    whose **named** checkpoints are journaled, with a durable `step`, `sleep`, external `awakeable`,
    and child workflows, plus a crash-recovery resume worker. Unlike Temporal's positional history,
    step identity is by **name**, so it is stable across source reordering. It layers on the v1
    **process managers** + **durable timers** (`keiro_timers`), which remain the right tool for
    fundamentally event-driven, reactive coordination. It also ships **continue-as-new** journal
    rotation (`continueAsNew` / `restoreSeed`, for unbounded histories) and a **versioning** API
    (`patch`, for cross-cutting logic changes — prefer renaming a step for single-step changes). See
    [Understanding durable execution](/docs/keiro/explanation/durable-execution), the [durable
    workflows reference](/docs/keiro/reference/durable-workflows), and the [workflow
    roadmap](/docs/keiro/explanation/workflow-roadmap). If the workflow comes from a `.keiro`
    specification, follow [Evolve a workflow with patch and
    continue-as-new](/docs/keiro/how-to/evolve-a-workflow-with-patch-and-continue-as-new).
  </Accordion>

  <Accordion title="How do I change a workflow's logic while instances are in flight?">
    Two tools, and the common one needs no API. Because step identity is the **name**, a single
    step's change is just a **rename** — the new `StepName` has no journaled history, so it runs fresh.
    For a change that *cross-cuts several steps*, `patch (PatchId "…")` returns a stable, journaled
    `Bool`: `True` for instances fresh at that point, `False` for ones already in flight, replayed
    verbatim forever. Reach for `patch` only when an in-flight instance would otherwise be left
    incoherent, and never reuse a retired `PatchId`. See [Write a durable workflow →
    Versioning](/docs/keiro/how-to/write-a-durable-workflow#versioning-a-running-workflow).
  </Accordion>

  <Accordion title="How do I scale the background workers?">
    Two additive primitives that leave the durable model untouched. **Push delivery** (`Keiro.Wake`,
    `runWorkflowResumeWorkerPush`) makes a poll-loop worker react within sub-second of an append by
    riding kiroku's existing `LISTEN/NOTIFY` notifier — no new connections, and a durable poll fallback
    so a dropped notification only costs latency. **Consumer-group sharding**
    (`Keiro.Subscription.Shard`, `runShardedSubscriptionGroupAck`) drains a high-volume category across a
    pool of identical workers that lease buckets cooperatively. Delivery remains at-least-once:
    `ShardAck` couples success, bounded retry, or atomic Kiroku dead-letter advancement to the member
    checkpoint, and idempotency absorbs overlap during failover. See
    [Scaling the workers](/docs/keiro/explanation/scaling-the-workers).
  </Accordion>

  <Accordion title="How do I prove a background item was not silently dropped?">
    Start with durable state, not absence from a backlog. Workflow journals and
    `keiro.keiro_workflows`, Kiroku member checkpoints / `kiroku.dead_letters`,
    `keiro.keiro_inbox`, `keiro.keiro_outbox`, `keiro.keiro_timers`, and
    `keiro.keiro_dead_letters` distinguish success, retry, and terminal failure. The defaults halt or
    retry; metric-only consumption occurs only under explicit policies such as `RejectedSkip` or
    `PoisonSkip`. The [durable worker outcome checklist](/docs/keiro/how-to/verify-durable-worker-outcomes)
    maps each path to its table and alert.
  </Accordion>

  <Accordion title="When should I use a Router vs. a process manager?">
    Ask one question: **does the next action depend on a history the coordinator must remember?** If
    *no* — each event maps to its targets independently (page the current on-call roster; record a
    transaction against every overlapping chapter) — use a **`Keiro.Router`**: it is stateless,
    resolves the target set from a read model per event, and dispatches one idempotent command to
    each. If *yes* — the reaction depends on what came before, or needs a durable timer — use a
    **process manager**, which folds its own event-sourced state. A **saga** is just a process
    manager whose job is compensation, not a third primitive. They also compose: a process manager
    can hand a fan-out off to a router. See [Routers and content-based
    dispatch](/docs/keiro/explanation/routers-and-content-based-dispatch) and [Process managers and
    sagas](/docs/keiro/explanation/process-managers-and-sagas).
  </Accordion>

  <Accordion title="What goes in a Router's resolve function?">
    A **read, not a write.** `resolve :: input -> Eff es [PMCommand targetCi]` computes the target
    set — it does *not* dispatch. Typically it `runQuery`s a read model and returns one `PMCommand`
    (target stream + command) per resolved target; keiro does the idempotent dispatch afterward.
    Returning `[]` means "this event routes nowhere". It can run more than once for the same event
    (the worker retries), which is safe because it only reads and the dispatch is keyed by a
    deterministic id. See [the `resolve` seam](/docs/keiro/reference/router#the-resolve-seam).
  </Accordion>

  <Accordion title="How do I know an event stream is safe to replay?">
    Validate it. `Keiro.EventStream.Validate` lifts keiki's pure umbrella check
    (`validateTransducer`) to the `EventStream` boundary: `validateEventStream label es` returns a
    list of labelled `EventStreamWarning`s (empty when sound), and `mkEventStream label es` is the
    fail-fast smart constructor returning `Left warnings` for an unsafe stream or
    `Right ValidatedEventStream` for a sound one. Public runners accept the validated wrapper, not the
    bare record. It checks hidden input, first-event recoverability, inversion ambiguity, guarded
    input reads, state-changing output-free edges, determinism, and dead edges — statically, **with
    no SMT solver and no database**. Keiro also rejects incoherent snapshot configuration and an
    initial state/register pair the snapshot codec cannot encode. The durable boundary always forces
    head-recoverability and state-changing-output-free checks on.
    Run it in a test or at start-up over your streams. See the [event stream
    reference](/docs/keiro/reference/event-stream-and-stream#validatedeventstream-and-replay-safety-validation)
    and [Why SymTransducer, not Decider](/docs/keiro/explanation/why-symtransducer-not-decider).
  </Accordion>

  <Accordion title="Can a bad snapshot fail or corrupt a command?">
    A snapshot is a **cache**, never durable truth. No stream, no compatible row, or undecodable JSON
    falls back to full event replay; `lookupSnapshotSeed` retains those miss reasons for telemetry.
    After an event append commits, keiro strictly encodes and writes a snapshot only if the policy
    fires. Encode or store failure is counted and the command remains successful. A failure to replay
    the just-appended batch is also post-commit: it records `keiro.snapshot.apply.divergence` and the
    `keiro.replay.divergence` span attribute so you can stop further writes and investigate. See the
    [snapshot outcome table](/docs/keiro/reference/snapshot#caller-visible-and-operational-outcomes).

    Durable workflows use the same boundary: their journal append is truth; a later snapshot-store
    failure increments `keiro.snapshot.write.failures` and does not change the completed step,
    completion, or continue-as-new outcome. That advisory catch is why workflow runners require the
    typed `Error StoreError` effect even when the workflow body itself does not.
  </Accordion>

  <Accordion title="Is keiro Postgres-only?">
    Yes. keiro persists everything — events, snapshots, read models, timers, the outbox, the inbox —
    to **one** PostgreSQL database. That single-substrate design is deliberate: no second datastore to
    keep consistent. See [Run the migrations](/docs/keiro/how-to/run-migrations) for the nine tables
    keiro owns.
  </Accordion>

  <Accordion title="How do I get exactly-once processing with Kafka?">
    You don't get distributed exactly-once *delivery* — no system does. You get **at-least-once**
    delivery plus the **inbox** dedupe table, keyed by `(source, dedupe_key)`, which makes
    *processing* effectively-once: a redelivered message is recognized and skipped. On the send side,
    the **outbox** publishes transactionally so a decided event is never lost. See [The inbox
    pattern](/docs/keiro/explanation/the-inbox-pattern), [The outbox
    pattern](/docs/keiro/explanation/the-outbox-pattern), [Choose an inbox dedupe
    policy](/docs/keiro/how-to/choose-an-inbox-dedupe-policy), and [Choose inbox
    persistence](/docs/keiro/how-to/choose-inbox-persistence).
  </Accordion>

  <Accordion title="Why are timers a first-class primitive instead of a delayed PGMQ message?">
    Because a saga's wait is **addressable**, and a queued message is not. A timer is a row in
    `keiro_timers` with a **caller-chosen `timerId`**, which buys four things PGMQ's per-message
    *delay* (a visibility offset applied at send time) cannot: **idempotent scheduling** — re-arming
    the same id on a redelivered event updates the existing row instead of enqueuing a second message
    that fires twice; **cancellation and rescheduling by id** (`cancelTimer`, the idempotent upsert) —
    an order paid before its timeout simply withdraws the row, whereas a delayed message sits invisible
    in the queue with no "cancel timer X" operation; **atomicity with the decision** — the
    `TimerRequest` is part of the process manager's action and commits in the *same* transaction as the
    event append, with no dual-write, where `enqueue` is its own PGMQ transaction; and
    **observability** — timers are queryable rows (backlog, stuck, due) with a `keiro.timer.*` metric
    set and an explicit `findStuckTimers` → requeue/cancel/dead-letter recovery API, where a delayed
    message is opaque until its visibility window elapses. A fired timer also routes back into the
    domain through the command cycle via its `processManagerName`/`correlationId`/`payload`, and
    durable-workflow `sleepNamed` reuses the very same table — one timer worker drains both. PGMQ's
    delay is built for retry back-off, not for an arbitrary, cancellable, far-future wait. See
    [Understanding durable timers](/docs/keiro/explanation/durable-timers) and the [Keiro.Timer
    reference](/docs/keiro/reference/timers).
  </Accordion>

  <Accordion title="Does keiro have a background job queue?">
    Yes — `keiro-pgmq`, a **separate package** in the keiro repo (module `Keiro.PGMQ`). You declare a
    typed `Job` bundling a queue, a payload codec, and a retry/dead-letter policy, then write a plain
    handler `p -> Eff es JobOutcome` (`Done | Retry delay | RetryDefault | Dead reason`) that never
    touches shibuya or PGMQ wire types. Run many queues continuously with `runJobWorkers`, or drain on
    a schedule with `runJobOnce`. It is the *worker* primitive for **transient work that is not a domain event** — for
    domain events crossing contexts, use the [outbox](/docs/keiro/explanation/the-outbox-pattern) and
    [inbox](/docs/keiro/explanation/the-inbox-pattern) instead. See [Background jobs with
    PGMQ](/docs/keiro/explanation/background-jobs-with-pgmq), the
    [reference](/docs/keiro/reference/pgmq-jobs), and [Configure FIFO queue ordering and
    provisioning](/docs/keiro/how-to/configure-fifo-queue-ordering-and-provisioning).
  </Accordion>

  <Accordion title="Is keiro-pgmq's enqueue transactional with my database write?">
    **No.** `enqueue` runs in its own PGMQ transaction, so it is not atomic with a domain write — a
    crash between the two commits leaves them inconsistent. For exactly-once enqueue tied to a row,
    send onto the queue via raw SQL inside the *same* `hasql` transaction as your write (plus a dedupe
    claim), the way the outbox does. See [Enqueue a job
    transactionally](/docs/keiro/cookbook/transactional-job-enqueue). If the thing you are sending is
    actually a domain event, that pattern *is* the [outbox](/docs/keiro/explanation/the-outbox-pattern) —
    use it.
  </Accordion>

  <Accordion title="What is keiro-dsl and do I need it to use keiro?">
    `keiro-dsl` is a separate **authoring** package and CLI — you do **not** need it to use keiro.
    A `.keiro` file is a typed *specification* of a keiro service; `keiro-dsl` turns its aggregate,
    projection, read-model, process, router, integration, queue, dispatch, and workflow nodes into
    compiling Haskell: a symbol-free `-- @generated` deterministic layer plus precisely typed
    **holes** for the behaviour-bearing code you fill by hand. The code it scaffolds depends only on
    keiro/keiki, never on keiro-dsl. It earns its keep on regular, multi-node services — especially
    when a coding agent is doing the filling — because the spec forces the dangerous decisions to be
    explicit. See [The keiro-dsl service
    toolchain](/docs/keiro/explanation/the-keiro-dsl-toolchain) and the
    [tutorial](/docs/keiro/tutorials/author-a-service-with-keiro-dsl). The [notation
    reference](/docs/keiro/reference/keiro-dsl-notation) inventories all twelve top-level nodes.
  </Accordion>

  <Accordion title="Does keiro-dsl generate the whole service, including the transducer?">
    **No — and that is the point.** The scaffolder emits only the deterministic ~67–96% (the boilerplate);
    it never emits the symbolic transducer logic (keiki's `./=` / `lit` / `B.slot` / `B.requireGuard`).
    That is the most brittle, framework-coupled code, and exactly what a human or coding agent writes
    well from a spec plus examples — so it is left as a **hole** and pinned by a generated harness. The
    **firewall invariant** — no generated module ever contains a keiki symbolic operator — keeps the
    two ownership domains separate. Re-scaffold overwrites only a target carrying the
    `-- @generated` banner; a bannerless collision aborts the entire write plan unless you explicitly
    adopt that path with `--force-generated-overwrite`. Filled holes are create-if-absent. See [The
    keiro-dsl service
    toolchain](/docs/keiro/explanation/the-keiro-dsl-toolchain#the-load-bearing-scope-decision-the-firewall).
  </Accordion>

  <Accordion title="How do I build an aggregate's stream name safely?">
    Declare the **category** once with `Keiro.Stream.categoryUnsafe` (or `category` for validated
    input) and derive both the per-entity stream and the matching `CategoryName` from it —
    `entityStream` / `entityStreamId` render `"<category>-<id>"`, and `categoryName` gives the
    `CategoryName` for category-scoped reads and subscriptions. This single-sources the category prefix
    and rejects a prefix containing `-` (kiroku's category/id boundary), which would otherwise silently
    fan a saga stream like `"hospital-surge-…"` into the wrong aggregate. Write a compound category in
    camelCase (e.g. `hospitalSurge`), never with `-`. See [Safe, category-based
    construction](/docs/keiro/reference/event-stream-and-stream#safe-category-based-construction-streamcategory).
  </Accordion>
</Accordions>
