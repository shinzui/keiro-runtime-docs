## The definition separates application data from framework metadata

```haskell
data ReadModel q r = ReadModel
  { name               :: Text
  , tableName          :: Text
  , schema             :: Text
  , subscriptionName   :: Text
  , version            :: Int
  , shapeHash          :: Text
  , defaultConsistency :: ConsistencyMode
  , strongScope        :: StrongScope
  , query              :: q -> Tx.Transaction r
  }
```

`schema` and `tableName` identify an application-owned data table. `qualifiedTableName` combines
them through `Keiro.Connection.qualifyTable`, producing a double-quoted name such as
`"app_reads"."order_summary"`. Keiro does not rewrite the SQL in `query`, so the application must
use that qualified table in its create/read/write statements.

The registry is separate: `name`, `version`, `shapeHash`, and status live in
`keiro.keiro_read_models`. The application schema is deliberately not persisted as registry
identity.

## Registration is startup work

The public query path calls `ensureReadModel`, but “ensure” means *look up and validate*, not insert.
Applications call this once during startup:

```haskell
registerReadModel
  (readModel ^. #name)
  (readModel ^. #version)
  (readModel ^. #shapeHash)
```

`registerReadModel` inserts a `Live` row when absent. Its conflict clause updates only `name`, so an
existing version, hash, and status remain unchanged. This makes startup idempotent without erasing
evidence that the code definition drifted from stored metadata.

A query never creates the row. The first gate can therefore return:

```haskell
data ReadModelError
  = ReadModelUnregistered Text
  | ReadModelStaleSchema Text Int Int Text Text
  | ReadModelWaitTimeout Text GlobalPosition GlobalPosition
  | ReadModelNotLive Text ReadModelStatus
```

Tests exercise the boundary directly: repeated queries do not change registry state, concurrent
explicit registration converges on one row, and an unregistered query leaves the registry empty.

## `runQueryWith`: validate, wait, query

```haskell
runQueryWith metrics consistency readModel input = do
  schemaCheck <- ensureReadModel readModel
  case schemaCheck of
    Left err -> pure (Left err)
    Right () -> do
      waitResult <- waitIfNeeded metrics consistency readModel
      case waitResult of
        Left err -> pure (Left err)
        Right () -> Right <$> runTransaction ((readModel ^. #query) input)
```

The order is a correctness boundary. Missing registration, stale schema, or non-`Live` lifecycle
status fails before any wait or application-table read. `runQuery metrics` selects
`defaultConsistency`; `runQueryWith` overrides it.

## Strong waits now have a scope

```haskell
data ConsistencyMode = Strong | Eventual | PositionWait PositionWaitOptions

data StrongScope
  = EntireLog
  | CategoryHead Text
```

`Eventual` skips the wait. `PositionWait` uses a caller-supplied target, and `target = Nothing` also
skips it. `Strong` selects a target from the model:

```haskell
waitIfNeeded metrics Strong readModel = do
  target <- case readModel ^. #strongScope of
    EntireLog -> storeHeadPosition
    CategoryHead category -> categoryHeadPosition category
  waitFor metrics (defaultStrongWaitOptions & #target ?~ target) readModel target
```

`EntireLog` reads the `$all` head. `CategoryHead` finds the maximum `$all` membership position whose
original stream belongs to the chosen category:

```sql
SELECT COALESCE(max(se.stream_version), 0)
FROM streams s
JOIN stream_events se
  ON se.original_stream_id = s.stream_id
 AND se.stream_id = 0
WHERE s.category = $1;
```

The `$all` membership `stream_version` is the global position. This target advances only when the
category does, so unrelated traffic cannot hold a category worker behind. That matters because
Kiroku currently does not advance category checkpoints on empty fetches.

`waitFor` polls the minimum `subscriptions.last_seen` for `subscriptionName`. Defaults are five
seconds with a 10 ms poll. On deadline it records `keiro.projection.wait.timeouts` and returns the
target plus last observed position.

## Registry status gates rebuilds

```haskell
data ReadModelStatus
  = Live | Rebuilding | Paused | Abandoned | UnknownStatus Text
```

Only `Live` passes `ensureReadModel`. Unknown database text remains visible as `UnknownStatus raw`
instead of failing during decoding. The next chapter shows how `startRebuild` first transitions this
row to `Rebuilding`, making the same registry row gate queries and fence concurrent live writers.
