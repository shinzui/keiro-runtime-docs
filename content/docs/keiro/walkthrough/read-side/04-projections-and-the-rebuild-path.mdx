## Inline writes ride the append transaction

```haskell
data InlineProjection co = InlineProjection
  { name  :: Text
  , apply :: co -> RecordedEvent -> Tx.Transaction ()
  }
```

`runCommandWithProjections` delegates to `runCommandWithSqlEvents`, which requires
`KirokuStoreResource`. Kiroku enriches events before the callback observes them; then every
projection receives every emitted `(decoded event, RecordedEvent)` pair inside the append
transaction. A failed application update aborts the events as well.

This is why an inline model normally uses `Eventual`: after successful return, both durable facts are
already committed. Its table SQL is still application-owned and should use
`qualifiedTableName`/`qualifyTable`.

## Async definitions name the fenced model

```haskell
data AsyncProjection = AsyncProjection
  { name             :: Text
  , readModelName    :: Text
  , subscriptionName :: Text
  , applyRecorded    :: RecordedEvent -> Tx.Transaction ()
  , idempotencyKey   :: RecordedEvent -> EventId
  }

data AsyncApplyOutcome
  = AsyncApplied
  | AsyncDuplicate
  | AsyncFenced
```

The added `readModelName` connects a writer to the registry row from chapter 03. The application
still owns polling, batching, and acknowledgement, but it no longer has to invent its own dedup or
rebuild race protocol.

## `applyAsyncProjection`: fence and dedup in one transaction

The live entry point first locks the registry status:

```sql
SELECT status
FROM keiro.keiro_read_models
WHERE name = $1
FOR SHARE;
```

Only a present `live` row continues. The unfenced inner operation inserts the projection/event key;
on a new key it runs `applyRecorded` and returns `AsyncApplied`, while a conflict returns
`AsyncDuplicate`. A missing or non-live row returns `AsyncFenced` before either dedup or model data
is touched.

The worker contract follows directly:

```haskell
case outcome of
  AsyncApplied   -> acknowledgeAndCheckpoint
  AsyncDuplicate -> acknowledgeAndCheckpoint
  AsyncFenced    -> failOrParkWithoutCheckpoint
```

Checkpointing `AsyncFenced` would skip the event while a rebuild owns the table. Kiroku's
acknowledgement-coupled delivery preserves the position when the handler does not acknowledge
success.

## `startRebuild`: establish ownership before destruction

```haskell
startRebuild ::
  ReadModel q r -> [Text] -> GlobalPosition -> Eff es ReadModelMetadata
```

One transaction performs these operations in order:

1. transition the registry row to `Rebuilding`;
2. `TRUNCATE TABLE` using `qualifiedTableName`;
3. delete dedup rows for the named async projections; and
4. reset all members of the model subscription to `replayFrom`.

The transition locks the same registry row used by live `FOR SHARE` checks. A live writer cannot
straddle the truncate: it commits before the exclusive rebuild transition, or resumes after it and
returns `AsyncFenced`. The checkpoint reset uses SQL directly because Kiroku's public checkpoint
save is monotonic and cannot move backward.

## Replay has one designated bypass

```haskell
applyAsyncProjectionUnfenced ::
  AsyncProjection -> RecordedEvent -> Tx.Transaction AsyncApplyOutcome
```

The rebuilder uses this entry point while status is `Rebuilding`. It retains the normal dedup insert
and therefore returns `AsyncApplied` or `AsyncDuplicate`; it bypasses only the live registry fence.
Live workers must never use it.

Keiro does not select or stream replay events for the application. The rebuilder owns fetching,
batching, cursor advancement, and domain verification of the reconstructed table.

## `finishRebuild`: guard promotion in the transition transaction

```haskell
data RebuildError
  = RebuildProducedNoApplies Text GlobalPosition

finishRebuild ::
  ReadModel q r -> [Text] -> GlobalPosition
  -> Eff es (Either RebuildError ReadModelMetadata)
```

For a non-empty projection-name list, `finishRebuild` reads the log head and counts the named dedup
rows created after `startRebuild` cleared them. If the head is beyond `replayFrom` and the count is
zero, it returns `RebuildProducedNoApplies` and leaves the row `Rebuilding`. Otherwise it transitions
to `Live` in the same transaction. Passing `[]` denotes an inline-only model and skips the dedup
guard.

The check catches a completely inert async replay; it is not proof that every relevant event was
applied. Operators must verify the table and cursor before calling it.

<Callout type="warn">
  `rebuild` and `promote` are low-level status-only functions. `promote` bypasses the zero-apply
  guard. The supported lifecycle is `startRebuild`, unfenced replay, verification, then
  `finishRebuild`; use `abandonRebuild` when recovery cannot safely continue.
</Callout>
