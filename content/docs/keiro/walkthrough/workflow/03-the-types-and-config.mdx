## `ProcessManager` — the configuration record

The whole engine is configured by one record with **seven** fields:

```haskell
-- keiro/src/Keiro/ProcessManager.hs
data ProcessManager input phi rs s ci co targetPhi targetRs targetState targetCi targetCo = ProcessManager
  { name              :: !Text
  , correlate         :: !(input -> Text)
  , eventStream       :: !(EventStream phi rs s ci co)
  , streamFor         :: !(Text -> Stream (EventStream phi rs s ci co))
  , targetEventStream :: !(EventStream targetPhi targetRs targetState targetCi targetCo)
  , targetProjections :: !(Stream targetCi -> [InlineProjection targetCo])
  , handle            :: !(input -> ProcessManagerAction ci targetCi)
  }
```

<TypeTable
  type={{
    name: { type: "Text", description: "Stable identity; part of every deterministic write id. jitsurei: \"jitsurei-fulfillment\" / \"jitsurei-escalation\"." },
    correlate: { type: "input -> Text", description: "Derives the correlation key, selecting which manager instance handles the event. jitsurei: orderIdText . eventOrderId / incidentIdText . escalationInputIncidentId." },
    eventStream: { type: "EventStream phi rs s ci co", description: "The manager's OWN EventStream; its events record the saga's progress. jitsurei: fulfillmentEventStream / escalationEventStream." },
    streamFor: { type: "Text -> Stream (EventStream phi rs s ci co)", description: "Maps a correlation key to the manager's Stream handle. jitsurei: fulfillmentStream . OrderId (fulfillment-<id>) / escalationStream . IncidentId (esc-<id>)." },
    targetEventStream: { type: "EventStream targetPhi targetRs targetState targetCi targetCo", description: "The aggregate dispatched commands are sent to. jitsurei: orderEventStream / incidentEventStream." },
    targetProjections: { type: "Stream targetCi -> [InlineProjection targetCo]", description: "Inline projections run in the same transaction as each dispatched command's append. jitsurei: const [orderSummaryInlineProjection] / const []." },
    handle: { type: "input -> ProcessManagerAction ci targetCi", description: "The pure reaction: the manager-state command, the target commands, and the timers to schedule." },
  }}
/>

The eleven type parameters split into the **manager's own** aggregate (`phi rs s ci co`: its
predicate algebra, registers, state, command, event) and the **target** aggregate (`targetPhi
targetRs targetState targetCi targetCo`), plus the `input` event type the manager reacts to. You
never spell them by hand; jitsurei fixes them with a `type` alias (e.g.
`FulfillmentProcessManager`).

### The jitsurei record literal

To see real values flow into all seven fields, here is `fulfillmentProcessManager` verbatim:

```haskell
-- keiro/jitsurei/src/Jitsurei/FulfillmentProcess.hs
fulfillmentProcessManager :: FulfillmentProcessManager
fulfillmentProcessManager = ProcessManager
  { name = "jitsurei-fulfillment"
  , correlate = orderIdText . eventOrderId
  , eventStream = fulfillmentEventStream
  , streamFor = fulfillmentStream . OrderId
  , targetEventStream = orderEventStream
  , targetProjections = const [orderSummaryInlineProjection]
  , handle = \event ->
      let orderId = eventOrderId event
          status = fulfillmentStatus event
       in ProcessManagerAction
            { command =
                ObserveFulfillmentEvent ObserveFulfillmentEventData
                  { orderId = orderId
                  , status = status
                  }
            , commands =
                case event of
                  PaymentApproved{} ->
                    [ PMCommand
                        { target = orderCommandStream orderId
                        , command = MarkPacked (MarkPackedData orderId)
                        }
                    ]
                  _ -> []
            , timers = []
            }
  }
```

Note how `handle` **always** advances the manager's own state (`command =
ObserveFulfillmentEvent …`) and, **only on `PaymentApproved`**, dispatches one `MarkPacked` to the
order; it schedules no timers. The escalation manager is the mirror image: it schedules a timer on
`IncidentReported` and dispatches a command on `ResponderAcked` (see
[chapter 05](/docs/keiro/walkthrough/workflow/05-the-timer-worker)).

## `PMCommandResult` — the outcome of one dispatch

```haskell
-- keiro/src/Keiro/ProcessManager.hs
data PMCommandResult target
  = PMCommandAppended  !(CommandResult target)  -- the dispatch appended events
  | PMCommandDuplicate !EventId                 -- idempotent replay; id already existed
  | PMCommandFailed    !CommandError            -- worker classifies transient retry vs deterministic halt
```

<TypeTable
  type={{
    PMCommandAppended: { type: "CommandResult target", description: "The dispatch appended events; carries the command's result. The happy path." },
    PMCommandDuplicate: { type: "EventId", description: "The command id already existed (pre-check hit or a DuplicateEvent) — an idempotent no-op." },
    PMCommandFailed: { type: "CommandError", description: "Any other CommandError. Transient failures retry; deterministic failures halt. The PMCommandFailed hazard (chapter 02): make target commands total to avoid benign deterministic failures." },
  }}
/>

## `PMStateResult` — the outcome of the manager append

```haskell
-- keiro/src/Keiro/ProcessManager.hs
data PMStateResult target
  = PMStateAppended  !(CommandResult target)
  | PMStateDuplicate !EventId
```

<TypeTable
  type={{
    PMStateAppended: { type: "CommandResult target", description: "The manager-state append landed; carries its result." },
    PMStateDuplicate: { type: "EventId", description: "The manager append id already existed — a benign replay. finish still runs the whole dispatch loop." },
  }}
/>

There is **no failure constructor**. A manager-state append that genuinely errors does not become
a `PMStateResult`; it aborts the whole reaction via the outer `Left CommandError`
([chapter 01](/docs/keiro/walkthrough/workflow/01-the-process-manager-dispatch-loop)).

## The timer types

### `TimerId`

```haskell
-- keiro/src/Keiro/Timer/Types.hs
newtype TimerId = TimerId UUID
```

A caller-chosen UUID. Because the schedule SQL upserts on `timer_id`, **choosing the id
deterministically is what makes scheduling idempotent** — jitsurei's `escalationTimerRequest`
derives it as a UUIDv5 of the incident id, so a redelivered `IncidentReported` re-arms the *same*
row instead of duplicating it.
