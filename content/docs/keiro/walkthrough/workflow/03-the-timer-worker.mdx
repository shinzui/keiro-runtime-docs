This chapter reads `runTimerWorker` in `keiro/src/Keiro/Timer.hs` тАФ the bare worker that turns a due
timer into a side effect. Read [02 тАФ The timer
schema](/docs/keiro/walkthrough/workflow/02-the-timer-schema) first.

## `runTimerWorker`: claim one, fire, maybe mark fired

```haskell
-- keiro/src/Keiro/Timer.hs
runTimerWorker ::
  (Store :> es) =>
  UTCTime ->
  (TimerRow -> Eff es (Maybe EventId)) ->
  Eff es (Maybe TimerRow)
runTimerWorker now fire = do
  due <- claimDueTimer now
  case due of
    Nothing -> pure Nothing
    Just timer -> do
      fired <- fire timer
      for_ fired (markTimerFired (timer ^. #timerId))
      pure (Just timer)
```

The entire worker is eleven lines, and that is the point. It:

1. **claims at most one** due timer with `claimDueTimer now` (the `SKIP LOCKED` CTE from chapter 02);
2. returns `Nothing` immediately when nothing is due тАФ the caller's loop then sleeps;
3. otherwise runs the caller's `fire timer` action; and
4. calls `markTimerFired` **only when `fire` returned `Just eventId`** тАФ `for_ fired` is a no-op on
   `Nothing`.

There is no loop, no clock (the caller passes `now`), and no supervisor. A `fire` that returns
`Nothing`, or a crash before `markTimerFired`, leaves the row in `firing` тАФ and a `firing` row is
re-claimable on a later pass, which is precisely the [at-least-once
guarantee](/docs/keiro/explanation/durable-timers#at-least-once-firing).

## How `fire` composes with the command cycle

`fire :: TimerRow -> Eff es (Maybe EventId)` is where a timer rejoins the rest of keiro. The typical
shape reads the row's routing fields, dispatches a command through the ordinary command cycle, and
returns the produced event id so the timer is marked `Fired`. The jitsurei escalation worker
(`Jitsurei/EscalationProcess.hs`) is the canonical example:

```haskell
runEscalationTimerWorker options now =
  runTimerWorker now $ \timer ->
    case incidentIdFromTimer timer of
      Nothing         -> pure Nothing
      Just incidentId -> do
        _ <- runCommand options incidentEventStream (incidentStream incidentId)
               (EscalateIncident (EscalateIncidentData incidentId))
        pure (Just firedEventId)
```

Two things compose here. First, **routing**: `incidentIdFromTimer` reads `timer ^.
#processManagerName` and `timer ^. #correlationId` to recover the incident id тАФ the timer carries its
own routing, so the worker needs no extra state. Second, **idempotency**: the dispatched
`EscalateIncident` runs through `runCommand`, and the incident aggregate accepts it only from
`Triaging` (chapter on [keeping commands
total](/docs/keiro/how-to/keep-target-commands-total)). So if the incident was already acknowledged,
the escalate is a benign rejection тАФ and because the worker returns `Just firedEventId` regardless,
the timer is still marked `Fired` and will not be re-claimed. The at-least-once worker and the total
target command together make the whole escalation race safe without any locking beyond `SKIP LOCKED`.

## What you have seen

Across the tour: `runProcessManagerOnce` reacts to an event by appending manager state and timers in
one transaction and dispatching each command in its own, all under deterministic ids; the timer
schema claims due rows atomically with `FOR UPDATE SKIP LOCKED` and re-arms only while scheduled; and
`runTimerWorker` fires one claimed timer per tick, marking it fired only on success. That is keiro's
entire shipped workflow engine тАФ process managers and durable timers, built from the same command
cycle and Postgres substrate as everything else.
