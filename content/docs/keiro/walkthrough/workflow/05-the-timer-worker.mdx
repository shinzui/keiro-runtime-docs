The last chapter. [Chapter 04](/docs/keiro/walkthrough/workflow/04-the-timer-schema) walked the
`keiro_timers` storage; here we walk the **worker** that drives it ‚Äî `runTimerWorker` in
`keiro/src/Keiro/Timer.hs` ‚Äî and show how its `fire` action rejoins the command cycle, using
jitsurei's escalation and payment-timeout workers as the worked example.

## `runTimerWorkerWith` ‚Äî the whole body

`runTimerWorker` is now a one-liner over `runTimerWorkerWith`, which carries stale-row recovery,
metrics recording, and the optional attempt ceiling:

```haskell
-- keiro/src/Keiro/Timer.hs
runTimerWorkerWith metrics options now fire = do
  for_ (options ^. #requeueStuckAfter) $ \ttl -> do
    requeued <- requeueStuckTimers ttl now
    recordTimerRequeued metrics (fromIntegral requeued)
  backlog <- countDueTimers now
  recordTimerBacklog metrics (fromIntegral backlog)
  stuck <- countStuckTimers now anyStuckTimer
  recordTimerStuck metrics (fromIntegral stuck)
  due <- claimDueTimer now
  case due of
    Nothing -> pure Nothing
    Just timer -> do
      recordTimerFireLag metrics (realToFrac (now `diffUTCTime` (timer ^. #fireAt)) * 1000)
      recordTimerAttempts metrics (fromIntegral (timer ^. #attempts))
      case options ^. #maxAttempts of
        Just attemptCeiling
          | (timer ^. #attempts) > attemptCeiling -> do
              _ <-
                deadLetterTimer
                  (timer ^. #timerId)
                  ("timer exceeded attempt ceiling of " <> Text.pack (show attemptCeiling))
              pure (Just timer)
        _ -> do
          fired <- fire timer
          for_ fired (markTimerFired (timer ^. #timerId))
          pure (Just timer)

runTimerWorker metrics = runTimerWorkerWith metrics defaultTimerWorkerOptions
```

The signatures:

```haskell
-- keiro/src/Keiro/Timer.hs
runTimerWorkerWith ::
  (IOE :> es, Store :> es) =>
  Maybe KeiroMetrics -> TimerWorkerOptions -> UTCTime ->
  (TimerRow -> Eff es (Maybe EventId)) -> Eff es (Maybe TimerRow)

runTimerWorker ::
  (IOE :> es, Store :> es) =>
  Maybe KeiroMetrics -> UTCTime ->
  (TimerRow -> Eff es (Maybe EventId)) -> Eff es (Maybe TimerRow)
```

Line by line:

1. **Stale-row recovery.** When `requeueStuckAfter = Just ttl`, the pass first moves every `Firing`
   row whose `updated_at` is at least `ttl` old back to `Scheduled`, and records the count in
   `keiro.timer.requeued`. The default is `Just 300`; `Nothing` disables this automatic recovery.
2. **The two gauges, once per pass.** Before claiming, the worker reads `countDueTimers` (the
   backlog) and `countStuckTimers now anyStuckTimer` (rows stranded in `firing` by earlier passes)
   and records each on the `Maybe KeiroMetrics` handle as `keiro.timer.backlog` and
   `keiro.timer.stuck`. Under a `Nothing` handle every `record*` call is a no-op, so an unmetered
   worker pays only for the two extra counts. (See the
   [telemetry reference](/docs/keiro/reference/telemetry#metric-catalogue).)
3. **`due <- claimDueTimer now`** ‚Äî claim **at most one** due timer using the
   `FOR UPDATE SKIP LOCKED` CTE from
   [chapter 04](/docs/keiro/walkthrough/workflow/04-the-timer-schema). This atomically moves the
   claimed row to `firing` and bumps `attempts`.
4. **`Nothing -> pure Nothing`** ‚Äî nothing was due. The worker returns immediately; the caller's
   loop typically sleeps until the next tick.
5. **The two histograms, per claimed timer.** `keiro.timer.fire.lag` is `(now ‚àí fire_at)` in
   **milliseconds** (the `* 1000` scales the seconds `diffUTCTime` yields; the lag is non-negative
   because only `fire_at <= now` rows are claimable), and `keiro.timer.attempts` is the **post-claim**
   attempt count.
6. **The attempt-ceiling branch.** If `options.maxAttempts = Just n` and the post-claim `attempts`
   **exceeds** `n`, the worker calls `deadLetterTimer` ‚Äî moving the row to the terminal `dead` state
   with an explanatory `last_error` ‚Äî and **does not** run `fire`. With `defaultTimerWorkerOptions`
   (`maxAttempts = Nothing`) this branch never fires and behavior is the historical claim-fire-mark.
7. **`fired <- fire timer`** ‚Äî otherwise hand the whole `TimerRow` (with its `processManagerName`,
   `correlationId`, and `payload`) to the caller-supplied `fire`. `fire` returns `Maybe EventId`:
   `Just` the id of the event its firing produced, or `Nothing` if it did nothing.
8. **`for_ fired (markTimerFired (timer ^. #timerId))`** ‚Äî `markTimerFired` runs **only when
   `fire` returned `Just`**. `for_` over a `Maybe` is a no-op on `Nothing`, so a `fire` that
   returns `Nothing` leaves the row `firing` and does **not** mark it `fired`.
9. **`pure (Just timer)`** ‚Äî return the claimed row (so the caller can log/inspect it),
   regardless of whether `fire` produced an event or the row was dead-lettered.

<Callout type="info">
  Because `claimDueTimer` bumps `attempts` **before** the ceiling check, the comparison sees the
  post-claim count: `maxAttempts = Just 0` dead-letters on the very first claim; `Just 2` on the
  third. This is the bounded counterpart to at-least-once firing ‚Äî a timer whose `fire` never
  succeeds is parked in `dead` instead of being re-claimed forever.
</Callout>

## At-least-once firing

The `Just`-marks-fired rule (step 8) is exactly what gives **at-least-once** semantics:

- if `fire` returns `Nothing`, the row stays `firing` and is never marked `fired`;
- if the process **crashes** after the claim but before `markTimerFired`, the row is left `firing`
  too.

In both cases the work was not durably completed, so the configured stale-row sweep (five minutes by
default) puts the row back to `scheduled` and it becomes claimable again ‚Äî the timer fires again
rather than being silently lost. A `fire` action that runs longer than `requeueStuckAfter` can overlap
that recovery and be invoked concurrently. The cost is that a `fire` action may run **more than once**
for the same logical timer, so
**`fire` must be idempotent** (the firing typically dispatches a command through the command
cycle, where the target's totality guard absorbs the repeat ‚Äî below). For the conceptual treatment
see [Understanding durable timers](/docs/keiro/explanation/durable-timers).

## The bare-worker reality

<Callout type="info">
  `runTimerWorker` is **bare by design**: no loop, no clock, and no supervisor. It claims and
  fires **one** timer for the `now` you pass and returns. The *caller* owns the schedule ‚Äî it
  decides `now`, how often to tick, and how to back off when `claimDueTimer` returns `Nothing`.
  See [drive the timer worker](/docs/keiro/how-to/drive-the-timer-worker) for the polling loop you
  wrap around it.
</Callout>

This is why the signature takes `now :: UTCTime` rather than reading a clock, and why it returns
after one timer rather than looping: keiro ships the *mechanism* (claim-fire-mark) and leaves the
*policy* (cadence, concurrency, backoff) to you. Running several copies of the loop concurrently is
safe ‚Äî `FOR UPDATE SKIP LOCKED` guarantees they claim disjoint timers.

## How `fire` rejoins the command cycle

The `fire` action is where a timer rejoins the rest of keiro. jitsurei shows two shapes.

**The full shape ‚Äî `runEscalationTimerWorker`.** When an escalation timer fires, the worker
dispatches `EscalateIncident` to the incident aggregate through the command cycle:

```haskell
-- keiro/jitsurei/src/Jitsurei/EscalationProcess.hs
runEscalationTimerWorker options now =
  runTimerWorker Nothing now $ \timer ->
    case incidentIdFromTimer timer of
      Nothing -> pure Nothing
      Just incidentId -> do
        _ <-
          runCommand
            options
            incidentEventStream
            (incidentStream incidentId)
            (EscalateIncident (EscalateIncidentData {incidentId = incidentId}))
        pure (Just (EventId (namedUuid ("jitsurei-escalation-fired:" <> incidentIdText incidentId))))
```

`incidentIdFromTimer` reconstructs the target from the timer's own `processManagerName` and
`correlationId` ‚Äî the worker needs **no extra state**, because the routing fields ride along on the
row. It then dispatches the command via `runCommand` and returns a **deterministic** fired event
id (a UUIDv5 of the incident id), so `markTimerFired` records a stable id even across retries.

**The minimal shape ‚Äî `runPaymentTimeoutWorker`.** The smallest possible `fire` just returns a
fixed `Just`:

```haskell
-- keiro/jitsurei/src/Jitsurei/Timers.hs
runPaymentTimeoutWorker now =
  runTimerWorker Nothing now (\_ -> pure (Just (EventId paymentTimeoutEventId)))
```

(Both pass `Nothing` for the metrics handle; wire `Just metrics` to populate the `keiro.timer.*`
instruments.)

It ignores the row and always reports `Just`, so the timer is marked `fired` on the first
successful pass. (`paymentTimeoutRequest` builds the matching `TimerRequest` for
`"jitsurei-fulfillment"`.)

Two properties to take away:

- **Routing.** The timer carries its own `processManagerName`, `correlationId`, and `payload`, so
  `fire` can address the dispatch with nothing but the claimed `TimerRow` ‚Äî no side table, no
  in-memory map.
- **Idempotency.** Because `fire` dispatches through the command cycle, the target's totality
  guard makes a *late* or *repeated* firing safe. In jitsurei, `EscalateIncident` is legal only
  from `Triaging`; if the incident was already acknowledged, a late escalate is a benign
  `CommandRejected`, so the escalation-timer-vs-ack race is safe and the at-least-once retry is
  harmless. This is the same discipline as the dispatch loop ‚Äî see
  [keep target commands total](/docs/keiro/how-to/keep-target-commands-total).

## Recap

You have now read keiro's workflow engine end to end. A process manager reacts to one event by
appending its own state and arming its timers in **one** transaction
([chapter 02](/docs/keiro/walkthrough/workflow/02-the-transaction-model)) and dispatching each
target command in **its own** ([chapter 01](/docs/keiro/walkthrough/workflow/01-the-process-manager-dispatch-loop)) ‚Äî
crash-safe not through atomicity but through **deterministic write ids** that make replay a no-op.
A separate, caller-driven worker claims due timers with `FOR UPDATE SKIP LOCKED`
([chapter 04](/docs/keiro/walkthrough/workflow/04-the-timer-schema)) and fires them at-least-once,
the firing rejoining the command cycle through a `fire` action you supply. The worker now also
records the `keiro.timer.*` metrics each pass and can bound runaway timers with
`maxAttempts`/`deadLetterTimer`; its default also requeues a stale `Firing` row after five minutes.
The remaining honest gaps: no backoff between claims, and the worker is still bare (no loop, clock,
or supervisor) ‚Äî but cancellation and dead-lettering now ship as the
[recovery API](/docs/keiro/reference/timers#recovery-operations).
