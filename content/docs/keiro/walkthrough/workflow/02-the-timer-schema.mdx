This chapter reads the `keiro_timers` storage in `keiro/src/Keiro/Timer/Schema.hs` — the SQL that
makes durable timers safe under many concurrent workers. Read [00 — Start
here](/docs/keiro/walkthrough/workflow/00-start-here) first.

## `scheduleTimerStmt`: insert, or re-arm only while still scheduled

```sql
-- keiro/src/Keiro/Timer/Schema.hs (scheduleTimerStmt)
INSERT INTO keiro_timers
  (timer_id, process_manager_name, correlation_id, fire_at, payload, status)
VALUES
  ($1, $2, $3, $4, $5, $6)
ON CONFLICT (timer_id) DO UPDATE
  SET process_manager_name = EXCLUDED.process_manager_name,
      correlation_id = EXCLUDED.correlation_id,
      fire_at = EXCLUDED.fire_at,
      payload = EXCLUDED.payload,
      status = EXCLUDED.status,
      updated_at = now()
  WHERE keiro_timers.status = 'scheduled'
```

`scheduleTimerTx` always inserts with `status = 'scheduled'`. The interesting part is the conflict
clause: `ON CONFLICT (timer_id) DO UPDATE … WHERE keiro_timers.status = 'scheduled'`. The `WHERE`
guard means a conflicting row is re-armed **only while it is still `Scheduled`**. A timer that has
already moved to `firing`, `fired`, or `cancelled` is *not* resurrected by a re-schedule — the
`DO UPDATE` simply matches no row. This is what makes "reschedule the same `timerId`" idempotent and
safe: a redelivered `IncidentReported` re-arms a pending escalation timer but cannot un-fire one that
already fired.

Because `scheduleTimerTx` returns a `Tx.Transaction ()`, it composes into the process manager's
manager-append transaction (chapter 01) rather than running on its own connection.

## `claimDueTimerStmt`: the atomic claim

```sql
-- keiro/src/Keiro/Timer/Schema.hs (claimDueTimerStmt)
WITH due AS (
  SELECT timer_id
  FROM keiro_timers
  WHERE status = 'scheduled'
    AND fire_at <= $1
  ORDER BY fire_at, timer_id
  LIMIT 1
  FOR UPDATE SKIP LOCKED
)
UPDATE keiro_timers kt
SET status = 'firing',
    attempts = kt.attempts + 1,
    updated_at = now()
FROM due
WHERE kt.timer_id = due.timer_id
RETURNING kt.timer_id, kt.process_manager_name, kt.correlation_id, kt.fire_at,
  kt.payload, kt.status, kt.attempts, kt.fired_event_id
```

The `due` CTE selects the single earliest claimable timer — `status = 'scheduled' AND fire_at <= now`
— ordered by `(fire_at, timer_id)` for a stable, fair pick, `LIMIT 1`, and locked with
`FOR UPDATE SKIP LOCKED`. `SKIP LOCKED` is the whole trick: a concurrent worker's transaction skips
any row already locked by another claim and takes the next one, so two workers never claim the same
timer and no worker blocks waiting on another. The outer `UPDATE … FROM due` flips the claimed row to
`firing` and bumps `attempts` in the same statement, then `RETURNING` hands back the full row. The
`keiro_timers_due_idx (status, fire_at, process_manager_name)` index is what keeps this claim cheap.

## `timerRowDecoder` and the status fallback

```haskell
timerRowDecoder :: D.Row TimerRow
timerRowDecoder =
  TimerRow
    <$> (TimerId <$> D.column (D.nonNullable D.uuid))
    <*> D.column (D.nonNullable D.text)
    <*> D.column (D.nonNullable D.text)
    <*> D.column (D.nonNullable D.timestamptz)
    <*> D.column (D.nonNullable D.jsonb)
    <*> (statusFromText <$> D.column (D.nonNullable D.text))
    <*> (fromIntegral <$> D.column (D.nonNullable D.int8))
    <*> (fmap EventId <$> D.column (D.nullable D.uuid))

statusFromText :: Text -> TimerStatus
statusFromText = \case
  "scheduled" -> Scheduled
  "firing"    -> Firing
  "fired"     -> Fired
  "cancelled" -> Cancelled
  _           -> Cancelled
```

The decoder maps the row's columns onto `TimerRow` in order. Note `statusFromText`: the four known
strings map to their constructors, and **any unrecognized string falls back to `Cancelled`**. So
`Cancelled` is doing double duty — it is both the "withdrawn before firing" state and the
defensive decode fallback for an unexpected stored value. (There is no `cancelTimer` SQL in keiro
0.1.0.0, so in practice `cancelled` rows arise only from this fallback or a future migration.)
