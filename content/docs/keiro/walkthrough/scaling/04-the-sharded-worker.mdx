The [lease layer](/docs/keiro/walkthrough/scaling/03-the-shard-lease) decides which buckets this
process currently owns. `Keiro.Subscription.Shard.Worker` turns that set into Kiroku readers without
ever acknowledging an in-flight event merely because ownership changed.

## One reconcile pass

`reconcileShardsOnce` is the testable ownership unit:

```haskell
reconcileShardsOnce
  :: KirokuStore
  -> ShardLease
  -> ShardedWorkerOptions
  -> IORef (Map Int RunningReader)
  -> ShardEventHandler
  -> IO (Set Int)
```

It estimates live owners, calls `acquireOwnedBuckets`, sheds buckets above the fair share, then diffs
the owned set against the running-reader map. New buckets start readers; lost buckets run
`stopReader`.

Two failure choices are deliberate:

- an ownership-snapshot error reports `ShardSnapshotFailed` and estimates from an empty snapshot;
- an acquisition error reports `ShardAcquireFailed` and preserves the **previous** local ownership
  set rather than pretending every bucket was lost.

A reader that ends unexpectedly removes itself from the map and reports `ShardReaderDied`; a later
pass can restart it while the lease remains owned. Ordinary synchronous handler exceptions do not
kill the reader, because they enter the delivery retry path below.

## `startReader` couples handler outcome to Kiroku

Each bucket becomes a Kiroku consumer-group member. The important fields are the member, the bounded
retry policy, and the bridge choice:

```haskell
let subConfig =
      (defaultSubscriptionConfig subName target (\_ -> pure Continue))
        { Sub.batchSize = opts ^. #batchSize
        , Sub.consumerGroup =
            Just (ConsumerGroup
              { member = fromIntegral bucket
              , size = fromIntegral (opts ^. #shardCount)
              })
        , Sub.retryPolicy = opts ^. #retryPolicy
        }

(stream, cancelAction) <-
  subscriptionAckStream store subConfig (opts ^. #bufferSize)
```

`subscriptionAckStream` yields an `AckItem` and waits on its one-shot reply. Keiro presents a smaller
domain to the application:

```haskell
data ShardDelivery = ShardDelivery
  { event   :: RecordedEvent
  , attempt :: Word
  , bucket  :: Int
  }

data ShardAck
  = ShardAckOk
  | ShardAckRetry RetryDelay
  | ShardAckDeadLetter DeadLetterReason
```

`attempt` comes from the bridge and is zero-based. `toSubscriptionResult` maps the three
constructors to Kiroku's `Continue`, `Retry`, and `DeadLetter`.

Before replying, `handleItem` distinguishes exceptions:

```haskell
handlerException err =
  case fromException err of
    Just async -> throwIO (async :: SomeAsyncException)
    Nothing    -> pure (ShardAckRetry (opts ^. #handlerRetryDelay))
```

A synchronous exception becomes bounded retry. An asynchronous exception from cancellation,
shedding, or shutdown escapes without filling the reply. Kiroku therefore cannot checkpoint an
event merely because Keiro stopped its reader.

## The Kiroku disposition ladder

Kiroku owns the durable consequences:

```text
ShardAckOk
  -> Continue
  -> continue through the batch
  -> save the member checkpoint at the batch tail

ShardAckRetry delay
  -> Retry delay
  -> leave checkpoint before this event
  -> redeliver with attempt + 1

ShardAckDeadLetter reason
  -> DeadLetter reason
  -> INSERT kiroku.dead_letters + checkpoint advance in one statement

Retry on the final allowed delivery
  -> DeadLetterMaxAttempts totalDeliveries
  -> the same atomic insert-and-advance
```

If the atomic dead-letter statement fails, Kiroku throws and leaves the checkpoint unadvanced. The
reader may die and be restarted, but the event was not silently skipped.

Successful events before a batch tail may replay after a crash. That is expected at-least-once
behavior, so application effects still need event-ID idempotency even when no rebalance occurs.

## Rebalance while a handler is running

When reconciliation sheds a bucket, `stopReader` cancels the Kiroku subscription and kills its drain
thread. If its handler had not replied, there is no acknowledgment. The new owner starts from the
member checkpoint and sees that event again.

A stale worker can also miss renewals and keep reading after its lease expires. The new owner may
then deliver the same event concurrently. Kiroku checkpoint updates are monotonic, so the stale
worker cannot move the cursor backward; only an idempotent handler prevents duplicated effects.

This is the tested acceptance boundary: a batch-tail handler killed in flight is redelivered, a
bucket shed mid-drain loses no event, and a zombie overlap may duplicate delivery without losing
progress.

## Loop and compatibility wrapper

`runShardedSubscriptionGroupAck` mints a worker ID, calls `ensureShards`, and runs reconciliation
every `renewInterval`. Its `finally` stops current readers and calls `relinquish`, so graceful
shutdown exposes buckets immediately; a process crash falls back to lease expiry.

`runShardedSubscriptionGroup` wraps the explicit API:

```haskell
runShardedSubscriptionGroup store subName opts handler =
  runShardedSubscriptionGroupAck store subName opts $ \delivery -> do
    handler (delivery ^. #event)
    pure ShardAckOk
```

It remains source-compatible for always-successful handlers. Escaping synchronous exceptions are
still retried, but only the explicit entry point can return an immediate poison dead letter.

## The rebalance signal remains a seam

The delay between reconcile passes is a poll. `Keiro.Wake` signals Kiroku event appends, not shard
claims or releases. A dedicated `keiro_shard_rebalance` notification could shorten voluntary
handoff, but it would be a latency optimization; lease expiry and the bounded poll remain the
durable recovery path.
