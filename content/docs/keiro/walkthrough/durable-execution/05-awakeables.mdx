`keiro/src/Keiro/Workflow/Awakeable.hs` is the second wake source: a durable promise an **external**
system resolves — a webhook, a human approver, an LLM tool call. The workflow allocates an id, hands
it out, and suspends; an external `signalAwakeable` records the result and the next run continues.
Where a sleep's wake source is a timer the framework drains, an awakeable's wake source is *someone
else's code*, which puts the weight on handing out the right opaque id and making the signal
crash-safe.

## The journaled id

```haskell
newtype AwakeableId = AwakeableId UUID
  deriving stock (Eq, Show, Generic)
  deriving newtype (ToJSON, FromJSON)

deterministicAwakeableId (WorkflowName name) (WorkflowId wid) label =
  AwakeableId $ UUID.V5.generateNamed UUID.V5.namespaceURL $ ...
    Text.intercalate ":" ["keiro", "awakeable", name, wid, label]
```

New allocations are random v4 UUIDs that the workflow journals under `awkid:<label>`. Replay reads
that recorded id, so the workflow reuses the exact id it already handed to the external system without
making the id guessable from `(workflow name, workflow id, label)`. The `ToJSON`/`FromJSON` instances
are over the inner UUID, so a workflow can journal the id and webhook payloads can carry it.

`deterministicAwakeableId` remains exported only as a legacy generation-0 adoption helper. If an older
in-flight workflow already registered a row under the v5 id, the first post-change allocation adopts
that row; new workflows should use the `AwakeableId` returned by `awakeableNamed`.

## `awakeableNamed`: allocate id + await

```haskell
awakeableNamed (StepName label) = do
  (name, wid) <- currentWorkflow
  gen <- currentRunGeneration
  aid <- step (StepName (awakeableAllocStepPrefix <> label)) $ do
           allocated <- allocateAwakeableId name wid gen label
           runTransaction $
             registerAwakeableTx (awakeableIdToUuid allocated)
                                 (unWorkflowName name) (unWorkflowId wid)
           pure allocated
  let stepNm = StepName (awakeableStepPrefix <> awakeableIdText aid)   -- "awk:<uuid>"
      await = awaitCancellable name wid aid stepNm
  pure (aid, await)
```

It returns the id (hand it to the external system) and an `await` action. The journaled step name is
`awk:<uuid>` — the awakeable id, not the label, so two awakeables never collide. `awakeable` is the
ordinal convenience (`ord:N` labels) the module steers you away from.

The registration lives **inside the allocation step's action**, so the `pending` row commits before
the id can be appended, returned, or handed to anyone. Registering it later — at the await arm — left
a window where an eager external system could signal an id whose row did not exist yet, and get back
"unknown id". The arm keeps its own idempotent registration as a **repair path** for historical
allocations whose journal entry exists without a row.

A crash between row registration and allocation journaling can leave an unreachable pending row.
That is acceptable: workflow GC removes awakeables by owner coordinates, and the id was never exposed.

## `awaitCancellable`: the arm that registers a pending row

```haskell
awaitCancellable name wid aid stepNm =
  awaitStep stepNm $ do
    existing <- lookupAwakeable (awakeableIdToUuid aid)
    case existing of
      Just row | row ^. #status == Cancelled -> throwIO (WorkflowAwakeableCancelled aid)
      _ -> runTransaction $ registerAwakeableTx (awakeableIdToUuid aid) (unWorkflowName name) (unWorkflowId wid)
```

This is `awaitStep` with a richer arm. Because `awaitStep` runs `arm` only on the miss path and re-runs
it on every resume until the step is journaled, the arm does two jobs each time it runs: (1) if the
awakeable was **cancelled**, throw `WorkflowAwakeableCancelled` instead of suspending forever; (2)
if a completed row already stores a payload, append the journal entry from that stored payload; (3)
otherwise (re-)insert the idempotent `pending` row and suspend. On the **hit** path (the signal already
journaled the result) the arm never runs — so a signalled-then-cancelled race still returns the
signalled value: a resolved promise cannot be un-resolved.

```haskell
newtype WorkflowAwakeableCancelled = WorkflowAwakeableCancelled AwakeableId
  deriving stock (Eq, Show)
instance Exception WorkflowAwakeableCancelled
```

The author can `catch` this for compensation, and the resume worker records the run as a failure.

## The table and its `pending`-guarded transitions

```sql
CREATE TABLE IF NOT EXISTS keiro_awakeables (
  awakeable_id        UUID PRIMARY KEY,
  owner_workflow_name TEXT NOT NULL,
  owner_workflow_id   TEXT NOT NULL,
  status              TEXT NOT NULL DEFAULT 'pending',  -- pending | completed | cancelled
  payload             JSONB,
  ...
);
```

`Keiro.Workflow.Awakeable.Schema` owns the row type and three transitions. `registerAwakeableTx` is an
`INSERT ... ON CONFLICT (awakeable_id) DO NOTHING` (the idempotent arm). Both transitions guard on
`status = 'pending'` and report whether they actually moved a row, via `rowsAffected`:

```sql
-- completeAwakeableTx
UPDATE keiro_awakeables SET status = 'completed', payload = $2, completed_at = $3, updated_at = now()
WHERE awakeable_id = $1 AND status = 'pending'
-- cancelAwakeableTx
UPDATE keiro_awakeables SET status = 'cancelled', updated_at = now()
WHERE awakeable_id = $1 AND status = 'pending'
```

So `completeAwakeableTx` returns `True` only on the genuine `pending → completed` transition; a second
signal matches no `pending` row and returns `False`.

## `signalAwakeable`: idempotent *and* crash-safe

This is the subtlest function in the module. It must both record the payload **and** append the
journal entry — two writes that a crash could split. It heals that split:

```haskell
signalAwakeable aid result =
  lookupAwakeable (awakeableIdToUuid aid) >>= \case
    Nothing  -> pure False
    Just row -> signalAwakeableFrom row result

signalAwakeableFrom row result
  | row ^. #status == Cancelled = pure False
  | otherwise = do
      now <- liftIO getCurrentTime
      let aid       = AwakeableId (row ^. #awakeableId)
          ownerName = WorkflowName (row ^. #ownerWorkflowName)
          ownerId   = WorkflowId   (row ^. #ownerWorkflowId)
          payload   = if row ^. #status == Completed then row ^. #payload else Just (toJSON result)
      gen <- currentGeneration ownerName ownerId
      case payload of
        Nothing -> pure False
        Just payloadValue -> do
          appendTx <- prepareJournalAppend ownerName ownerId gen StepRecorded { ... payloadValue ... }
          (shouldAppend, ...) <- runTransaction $ do
            transitioned <-
              if row ^. #status == Pending
                then completeAwakeableTx (awakeableIdToUuid aid) (toJSON result) now
                else pure False
            -- The read row may be stale. If the guarded UPDATE lost, re-read status
            -- *inside* this transaction: append only if another signal completed it.
            ...
```

The key insight is the middle guard. The journal entry is **(re-)appended whenever the row is
`completed`** — using the value just written, or, for an *already*-completed row whose journal append
never happened before the atomic path existed, the **stored payload**. For a pending row, the row
transition and journal append commit in one transaction; for an already-completed row, the re-append
repairs old wedges. The append path is itself idempotent (deterministic id, duplicate treated as
success), so the re-append collapses to a no-op once the entry is present. The consequence the Haddock
flags:

> A `False` return therefore does not mean "nothing happened": the journal may still have been repaired.

### Why the status is re-read inside the transaction

The row read at the top of `signalAwakeable` happens *before* the transaction opens, so it may be
stale by the time the guarded `pending → completed` UPDATE runs. Deciding whether to append from that
pre-transaction snapshot let a concurrent cancel and signal both appear to win: cancellation flipped
the row while the signal still believed it was `pending`, appended a result, and the workflow then
both fired compensation **and** received a value.

So the transaction re-reads status when its UPDATE loses, and appends in exactly three cases: it
performed the transition itself, it is repairing an already-completed row from the stored payload, or
the in-transaction read shows another *signal* completed the row. If **cancellation** won, it appends
nothing and returns `False`. Cancellation and completion stay mutually exclusive terminal outcomes
from the workflow's point of view.

`signalAwakeableFrom` is public purely as a narrow, deterministic race-test seam — it lets a test
interpose between the initial read and the guarded completion. Ordinary callers use `signalAwakeable`.

`cancelAwakeable` is the simple counterpart — flip a still-`pending` row to `cancelled`, write **no**
journal entry (there is no result to record), and let the next resume's arm throw. The pending-row
count behind the `keiro.workflow.awakeables.pending` gauge comes from `countPendingAwakeables`
(`SELECT count(*) ... WHERE status = 'pending'`), which the resume worker samples each pass.
