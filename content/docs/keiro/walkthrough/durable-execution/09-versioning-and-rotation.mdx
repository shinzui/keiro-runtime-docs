The previous chapters built journal-and-replay over named steps. This one reads the two MasterPlan-6
additions that evolve a *running* workflow: **continue-as-new** rotation (EP-48) keeps an unbounded
history bounded, and the **patch** API (EP-49) decides a cross-cutting branch once and journals it. Both
fall out of the existing model rather than bolting onto it — no new journal-event types for `patch`, one
additive marker for rotation.

```text
keiro/src/Keiro/Workflow/Types.hs    -- PatchId, workflowGenerationStreamName, WorkflowContinuedAsNew, the reserved names
keiro/src/Keiro/Workflow/Schema.hs   -- the generation column, currentGeneration, the current-gen discovery CTE
keiro/src/Keiro/Workflow.hs          -- continueAsNew/restoreSeed/patch, the ContinueAsNew/Patch handlers, rotateGeneration
```

## Generations: one logical id, many physical journals

A rotation does not lose the workflow's identity — it rotates the *physical* stream the journal lives
on. `Keiro.Workflow.Types` keeps generation 0 byte-for-byte compatible and only suffixes higher ones:

```haskell
workflowGenerationStreamName :: WorkflowName -> WorkflowId -> Int -> StreamName
workflowGenerationStreamName name wid gen
  | gen <= 0 = workflowStreamName name wid
  | otherwise =
      let StreamName base = workflowStreamName name wid
       in StreamName (base <> "#" <> Text.pack (show gen))
```

So an instance that never rotates stays on `wf:<name>-<id>` and needs zero data migration; a rotated one
lives on `wf:<name>-<id>#1`, `#2`, … The `#` is a *new* structural separator, distinct from the `:` and
`-` that `workflowStreamName` already reserves, so it cannot collide with an existing boundary. The
**logical** identity `(WorkflowName, WorkflowId)` the author and the resume registry see is stable across
rotations; only the physical stream rotates underneath it.

The step index gains a matching `generation` column ([chapter 03](/docs/keiro/walkthrough/durable-execution/03-the-step-index-and-discovery)),
and the lookup is the highest generation recorded for the logical id:

```haskell
currentGenerationStmt :: Statement (Text, Text) Int32
currentGenerationStmt =
  preparable
    """
    SELECT COALESCE(MAX(generation), 0)::int4
    FROM keiro_workflow_steps
    WHERE workflow_id = $1 AND workflow_name = $2
    """
    ...
```

`runWorkflowWith` resolves `gen <- currentGeneration name wid` **once per run** and operates only on it.
A never-rotating workflow resolves to 0, so naming, load, and append are exactly as the earlier chapters
described; a rotated one resolves to its newest generation, so discovery and resume transparently
continue there. The `gen` is threaded through every lookup (`stepExists`, `loadStepIndex`,
`recordStep`, `appendJournalTx`) and into the deterministic id, which now namespaces by generation so
`"s1"` on generation 0 and `"s1"` on generation 1 produce *different* kiroku event ids.

## Rotation: a non-returning unwind, ordered for crash-safety

`continueAsNew` mirrors the `WorkflowSuspend` machinery from [chapter 02](/docs/keiro/walkthrough/durable-execution/02-the-effect-and-replay-loop):
the handler throws a sentinel that `runWorkflowWith` catches and turns into an outcome. The result type
is fully polymorphic because control never returns to the caller within *this* run.

```haskell
ContinueAsNew seed -> throwIO (WorkflowRotate (Aeson.toJSON seed))
```

```haskell
outcome <-
  (Completed <$> runHandler)
    `catch` (\WorkflowSuspend -> pure Suspended)
    `catch` (\(WorkflowRotate seedJson) -> rotateGeneration name wid gen seedJson)
```

`rotateGeneration` is where the crash-safety lives. It appends **two** entries, and the *order* is the
whole argument:

```haskell
rotateGeneration name wid gen seedJson = do
  let nextGen = gen + 1
  now <- liftIO getCurrentTime
  -- 1. Seed step on the NEXT generation first (advances the current generation).
  seedExists <- stepExists name wid nextGen continueSeedStepName
  unless seedExists $ do
    (_, appendResult) <-
      appendJournalTx name wid nextGen (StepRecorded continueSeedStepName seedJson now)
    writeWorkflowSnapshot
      (appendResult ^. #streamId) (appendResult ^. #streamVersion)
      (Map.singleton continueSeedStepName seedJson)
  -- 2. Terminal rotation marker on the CURRENT generation (audit + closes it).
  markerExists <- stepExists name wid gen continuedAsNewStepName
  unless markerExists $
    void (appendJournalTx name wid gen (WorkflowContinuedAsNew nextGen now))
  pure ContinuedAsNew
```

The seed step is appended to generation `gen + 1` **first**. That single append advances
`MAX(generation)` — and therefore `currentGeneration` — to `gen + 1`. After it commits, any re-run
resolves the current generation to `gen + 1`, hydrates from the seed, and never re-enters generation
`gen`. So even a crash *between* the two appends converges to "continue from the seed", never re-running
the closed generation's work. Both appends are existence-guarded with deterministic, generation-namespaced
ids, so the whole rotation is idempotent. The one-entry seed map is snapshotted unconditionally (rotation
is exactly when a fresh snapshot earns its keep), so the next generation hydrates in O(1).

`restoreSeed` reads it back with no new machinery at all — it is an ordinary journaled `step`:

```haskell
restoreSeed def = step (StepName continueSeedStepName) (pure def)
```

On a generation that was rotated *into*, the seed step is already journaled, so this `step` hits it and
returns the carried value without running `pure def`. On the very first generation it misses and records
`def`. The new `WorkflowContinuedAsNew` marker is the only added journal-event constructor; it is
additive within `schemaVersion = 1` (old journals never carry the tag, so no upcaster is needed), and
`loadJournal` folds it as carrying no step result.

## Discovery scopes to the current generation

A rotated workflow is **not** finished — its work continues on the next generation — so
`findUnfinishedWorkflowIds` must treat it as unfinished. Current discovery reads the lifecycle row,
not an anti-join over step rows:

```sql
SELECT workflow_id, workflow_name
FROM keiro_workflows
WHERE status NOT IN ('completed', 'cancelled', 'failed')
  AND (wake_after IS NULL OR wake_after <= $1)
```

`__workflow_continued_as_new__` is deliberately **not** a terminal workflow status. A rotation appends
the next generation's seed step first, so `currentGeneration` moves forward while the `keiro_workflows`
row remains `running`. The resume worker therefore keeps driving a rotating workflow forward, one
generation at a time — the `ContinuedAsNew` outcome counts toward `resumed`, exactly like a
`Suspended` one.

## Patch: the step hit/miss shape, deciding from the recorded active set

`patch` reuses the `Step` hit/miss handler shape, but its *miss* path computes a decision instead of
running a user action. When a fresh workflow generation starts, `runWorkflowWith` records the deployed
`activePatches` set once under `patchSetStepName`:

```haskell
recordPatchSetIfFresh runGen initial = do
  let patches = options ^. #activePatches
      freshStart = Map.keysSet initial `Set.isSubsetOf` Set.singleton continueSeedStepName
  when (freshStart && not (Set.null patches)) $
    appendJournal name wid runGen (StepRecorded patchSetStepName (toJSON patches) now)
```

An in-flight generation that already has ordinary history will not get that new active-set row. A fresh
generation records the set before user logic runs, so a workflow that executes a step before its first
`patch` call is still classified from the start-of-generation set rather than from the live mutating
journal.

```haskell
Patch pid -> do
  let key = patchStepName pid
  journal <- liftIO (readIORef journalRef)
  case Map.lookup key journal of
    Just stored -> decodeStored key stored          -- hit: replay the recorded decision verbatim
    Nothing -> do                                    -- miss: first encounter
      recordedSet <- maybe (pure []) (decodeStored patchSetStepName) (Map.lookup patchSetStepName journal)
      let decision = unPatchId pid `elem` recordedSet
          encoded = Aeson.toJSON decision
      _ <- appendJournal name wid gen (StepRecorded key encoded now)
      liftIO (atomicModifyIORef' journalRef (\m -> (Map.insert key encoded m, ())))
      pure decision
```

The decision is journaled under `patch:<patchId>` as an ordinary `StepRecorded`, so the replay loop and
the step index stay uniform — **no new journal-event constructor for patch**. Every later replay hits the
stored `Bool` and returns it verbatim, forever. That is the whole guarantee: an instance fresh at the
generation start, with the patch id active, runs the new branch; one already in flight when the patch
shipped keeps the old branch; and neither ever flip-flops across resumes.

The design note worth carrying away: `patch` is the escape hatch for a change that *cross-cuts several
steps*. For one step changed, **rename the step** — a renamed `StepName` has no journaled history under
its new name, so its action runs fresh on the next replay, which is exactly right and needs no API at all.
