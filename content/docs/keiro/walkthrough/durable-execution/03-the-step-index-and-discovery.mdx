`keiro/src/Keiro/Workflow/Schema.hs` owns the `keiro_workflow_steps` table — the **derived index** of
journaled steps. `Keiro.Workflow.Instance` owns `keiro_workflows` — one lifecycle row per logical
workflow instance. The journal stream is still the source of truth for replay; these tables are
fast-lookup and operator-visible views kept in sync inside journal append transactions.

`keiro_workflow_steps` answers "does this step exist?" without rescanning streams. `keiro_workflows`
answers "which workflow instances are due for resume or GC?" without deriving lifecycle from step rows
on every pass.

## The table

```sql
CREATE TABLE IF NOT EXISTS keiro_workflow_steps (
  workflow_id    text        NOT NULL,
  workflow_name  text        NOT NULL,
  generation     integer     NOT NULL DEFAULT 0,
  step_name      text        NOT NULL,
  result         jsonb       NOT NULL,
  recorded_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (workflow_id, workflow_name, generation, step_name)
);
CREATE INDEX IF NOT EXISTS keiro_workflow_steps_workflow_idx
  ON keiro_workflow_steps (workflow_id, workflow_name, generation);
```

The primary key is what makes a step's identity its **name** within a logical workflow generation: one
row per named step per `(workflow_id, workflow_name, generation)`. The terminal markers are ordinary
rows whose `step_name` is one of `__workflow_completed__`, `__workflow_cancelled__`,
`__workflow_failed__`, or `__workflow_continued_as_new__`.

The row type mirrors the table:

```haskell
data WorkflowStepRow = WorkflowStepRow
  { workflowId :: !Text, workflowName :: !Text, generation :: !Int, stepName :: !Text, result :: !Value, recordedAt :: !UTCTime }
```

## `recordStepTx`: idempotent upsert

```haskell
recordStepStmt =
  preparable
    """
    INSERT INTO keiro_workflow_steps
      (workflow_id, workflow_name, generation, step_name, result, recorded_at)
    VALUES ($1, $2, $3, $4, $5, $6)
    ON CONFLICT (workflow_id, workflow_name, generation, step_name) DO NOTHING
    """
    ...
```

`ON CONFLICT DO NOTHING` makes a replayed or raced write a no-op — a second attempt to record the same
step changes nothing. Because `recordStepTx` runs **in the same transaction** as the journal append,
the index and the journal can never disagree: either both commit or neither does.

## `stepExists`: the idempotency pre-check

```haskell
stepExistsStmt =
  preparable
    """
    SELECT EXISTS (
      SELECT 1 FROM keiro_workflow_steps
      WHERE workflow_id = $1 AND workflow_name = $2 AND generation = $3 AND step_name = $4
    )
    """
    ...
```

This is the workhorse behind the wake-source helpers' idempotence. `appendJournalEntryReturningId`,
`appendCompletion`, and the cancellation short-circuit all call `stepExists` to decide whether to
append — relying on the index rather than the event store's duplicate-id rejection, so the check is a
cheap indexed lookup.

## `keiro_workflows`: the lifecycle row

The resume worker no longer derives lifecycle by anti-joining step rows on every pass. The
`keiro-workflows-instances` migration backfills and maintains one row per logical workflow:

```sql
CREATE TABLE IF NOT EXISTS keiro_workflows (
  workflow_id      TEXT        NOT NULL,
  workflow_name    TEXT        NOT NULL,
  generation       INTEGER     NOT NULL DEFAULT 0,
  status           TEXT        NOT NULL DEFAULT 'running',
  attempts         INTEGER     NOT NULL DEFAULT 0,
  last_error       TEXT,
  next_attempt_at  TIMESTAMPTZ,
  leased_by        TEXT,
  lease_expires_at TIMESTAMPTZ,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  completed_at     TIMESTAMPTZ,
  wake_after       TIMESTAMPTZ,
  PRIMARY KEY (workflow_id, workflow_name)
);
```

`upsertInstanceTx` runs beside journal-index writes: ordinary `StepRecorded` rows keep status
`running`; `WorkflowCompleted`, `WorkflowCancelled`, and `WorkflowFailed` make the instance terminal
and set `completed_at`; a suspended run marks the row `suspended`.

The lease and attempt fields are operational state for the resume worker. `claimInstance` sets
`leased_by` and `lease_expires_at` only for `running`/`suspended` rows whose lease is expired and whose
`next_attempt_at` is due. `recordCrashTx` increments `attempts`, stores `last_error`, and sets
`next_attempt_at` to an exponential backoff. `releaseInstance owner progressed ...` clears the lease;
when `progressed` is true it also resets attempts and errors.

## `findUnfinishedWorkflowIds`: due instance discovery

The query the resume worker is built on is now simple lifecycle discovery:

```sql
SELECT workflow_id, workflow_name
FROM keiro_workflows
WHERE status NOT IN ('completed', 'cancelled', 'failed')
  AND (wake_after IS NULL OR wake_after <= $1)
ORDER BY workflow_name, workflow_id
```

The explicit timestamp makes `wake_after` a self-expiring skip. `sleepNamed` stores the first timer
`fire_at` in `keiro_workflows.wake_after`, so a workflow parked only on a not-yet-due sleep disappears
from discovery until the timer is due. Wake sources that append a completion keep using the same
journal append path, so the next due resume pass replays past them.

`loadStepIndex` rounds out the read surface; it is exposed for completeness, though the replay loop
pre-loads from the journal stream instead.
