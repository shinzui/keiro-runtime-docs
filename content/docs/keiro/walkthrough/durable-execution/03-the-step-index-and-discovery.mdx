`keiro/src/Keiro/Workflow/Schema.hs` owns the `keiro_workflow_steps` table — the **derived index** of
journaled steps. The journal stream is the source of truth for replay; this table is a fast-lookup view
kept in sync inside the same transaction as each journal append (you saw `recordStepTx` composed into
`appendJournalTx` in [chapter 02](/docs/keiro/walkthrough/durable-execution/02-the-effect-and-replay-loop)).
It exists so the hot path and the resume worker can answer "does this step exist?" and "which workflows
are unfinished?" without rescanning streams.

## The table

```sql
CREATE TABLE IF NOT EXISTS keiro_workflow_steps (
  workflow_id    text        NOT NULL,
  workflow_name  text        NOT NULL,
  step_name      text        NOT NULL,
  result         jsonb       NOT NULL,
  recorded_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (workflow_id, step_name)
);
CREATE INDEX IF NOT EXISTS keiro_workflow_steps_workflow_idx ON keiro_workflow_steps (workflow_id);
```

The primary key `(workflow_id, step_name)` is what makes a step's identity its **name** at the storage
layer too: one row per named step per instance. The terminal completion marker is just a row whose
`step_name` is `__workflow_completed__` and whose `result` is JSON `null`. The migration comment ties
the schema to the runtime contract:

> The reserved step name `__workflow_completed__` is written when a workflow finishes … its absence is
> how `findUnfinishedWorkflowIds` distinguishes an in-flight workflow from a completed one.

The row type mirrors the table:

```haskell
data WorkflowStepRow = WorkflowStepRow
  { workflowId :: !Text, workflowName :: !Text, stepName :: !Text, result :: !Value, recordedAt :: !UTCTime }
```

## `recordStepTx`: idempotent upsert

```haskell
recordStepStmt =
  preparable
    """
    INSERT INTO keiro_workflow_steps (workflow_id, workflow_name, step_name, result, recorded_at)
    VALUES ($1, $2, $3, $4, $5)
    ON CONFLICT (workflow_id, step_name) DO NOTHING
    """
    ...
```

`ON CONFLICT DO NOTHING` makes a replayed or raced write a no-op — a second attempt to record the same
step changes nothing. Because `recordStepTx` runs **in the same transaction** as the journal append,
the index and the journal can never disagree: either both commit or neither does.

## `stepExists`: the idempotency pre-check

```haskell
stepExistsStmt =
  preparable
    """
    SELECT EXISTS (SELECT 1 FROM keiro_workflow_steps WHERE workflow_id = $1 AND step_name = $2)
    """
    ...
```

This is the workhorse behind the wake-source helpers' idempotence. `appendJournalEntryReturningId`,
`appendCompletion`, and the cancellation short-circuit all call `stepExists` to decide whether to
append — relying on the index rather than the event store's duplicate-id rejection, so the check is a
cheap indexed lookup.

## `findUnfinishedWorkflowIds`: the discovery anti-join

The query the resume worker is built on returns every workflow that has steps but **no terminal
marker**:

```sql
SELECT DISTINCT s.workflow_id, s.workflow_name
FROM keiro_workflow_steps s
WHERE NOT EXISTS (
  SELECT 1 FROM keiro_workflow_steps c
  WHERE c.workflow_id = s.workflow_id
    AND c.step_name IN ('__workflow_completed__', '__workflow_cancelled__')
)
```

A correlated `NOT EXISTS` anti-join: a workflow drops out of discovery the moment *either* terminal
marker row exists, so a **completed** workflow and a **cancelled** workflow both look "finished". The
source comment is blunt about the coupling:

> The literals `__workflow_completed__` / `__workflow_cancelled__` must match
> `Keiro.Workflow.Types.completedStepName` / `.cancelledStepName`.

Note the query groups by `workflow_id` alone (via the `c.workflow_id = s.workflow_id` correlation) — it
does **not** also match on name. That is the reason a parent and child must use **distinct** ids: a
child's terminal marker under a shared id would satisfy the `NOT EXISTS` and hide the still-unfinished
parent. The worked example guards against this by suffixing the child id (`<order>-ship`).

`loadStepIndex` (a full `SELECT step_name, result WHERE workflow_id = $1`) rounds out the read surface;
it is exposed for completeness, though the replay loop pre-loads from the journal stream instead.
