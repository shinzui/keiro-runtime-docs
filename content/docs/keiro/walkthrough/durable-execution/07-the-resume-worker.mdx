`keiro/src/Keiro/Workflow/Resume.hs` is what makes a suspended workflow actually resume. Nothing in the
runtime *notices* that a workflow is due for work — that it crashed mid-run, or is parked on a wait
whose source has since resolved. This module notices: a background worker that each pass asks the
database "which workflow instances are unfinished and due?", claims each through a row lease, and
re-invokes it so it proceeds.

## Why a registry

A workflow's *body* is application Haskell; only its *journal* lives in the database. To re-invoke a
workflow the worker must turn a stored name into a function `WorkflowId -> Eff (Workflow : es) a` — and
there is no way to materialize a closure from a string. So the application supplies one:

```haskell
data WorkflowDef es = forall a. (Aeson.ToJSON a) => WorkflowDef { runDef :: WorkflowId -> Eff (Workflow : es) a }
type WorkflowRegistry es = Map WorkflowName (WorkflowDef es)
```

The result type `a` is **existential** — the worker discards it (it cares only `Completed` vs
`Suspended`), so one registry holds workflows of different return types. The module frames this as the
resume-worker analogue of the `fire` action `runTimerWorker` takes:

> the worker owns the discovery loop and the database access; the application owns the domain behaviour.

## `resumeWorkflowsOnce`: one discover-and-reinvoke pass

```haskell
resumeWorkflowsOnce opts registry = do
  pending <- countPendingAwakeables                       -- EP-44 gauge sample, once per pass
  recordWorkflowAwakeablesPending mMetrics (fromIntegral pending)
  now <- getCurrentTime
  unfinished <- findUnfinishedWorkflowIds now             -- non-terminal keiro_workflows rows due now
  runningChildren <- findRunningChildIds                  -- freshly-spawned, zero-step children
  let pairs = nub (unfinished <> runningChildren)
      seed = emptyResumeSummary { discovered = length pairs }
  owner <- UUIDv4.nextRandom
  foldM (advance owner) seed pairs
```

Discovery **unions two sources**: the due instance query from
[chapter 03](/docs/keiro/walkthrough/durable-execution/03-the-step-index-and-discovery), and the
running-children query from [chapter 06](/docs/keiro/walkthrough/durable-execution/06-child-workflows)
— so a child with a link row but no step rows yet is still driven. `nub` collapses a child that appears
in both. The comment names the discovery boundary that gives this surface zero upstream dependency:

> Discovery is the `findUnfinishedWorkflowIds` index query only — **no kiroku `wf:` prefix
> subscription** is used or needed.

## `advance`: registry lookup, child routing, re-invoke

```haskell
advance owner acc (widText, wnameText) =
  case Map.lookup (WorkflowName wnameText) registry of
    Nothing -> do
      liftIO $ hPutStrLn stderr ("keiro resume worker: no registry entry for workflow " <> ... <> "; skipping")
      pure acc { unknownName = unknownName acc + 1 }
    Just (WorkflowDef runDef) -> do
      let wid = WorkflowId widText; name = WorkflowName wnameText
      claimed <- claimInstance owner (leaseTtl opts) name wid
      if not claimed
        then pure acc { leaseSkipped = leaseSkipped acc + 1 }
        else finally
          (drive and classify the instance)
          (releaseInstance owner progressed name wid)
```

Four things to notice. **(1)** A name absent from the registry is surfaced as `unknownName` and logged
— never silently dropped — so a deploy that removed a workflow while instances were in flight is
visible. **(2)** A live foreign lease skips only that instance and increments `leaseSkipped`; a dead
worker's lease becomes claimable after `leaseTtl`. **(3)** A workflow that is some parent's child is
driven through `runChildWorkflow` (so its result propagates to the parent on completion), everything
else through bare `runWorkflowWith`. **(4)** Re-invocation threads `runOptions` (the snapshot and
telemetry options), so a resumed run honours the same options as its first run, and the
per-re-invocation `keiro.workflow.resumed` counter ticks.

```haskell
bumpForOutcome outcome acc = case outcome of
  Completed _ -> acc { resumed = resumed acc + 1, completed = completed acc + 1 }
  Suspended   -> acc { resumed = resumed acc + 1, stillSuspended = stillSuspended acc + 1 }
  Cancelled   -> acc { resumed = resumed acc + 1 }
  Failed      -> acc { resumed = resumed acc + 1 }
  ContinuedAsNew -> acc { resumed = resumed acc + 1 }
```

Synchronous exceptions are isolated per instance. `recordCrashTx` increments `attempts`, stores
`last_error`, and sets `next_attempt_at` with exponential backoff; below `maxAttempts` the worker
releases the lease and leaves the instance for a later pass. At the ceiling it appends
`WorkflowFailed` and counts `failed`. Store errors while advancing are counted as `transientErrors`
without marking the workflow failed.

## The summary

```haskell
data ResumeSummary = ResumeSummary
  { discovered :: !Int, resumed :: !Int, completed :: !Int, stillSuspended :: !Int
  , unknownName :: !Int, failed :: !Int, transientErrors :: !Int, leaseSkipped :: !Int
  }
```

The per-pass observability record. A steady non-zero `unknownName` means an unregistered workflow has
live instances; a `discovered` that never drains to zero means some wake source is not resolving;
`leaseSkipped` is expected when multiple workers share the backlog.

## The loop, leases, and concurrency

```haskell
runWorkflowResumeWorkerWith opts registry = forever $ do
  _summary <- resumeWorkflowsOnce opts registry
  liftIO (threadDelay (pollInterval opts))
```

The poll loop is deliberately trivial — `resumeWorkflowsOnce` is the testable unit, the loop just
repeats it (mirroring how an application schedules `runTimerWorker`). Concurrency starts with the
`keiro_workflows` lease: steady-state workers do not advance the same instance at the same time. If a
lease expires during a long or crashed advance, the journal append path is still deterministic and
serialized per step, so racing workers converge on the existing recorded step.

`resumeWorkflowsOnce` is idempotent across passes: a workflow that completes gets terminal
`keiro_workflows` status and drops out of discovery; a sleeper with future `wake_after` drops out until
the timestamp is due; a failed instance backs off through `next_attempt_at`.

The [how-to](/docs/keiro/how-to/run-the-workflow-resume-worker) wires this into a production deployment
(registry, poll interval, and draining the sleep timers the worker depends on).
