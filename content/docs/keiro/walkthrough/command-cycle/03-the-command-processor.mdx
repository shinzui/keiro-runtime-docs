This chapter reads `runCommand` and its post-hydrate helpers in `keiro/src/Keiro/Command.hs` — the
append-only runner. [Hydration](/docs/keiro/walkthrough/command-cycle/02-hydration) now lives in its
own chapter, so here we pick the command up *after* state is rebuilt: transduce it, encode the result,
append, and maybe snapshot. Read
[02 — Hydration](/docs/keiro/walkthrough/command-cycle/02-hydration) first.

## `runCommand`: span, attempt loop, outcome

```haskell
-- keiro/src/Keiro/Command.hs
runCommand ::
  RunCommandOptions ->
  ValidatedEventStream phi rs s ci co ->
  Stream (EventStream phi rs s ci co) ->
  ci ->
  Eff es (Either CommandError (CommandResult (EventStream phi rs s ci co)))
runCommand options validatedEventStream targetStream command =
  withCommandSpan (options ^. #tracer) (resolvedStreamName eventStream targetStream) Nothing $ \mSpan -> do
    (result, attemptNo) <- attempt 1 Nothing
    recordCommandOutcome mSpan (^. #eventsAppended) attemptNo result
    pure result
  where
    eventStream = unvalidated validatedEventStream

    attempt attemptNo lastConflict = do
      hydrated <- hydrate options eventStream targetStream
      either (\err -> pure (Left err, attemptNo)) (runPlan attemptNo lastConflict) hydrated
```

The first important line is not telemetry; it is `eventStream = unvalidated validatedEventStream`.
The public API accepts only `ValidatedEventStream`, whose constructor is hidden in
`Keiro.EventStream.Validate`. `runCommand` unwraps it only after the type system has forced callers
to prove the stream with `mkEventStream`, `mkEventStreamWith`, or `mkEventStreamOrThrow`. From that
point on the implementation uses the raw `EventStream` fields for hydration, `Keiki.stepEither`, encoding,
name resolution, and snapshotting.

Then the body opens with the telemetry bookends — worth naming explicitly rather than skimming past.
`withCommandSpan` opens an *optional* OpenTelemetry span: a `Nothing` tracer (the default) means no
span and zero overhead; a `Just` tracer opens an internal-kind span named after the resolved stream.
After the work, `recordCommandOutcome` stamps the outcome onto that span:

```haskell
recordCommandOutcome Nothing _ _ _ = pure ()
recordCommandOutcome (Just sp) eventsOf attemptNo result = do
  addAttribute sp (unkey db_system_name) ("postgresql" :: Text)
  addAttribute sp (unkey keiro_retry_attempt) (fromIntegral attemptNo :: Int64)
  case result of
    Right v  -> addAttribute sp (unkey keiro_events_appended) (Prelude.fromIntegral (eventsOf v) :: Int64)
    Left err -> do
      addAttribute sp (unkey error_type) (commandErrorClass err)
      setStatus sp (Error (Text.pack (show err)))
```

It is a pure no-op without a span. With one, success stamps `db.system.name`,
`keiro.retry.attempt`, and `keiro.events.appended`; failure stamps the retry attempt and a
low-cardinality `error.type` from `commandErrorClass`, then sets the span status to `Error` carrying
the rendered `CommandError`:

```haskell
commandErrorClass = \case
  HydrationDecodeFailed{} -> "hydration_decode_failed"
  HydrationReplayFailed _ HydrationNoInvertingEdge -> "hydration_replay_no_inverting_edge"
  HydrationReplayFailed _ HydrationAmbiguousInversion -> "hydration_replay_ambiguous_inversion"
  HydrationReplayFailed _ HydrationQueueMismatch -> "hydration_replay_queue_mismatch"
  HydrationReplayFailed _ HydrationTruncatedChain -> "hydration_replay_truncated_chain"
  HydrationGapDetected{} -> "hydration_gap_detected"
  CommandRejected -> "command_rejected"
  CommandAmbiguous{} -> "command_ambiguous"
  EncodeFailed{} -> "encode_failed"
  StoreFailed{} -> "store_failed"
  RetryExhausted{} -> "retry_exhausted"
  ConflictFixpoint{} -> "conflict_fixpoint"
```

Telemetry is cross-cutting; the [operations tour](/docs/keiro/walkthrough/operations/00-start-here) and the
[telemetry reference](/docs/keiro/reference/telemetry) own the full span story. Here it is enough to
know every runner is wrapped in these two bookends. Between them, `attempt` runs:
[Hydrate](/docs/keiro/walkthrough/command-cycle/02-hydration) first, and a hydration error
short-circuits to `Left`; otherwise control passes to `runPlan`.

## `runPlan` → `prepareCommandPlan`: Transduce, then encode

```haskell
    runPlan attemptNo lastConflict current =
      case conflictFixpoint lastConflict (current ^. #streamVersion) of
        Just err -> pure (Left err, attemptNo)
        Nothing ->
          case prepareCommandPlan options eventStream targetStream current command of
            Left err -> pure (Left err, attemptNo)
            Right (CommandNoOp result) -> pure (Right result, attemptNo)
            Right (CommandAppend current' events encoded) ->
              appendOnce attemptNo current' events encoded
```

`prepareCommandPlan` runs the **Transduce** phase and the encode step, producing the `CommandPlan` from
chapter 01:

```haskell
prepareCommandPlan options eventStream targetStream current command =
  case evaluateCommand eventStream current command of
    Left err -> Left err
    Right events -> toPlan events
  where
    toPlan [] = Right (CommandNoOp (noOpResult targetStream current))
    toPlan events =
      CommandAppend current events
        . assignEventIds (options ^. #eventIds)
        <$> encodeEvents (eventStream ^. #eventCodec) (options ^. #metadata) events
```

The Transduce step is `evaluateCommand`, which steps the keiki transducer with the command:

```haskell
evaluateCommand eventStream current command =
  case Keiki.stepEither (eventStream ^. #transducer) (state current, registers current) command of
    Left Keiki.NoOutgoingEdges{} -> Left CommandRejected
    Left Keiki.NoMatchingEdge{} -> Left CommandRejected
    Left (Keiki.AmbiguousEdges _ matches) ->
      Left (CommandAmbiguous [Keiki.edgeIndex (Keiki.matchedEdge m) | m <- matches])
    Right (_, _, events) -> Right events
```

This is where the **no-op vs. rejected vs. ambiguous** boundary lives. No outgoing edge or no
matching edge is `CommandRejected`. Multiple matches are `CommandAmbiguous` with zero-based edge
indices in declaration order; treating that as rejection would hide a nondeterministic aggregate
definition. A no-op is the opposite: `evaluateCommand` succeeds with `Right []`, and `toPlan []`
turns it into `Right (CommandNoOp …)`. Same empty-handed feeling, different control flow.

When events *are* emitted, `toPlan events` encodes them and assigns ids:

```haskell
encodeEvents :: Codec co -> Maybe Value -> [co] -> Either CommandError [EventData]
encodeEvents codec md =
  Prelude.mapM (mapLeft EncodeFailed . encodeForAppendWithMetadata codec md)

-- Caller ids are assigned POSITIONALLY (zip-style). [] is identity.
assignEventIds :: [EventId] -> [EventData] -> [EventData]
assignEventIds [] events = events
assignEventIds _ [] = []
assignEventIds (supplied : suppliedRest) (event : eventRest) =
  (event & #eventId .~ Just supplied) : assignEventIds suppliedRest eventRest
```

`encodeEvents` maps each emitted event through `encodeForAppendWithMetadata`, lifting any `CodecError`
to `EncodeFailed` (the encode side of [chapter 05](/docs/keiro/walkthrough/command-cycle/05-the-codec-on-the-boundary)).
`assignEventIds` then assigns the caller's `eventIds` **positionally** — the first supplied id to the
first event, and so on — with `[]` as the identity (no ids supplied, events untouched). That positional
assignment is the basis of idempotent appends: a deterministic id on the first emitted event lets a
re-run be rejected as a duplicate rather than re-applied (see
[Make commands idempotent](/docs/keiro/how-to/make-commands-idempotent)).

## `appendOnce`: optimistic append, then verify replay

```haskell
    appendOnce mSpan attemptNo current events encoded = do
      liftIO (options ^. #beforeAppend)
      appended <- tryError @StoreError $
        appendToStream
          ((eventStream ^. #resolveStreamName) targetStream)
          (expectedVersion (current ^. #streamVersion))
          encoded
      case appended of
        Right appendResult -> do
          verifyAndSnapshot options mSpan eventStream current events appendResult
          pure (Right (appendedResult targetStream appendResult (Prelude.length encoded)), attemptNo)
        Left (_, storeError) ->
          retryOrFail options attempt attemptNo (current ^. #streamVersion) storeError
```

The append asserts an expected version derived from the hydrated version:

```haskell
expectedVersion :: StreamVersion -> ExpectedVersion
expectedVersion (StreamVersion 0) = NoStream
expectedVersion version = ExactVersion version
```

A hydrated version of `0` means "this stream does not exist yet," so the append asserts `NoStream`; any
other version asserts `ExactVersion v`. `tryError @StoreError` captures the store error as a `Left`
rather than throwing it, handing control to `retryOrFail`. On success, `verifyAndSnapshot` runs
**after the append has committed**:

```haskell
verifyAndSnapshot options mSpan eventStream current events appendResult
  | not (options ^. #verifyReplayOnAppend)
  , Nothing <- eventStream ^. #stateCodec = pure ()
  | otherwise =
      case Keiki.applyEventsEither transducer (state current, registers current) events of
        Left failure -> do
          recordSnapshotApplyDivergence (options ^. #metrics) 1
          for_ mSpan $ \sp ->
            addAttribute sp "keiro.replay.divergence" (renderReplayFailure failure)
        Right finalState ->
          case eventStream ^. #stateCodec of
            Nothing -> pure ()
            Just codec -> when (snapshotPolicyFires finalState) $ do
              encoded <- liftIO (encodeSnapshotStrict codec finalState)
              case encoded of
                Left _ -> recordSnapshotEncodeFailures (options ^. #metrics) 1
                Right value -> do
                  outcome <- tryError @StoreError (writeSnapshotEncoded ... value)
                  case outcome of
                    Left _ -> recordSnapshotWriteFailures (options ^. #metrics) 1
                    Right () -> pure ()
```

By default, the runner replays every just-appended batch from its pre-command state. If replay
diverges, keiro increments `keiro.snapshot.apply.divergence`, adds a compact
`keiro.replay.divergence` span attribute, and still returns success: the event transaction already
committed, so reporting rollback would be false. The next hydration will surface the matching typed
replay failure.

Setting `verifyReplayOnAppend = False` skips this fold only when `stateCodec = Nothing`.
Snapshot-enabled streams always need the fold to compute post-command state. When the policy fires,
encoding is forced by `encodeSnapshotStrict`; encode and store failures are counted and swallowed for
the same post-commit reason. The detailed cache behavior belongs to the
[snapshot reference](/docs/keiro/reference/snapshot).

Both exits return a `CommandResult` through one of two constructors — `appendedResult` after an append,
`noOpResult` for the `CommandNoOp` branch above:

```haskell
noOpResult targetStream current = CommandResult
  { target = targetStream, streamVersion = current ^. #streamVersion
  , globalPosition = Nothing, eventsAppended = 0 }

appendedResult targetStream appendResult count = CommandResult
  { target = targetStream, streamVersion = appendResult ^. #streamVersion
  , globalPosition = Just (appendResult ^. #globalPosition), eventsAppended = count }
```

`noOpResult` reports the hydrated stream version, `eventsAppended = 0`, and no global position;
`appendedResult` reports the post-append version and a `Just` position with the real count.

## `retryOrFail` + `isRetryableConflict`: the bounded conflict loop

On a store error, the conflict loop decides whether to retry:

```haskell
retryOrFail options retry attemptNo observedVersion storeError
  | isRetryableConflict storeError, attemptNo <= options ^. #retryLimit = retry (attemptNo + 1)
  | isRetryableConflict storeError = pure (Left (RetryExhausted attemptNo storeError))
  | otherwise = pure (Left (StoreFailed storeError))
```

A retryable conflict with budget left re-enters `attempt` (which re-hydrates and re-transduces from the
current head); a retryable conflict with no budget becomes `RetryExhausted` carrying the attempts made
and the last error; anything else is `StoreFailed`. If a `StreamAlreadyExists` retry rehydrates to the
same stream version as the previous attempt, the runner stops with `ConflictFixpoint` rather than
looping on an invisible conflict. The retryable set is the sharp part:

```haskell
isRetryableConflict = \case
  WrongExpectedVersion{} -> True
  StreamAlreadyExists{} -> True   -- a lost new-stream race is RETRIED, not surfaced
  _ -> False
```

<Callout type="warn">
`StreamAlreadyExists` is **retryable**, so a lost race to *create* a new stream is silently retried —
the re-hydrate now sees the stream exists and asserts `ExactVersion` instead — never surfaced to the
caller as `StreamAlreadyExists`. Tune the budget with
[Configure concurrency retries](/docs/keiro/how-to/configure-concurrency-retries).
</Callout>
