This chapter reads the hydration machine in `keiro/src/Keiro/Command.hs` — the half of the command
cycle that rebuilds current state from stored events before a command is transduced. Read
[01 — Command types and errors](/docs/keiro/walkthrough/command-cycle/01-command-types-and-errors)
first, since this chapter uses `Hydrated`, `HydrationReplayReason`, and the `CommandError`
constructors throughout.

Hydration must recover `(state, registers, streamVersion)` from the event log. A snapshot may provide
the starting point, but events remain authoritative.

## The two doors: snapshot seed or initial state

```haskell
hydrate options eventStream targetStream =
  snapshotSeed >>= \case
    Nothing -> hydrateFull options eventStream targetStream
    Just seed -> do
      replayed <-
        hydrateSeeded options eventStream targetStream
          (seed ^. #state)
          (seed ^. #registers)
          (seed ^. #streamVersion)
      case replayed of
        Left _ -> hydrateFull options eventStream targetStream
        Right hydrated -> pure (Right hydrated)
```

When the stream has no `StateCodec`, `snapshotSeed` is `Nothing`. With a codec,
`lookupSnapshotSeed` either returns `SnapshotHit seed` or `SnapshotUnavailable reason`. Hits increment
`keiro.snapshot.read.hits`; unavailable results increment `keiro.snapshot.read.misses`, and a decode
failure also increments `keiro.snapshot.decode.failures`.

Any replay error from a snapshot seed falls back to `hydrateFull`, which calls the same replay engine
from `initialState`, `initialRegisters`, and `StreamVersion 0`. That does not hide a real history
problem: if full replay encounters it too, that typed error is returned. It only prevents a stale or
incorrect cache entry from becoming authoritative.

## Decode contiguous pages, then replay the valid prefix

`hydrateSeeded` reads forward after the seed in bounded pages. `pageSize` is clamped to at least one,
and each page is decoded into the longest valid, contiguous prefix:

```haskell
decodePrefix previousRecorded = go [] [] startingVersion
  where
    go recordedAcc eventAcc lastSeen = \case
      [] -> (reverse recordedAcc, reverse eventAcc, Nothing)
      recorded : rest ->
        let observed = recorded ^. #streamVersion
            expected = nextStreamVersion lastSeen
        in if observed /= expected
             then (reverse recordedAcc, reverse eventAcc,
                   Just (HydrationGapDetected expected observed))
             else case decodeRecorded (eventStream ^. #eventCodec) recorded of
               Left err ->
                 (reverse recordedAcc, reverse eventAcc,
                  Just (HydrationDecodeFailed err))
               Right event ->
                 go (recorded : recordedAcc) (event : eventAcc) observed rest
```

The decoded prefix is replayed **before** the pending gap or decode error is returned. That ordering
is intentional: if an earlier valid event already contradicts the transducer, its replay failure is
more precise than a later bad event in the same page.

The continuity check matters because Kiroku supports per-stream truncation markers. If a marker hides
versions that the initial state or snapshot does not cover, the first visible event has a higher
version than expected and hydration returns `HydrationGapDetected expected observed`. Restore stream
visibility with `clearStreamTruncateBefore`, or provide a retained snapshot at least as new as the
last hidden event. Truncation hides events from per-stream reads; it does not delete them from `$all`
or category/subscription reads.

<Callout type="warn">
A truncation marker above the current stream head can make the per-stream read appear empty. A later
append then collides with the still-existing stream; repeated hydration with no visible progress ends
as `ConflictFixpoint`, not `HydrationGapDetected`.
</Callout>

## `replayEvents` preserves the reason

Each decoded page is passed to keiki's structured replay API with the previous page's
`InFlight`/`Settled` carrier and register file:

```haskell
case Keiki.replayEvents transducer (wrapper, registers) decodedEvents of
  Left replayFailure ->
    Left (hydrationReplayError previousRecorded decodedRecorded replayFailure)
  Right (nextWrapper, nextRegisters) ->
    case pendingInputFailure of
      Just err -> Left err
      Nothing -> Right (nextWrapper, nextRegisters, latestRecorded ...)
```

Keiro maps `ReplayFailureReason` to its public, monomorphic vocabulary:

| keiki reason | `HydrationReplayReason` | Meaning |
|---|---|---|
| `ReplayNoInvertingEdge` | `HydrationNoInvertingEdge` | No edge's first output could have produced the event |
| `ReplayAmbiguousInversions` | `HydrationAmbiguousInversion` | More than one edge could have produced it |
| `ReplayQueueMismatch` | `HydrationQueueMismatch` | A later event did not equal the next output expected from an in-flight chain |
| `ReplayLogTruncated` | `HydrationTruncatedChain` | Input ended before a multi-event output chain completed |

The result is `HydrationReplayFailed failureVersion reason`. For an event failure, `failureVersion`
is the stored version at `replayFailedIndex`. For a truncated chain, it is the latest stored version
consumed before the missing tail. Callers no longer have to infer these cases from one unstructured
"replay failed" value.

## Why replay runs backward from events

The transducer's forward direction turns a command into events. Hydration has only those persisted
events, so replay uses the first event as a witness of which edge originally fired. It solves the
edge's first output back to candidate command data, checks the guard, applies the register update,
and queues any remaining outputs from that edge.

This explains all four reasons. No candidate is `HydrationNoInvertingEdge`; multiple candidates are
`HydrationAmbiguousInversion`; a queued output mismatch is `HydrationQueueMismatch`; and ending while
the queue is non-empty is `HydrationTruncatedChain`. The corresponding keiki checks catch many such
definitions before startup, which is why public runners require a `ValidatedEventStream`.

For the inversion mechanics, see keiki's
[Structured step and replay failures](/docs/keiki/reference/step-failure) and
[Validation](/docs/keiki/reference/validate).

## The final `Settled` rule

A page may end `InFlight`; `hydrateSeeded` carries that wrapper into the next page. Only the end of
the entire stream must be settled:

```haskell
finishReplay = \case
  Left err -> Left err
  Right (Keiki.Settled finalState, finalRegisters, lastRecorded) ->
    Right Hydrated
      { state = finalState
      , registers = finalRegisters
      , streamVersion = maybe seedVersion (^. #streamVersion) lastRecorded
      }
  Right (Keiki.InFlight{}, _, lastRecorded) ->
    Left
      (HydrationReplayFailed
        (maybe seedVersion (^. #streamVersion) lastRecorded)
        HydrationTruncatedChain)
```

A healthy stream ends `Settled`: every output promised by its final transition is present. A final
`InFlight` value has no valid current aggregate state, so hydration reports the typed truncated-chain
failure rather than returning partial state.
