## What this part covers

`keiro/src/Keiro/Command.hs` — the real call site that runs the
[transducer step](/docs/keiro/walkthrough/foundation/04-the-symtransducer-and-step) from the previous
chapter against a hydrated stream. This is the linchpin the whole tour builds to: it shows *both* halves
of the `(state, registers)` pair being passed into `Keiki.stepEither`.

## The hydrated record

When keiro hydrates a stream it produces a `Hydrated` record holding the recovered machine
configuration plus bookkeeping:

```haskell
-- keiro/src/Keiro/Command.hs
data Hydrated rs s = Hydrated
  { state          :: !s
  , registers      :: !(RegFile rs)
  , streamVersion  :: !StreamVersion
  }
  deriving stock (Generic)
```

The two fields that matter here are `state` and `registers` — the same `(s, RegFile rs)` pair the
transducer steps on. `streamVersion` is the per-stream coordinate the append path uses; it is not
part of the machine's configuration. Hydration does not invent a global position from a per-stream
read, which is why a no-op `CommandResult` reports `globalPosition = Nothing`.

## The call site

```haskell
-- keiro/src/Keiro/Command.hs
evaluateCommand ::
  BoolAlg phi (RegFile rs, ci) =>
  EventStream phi rs s ci co ->
  Hydrated rs s ->
  ci ->
  Either CommandError [co]
evaluateCommand eventStream current command =
  case Keiki.stepEither (eventStream ^. #transducer) (state current, registers current) command of
    Left Keiki.NoOutgoingEdges{} -> Left CommandRejected
    Left Keiki.NoMatchingEdge{} -> Left CommandRejected
    Left (Keiki.AmbiguousEdges _ matches) ->
      Left (CommandAmbiguous [Keiki.edgeIndex (Keiki.matchedEdge m) | m <- matches])
    Right (_, _, events) -> Right events
```

This is the line the tour exists to explain:

```haskell
Keiki.stepEither (eventStream ^. #transducer) (state current, registers current) command
```

It reads the `transducer` out of the `EventStream`, builds the pair `(state current, registers
current)` from the hydrated record — **both halves** — and passes the command. The result maps
directly onto `stepEither`'s return shape from the previous chapter:

- `NoOutgoingEdges` and `NoMatchingEdge` become `Left CommandRejected`.
- `AmbiguousEdges` becomes `Left (CommandAmbiguous indices)`, preserving the zero-based declaration
  indices instead of hiding a definition bug as a business rejection.
- `Right (_, _, events)` becomes `Right events` — the emitted events to encode and append. (The next
  state and next registers in the first two slots are discarded *here*; they are recomputed on the
  write path when a snapshot needs to be written.)

## Why both halves must be carried

The registers are **not recoverable from the control state** — the point established in the
[previous chapter](/docs/keiro/walkthrough/foundation/04-the-symtransducer-and-step). Two streams at the
same vertex can hold different register values, so knowing the vertex tells you nothing about the
register file. That single fact forces the whole pair through every stage:

- **Hydration** replays the stored events to rebuild *both* the state and the registers (via keiki's
  structured `replayEvents`), producing the `Hydrated` record above.
- **The step** consumes the whole pair: `(state current, registers current)`.
- **A snapshot** must persist the whole pair, because a snapshot that stored only the state could never
  reconstruct the registers.

If keiro carried only the state, a transition whose guard reads a register — or whose `output` is built
from a register — would have nothing to read. So the command cycle hydrates and carries registers, not
just state, all the way through.
