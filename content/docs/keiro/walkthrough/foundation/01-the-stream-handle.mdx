## What this part covers

`keiro-core/src/Keiro/Stream.hs` — the whole file. It defines the **typed handle** every command
targets: a `Stream a`. Everything else in the foundation (the `EventStream`, the command cycle) takes
one of these handles and resolves it to the physical stream the event store reads and appends to. The
file has two halves: the bare newtype + its three functions (this section), and a safe,
category-based construction layer that single-sources the stream's category prefix (below).

## The newtype

```haskell
-- keiro-core/src/Keiro/Stream.hs
-- | A 'StreamName' tagged with the phantom type @a@ of the stream it names.
newtype Stream a = Stream
  { name :: StreamName
  }
  deriving stock (Generic, Eq, Ord, Show)
```

A `Stream a` is *nothing but* a `StreamName` (kiroku's `Kiroku.Store.Types.StreamName`) wrapped in a
`newtype` — a zero-cost wrapper that vanishes at runtime. What it adds is the **phantom type
parameter** `a`. "Phantom" means `a` appears in the type but never in a field: there is no value of
type `a` anywhere in the record. It exists purely so the type checker can demand, say, a
`Stream OrderEventStream` rather than a bare `StreamName`, so a name built for one aggregate cannot be
passed where a name for another is expected. The wrapper is otherwise transparent — at runtime a
`Stream a` *is* its `StreamName`.

## The three functions

The module exports exactly one constructor and two accessors:

```haskell
-- keiro-core/src/Keiro/Stream.hs
-- | Build a 'Stream' handle from a raw stream-name 'Text'.
stream :: Text -> Stream a
stream name = Stream { name = StreamName name }

-- | Recover the underlying 'StreamName' from a 'Stream' handle.
streamName :: Stream a -> StreamName
streamName value = value ^. #name

-- | Transform the underlying 'StreamName' while preserving the phantom type.
mapStreamName :: (StreamName -> StreamName) -> Stream a -> Stream a
mapStreamName f value = value & #name %~ f
```

Read them in order:

- `stream` builds a handle from a raw `Text`. Note the phantom `a` is *free* in the result type — the
  caller decides which tag the handle carries, usually by a type annotation on the binding it feeds
  (see `orderStream` below).
- `streamName` recovers the underlying `StreamName`. This is the projection the runtime uses when it
  needs the physical name to hand to the store.
- `mapStreamName` rewrites the name *while keeping the same phantom tag* — handy for namespacing or
  prefixing a stream name without losing the compile-time `a`. Because the tag is preserved, a
  prefixed `Stream OrderEventStream` is still an `OrderEventStream` handle.

The implementations are tiny because the type is the whole point: the work is at compile time, in the
tag `a`, not at runtime.

## The jitsurei anchor

The worked example (`jitsurei/src/Jitsurei/OrderStream.hs`) builds a handle for an order aggregate.
It no longer hand-assembles the `"order-"` prefix; it declares the category once and derives the
stream from it (the category API, below):

```haskell
-- jitsurei/src/Jitsurei/OrderStream.hs
orderCategory :: Stream.StreamCategory a
orderCategory = Stream.categoryUnsafe "order"

orderStream :: OrderId -> Stream OrderEventStream
orderStream = Stream.entityStream orderCategory . orderIdText   -- renders "order-<id>"
```

`orderStream` takes a domain `OrderId`, renders it into a `Text` name (`"order-…"`), and tags the
resulting handle as a `Stream OrderEventStream`. From here on the type checker treats this handle as
*the order stream's* — you cannot accidentally pass it where a different aggregate's handle is
expected.

## Safe, category-based construction

The second half of the file exists because `stream` takes a *raw* `Text`, and the prefix you put in
front of the first `-` is not cosmetic. In kiroku a stream name's **category** is the substring before
its first `-`, and that category is the join key for [`readCategory`](/docs/kiroku/reference/core-types)
and category subscriptions — every stream sharing the prefix fans in. Hand-writing `"order-" <> …` at
each call site duplicates that join key and invites a subtler bug: a "category" that itself contains a
`-` (a saga prefix like `"hospital-surge-"`) silently parses as `hospital`, fanning the saga's stream
into the wrong aggregate.

So `Keiro.Stream` adds a validated, phantom-typed category:

```haskell
-- keiro-core/src/Keiro/Stream.hs
newtype StreamCategory a = StreamCategory { categoryTextOf :: Text }

category       :: Text -> Either CategoryError (StreamCategory a)  -- rejects "", "$all", and any text with `-`
categoryUnsafe :: Text -> StreamCategory a                         -- for static literals; `error`s on bad input
categoryName   :: StreamCategory a -> CategoryName                 -- the CategoryName for category reads/subs
entityStream   :: StreamCategory a -> Text -> Stream a             -- renders "<category>-<id>"
entityStreamId :: StreamIdSegment i => StreamCategory a -> i -> Stream a
```

Two things make this more than a convenience. First, `category` **rejects** a prefix containing `-`,
turning the silent mis-parse into a definition-site failure. Second, `entityStream` delegates the
actual name mechanics to kiroku's `streamNameInCategory`, and `categoryName` reads back the matching
`CategoryName` — so `categoryName cat` always equals kiroku's own category-of-stream-name for any
stream `entityStream cat id` produces. The per-entity stream and the category read are derived from
*one* declaration and cannot drift. The [EventStream and Stream reference](/docs/keiro/reference/event-stream-and-stream#safe-category-based-construction-streamcategory)
covers the full surface; the rule it enforces is owned by kiroku
([core types](/docs/kiroku/reference/core-types)).

## How a handle becomes a physical name

A `Stream a` is a *typed* handle. The store, however, reads and appends to a physical `StreamName`.
The bridge is the `EventStream`'s `resolveStreamName` field, walked in the
[next chapter](/docs/keiro/walkthrough/foundation/02-the-event-stream): it is a function
`Stream (EventStream …) -> StreamName` that maps the typed handle to the physical name. For the order
stream it is simply `Stream.streamName` — the projection above — so the handle's name *is* the physical
name.
