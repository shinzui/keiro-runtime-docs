This chapter reads the inbox receive path in `keiro/src/Keiro/Inbox.hs`,
`keiro/src/Keiro/Inbox/Types.hs`, and `keiro/src/Keiro/Inbox/Schema.hs`. Read [00 — Start

## `dedupeKeyFor`: choosing the identity

```haskell
-- keiro/src/Keiro/Inbox/Types.hs
dedupeKeyFor policy event kafka = case policy of
  PreferIntegrationMessageId ->
    let mid = event ^. #messageId
     in if Text.null mid then Left (DedupePolicyUnsatisfied policy) else Right mid
  PreferSourceEventIdentity ->
    case event ^. #sourceEventId of
      Just (EventId u) -> Right (UUID.toText u)
      Nothing -> case event ^. #sourceGlobalPosition of
        Just (GlobalPosition p) -> Right (Text.pack (show p))
        Nothing -> Left (DedupePolicyUnsatisfied policy)
  KafkaDeliveryIdentity ->
    case kafka of
      Just ref -> Right ((ref ^. #topic) <> ":" <> Text.pack (show (ref ^. #partition)) <> ":" <> Text.pack (show (ref ^. #offset)))
      Nothing -> Left (DedupePolicyUnsatisfied policy)
  CustomDedupeKey k -> if Text.null k then Left (DedupePolicyUnsatisfied policy) else Right k
```

Each branch produces the `Text` dedupe key or fails with `DedupePolicyUnsatisfied` when its field
is absent. Note the layering: `PreferSourceEventIdentity` falls back from `sourceEventId` to
`sourceGlobalPosition` before giving up. The key it returns, paired with `source`, is the inbox
row's primary key.

## `runInboxTransaction`: resolve the key, then one transaction

```haskell
-- keiro/src/Keiro/Inbox.hs
runInboxTransaction policy event kafka handler =
  case dedupeKeyFor policy event kafka of
    Left err -> pure (Left err)
    Right dedupe ->
      Right <$> runInboxTransactionWithKey (event ^. #source) dedupe event kafka handler

runInboxTransactionWithKey src dedupe event kafka handler = do
  now <- liftIO getCurrentTime
  runTransaction $ do
    inserted <- tryInsertProcessingTx src dedupe event kafka now
    case inserted of
      Right () -> do
        result <- handler event
        markCompletedTx src dedupe now
        pure (InboxProcessed result)
      Left row -> case row ^. #status of
        InboxCompleted -> pure InboxDuplicate
        InboxProcessing -> pure InboxInProgress
        InboxFailed -> pure (InboxPreviouslyFailed (row ^. #lastError))
```

The whole body runs inside one `runTransaction`. The control flow is the entire idempotency
contract:

- `tryInsertProcessingTx` returns `Right ()` for a **new** row → run `handler event`, then
  `markCompletedTx`, and return `InboxProcessed`. The insert, the handler's writes, and the
  completion update all commit **together**.
- It returns `Left existingRow` for a **duplicate** → branch on the row's status. A `completed` row
  is `InboxDuplicate` (the common redelivery case); `processing`/`failed` map to `InboxInProgress` /
  `InboxPreviouslyFailed` (reachable only from future async paths).

If the handler throws or condemns the transaction, the `runTransaction` rolls back — the inbox row
insert included — so the next delivery sees no row and starts fresh.

## `tryInsertProcessingTx`: `ON CONFLICT DO NOTHING RETURNING TRUE`

```haskell
-- keiro/src/Keiro/Inbox/Schema.hs (tryInsertStmt, abridged)
INSERT INTO keiro_inbox ( source, dedupe_key, … , received_at )
VALUES ( $1, $2, … , $25 )
ON CONFLICT (source, dedupe_key) DO NOTHING
RETURNING TRUE
```

```haskell
tryInsertProcessingTx src dedupe event kafka now = do
  inserted <- Tx.statement (toEncodedInsert src dedupe event kafka now) tryInsertStmt
  if inserted
    then pure (Right ())
    else do
      existing <- Tx.statement (src, dedupe) selectByKeyStmt
      case existing of
        Just row -> pure (Left row)
        Nothing  -> pure (Right ())   -- race: inserted-then-deleted between attempt and lookup; treat as new
```

The `INSERT … ON CONFLICT (source, dedupe_key) DO NOTHING RETURNING TRUE` is the atomic dedupe: a
new row returns `TRUE` (decoded to `inserted = True`); a conflict returns no row (`inserted =
False`). On a conflict the code re-selects the existing row to branch on its status. The `Nothing`
fall-through handles a rare race (the row was inserted then GC-deleted between the attempt and the
lookup) by treating it as new — the next insert in this transaction reflects the committed state.

Note `markCompletedTx` sets `status = 'completed'`, `completed_at`, and clears `last_error`;
`markFailedTx` exists but the v1 wrapper never calls it (there is no failure branch in the
single-transaction path — a failure is a rollback).
