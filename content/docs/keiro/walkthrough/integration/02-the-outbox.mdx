This chapter reads the outbox publish path in `keiro/src/Keiro/Outbox.hs`,
`keiro/src/Keiro/Outbox/Types.hs`, and `keiro/src/Keiro/Outbox/Schema.hs`. Read [01 — The
inbox](/docs/keiro/walkthrough/integration/01-the-inbox) first.

## `enqueueProducerEventTx`: mint and enqueue in one transaction

```haskell
-- keiro/src/Keiro/Outbox.hs
enqueueProducerEventTx producer outboxId draft = do
  event <- mintIntegrationEvent producer draft     -- mints a prefixed TypeID as messageId
  pure (enqueueOutboxTx (OutboxMessage {outboxId, event}))
```

The producer mints a fresh `messageId` (a TypeID with the producer's prefix) and builds the full
envelope from the draft, then returns the `Tx.Transaction ()` that inserts the row. The insert is
idempotent on `(source, message_id)`:

```haskell
-- keiro/src/Keiro/Outbox/Schema.hs (enqueueOutboxStmt, abridged)
INSERT INTO keiro_outbox ( outbox_id, message_id, source, … , occurred_at )
VALUES ( $1, $2, $3, … , $22 )
ON CONFLICT (source, message_id) DO NOTHING
```

The TypeID is minted *before* the insert; if the transaction rolls back, the id is simply discarded
and the next attempt mints a new one. Row-level idempotency hinges on a stable `OutboxId`, not the
minted message id.

## `claimOutboxBatch`: the claim CTE

The claim is one statement: select ready rows under the ordering predicate, lock them with
`SKIP LOCKED`, and flip them to `publishing`:

```sql
-- keiro/src/Keiro/Outbox/Schema.hs (claimSql, abridged)
WITH ready AS (
  SELECT r.outbox_id FROM keiro_outbox r
  WHERE r.status IN ('pending', 'failed')
    AND r.next_attempt_at <= $2
    AND ( <ordering predicate> )
  ORDER BY r.created_at
  LIMIT $1
  FOR UPDATE SKIP LOCKED
)
UPDATE keiro_outbox kt
SET status = 'publishing', attempt_count = kt.attempt_count + 1, updated_at = $2
WHERE kt.outbox_id IN (SELECT outbox_id FROM ready)
RETURNING <row columns>
```

Three things to see. First, the candidate set is `status IN ('pending', 'failed')` — **`publishing`
is not selected**, which is the [stale-row
gap](/docs/keiro/how-to/choose-an-outbox-ordering-policy#recovering-stranded-publishing-rows): a
crash between this claim and the later mark strands a row in `publishing` with no automatic
recovery. Second, `FOR UPDATE SKIP LOCKED` lets many workers claim disjoint batches without
blocking. Third, the ordering predicate is spliced in per policy:

```haskell
policyPredicate = \case
  PerKeyHeadOfLine -> perKeyPredicate
  PerSourceStream  -> perSourcePredicate
  StopTheLine      -> perKeyPredicate     -- same claim predicate; halts at the worker level
  BestEffort       -> "TRUE"

perKeyPredicate =
  "( r.message_key IS NULL OR NOT EXISTS (
       SELECT 1 FROM keiro_outbox earlier
       WHERE earlier.source = r.source
         AND earlier.message_key = r.message_key
         AND earlier.created_at < r.created_at
         AND earlier.status NOT IN ('sent', 'dead') ) )"
```

`perKeyPredicate` skips a row if an *earlier* same-key row is still non-terminal (`status NOT IN
('sent', 'dead')`) — so a stuck row head-of-line-blocks later rows with its key, while null-keyed
rows bypass the check. (Note `publishing` counts as non-terminal here, which is why a stranded
`publishing` row blocks its key.)

## `drainBatch`: publish, then mark

```haskell
-- keiro/src/Keiro/Outbox.hs (publishClaimedOutbox / drainBatch, abridged)
drainBatch (row : rest) acc = do
  outcome <- withProducerSpan … (publish row)
  now <- liftIO getCurrentTime
  case outcome of
    PublishSucceeded -> do
      markOutboxSent (row ^. #outboxId) now
      drainBatch rest (acc & #claimed +~ 1 & #published +~ 1)
    PublishFailed errMsg -> do
      let delay = nextDelay (options ^. #backoff) (row ^. #attemptCount)
      resultStatus <- runTransaction $
        markOutboxFailedTx (row ^. #outboxId) errMsg (options ^. #maxAttempts) delay now
      let acc' = case resultStatus of { OutboxDead -> …dead+1 ; _ -> …retried+1 }
      case options ^. #orderingPolicy of
        StopTheLine -> pure (acc' & #haltedOn ?~ row ^. #outboxId)
        _           -> drainBatch rest acc'
```

For each claimed row the worker calls your `publish`, then reflects the outcome: a success is
`markOutboxSent` (status `sent`, `published_at` set); a failure is `markOutboxFailedTx`, which reads
`attempt_count` and decides `OutboxDead` (`>= maxAttempts`) versus `OutboxFailed` (with
`next_attempt_at = now + delay`). Under `StopTheLine` the worker halts on the first failure and
records `haltedOn`; under the other policies it continues. The function processes **one claimed
batch and returns** — the application schedules it again on the next tick.
