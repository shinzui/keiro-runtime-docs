## `publishClaimedOutbox`: one pass per call

```haskell
-- keiro/src/Keiro/Outbox.hs
data PublishOutcome = PublishSucceeded | PublishFailed !Text
  deriving stock (Generic, Eq, Show)

publishClaimedOutbox ::
  (IOE :> es, Store :> es) =>
  ([OutboxRow] -> Eff es [(OutboxId, PublishOutcome)]) -> OutboxPublishOptions -> Maybe KeiroMetrics ->
  Eff es OutboxPublishSummary
publishClaimedOutbox publish options mMetrics = do
  now <- liftIO getCurrentTime
  rows <- claimOutboxBatch (options ^. #orderingPolicy) (options ^. #batchSize) now
  summary <- publishBatch rows
  -- Counters from the aggregated pass summary.
  recordOutboxPublished mMetrics (fromIntegral (summary ^. #published))
  recordOutboxRetried mMetrics (fromIntegral (summary ^. #retried))
  recordOutboxDeadlettered mMetrics (fromIntegral (summary ^. #dead))
  pure summary
  where
    …
```

<Callout type="info">
`publishClaimedOutbox` processes **one claimed batch and returns** — it does NOT loop. The
application schedules it repeatedly (e.g. once per process-compose tick). It is also
transport-neutral: it takes a caller-supplied
`publish :: [OutboxRow] -> Eff es [(OutboxId, PublishOutcome)]`, so the broker library never appears
in keiro itself.
</Callout>

It claims a batch under the active `OrderingPolicy`, hands the claimed rows to the publish callback,
and marks outcomes in bulk where possible. The trailing `mMetrics :: Maybe KeiroMetrics` is the
opt-in [metrics](/docs/keiro/reference/telemetry#metrics-surface) handle; after the publish pass it
records the `published` / `retried` / `dead` counts from the aggregated summary as the
`keiro.outbox.published`, `keiro.outbox.retried`, and `keiro.outbox.deadlettered` counters. Backlog
gauging and stale-row reclaiming moved to `outboxMaintenancePass`, so the publish hot path does not
run an extra `COUNT(*)` or sweep on every call.

## `publishBatch`: publish, then mark ordered groups

```haskell
publishBatch [] = pure emptySummary
publishBatch batch =
  case options ^. #orderingPolicy of
    StopTheLine -> publish singleton batches until the first failure
    policy -> do
      outcomes <- publishRows batch
      markProcessedOutcomes policy batch outcomes Nothing
```

For non-`StopTheLine` policies, the worker calls your publisher once with the whole claimed batch
inside an optional OpenTelemetry producer span. The span records `keiro.outbox.batch.size`, and a
publish failure sets `error.type = publish_failed`. The callback returns outcomes keyed by
`OutboxId`; missing outcomes become `PublishFailed "publisher returned no outcome"`, and a thrown
exception fails every row in the callback.

Then keiro groups the claimed rows according to the active ordering policy:

- `PerKeyHeadOfLine` groups by `(source, key)` and leaves null-keyed rows independent.
- `PerSourceStream` groups by `source`.
- `BestEffort` treats every row independently.
- `StopTheLine` is handled as singleton batches and halts after the first failed row.

Within each ordered group, keiro marks the successful prefix sent with `markOutboxSentBatch`, marks
the failed pivot retryable or dead with `markOutboxFailedTx`, and returns the suffix to retryable
state without consuming attempts. That preserves per-key/per-source order while allowing a claimed
batch to contain long contiguous runs for the same key.

<Callout type="warn">
Publish is **at-least-once**: mark a row sent only *after* the broker acknowledges. A crash after the
broker ack but before `markOutboxSent` re-publishes the row on the next pass — the consumer's inbox
([chapter 02](/docs/keiro/walkthrough/integration/02-the-inbox)) dedupes the double-send on the
stable `messageId`.
</Callout>

## `markOutboxFailedTx`: the dead-letter decision

```haskell
-- keiro/src/Keiro/Outbox/Schema.hs
markOutboxFailedTx :: OutboxId -> Text -> Int -> NominalDiffTime -> UTCTime -> Tx.Transaction OutboxStatus
markOutboxFailedTx outboxId errMsg maxAttempts delay now = do
  currentAttempt <- Tx.statement (unOutboxId outboxId) readAttemptCountStmt
  let attempt = fromMaybe 0 currentAttempt
      shouldDie = attempt >= maxAttempts
      nextStatus = if shouldDie then OutboxDead else OutboxFailed
      nextAttempt = addUTCTime delay now
  Tx.statement (unOutboxId outboxId, statusText nextStatus, errMsg, nextAttempt, now) markFailedStmt
  pure nextStatus
```

```sql
-- markSentStmt
UPDATE keiro_outbox SET status = 'sent', published_at = $2, last_error = NULL, updated_at = $2 WHERE outbox_id = $1
-- markFailedStmt
UPDATE keiro_outbox SET status = $2, last_error = $3, next_attempt_at = $4, updated_at = $5 WHERE outbox_id = $1
```

`markOutboxFailedTx` reads the row's current `attempt_count`: `>= maxAttempts` → `OutboxDead`
(terminal; the row stays in the table for operator inspection), otherwise `OutboxFailed` with
`next_attempt_at = now + delay` (so the claim picks it up again after the backoff). It runs inside the
caller's transaction so the "read count → write status" pair is atomic across concurrent workers. It
returns the resulting `OutboxStatus` so the worker can update its counters. `markOutboxSent` is the
success counterpart: `status = 'sent'`, `published_at`, cleared `last_error`.

## Backoff: `BackoffSchedule`, `nextDelay`, `ExponentialBackoffOptions`

```haskell
-- keiro/src/Keiro/Outbox/Types.hs
data ExponentialBackoffOptions = ExponentialBackoffOptions
  { initial :: !NominalDiffTime, maxDelay :: !NominalDiffTime, multiplier :: !Double }
  deriving stock (Generic, Eq, Show)

data BackoffSchedule = ConstantBackoff !NominalDiffTime | ExponentialBackoff !ExponentialBackoffOptions
  deriving stock (Generic, Eq, Show)

nextDelay :: BackoffSchedule -> Int -> NominalDiffTime
nextDelay (ConstantBackoff delay) _ = delay
nextDelay (ExponentialBackoff opts) attempt =
  let raw = (opts ^. #initial) * realToFrac ((opts ^. #multiplier) ** fromIntegral (max 0 (attempt - 1)))
   in min (opts ^. #maxDelay) raw

defaultPublishOptions :: OutboxPublishOptions
defaultPublishOptions = OutboxPublishOptions
  { batchSize = 32, maxAttempts = 10, backoff = ConstantBackoff 2
  , orderingPolicy = PerKeyHeadOfLine, publishingTimeout = 300, tracer = Nothing }
```

`ConstantBackoff` is a fixed delay between retries. `ExponentialBackoff` grows as `min maxDelay
(initial * multiplier ^ (attempt - 1))`, with `attempt` 1-based (`1` = first failure, so the first
retry waits `initial`). `defaultPublishOptions` ships a batch of 32, ten attempts, a two-second
constant backoff, and per-key head-of-line ordering. For tuning, see [Choose an outbox ordering
policy](/docs/keiro/how-to/choose-an-outbox-ordering-policy).

## `StopTheLine` and the summary

```haskell
data OutboxPublishOptions = OutboxPublishOptions
  { batchSize :: !Int, maxAttempts :: !Int, backoff :: !BackoffSchedule
  , orderingPolicy :: !OrderingPolicy, publishingTimeout :: !NominalDiffTime
  , tracer :: !(Maybe Tracer) }
  deriving stock (Generic)

data OutboxPublishSummary = OutboxPublishSummary
  { claimed :: !Int, published :: !Int, retried :: !Int, dead :: !Int, haltedOn :: !(Maybe OutboxId) }
  deriving stock (Generic, Eq, Show)
```

Under `StopTheLine`, keiro sends singleton batches, halts on the first `PublishFailed`, records the
offending row in `haltedOn`, and marks the remaining claimed rows skipped without consuming attempts.
Every other policy continues through the claimed batch. The summary's invariant is
`published + retried + dead == claimed`; `haltedOn` is the failed pivot row already counted in
`retried` or `dead`.

A crash anywhere in this drain — after the claim flipped a row to `publishing` but before the mark —
is recovered by the separately scheduled `outboxMaintenancePass` after `publishingTimeout`: the row
moves back to `failed` when it is still below the attempt ceiling, or to `dead` when it has exhausted
attempts. See
[chapter 03](/docs/keiro/walkthrough/integration/03-the-outbox-enqueue-and-claim) for the reclaim
path and [Choose an outbox ordering
policy](/docs/keiro/how-to/choose-an-outbox-ordering-policy#recovering-stale-publishing-rows) for
tuning.
