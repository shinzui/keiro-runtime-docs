title: "03 — The migration runner"
description: "Reading Keiro.Migrations: manifest embedding, dependency-ordered components, standard CLI dispatch, dedicated schema, and legacy-history import."
---

Keiro's migration package separates three responsibilities: internal exact-byte component
construction, a small public composition facade, and an application-owned CLI executable.

## Exact bytes become the `keiro` component

`Keiro.Migrations.Internal.Definition` embeds the ordered manifest and delegates validation to
pg-migrate:

```haskell
embeddedMigrationEntries =
  $(embedMigrationManifest "migrations/manifest")

keiroMigrations =
  migrationComponentFromEmbeddedSql
    "keiro"
    (Set.singleton "kiroku")
    embeddedMigrationEntries
```

The stable component name becomes part of every durable identity. The declared dependency means no
valid plan can place Keiro before Kiroku or omit Kiroku.

The manifest currently contains eighteen entries. Positions `0001`–`0016` preserve historical
payloads, `0017-schema-management-comment` is a non-destructive native canary, and `0018` creates
`keiro.keiro_dead_letters` for rejected dispatch evidence.

## The public facade composes concrete values

`Keiro.Migrations` does not expose embedded bytes or runner actions:

```haskell
frameworkMigrationPlan kiroku keiro =
  migrationPlan (kiroku :| [keiro])
```

The caller obtains Kiroku's component from `Kiroku.Store.Migrations`, obtains Keiro's from this
facade, and passes both concrete values. Applications adding PGMQ or their own component construct a
larger plan themselves.

## The CLI remains application-owned

`app/Main.hs` builds that plan, mounts `migrationCommandParser`, resolves `DATABASE_URL` into Hasql
settings, dispatches with `runMigrationCommand`, selects text or JSON from the parsed command, and
maps `ExitClass` to process exit.

It therefore exposes the standard command algebra rather than a Keiro-specific runner:

```text
plan  list  check --manifest PATH  status  verify  up  repair  new
```

No command reads Codd settings, an expected-schema tree, or a runtime SQL directory. Execution uses
the pg-migrate dedicated-connection/advisory-lock lifecycle and ledger v1.

## Schema ownership is physical

The SQL qualifies framework objects under `keiro`, matching `Keiro.Schema.keiroSchema`. Kiroku's
event-store objects remain in `kiroku`; application projection/read-model schemas are configured and
owned separately. This is not merely a `search_path` convention.

`0018` adds `keiro_dead_letters`, distinct from `kiroku.dead_letters`: the former records rejected
target dispatches after a source event was handled, while the latter records terminal source
subscription delivery.

## Codd survives only as cutover evidence

`Keiro.Migrations.History.Codd` pairs the seven Kiroku and sixteen Keiro legacy names with their
native component identities. `frameworkCoddSourceConfig` combines exact payload maps and manifests;
`frameworkCoddHistoryMappings` covers both gap-free prefixes.

The generic importer reads and locks the predecessor, verifies those 23 mappings, and atomically
writes target ledger/audit rows without executing the target actions. The first native `up` then
applies only Kiroku `0008` and Keiro `0017`–`0018`.

This distinction is the handoff:

| Database | First operation |
| --- | --- |
| fresh | run the native combined plan |
| historical Codd, already in dedicated schemas | import combined history, then native `up` |
| Keiro 0.1 tables still in `kiroku` | relocate tables, import history, then native `up` |

Continue with [Run the migrations](/docs/keiro/how-to/run-migrations) for the operator path or
[Upgrade a 0.1 database](/docs/keiro/how-to/upgrade-a-0-1-database) for the cutover.
