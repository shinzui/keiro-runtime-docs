This chapter reads `keiro-migrations/src/Keiro/Migrations.hs` (the runner),
`keiro-migrations/app/Main.hs` (the `keiro-migrate` CLI), and the three embedded SQL files. It is
self-contained: keiro bakes its schema into the binary and applies it through
[`codd`](https://github.com/mzabani/codd), a Haskell migration tool that keeps a **ledger** of
applied migrations so re-running is idempotent.

## Compile-time embedding with `embedDir`

The SQL is not read from disk at run time — it is inlined into the binary at **compile** time with a
Template Haskell splice:

```haskell
-- keiro-migrations/src/Keiro/Migrations.hs
embeddedMigrationFiles :: [(FilePath, ByteString)]
embeddedMigrationFiles = $(embedDir "sql-migrations")
```

`embedDir` (from `Data.FileEmbed`) reads every file in `sql-migrations/` at compile time and produces
a `[(FilePath, ByteString)]` of their names and bytes. The deployed `keiro-migrate` binary therefore
carries the schema with it — no loose `.sql` files to ship or lose.

## Parsing each file into a codd migration

`keiroFrameworkMigrations` turns the embedded bytes into codd's `AddedSqlMigration` values, one per
file, ordered by their timestamped filenames:

```haskell
-- keiro-migrations/src/Keiro/Migrations.hs
keiroFrameworkMigrations :: (MonadFail m, EnvVars m) => m [AddedSqlMigration m]
keiroFrameworkMigrations =
  traverse parseEmbeddedMigration embeddedMigrationFiles
 where
  parseEmbeddedMigration (name, bytes) = do
    let stream = PureStream $ Streaming.yield (TE.decodeUtf8 bytes)
    result <- parseAddedSqlMigration name stream
    case result of
      Left err -> fail ("Invalid Keiro migration " <> name <> ": " <> err)
      Right migration -> pure migration
```

Each file's bytes are decoded to UTF-8, wrapped in a `PureStream`, and handed to
`parseAddedSqlMigration`. A parse failure is **fatal**: the helper `fail`s with the offending
filename, so a malformed migration aborts the run rather than silently skipping. On success it yields
the parsed `AddedSqlMigration`.

## Three accessors: framework-only, the alias, and the full set

```haskell
-- keiro-migrations/src/Keiro/Migrations.hs
-- Compatibility alias for Keiro-owned framework migrations only.
keiroMigrations :: (MonadFail m, EnvVars m) => m [AddedSqlMigration m]
keiroMigrations = keiroFrameworkMigrations

-- Kiroku event-store migrations followed by Keiro framework migrations.
allKeiroMigrations :: (MonadFail m, EnvVars m) => m [AddedSqlMigration m]
allKeiroMigrations = do
  kiroku <- Kiroku.kirokuMigrations
  keiro <- keiroFrameworkMigrations
  pure (kiroku <> keiro)
```

`keiroMigrations` is a plain alias of `keiroFrameworkMigrations` (keiro-owned SQL only) — use it when
composing keiro's schema into a service that owns its own migration list. `allKeiroMigrations`
concatenates **kiroku's** event-store migrations **then** keiro's framework migrations.

<Callout type="warn">
The concatenation order in `allKeiroMigrations` (kiroku, *then* keiro) is **load-bearing**. keiro's
tables live in the `kiroku` schema and reference the event-store tables, so kiroku's migrations must
run first — flip the order and the framework tables would reference objects that do not yet exist.
</Callout>

## The four runners

Each accessor is paired with two runners — one that verifies the resulting schema, one that skips the
check — giving four entry points:

```haskell
-- keiro-migrations/src/Keiro/Migrations.hs
runKeiroMigrations           :: CoddSettings -> DiffTime -> VerifySchemas -> IO ApplyResult
runKeiroMigrationsNoCheck    :: CoddSettings -> DiffTime -> IO ()
runAllKeiroMigrations        :: CoddSettings -> DiffTime -> VerifySchemas -> IO ApplyResult
runAllKeiroMigrationsNoCheck :: CoddSettings -> DiffTime -> IO ()
```

The `runKeiro*` pair applies keiro-owned SQL only; the `runAll*` pair applies the kiroku-then-keiro
set. Their bodies all share one shape — wrap codd's logger, fetch the migration list, then apply it:

```haskell
-- keiro-migrations/src/Keiro/Migrations.hs (runAllKeiroMigrations)
runAllKeiroMigrations settings connectTimeout verifySchemas =
  runCoddLogger $ do
    migrations <- allKeiroMigrations
    applyMigrations settings (Just migrations) connectTimeout verifySchemas
```

```haskell
-- keiro-migrations/src/Keiro/Migrations.hs (runAllKeiroMigrationsNoCheck)
runAllKeiroMigrationsNoCheck settings connectTimeout =
  runCoddLogger $ do
    migrations <- allKeiroMigrations
    applyMigrationsNoCheck settings (Just migrations) connectTimeout (\_ -> pure ())
```

`runCoddLogger` supplies codd's logging context. The `*NoCheck` runners call
`applyMigrationsNoCheck`, which skips the expected-schema comparison (`VerifySchemas`) — useful for a
local dev database that keeps no checked-in expected-schema representation — but it **keeps codd's
migration ledger and locking**, so it is still idempotent. The `DiffTime` argument is the connect
timeout.

## Idempotency

Re-running the migrations is safe by two independent mechanisms. First, codd's **ledger**: it records
which migrations have been applied and skips them on the next run (this holds even for the `*NoCheck`
runners). Second, the SQL itself is written re-runnably — every `CREATE TABLE IF NOT EXISTS` and
`CREATE INDEX IF NOT EXISTS`. Either alone makes a second run a no-op; together they make the runner
safe to call on every deploy.

## The `keiro-migrate` CLI

The executable (`app/Main.hs`) is a thin dispatcher over the `runAll*` pair:

```haskell
-- keiro-migrations/app/Main.hs
main :: IO ()
main = do
  settings <- getCoddSettings
  noCheck <- lookupEnv "KEIRO_MIGRATE_NO_CHECK"
  case noCheck of
    Just _ ->
      runAllKeiroMigrationsNoCheck settings (secondsToDiffTime 5)
    Nothing -> do
      _ <- runAllKeiroMigrations settings (secondsToDiffTime 5) LaxCheck
      pure ()
```

`getCoddSettings` reads codd's environment (the connection string and the managed schema). The CLI
then dispatches on `KEIRO_MIGRATE_NO_CHECK`: if the variable is **set**, it runs the no-check path;
otherwise it runs the verifying path with `LaxCheck`. Both apply the **full** kiroku+keiro set with a
5-second connect timeout (`secondsToDiffTime 5`). For the invocation, read [Run the
migrations](/docs/keiro/how-to/run-migrations).

## The three files, the five tables

The embedded files apply in filename-timestamp order; each creates tables in the `kiroku` schema:

```text
2026-05-17-00-00-00-keiro-bootstrap.sql  -> keiro_snapshots, keiro_read_models, keiro_timers
2026-05-17-01-00-00-keiro-outbox.sql     -> keiro_outbox
2026-05-17-02-00-00-keiro-inbox.sql      -> keiro_inbox
```

Each table's purpose, with a cross-link to the subsystem that owns it:

- **`keiro_snapshots`** — the latest persisted state per stream, keyed by `stream_id`, with a
  compatibility index (`keiro_snapshots_compat_idx`) on the codec version and shape hash. See
  [Snapshot](/docs/keiro/reference/snapshot).
- **`keiro_read_models`** — the registry of read-model build status (version, shape hash,
  `last_built_at`). See [Read model](/docs/keiro/reference/read-model).
- **`keiro_timers`** — scheduled process-manager timers, with a due index
  (`keiro_timers_due_idx`) on `(status, fire_at, process_manager_name)`. See
  [Timers](/docs/keiro/reference/timers).
- **`keiro_outbox`** — the transactional outbox of pending integration events, with a pending index
  and a per-key head-of-line index. See [Outbox](/docs/keiro/reference/outbox).
- **`keiro_inbox`** — the consumer-side dedupe ledger keyed by `(source, dedupe_key)`. See
  [Inbox](/docs/keiro/reference/inbox).

For the column-level detail of every table, read the
[Migrations and schema reference](/docs/keiro/reference/migrations-and-schema) — this tour names the
tables and walks the runner rather than re-listing each field.
