---
title: "Your first read model"
description: "Create an application-schema table, register a read model explicitly, update it inline, and query it."
---

By the end of this tutorial you will have a small order-summary read model. Its table lives in an
application schema, an inline projection updates it in the event append transaction, and startup
registers its schema identity before the first query.

<Callout type="info">
  This tutorial starts after [Get started with Keiro](/docs/keiro/tutorials/getting-started): you
  already have a migrated Kiroku/Keiro database and a `ValidatedEventStream` for orders.
</Callout>

## 1. Create an application-owned table

Choose an application schema, such as `app_reads`, and create the table through your normal
migration process:

```sql
CREATE SCHEMA IF NOT EXISTS app_reads;

CREATE TABLE app_reads.order_summary (
  order_id text PRIMARY KEY,
  status text NOT NULL
);
```

Keiro's registry stays in `keiro.keiro_read_models`; projection data should not. In development or
tests, `ensureProjectionSchema "app_reads"` can create the schema, but Keiro never calls it
automatically. Use `keiroConnectionSettings connectionString "app_reads"` (or
`withProjectionSchema`) when the pool should also resolve that application schema.

## 2. Define qualified projection statements

Both the writer and reader should name the application table explicitly:

```sql
INSERT INTO "app_reads"."order_summary" (order_id, status)
VALUES ($1, $2)
ON CONFLICT (order_id) DO UPDATE SET status = EXCLUDED.status;

SELECT status
FROM "app_reads"."order_summary"
WHERE order_id = $1;
```

`qualifyTable "app_reads" "order_summary"` and, once the model exists,
`qualifiedTableName orderSummaryReadModel` both produce the safely quoted table reference. Keiro
does not rewrite the SQL inside `query` or `apply`.

## 3. Define the inline projection and read model

```haskell
orderSummaryInlineProjection :: InlineProjection OrderEvent
orderSummaryInlineProjection = InlineProjection
  { name = "order-summary-inline"
  , apply = \event recorded ->
      Tx.statement (summaryParams event recorded) upsertOrderSummaryStmt
  }

orderSummaryReadModel :: ReadModel OrderId (Maybe Text)
orderSummaryReadModel = ReadModel
  { name               = "order-summary"
  , tableName          = "order_summary"
  , schema             = "app_reads"
  , subscriptionName   = "order-summary-inline"
  , version            = 1
  , shapeHash          = "order-summary-v1"
  , defaultConsistency = Eventual
  , strongScope        = EntireLog
  , query              = \orderId -> Tx.statement orderId selectOrderSummaryStmt
  }
```

The model is inline-only, so `Eventual` is the correct default: its row commits with the event and
there is no async subscription cursor for `Strong` to wait on. `strongScope` is still an explicit
part of every `ReadModel`; it matters if this model later becomes async-fed.

## 4. Register the model at startup

After migrations have created the table, register the definition before serving commands or reads:

```haskell
_ <- registerReadModel
       (orderSummaryReadModel ^. #name)
       (orderSummaryReadModel ^. #version)
       (orderSummaryReadModel ^. #shapeHash)
```

Registration is idempotent. It does not overwrite an existing version or hash, which preserves
schema-drift detection. A query does not register on demand: missing startup wiring returns
`ReadModelUnregistered "order-summary"`.

## 5. Run a command with the projection

Under your configured `KirokuStoreResource` interpreter, run:

```haskell
result <- runCommandWithProjections
            defaultRunCommandOptions
            orderEventStream
            targetStream
            command
            [orderSummaryInlineProjection]
```

`runCommandWithProjections` uses the resource-aware SQL command path, so Kiroku enrichment happens
before projection callbacks observe the recorded events. If the projection fails, the append and
table update both roll back.

## 6. Query the registered model

```haskell
summary <- runQuery Nothing orderSummaryReadModel orderId
```

After a successful inline command, `summary` sees the committed row without waiting. Its result is
`Either ReadModelError (Maybe Text)`: an unregistered definition, stale version/hash, or non-`Live`
status fails before the table query runs.

You now have an application-schema read model with explicit startup registration and an atomic
inline update. Next, use [Choose a consistency mode](/docs/keiro/how-to/choose-a-consistency-mode)
when moving the projection to an asynchronous worker.
