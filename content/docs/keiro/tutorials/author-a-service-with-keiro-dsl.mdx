This tutorial builds the `transfer-routing` service from the current conformance corpus. It is a
small but realistic vertical: a registered read model selects hospitals, a router turns an accepted
transfer need into one command per selected hospital, and an aggregate records each routed need.

The finished source is `keiro-dsl/test/fixtures/transfer-routing.keiro` in the Keiro repository. Its
filled proof lives under `keiro-dsl/test/conformance-newsurface/`; it does not depend on the separate
example application.

<Callout type="info">
Run from the Keiro repository root. The installed command is shown as `keiro-dsl`; in the source tree
use `cabal run keiro-dsl --`, or put `keiro-dsl/bin` on `PATH`.
</Callout>

## 1. Start with the service topology

Create `transfer-routing.keiro`:

```keiro-dsl
context transfer-routing

readmodel hospital_load {
  table = "hospital_load"
  schema = "hospital_transfer"
  columns {
    hospital_id text required
    region text required
    available_beds int required
  }
  version = 1
  shape = "fnv1a:977395d28f254ddb"
  consistency = Eventual
  feed = subscription
  subscription = "transfer-routing-hospital-load-sub"
}

router HospitalTransferRouter
  name "hospital-transfer-router"
  input AcceptedHospitalTransferNeed { transferNeedId region }
  key input.transferNeedId via idText
  resolve stable via read-model hospital_load row { hospitalId }
  target Hospital
  projections [ ]
  dispatch-each RouteAcceptedTransferNeed {
    transferNeedId=input.transferNeedId hospitalId=resolved.hospitalId
  }
    on-appended AckOk ; on-duplicate AckOk ; on-failed Retry
  dispatch-id strategy=uuidv5 from=(name, key, sourceEventId, targetStreamName, occurrence)
  rejected => deadLetter
  poison => halt

aggregate Hospital
  regs
  states Accepting

  command RouteAcceptedTransferNeed { transferNeedId hospitalId }
  event AcceptedTransferNeedRouted = fields(RouteAcceptedTransferNeed)

  Accepting -- RouteAcceptedTransferNeed -->
    emit AcceptedTransferNeedRouted
    goto Accepting
```

The spec states the persistence and delivery decisions, not arbitrary application code. In
particular:

- the read-model table is schema-qualified and its ordered column surface is captured by `shape`;
- `resolve stable` acknowledges that retrying a source event must not discover a drifting recipient
  set when exact fan-out matters;
- the router identity includes its stable name, source key, source event, target stream, and
  occurrence, so two selected hospitals receive distinct deterministic dispatch IDs;
- rejection and poison handling are explicit worker policies; and
- the aggregate transition is durable because it emits an event before returning to `Accepting`.

## 2. Parse and check before generating Haskell

Normalize the syntax, then run the semantic gate:

```bash
keiro-dsl parse transfer-routing.keiro
keiro-dsl check transfer-routing.keiro
# OK
```

`parse` proves the file is valid notation and prints its canonical form. `check` resolves the router,
read-model, aggregate, command, input, row, and binding references. It also verifies the captured
read-model shape and the router's policy table. Either command exits nonzero on failure.

Use `check --emit` when you want both steps in one command:

```bash
keiro-dsl check transfer-routing.keiro --emit
```

Fix a failure in the `.keiro` file. Do not compensate for a rejected spec in generated Haskell.

## 3. Scaffold the two ownership domains

Generate into a disposable directory first:

```bash
keiro-dsl scaffold transfer-routing.keiro --out gen
```

The output is intentionally split:

| Node | Overwritable generated modules | Create-once, hand-owned modules |
|---|---|---|
| `Hospital` | `Domain`, `Codec`, `EventStream`, `Projection`, `Harness` | `Hospital/Holes.hs` |
| `hospital_load` | `ReadModelTable`, `ReadModel`, `ReadModelHarness` | `Hospital_load/ReadModelHoles.hs` |
| `HospitalTransferRouter` | `Router`, `RouterHarness` | `HospitalTransferRouter/RouterHoles.hs` |

Every generated file carries `-- @generated`. Before writing, the scaffolder validates the whole
output plan: module paths, type and policy lowering, firewall cleanliness, and existing generated
file banners. If a target exists without the banner, scaffold exits 1 and writes nothing. Move the
hand-owned file, or use `--force-generated-overwrite` only when deliberately adopting that path as
generated code.

A successful report also names the harnesses, confirms `firewall: OK …`, and writes:

- `keiro-dsl-manifest.transfer-routing.txt`, with Cabal `other-modules:` and `build-depends:` blocks;
- `keiro-dsl-scaffold-record.transfer-routing.txt`, used on later runs to report stale paths.

Re-scaffolding never deletes stale files. Review `stale: generated` candidates before deleting them,
and treat every `stale: hole` entry as hand-owned code.

## 4. Fill the behavior-bearing seams

Never edit a generated file. Fill the aggregate transducer in `Hospital/Holes.hs`, the qualified SQL
and projection behavior in `Hospital_load/ReadModelHoles.hs`, and the resolver/runtime assembly in
`HospitalTransferRouter/RouterHoles.hs` (the corpus keeps the completed router value in
`RouterValue.hs`).

The read-model hole imports the generated table constant:

```haskell
import Generated.TransferRouting.Hospital_load.ReadModelTable
  (hospitalLoadQualifiedTable)
```

Use that value in SQL instead of relying on PostgreSQL `search_path`. The router fill queries the
read model, constructs one `PMCommand` per row, builds targets with the generated `hospitalCategory`,
and passes the generated `hospitalTransferRouterWorkerOptions` to `runRouterWorkerWith`.

`on-duplicate AckOk` is safe only because the generated runtime confirms the attempted event ID in
the intended target stream. A hand-written dispatch path must call `confirmBenignDuplicate`; a bare
`DuplicateEvent` match is not evidence of success.

## 5. Run the cold-start proof

The checked-in proof compiles the generated and hand-owned sides together:

```bash
cabal test keiro-dsl-conformance-newsurface
```

It asserts the aggregate harness, validated event-stream construction, read-model facts, router
name/key/policy, stable resolver source, selected target stream, and dispatched command. A green
scaffold alone does not prove your fills; the harness does.

After any spec change, re-scaffold and rerun the same component. Generated modules are refreshed;
filled holes remain in place.

## 6. Gate evolution with `diff`

Commit a known-good spec, then compare the working copy with it:

```bash
keiro-dsl diff --since HEAD transfer-routing.keiro
```

Each line has this shape:

```text
<ADDITIVE|WARNING|BREAKING>: <node> <facet> <subject>: <detail> [optional-code]
```

Only `BREAKING` makes the command exit nonzero. For this service, changing the router `name`, key
derivation, read-model identity/table/subscription, target aggregate, or aggregate wire identity is
breaking because durable rows or retries still refer to the old identity. A forward-policy change
may be a visible `WARNING`; a safe new surface is `ADDITIVE`.

## You now own the loop

The repeatable lifecycle is:

```text
new/write → parse/check → scaffold → fill → harness → diff → re-scaffold
```

Keep the [notation and CLI reference](/docs/keiro/reference/keiro-dsl-notation) nearby. If you are
upgrading an older spec, use
[Migrate a keiro spec to 0.2](/docs/keiro/how-to/migrate-a-keiro-spec-to-0-2) before scaffolding.
