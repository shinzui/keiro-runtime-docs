---
title: "Your first durable workflow"
description: "Write a small durable workflow with a named step, a durable sleep, and an external awakeable; run it to suspension, resume it, and watch a journaled step replay without running again."
---

In this tutorial you write a **durable workflow** — an ordinary function whose checkpoints survive a
crash — run it to its first suspension, resume it past a durable sleep, signal an external awakeable,
and prove that a journaled step replays without running again. You will see the journal-and-replay
model do its work with your own output.

This is a learning lesson; for the concept first, skim [Understanding durable
execution](/docs/keiro/explanation/durable-execution). The runnable, end-to-end version of everything
here is the jitsurei `workflow` demo (`cabal run jitsurei:exe:jitsurei-demo -- workflow`), whose source
is `jitsurei/src/Jitsurei/DurableWorkflow.hs`.

## What you need

An open kiroku `Store` with keiro's migrations applied (the [getting-started
tutorial](/docs/keiro/tutorials/getting-started) covers opening a store; [Run the
migrations](/docs/keiro/how-to/run-migrations) covers the schema, which now includes the workflow
tables). The imports for this tutorial:

```haskell
import Keiro.Workflow           (StepName (..), WorkflowId (..), step, runWorkflow, WorkflowOutcome (..))
import Keiro.Workflow.Sleep     (sleepNamed, runWorkflowTimerWorker)
import Keiro.Workflow.Awakeable (AwakeableId, awakeableNamed, signalAwakeable)
import Keiro.Workflow.Resume    (resumeWorkflowsOnce, defaultWorkflowResumeOptions, WorkflowDef (..), WorkflowRegistry)
import Keiro.Workflow.Types     (WorkflowName (..))
import qualified Data.Map.Strict as Map
```

<Steps>
<Step>

### Write the workflow

A workflow is a `do`-block in the `Workflow` effect. Ours reserves inventory, waits out a short
cooling-off period, then waits for a payment webhook before charging:

```haskell
greetWorkflow :: (Workflow :> es, Store :> es, IOE :> es) => Text -> Eff es Text
greetWorkflow name = do
  _ <- step (StepName "reserve") $ do
         liftIO (putStrLn ("  reserve ran for " <> Text.unpack name))
         pure ("RES-" <> name)
  sleepNamed (StepName "cooling-off") 2          -- a durable 2-second wait
  (awkId, awaitPayment) <- awakeableNamed (StepName "payment")
  step (StepName "hand-out-payment-id") $
         liftIO (savePaymentAwakeable name awkId)
  payment <- awaitPayment                          -- SUSPEND until an external signal
  step (StepName "charge") $ do
    liftIO (putStrLn ("  charge ran with " <> Text.unpack payment))
    pure ("CHG-" <> name)
```

Read it top to bottom: that is the whole process. Each `step` prints **only when its action actually
runs**, so the transcript will show you exactly when work happens versus when it is replayed from the
journal.

</Step>
<Step>

### Run it once — it suspends

Run the workflow under a name and a stable instance id:

```haskell
let wfName = WorkflowName "greet"
    wfId   = WorkflowId "order-1"

outcome1 <- runWorkflow wfName wfId (greetWorkflow "order-1")
-- prints:  reserve ran for order-1
-- outcome1 == Suspended
```

The `reserve` step ran and was journaled, the cooling-off sleep armed a `keiro_timers` row, and the
run **suspended** — `runWorkflow` returns `Suspended`. Nothing past the sleep has run. The only durable
state of the pause is one timer row; the process could crash here and lose nothing.

</Step>
<Step>

### Fire the sleep timer, then resume

A suspended workflow does not resume itself — a worker turns the crank. First drain the sleep timer
(with a clock past the 2-second delay) so its completion is journaled:

```haskell
now <- getCurrentTime
let later = addUTCTime 5 now
_ <- runWorkflowTimerWorker Nothing later (\_ -> pure Nothing)   -- fires the cooling-off sleep
```

Now run a **resume pass**. It needs a registry mapping the workflow name to a function that rebuilds
the body from an id:

```haskell
let registry :: WorkflowRegistry es
    registry = Map.fromList
      [ (wfName, WorkflowDef (\(WorkflowId i) -> greetWorkflow i)) ]

summary <- resumeWorkflowsOnce defaultWorkflowResumeOptions registry
-- prints nothing from `reserve` (it replayed!), and the run parks on `payment`
```

Notice `reserve` did **not** print this time: it was replayed from the journal, short-circuited to its
recorded result. The run advanced past the now-resolved sleep and suspended again on the `payment`
awakeable.

</Step>
<Step>

### Signal the awakeable

An awakeable is a durable promise an external system resolves. The workflow handed the opaque
`AwakeableId` to the outside world before it suspended; in a real service that might be a webhook link,
an approval row, or a message to another service. Load that id and signal it with a payload:

```haskell
awkId <- loadPaymentAwakeable "order-1"  -- the id saved by savePaymentAwakeable
signalled <- signalAwakeable awkId ("pay_demo" :: Text)
-- signalled == True
```

`signalAwakeable` records the payload in the journal and flips the `keiro_awakeables` row to
`completed`. It is idempotent and crash-safe — signalling twice returns `False` the second time and
changes nothing.

</Step>
<Step>

### Resume to completion

One more resume pass drives the workflow to the end:

```haskell
final <- resumeWorkflowsOnce defaultWorkflowResumeOptions registry
-- prints:  charge ran with pay_demo
-- the workflow reached Completed "CHG-order-1"
```

Only `charge` printed — `reserve` and the `payment` await replayed silently. Run the same `(name, id)`
again and you will see **nothing print at all**: every step is journaled, so the completed workflow is
not re-executed from scratch. That is durability: the workflow's progress lives in the journal, not in
the process.

</Step>
</Steps>

## What just happened

You exercised the three behaviours that make durable execution work:

- **Run-and-journal.** Each journaled `step` replayed from its recorded result. `reserve` printed on
  the first run and never again.
- **Suspend-and-resume.** A `sleepNamed` and an `awakeableNamed` each suspended the run; an external
  wake source (a timer fire, a `signalAwakeable`) journaled the completion; a resume pass continued.
- **Crash safety by re-invocation.** Resuming is just running the same function again — the journal
  makes that cheap and side-effect-safe.
