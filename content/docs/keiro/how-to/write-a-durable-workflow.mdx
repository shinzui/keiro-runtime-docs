You want to write a long-running process as one function whose checkpoints survive a crash, a
redeploy, or an idle wait. This guide assembles a durable workflow from the four primitives and runs
it. For the concept, see [Understanding durable execution](/docs/keiro/explanation/durable-execution);
for every signature, the [durable workflows reference](/docs/keiro/reference/durable-workflows).

## Import the surface

The umbrella `Keiro` module does not re-export workflows — import the submodules directly:

```haskell
import Keiro.Workflow           -- step, runWorkflow/runWorkflowWith, WorkflowOutcome, the journal codec
import Keiro.Workflow.Sleep     -- sleepNamed / sleep
import Keiro.Workflow.Awakeable -- awakeableNamed, signalAwakeable, cancelAwakeable
import Keiro.Workflow.Child     -- spawnChild, awaitChild, cancelChild
```

## Compose the primitives

A workflow is an `Eff (Workflow : es) a`. Use `step` for any side effect, and the three waits in line:

```haskell
orderFulfillmentWorkflow :: (Workflow :> es, Store :> es, IOE :> es) => OrderId -> Eff es Text
orderFulfillmentWorkflow orderId = do
  _reservation <- step (StepName "reserve-inventory") (reserveInventory orderId)
  sleepNamed (StepName "cooling-off") 300                       -- a durable 5-minute wait
  (_awkId, awaitPayment) <- awakeableNamed (StepName "payment-webhook")
  payment <- awaitPayment                                       -- wait for an external signal
  _charge <- step (StepName "charge") (chargeCard orderId payment)
  childHandle <- spawnChild shipOrderWorkflowName (shipChildId orderId) (shipOrderWorkflow orderId)
  awaitChild childHandle                                        -- wait for a child workflow
```

- **`step name action`** runs `action` and journals its JSON result; a replay returns the recorded
  value without re-running. If the process crashes before the append commits, the action may run
  again, so external effects still need idempotency. Anything `ToJSON`/`FromJSON` works as a result.
- **`sleepNamed name delta`** durably pauses (a `keiro_timers` row) and resumes when a timer worker
  fires it. No thread is held.
- **`awakeableNamed name`** returns an opaque, journaled id and an `await`; the workflow suspends until
  an external caller runs `signalAwakeable`.
- **`spawnChild` / `awaitChild`** spawn and wait on a child workflow with its own journal.

<Callout type="warn">
**Name your steps, and keep names stable.** A step's identity is its `StepName`, not its position. The
named forms (`step`, `sleepNamed`, `awakeableNamed`) survive code edits that insert or reorder
unrelated code. The ordinal convenience forms (`sleep`, `awakeable`) derive names from call order and
are only conditionally safe — avoid them for anything that must survive a code change mid-flight.
</Callout>

## Run it

`runWorkflow` runs (or resumes) the workflow and returns a [`WorkflowOutcome`](/docs/keiro/reference/durable-workflows#running-a-workflow):

```haskell
outcome <- runWorkflow (WorkflowName "order-fulfillment") (workflowIdFor orderId) (orderFulfillmentWorkflow orderId)
case outcome of
  Completed tracking -> ...   -- finished; `tracking` is the function's result
  Suspended          -> ...   -- parked at a wait; a resume pass will continue it later
  Cancelled          -> ...   -- a parent cancelled this (child) workflow
  Failed             -> ...   -- a failure marker is already in the journal
  ContinuedAsNew     -> ...   -- rotated; a later pass drives the new generation
```

Running the same `(name, id)` again replays the journal and short-circuits completed steps, so calling
`runWorkflow` is itself the resume mechanism. In production you do not call it by hand on every wait —
you [run the resume worker](/docs/keiro/how-to/run-the-workflow-resume-worker) on a loop.

The runner's effect row must include `Error StoreError`:

```haskell
runOrder :: (IOE :> es, Store :> es, Error StoreError :> es) => OrderId -> Eff es (WorkflowOutcome Text)
runOrder orderId =
  runWorkflow (WorkflowName "order-fulfillment") (workflowIdFor orderId)
    (orderFulfillmentWorkflow orderId)
```

This is the failure boundary, not a requirement on every workflow body. Journal-store errors surface
through that channel. Once a journal append has committed, a snapshot write failure is advisory: the
runner counts `keiro.snapshot.write.failures`, returns the journal-determined outcome, and a later run
can reconstruct state by replaying the journal.

## Resolve and repair the waits

- **Sleeps** fire when a timer worker drains `keiro_timers`. Run `runWorkflowTimerWorker` (it handles
  both workflow sleeps and ordinary process-manager timers in one pass) on your timer polling loop.
- **Awakeables** resolve when an external system calls `signalAwakeable awkId payload`. Hand the
  `AwakeableId` returned by `awakeableNamed` to that system (for example, store it in the approval row
  or include it in the webhook link). New ids are random and journaled so replay recovers the same id;
  `deterministicAwakeableId` exists only for legacy generation-0 adoption. Repair one that will never
  arrive with `cancelAwakeable awkId`; the workflow's `await` then throws
  `WorkflowAwakeableCancelled`, which you can `catch` for compensation.
- **Children** must be **registered in the resume worker's registry** by name — `spawnChild` does not
  run the child inline; the worker drives it. Repair a parent stuck on a never-finishing child by
  driving or cancelling the child (`cancelChild`).

<Callout type="warn">
**A child id names one child execution for that child workflow name.** Reusing the same child
`(WorkflowName, WorkflowId)` attaches to the existing execution and re-delivers its stored result. To
run a fresh child after `continueAsNew`, derive a fresh child id from the carried seed.
</Callout>

## Set run options for long journals

Use `runWorkflowWith` to enable snapshots (so a resume does not replay every step) or telemetry. Set
fields with the **generic-lens label** — a bare record update on `snapshotPolicy` is ambiguous:

```haskell
let opts = defaultWorkflowRunOptions
             & #snapshotPolicy .~ Every 2     -- snapshot the step-result map every 2 appends
             & #metrics        .~ Just metrics
             & #tracer         .~ Just tracer
outcome <- runWorkflowWith opts name wid (orderFulfillmentWorkflow orderId)
```

The resume worker threads the same options through `WorkflowResumeOptions.runOptions`, so a resumed run
honours them too. Snapshots are advisory: a stale or corrupt one degrades to a full replay, and a
post-commit snapshot-store failure is counted without changing the successful step, completion, or
rotation outcome. Alert on `keiro.snapshot.write.failures`; use the workflow journal to verify the
durable result.

## Make step actions safe to re-run

Replay short-circuits a *journaled* step — but a step that **crashes before it journals** re-runs on
the next pass. So keep the action inside a `step` idempotent with respect to the outside world: pass
deterministic external ids, dedupe at the destination, or model the effect through keiro's
[outbox](/docs/keiro/explanation/the-outbox-pattern) / [idempotent
commands](/docs/keiro/how-to/make-commands-idempotent). The journal guarantees *skip after the step is
journaled*; your action should be safe under *at least once until it commits*.

## Versioning a running workflow

When you change a workflow whose instances are already in flight, you have **two** tools — and the
common one needs no API.

**Rename the step.** Because step durability is keyed on the *name*, not the source position, changing
one step's behaviour is just a rename. Change `step (StepName "charge") …` to
`step (StepName "charge-v2") …`: the new name has no journaled history, so its action runs fresh on the
next replay, exactly as you want. **The rule of thumb: if a single rename makes the change correct,
rename.**

**Patch — the escape hatch.** Only when a change *cross-cuts several steps* — an instance already past
some of them must take one stable branch for the rest of its life — reach for `patch`. It returns a
stable, journaled `Bool`: `True` for an instance fresh at that point, `False` for one already in flight
when the patch shipped. For example, wrapping two existing steps in a new fraud check:

```haskell
orderFlow orderId = do
  useFraudCheck <- patch (PatchId "fraud-check-v2")
  if useFraudCheck
    then do
      _ <- step (StepName "reserve-inventory") (reserveInventoryWithHold orderId)
      _ <- step (StepName "fraud-review")      (fraudReview orderId)
      _ <- step (StepName "charge")            (chargeWithHold orderId)
      pure ()
    else do
      _ <- step (StepName "reserve-inventory") (reserveInventory orderId)
      _ <- step (StepName "charge")            (chargeCard orderId)
      pure ()
```

An order already mid-flight (it reserved inventory under the old logic, with no hold to release)
observes `useFraudCheck == False` and finishes on the old path; a fresh order observes `True`. Reach
for `patch` *only* when an in-flight instance would be left incoherent by the new code, and **never
reuse a retired `PatchId`** — the decision is journaled under `patch:<id>` and replayed forever.

## Keep an unbounded history bounded with continue-as-new

A workflow that runs an *unbounded* number of steps — a poller, a per-day rolling process — would grow
its journal without limit. `continueAsNew seed` rotates it onto a fresh journal *generation* so the
per-generation history stays bounded; read the carried state back at the top of the body with
`restoreSeed`:

```haskell
pollerWorkflow :: (Workflow :> es, Store :> es, IOE :> es) => Eff es a
pollerWorkflow = do
  cursor <- restoreSeed (Cursor 0)               -- the carried value, or the default on generation 0
  batch  <- step (StepName "poll") (pollFrom cursor)
  step (StepName "handle") (handleBatch batch)
  sleepNamed (StepName "interval") 60
  continueAsNew (advance cursor batch)           -- rotate onto a fresh generation; never returns
```

`continueAsNew` never returns to its caller (its result type is polymorphic) — it unwinds the run, so
put it at the end of the body. A rotating run returns the `ContinuedAsNew` outcome, and the resume
worker drives the next generation forward automatically. Snapshots compact *within* a generation;
rotation bounds the history *across* generations.
