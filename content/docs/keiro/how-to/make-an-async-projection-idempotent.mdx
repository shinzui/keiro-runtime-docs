---
title: "Make an async projection idempotent"
description: "Use AsyncProjection's idempotency key and keiro_projection_dedup so at-least-once redelivery is safe."
---

An async projection runs off a subscription with **at-least-once** delivery, so the same event may be
applied twice. `AsyncProjection` now centralizes the first line of defense: `applyAsyncProjection`
inserts `(projection_name, event_id)` into `kiroku.keiro_projection_dedup` in the same transaction as
your read-model update, and skips the update when that row already exists.

<Callout type="info">
  Assumes you understand the projection primitives (see
  [Projections, read models, and snapshots](/docs/keiro/explanation/projections-read-models-and-snapshots)).
</Callout>

## Goal

Apply each event at most once per projection within the retained dedup window, even under redelivery.

## Steps

1. **Key the projection off the event id** via `idempotencyKey`, then put the real table update in
   `applyRecorded`:

   ```haskell
   myAsyncProjection :: AsyncProjection
   myAsyncProjection = AsyncProjection
     { name             = "orders-summary-async"
     , subscriptionName = "orders-summary-async"
     , applyRecorded    = \recorded -> Tx.statement (params recorded) upsertSummaryStmt
     , idempotencyKey   = \recorded -> recorded ^. #eventId
     }
   ```

   The subscription worker should run `applyAsyncProjection myAsyncProjection recorded`, not call
   `applyRecorded` directly. Keiro writes the dedup row first; only the first delivery for that
   `(projection, event)` runs `applyRecorded`.

2. **Keep table writes domain-idempotent when practical.** The central dedup table protects ordinary
   redelivery. Still prefer natural upserts in the read model when the business key allows it; they
   make manual rebuilds, operator replays after dedup pruning, and future migrations easier to reason
   about.

<Callout type="warn">
keiro ships **no** async-projection worker. `applyAsyncProjection myAsyncProjection recorded` hands
you the `Tx.Transaction ()`; you must wire it to a kiroku subscription and advance the checkpoint
(`subscriptionName`) yourself. The idempotency key keeps that loop safe under at-least-once delivery.
</Callout>

## Verify it worked

Apply the same `RecordedEvent` twice in a test through `applyAsyncProjection`: the first call writes
one `keiro_projection_dedup` row and updates the read model; the second call conflicts on the dedup
primary key and skips `applyRecorded`.
