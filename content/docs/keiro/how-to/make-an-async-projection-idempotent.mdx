---
title: "Make an async projection idempotent"
description: "Deduplicate redelivery, honor the read-model writer fence, and checkpoint every async outcome correctly."
---

An async projection receives at-least-once delivery. Keiro makes each application idempotent within
the retained dedup window and returns a distinct outcome when a rebuild fences the live worker.

## Define the projection

Register the read model at startup, then make the projection name that registry identity:

```haskell
ordersProjection :: AsyncProjection
ordersProjection = AsyncProjection
  { name             = "orders-summary-async"
  , readModelName    = "orders-summary"
  , subscriptionName = "orders-summary-sub"
  , applyRecorded    = \recorded ->
      Tx.statement (params recorded) upsertQualifiedSummaryStmt
  , idempotencyKey   = \recorded -> recorded ^. #eventId
  }
```

`upsertQualifiedSummaryStmt` should address the application table with a schema-qualified name such
as `"app_reads"."order_summary"`.

## Apply and classify every outcome

Run `applyAsyncProjection ordersProjection recorded` in the delivery transaction, then couple the
delivery acknowledgement/checkpoint to the result:

```haskell
case outcome of
  AsyncApplied ->
    acknowledgeAndCheckpoint
  AsyncDuplicate ->
    acknowledgeAndCheckpoint
  AsyncFenced ->
    failOrParkWithoutCheckpoint
```

- `AsyncApplied` inserts `(projection_name, event_id)` and runs `applyRecorded` in the same
  transaction.
- `AsyncDuplicate` finds that retained key and skips the table update; it is idempotent success.
- `AsyncFenced` means the registry row is missing or not `Live`. It performs neither dedup nor table
  writes.

<Callout type="warn">
  Never acknowledge or advance the subscription checkpoint for `AsyncFenced`. Doing so skips the
  event while the rebuild owns the model. Fail or park the delivery and retry it after guarded
  promotion returns the model to `Live`.
</Callout>

Keep natural-key upserts in `applyRecorded` when practical. The central dedup table handles normal
redelivery, while domain-idempotent writes make operator replay and deliberate dedup pruning easier
to reason about.

## Retain dedup keys long enough

Use `pruneAsyncProjectionDedupBefore` only before the oldest event the subscription system can
redeliver. Pruning reopens those events for application. It is not the rebuild mechanism:
`startRebuild` atomically clears only the named projections' keys while fencing writers, truncating
the model table, and resetting the checkpoint.

## Verify all three paths

In integration tests:

1. Apply an event while the model is `Live`; expect `AsyncApplied` and one table update.
2. Redeliver it; expect `AsyncDuplicate` and no second update.
3. Call `startRebuild`, then try the live entry point; expect `AsyncFenced`, no dedup row, no table
   update, and no checkpoint advance.

See [Projection](/docs/keiro/reference/projection) for the exact surface and
[Rebuild a read model](/docs/keiro/how-to/rebuild-a-read-model) for the unfenced replay entry point.
