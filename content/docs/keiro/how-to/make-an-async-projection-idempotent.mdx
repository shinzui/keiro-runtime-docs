---
title: "Make an async projection idempotent"
description: "Use a source-event-id unique index and ON CONFLICT DO NOTHING so at-least-once redelivery is safe."
---

An async projection runs off a subscription with **at-least-once** delivery, so the same event may be
applied twice. You make the apply idempotent so a redelivery writes nothing.

<Callout type="info">
  Assumes you understand the projection primitives (see
  [Projections, read models, and snapshots](/docs/keiro/explanation/projections-read-models-and-snapshots)).
</Callout>

## Goal

Apply each event at most once to the read-model table, even under redelivery.

## Steps

1. **Add a unique key on the source event id** to the read-model table:

   ```sql
   ALTER TABLE my_read_model ADD COLUMN source_event_id UUID;
   CREATE UNIQUE INDEX my_read_model_source_event_id_uniq ON my_read_model (source_event_id);
   -- in the projection's Tx, INSERT ... ON CONFLICT (source_event_id) DO NOTHING
   ```

2. **Key the projection off the event id** via `idempotencyKey`, and write with `ON CONFLICT DO
   NOTHING`:

   ```haskell
   myAsyncProjection :: AsyncProjection
   myAsyncProjection = AsyncProjection
     { name             = "orders-summary-async"
     , subscriptionName = "orders-summary-async"
     , applyRecorded    = \recorded -> Tx.statement (params recorded) upsertOnConflictDoNothingStmt
     , idempotencyKey   = \recorded -> recorded ^. #eventId
     }
   ```

<Callout type="warn">
keiro ships **no** async-projection worker. `applyAsyncProjection myAsyncProjection recorded` hands
you the `Tx.Transaction ()`; you must wire it to a kiroku subscription and advance the checkpoint
(`subscriptionName`) yourself. The idempotency key keeps that loop safe under at-least-once delivery.
</Callout>

## Verify it worked

Apply the same `RecordedEvent` twice in a test: the second `INSERT … ON CONFLICT DO NOTHING` affects
zero rows, and the table holds exactly one row for that `source_event_id`.
