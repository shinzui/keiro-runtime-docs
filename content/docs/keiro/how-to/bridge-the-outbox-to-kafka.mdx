---
title: "Bridge the outbox to Kafka"
description: "Write the batch-shaped outbox publisher callback: convert OutboxRows to KafkaProducerRecords, produce them, and return PublishSucceeded only after broker acks."
---

`publishClaimedOutbox` is transport-neutral — it hands you the claimed rows as a batch and asks for
one `PublishOutcome` per row. This guide shows how to write that callback for Kafka.

<Callout type="info">
Assumes you publish through the outbox (see [Publish with the
outbox](/docs/keiro/how-to/publish-with-the-outbox)) and have a Kafka producer from your broker
library (e.g. `hw-kafka-client` or `kafka-effectful`).
</Callout>

## Goal

Implement the `[OutboxRow] -> Eff es [(OutboxId, PublishOutcome)]` publish function so the worker
can hand Kafka a batch and reflect each broker acknowledgement back into the row's status.

## keiro stays free of librdkafka

`keiro` deliberately does not depend on a Kafka client library. `Keiro.Outbox.Kafka` gives you a
**neutral** record, and your integration package bridges it to the broker library's record type in
its own dependency scope:

```haskell
data KafkaProducerRecord = KafkaProducerRecord
  { topic :: !Text, key :: !(Maybe ByteString), payload :: !ByteString, headers :: ![(ByteString, ByteString)] }

outboxRowToKafkaRecord :: OutboxRow -> KafkaProducerRecord
```

`outboxRowToKafkaRecord` sets `topic` from the envelope's `destination`, UTF-8-encodes the `key`,
copies the `payloadBytes`, and UTF-8-encodes the canonical headers.

## Steps

1. **Convert each row to a `KafkaProducerRecord`.** This is pure:

   ```haskell
   let rec = outboxRowToKafkaRecord row    -- :: KafkaProducerRecord
   ```

2. **Produce each record and map the result.** Return `PublishSucceeded` **only after the
   broker acks**; any failure is `PublishFailed err` (the worker will retry it after the backoff):

   ```haskell
   publishViaKafka :: (IOE :> es) => Producer -> [OutboxRow] -> Eff es [(OutboxId, PublishOutcome)]
   publishViaKafka producer rows =
     traverse publishOne rows
    where
     publishOne row = do
       let rec = outboxRowToKafkaRecord row
       result <- liftIO (produceSync producer (toBrokerRecord rec))   -- your library's record type
       pure
         ( row ^. #outboxId
         , case result of
             Right () -> PublishSucceeded
             Left err -> PublishFailed (renderProduceError err)
         )
   ```

   `toBrokerRecord` is the small adapter in your integration package that maps the neutral
   `KafkaProducerRecord` (topic, key bytes, payload bytes, header bytes) to your broker library's
   producer record.

3. **Hand the callback to the worker.** Schedule it once per tick:

   ```haskell
   summary <- publishClaimedOutbox (publishViaKafka producer) defaultPublishOptions Nothing
   ```

<Callout type="warn">
Do not report `PublishSucceeded` for a row until the broker has accepted that row. If you
fire-and-forget and return success prematurely, the worker marks the row `sent` for a publish that may
not have landed — defeating the durability the outbox exists to provide. A late ack that races a
crash just means the row is republished, which the consumer's inbox deduplicates.
</Callout>

For higher throughput, replace the `traverse publishOne` body with your broker library's batch or
pipelined send API and return outcomes keyed by `OutboxId`. Under `StopTheLine`, keiro calls your
callback with singleton batches; under the other ordering policies it passes the whole claimed batch.

## Verify it worked

Point the producer at a test broker, enqueue a row, run one publisher pass, and confirm the row
moves to `sent` and the message appears on the topic with its `keiro-*` headers intact. Make the
broker reject a produce and confirm the row goes to `failed` with `next_attempt_at` in the future.
