---
title: "Run the migrations"
description: "Apply keiro's schema with the keiro-migrate tool, or compose keiro's migrations into a service's own codd ledger."
---

keiro owns nine PostgreSQL tables that must exist before the framework runs. They ship as **embedded
codd migrations** (codd is a Haskell migration tool; the SQL is baked into the binary). This guide
shows how to apply them — either with the ready-made `keiro-migrate` tool or by composing keiro's SQL
into your service's own migration ledger.

<Callout type="info">
  keiro's tables live in the **`kiroku` schema** (the same schema the event store uses; keiro
  extends it). You apply **kiroku's** event-store migrations and **keiro's** framework migrations
  together, in that order — `keiro-migrate` does both for you.
</Callout>

## Goal

Apply the kiroku + keiro schema to a database so the nine keiro tables (`keiro_snapshots`,
`keiro_read_models`, `keiro_timers`, `keiro_outbox`, `keiro_inbox`, `keiro_workflow_steps`,
`keiro_awakeables`, `keiro_workflow_children`, `keiro_subscription_shards`) exist.

## Steps

1. **Run `keiro-migrate` against your database.** It reads codd settings from the environment, then
   applies kiroku's migrations followed by keiro's. For a local dev database, skip schema
   verification with `KEIRO_MIGRATE_NO_CHECK`:

   ```bash
   # Apply kiroku + keiro schema (local dev: skip the expected-schema check).
   export CODD_CONNECTION="host=localhost dbname=myservice user=postgres"
   export CODD_SCHEMAS=kiroku
   export KEIRO_MIGRATE_NO_CHECK=1
   cabal run keiro-migrate
   ```

   <Callout type="info">
   Without `KEIRO_MIGRATE_NO_CHECK`, `keiro-migrate` verifies the resulting schema with codd's
   `LaxCheck` and a 5-second connect timeout. `LaxCheck` is the right default for production; the
   no-check path is for dev databases that keep no checked-in expected-schema representation.
   </Callout>

2. **(Alternative) Compose keiro's migrations into your own ledger.** If your service already runs
   codd, do not shell out — pull keiro's migrations into your own list. `keiroFrameworkMigrations`
   returns **only** keiro-owned SQL; `allKeiroMigrations` returns **kiroku then keiro** in the
   load-bearing order:

   ```haskell
   import Keiro.Migrations (keiroFrameworkMigrations, allKeiroMigrations)

   -- keiro-owned SQL only, to splice after your own kiroku setup:
   keiroOnly <- keiroFrameworkMigrations
   -- or the full, correctly-ordered set (kiroku event store, then keiro framework):
   everything <- allKeiroMigrations
   ```

   <Callout type="warn">
   The order in `allKeiroMigrations` is **load-bearing**: kiroku's event-store tables must exist
   before keiro's tables that reference them. If you compose by hand with
   `keiroFrameworkMigrations`, make sure kiroku's migrations run first.
   </Callout>

## Scaffold a new migration

When you add your own keiro-adjacent table, let the CLI stamp a correctly-named, correctly-pinned
skeleton instead of hand-numbering a file. From the `keiro-migrations` package root (or with
`KEIRO_MIGRATIONS_DIR` pointing at your `sql-migrations` dir):

```bash
cabal run keiro-migrate -- new "add widget projection table"
# Created sql-migrations/2026-06-05-14-32-07-keiro-add-widget-projection-table.sql
```

The file is named for the real current UTC time (so it can never collide with or sort before an
existing one) and opens with the `SET search_path TO kiroku, pg_catalog;` pin plus a TODO body. After
adding it, **touch the embed comment** in `src/Keiro/Migrations.hs` (or `cabal clean`) so Template
Haskell re-embeds the directory — the command prints this reminder. See the
[migration scaffold reference](/docs/keiro/reference/migrations-and-schema#the-migration-scaffold-keiromigrationsnew).

## Verify it worked

Connect to the database and confirm the nine keiro tables exist in the `kiroku` schema:

```bash
psql "$CODD_CONNECTION" -c "\dt kiroku.keiro_*"
```

You should see `keiro_snapshots`, `keiro_read_models`, `keiro_timers`, `keiro_outbox`, `keiro_inbox`,
`keiro_workflow_steps`, `keiro_awakeables`, `keiro_workflow_children`, and `keiro_subscription_shards`.
Re-running `keiro-migrate` is idempotent — codd's ledger skips already-applied migrations.
