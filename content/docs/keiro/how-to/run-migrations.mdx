---
title: "Run the migrations"
description: "Apply the native Kiroku-before-Keiro plan or compose those components into a larger application-owned pg-migrate plan."
---

Keiro runtime code never creates framework schema. Run the native plan as a deployment job before
opening the application.

<Callout type="info">
  Kiroku owns schema `kiroku`; Keiro owns schema `keiro`. The `keiro` component depends on
  `kiroku`, and the planner enforces that order.
</Callout>

## Apply the packaged framework plan

```console
export DATABASE_URL='host=localhost dbname=myservice user=service_owner'

cabal run keiro-migrate -- plan --json > plan.json
cabal run keiro-migrate -- status --json > status-before.json
cabal run keiro-migrate -- verify --json > verify-before.json
cabal run keiro-migrate -- up --lock-timeout 30000 --json > up.json
cabal run keiro-migrate -- verify --json > verify-after.json
```

On a fresh database, the preflight `verify` reports the reviewed pending plan. Stop if it reports
changed applied bytes, unresolved state, gaps, or unknown rows. Post-apply verification must have no
issues. Preserve the JSON with the release record.

## Compose the application plan

Libraries export components; the application owns the final plan and process policy:

```haskell
import Data.List.NonEmpty (NonEmpty (..))
import Database.PostgreSQL.Migrate (migrationPlan)
import Keiro.Migrations qualified as Keiro
import Kiroku.Store.Migrations qualified as Kiroku
import Pgmq.Migration qualified as Pgmq

servicePlan = do
  kiroku <- Kiroku.kirokuMigrations
  keiro <- Keiro.keiroMigrations
  pgmq <- Pgmq.pgmqMigrations
  application <- applicationMigrations
  migrationPlan (kiroku :| [keiro, pgmq, application])
```

Declare the application component's dependencies to match objects its SQL consumes. Mount the
standard CLI around `servicePlan` and execute the complete plan. Do not run each component during
library initialization or split one shared `pgmigrate` ledger across plans that omit known rows.

## Author the next Keiro migration

```console
cabal run keiro-migrate -- new \
  --manifest keiro-migrations/migrations/manifest \
  --description "add workflow lookup index"

cabal run keiro-migrate -- check \
  --manifest keiro-migrations/migrations/manifest
```

Review the new SQL and manifest append together. Use qualified `keiro.<object>` names, never edit a
released payload, and keep application-owned objects outside `kiroku` and `keiro`.

## Existing 0.1 databases

Do not run native `up` first. A database created by `keiro-migrations 0.1.0.0` needs its framework
tables relocated from `kiroku` to `keiro`, then its combined Codd history imported without replaying
DDL. Follow [Upgrade a 0.1 database](/docs/keiro/how-to/upgrade-a-0-1-database).

## Verify runtime readiness

Confirm the schemas and then start the application with runtime schema initialization disabled:

```console
psql "$DATABASE_URL" -c '\dt kiroku.*'
psql "$DATABASE_URL" -c '\dt keiro.*'
```

```haskell
withStore
  (defaultConnectionSettings connString
    & #schemaInitialization .~ SkipSchemaInitialization)
  app
```

See [Deploy a migration plan](/docs/pg-migrate/how-to/deploy-a-migration-plan) for backups,
quiescence, cleanup issues, and forward recovery.
