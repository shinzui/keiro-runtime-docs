---
title: "Scaffold and fill holes"
description: "Plan and emit generated modules safely, preserve create-once behavior, resolve protected and stale paths, fill typed seams, and verify the combined service with harnesses."
---

`scaffold` turns a checked spec into an overwritable deterministic layer and create-once hand-owned
behavior seams. It plans and verifies the complete output before its first write.

## Scaffold into the consuming source tree

```bash
keiro-dsl scaffold service.keiro --out src
```

Before writing, the command checks validation, exact/case-folded path collisions, faithful lowering,
the symbolic-operator firewall, and every existing generated target's `-- @generated` banner. A
refusal exits 1 and writes nothing.

<Callout type="warn">
If a generated path already contains hand-owned code, move or rename that file. Use
`--force-generated-overwrite` only after reviewing that exact path and intentionally adopting it as
generated output. The switch bypasses only missing-banner protection.
</Callout>

## Read the ownership report

The report distinguishes generated and hole dispositions:

```text
generated  Generated.TransferRouting.Hospital.Domain       (overwritten)
hole       TransferRouting.Hospital.Holes                  (created)
hole       TransferRouting.Hospital_load.ReadModelHoles    (skipped: already present)
firewall: OK (N generated modules scanned, 0 forbidden operators)
manifest: src/keiro-dsl-manifest.transfer-routing.txt
```

Generated families include aggregate `Domain`/`Codec`/`EventStream`/`Projection`/`Harness`,
read-model `ReadModelTable`/`ReadModel`/harness, process/router facts/runtime, contract/inbox/publisher,
queue/policy, and workflow facts/runtime. Exact output depends on nodes; the manifest is authoritative.

Create-once files include `Holes`, `ProcessHoles`, `RouterHoles`, and `ReadModelHoles`. Workflow
behavior remains in a separately hand-authored body module and has no generated hole stub.

## Reconcile the manifest and stale record

Paste the manifest's `other-modules:` and `build-depends:` blocks into the Cabal component that owns
the output. Re-run this step whenever the node inventory changes.

The per-context scaffold record powers later `stale:` output. Scaffold never deletes a stale path:

- inspect then delete a stale `generated` entry if nothing imports it;
- inspect/migrate a stale `hole` entry as hand-owned code; and
- separate output directories if a previous-spec-path note reveals two specs sharing one context.

## Fill only hand-owned seams

Read each typed signature and `-- HOLE …` annotation, then use generated names/constants:

- aggregate: symbolic guard/write/emit/goto and upcasters;
- read model: qualified-table SQL, apply, and query;
- process/router: resolver/handler, stream/category, projections, and runtime assembly;
- integration: decode/handler/bridge behavior;
- queue: fan-out, raw-SQL dedupe, and job handler; and
- workflow: ordered body using generated facts.

Never put Keiki symbolic operators in generated modules. A wrong generated file means the spec or
scaffolder is wrong.

For hand-written duplicate dispatch, call `confirmBenignDuplicate` with the target stream and
attempted event ID. Do not acknowledge a bare global `DuplicateEvent`.

## Run the matching harness

```bash
cabal test keiro-dsl-conformance-newsurface
```

Choose the component matching the surface (`snapshot`, `process-runtime`, `router-full`,
`readmodel-runtime`, `intake-full`, `queue-runtime`, `dispatch-full`, `workflow-full`, and so on).
Harnesses prove the generated facts and hand-owned behavior agree; scaffold alone cannot.

After every spec change: check, scaffold, reconcile stale/manifest output, compile, and rerun the
harness.
