---
title: "Add a snapshot"
description: "Enable snapshotting on an event stream with a state codec and a policy, and invalidate stale snapshots."
---

Hydrating a long stream by replaying every event gets slow. A snapshot caches the folded state so
hydration can start partway. You enable it on the `EventStream`.

<Callout type="info">
  Assumes you have an `EventStream` definition and know how to validate it into a
  `ValidatedEventStream` (see [Get started with keiro](/docs/keiro/tutorials/getting-started))
  and understand that snapshots are advisory (see
  [Consistency and snapshots](/docs/keiro/explanation/consistency-and-snapshots)).
</Callout>

## Goal

Turn on snapshotting for a stream and know how to invalidate old snapshots.

## Declare it in a `.keiro` aggregate

When keiro-dsl owns the event stream, put the policy after the aggregate projection:

```keiro-dsl
snapshot every 100
  state-codec version=1 shape-hash="<captured-live-hash>"
```

Use `snapshot on-terminal` instead when terminal transitions are the useful cache boundaries.
`every` must be at least 1; codec version must be at least 1; the hash must be non-empty.

The live hash is module-qualified Haskell state/register shape data, not the event wire hash. To
capture it safely:

1. scaffold `keiro-dsl/test/fixtures/reservation-snapshot.keiro` or your equivalent with a temporary
   hash;
2. run the generated snapshot harness (the upstream component is
   `keiro-dsl-conformance-snapshot`);
3. copy the printed live hash into the spec; and
4. re-scaffold and rerun the harness.

The snapshot clause changes the generated aggregate `EventStream` and `Harness`; it does not create
a new hole module. The existing hand-owned `Holes` still contains the transducer and projection
behavior. Generated construction lowers to `Every n`/`OnTerminal` and `defaultStateCodec version`,
then validates policy/codec coherence and the initial pair.

```bash
keiro-dsl check keiro-dsl/test/fixtures/reservation-snapshot.keiro
cabal test keiro-dsl-conformance-snapshot
```

## Steps

1. **Set a `stateCodec` and a `snapshotPolicy`.** For hand-authored wiring, derive a snapshot-enabled
   definition from the same aggregate stream definition:

   ```haskell
   snapshotOrderEventStreamDef :: OrderEventStream
   snapshotOrderEventStreamDef = orderEventStreamDef
     { snapshotPolicy = Every 2
     , stateCodec     = Just (defaultStateCodec @OrderRegs @OrderState 1)
     }

   snapshotOrderEventStream :: ValidatedOrderEventStream
   snapshotOrderEventStream =
     mkEventStreamOrThrow "orders-with-snapshots" snapshotOrderEventStreamDef
   ```

   `defaultStateCodec` encodes the joint `(state, registers)`; its `Int` argument is the
   `stateCodecVersion`. `Every 2` writes a snapshot whenever the stream version is a positive multiple
   of 2; a multi-event append also fires when its version span crosses such a boundary. Leaving
   `stateCodec = Nothing` disables snapshotting, and `mkEventStream` rejects that incoherent
   configuration when the policy is not `Never`.

2. **Validate the initial pair at startup.** For hand-authored wiring, handle `mkEventStream`
   explicitly:

   ```haskell
   snapshotOrderEventStream =
     case mkEventStream "orders" snapshotOrderEventStreamDef of
       Left warnings -> error (show warnings) -- replace with your startup-error path
       Right validated -> validated
   ```

   Validation forces the codec over `(initialState, initialRegisters)`. If `emptyRegFile` left an
   `uninit: <slot>` thunk in a persisted register, startup fails before the first command. Initialize
   every persisted slot or use a codec whose initial state is total.

3. **Invalidate by changing compatibility, not by relabelling bytes.** Compatibility is the triple
   `(stateCodecVersion, shapeHash, stateShapeHash)`; a mismatch on any component makes the old row
   `SnapshotNotFound` for the new codec, hydration replays events, and a later policy firing writes a
   compatible row. Two of the three move for you:

   - a **register-layout** change moves `regfile_shape_hash`;
   - a **control-state datatype** change moves `state_shape_hash` (keiki's `CanonicalStateShape`);
   - DSL-generated streams additionally fold a spec-visible **fold fingerprint** into
     `state_shape_hash` through `withFoldFingerprint`.

   Bump `stateCodecVersion` yourself when the state JSON or custom encoding changes without a
   structural change — **and whenever fold logic changes in a way none of the above can see.**

   Keiki 0.2 causes a one-time miss for every old non-empty 0.1 register shape hash; migration
   `0019-keiro-snapshots-state-shape-hash.sql` causes another for every pre-migration row. The empty
   `regfile:0` hash is unchanged. Do not update an old row's hash in place.

<Callout type="warn">
**The bump you will forget.** Editing a hand-written guard or `update` body — or logic that lives only
in a generated service's `Holes` module — moves *no* derived hash. The old seed still matches, so
tail-only replay derives state a full replay would never reach, and post-append verification may
persist it at the new head version.

Bump `stateCodecVersion`, or maintain your own token with
`withFoldFingerprint "orders-fold-2026-07"`. Then confirm with the
[replay audit](/docs/keiro/how-to/audit-real-logs-before-a-deploy) before deploying, and alert on
`keiro.snapshot.seed.divergence` afterwards — it is the sampled runtime backstop for exactly this
mistake.
</Callout>

4. **Discard a bad cache row only when necessary.** To evict one known stream snapshot:

   ```sql
   DELETE FROM keiro.keiro_snapshots WHERE stream_id = $1;
   ```

   `TRUNCATE keiro.keiro_snapshots` is appropriate only when you intentionally accept a full-replay
   surge for every aggregate, such as in an isolated test database.

<Callout type="warn">
A snapshot is **advisory**. Its fold, strict encode, and store write run synchronously after the event
append has committed. Missing, incompatible, or undecodable rows fall back to full replay. A
post-commit encode/write failure leaves the command successful and is counted; do not retry the
command as though its events rolled back.
</Callout>

With a fixed discriminator, a lower-version snapshot write is ignored. A row with a *different*
discriminator may be replaced even at a lower stream version, so a rolled-back deployment can recover
the single slot. Keep mixed codec deployments brief: incompatible writers can repeatedly replace one
another's cache rows, causing misses without corrupting event state.

## Verify it worked

For a `.keiro` aggregate, run its generated harness; the checked source fixture uses:

```bash
cabal test keiro-dsl-conformance-snapshot
```

For hand-authored wiring, append past the policy threshold in your integration test and confirm a row
appears in `keiro.keiro_snapshots`. Keiro's own `keiro-test` component covers the same threshold,
tail-hydration, corrupt-cache, strict-encode, and store-failure paths. Test these scenarios with
`RunCommandOptions.metrics` installed:

- corrupt the row's JSON and confirm the command succeeds through full replay while
  `keiro.snapshot.read.misses` and `keiro.snapshot.decode.failures` increase;
- make the snapshot table reject a write and confirm the events remain committed, the command stays
  successful, and `keiro.snapshot.write.failures` increases;
- use a partial encoder at a firing boundary and confirm `keiro.snapshot.encode.failures` increases
  without a snapshot row being written;
- set `RunCommandOptions.seedVerifySampleRate = 1` in a staging run so every accepted seed is checked
  against a full replay, and confirm `keiro.snapshot.seed.divergence` stays at zero.
