---
title: "Route events to commands"
description: "Resolve dynamic targets, dispatch with target-derived IDs, and choose a rejection policy for the live router worker."
---

Use `Router` when one source event must dispatch to a data-dependent target set and the coordinator
does not need its own history.

## 1. Define a read-only resolver

```haskell
pagingRouter :: (IOE :> es, Store :> es) => Router IncidentRaised TargetPhi ... es
pagingRouter = Router
  { name = "paging-router"
  , key = \raised -> incidentIdText (raised ^. #incidentId)
  , resolve = \raised -> do
      queryResult <- runQuery Nothing onCallReadModel (raised ^. #service)
      responders <- requireRouterTargets queryResult
      pure
        [ PMCommand
            { target = pageStream (raised ^. #incidentId) responder
            , command = SendPage (raised ^. #incidentId) responder
            }
        | responder <- responders
        ]
  , targetEventStream = pageEventStream
  , targetProjections = const []
  }
```

Register `onCallReadModel` explicitly at startup. `requireRouterTargets` represents your deliberate
failure boundary: do not turn `ReadModelUnregistered`, stale schema, rebuild state, or wait timeout
into `[]`, because that silently acknowledges “route nowhere.” Surface the failure so supervision
stops without accepting the source event.

Keep `resolve` read-only. It can run multiple times for the same event, and its target set may drift.
Each target is dispatched independently after resolution.

## 2. Understand the target identity

For each command, Keiro derives:

```haskell
deterministicRouterCommandId
  routerName correlation sourceEventId targetStreamName sameStreamOccurrence
```

Reordering distinct targets does not change their IDs. Two commands to the same target get
occurrences `0`, `1`, and so on. A target added on retry receives a fresh dispatch, so attempts
accumulate the union of resolved targets. Keep resolution stable when the exact recipient set must
be frozen.

During the upgrade window, the runner also checks the old positional ID in the same target. Keep
target order/membership stable across that window: the probe cannot guarantee one logical dispatch
when both old code and resolver output change together.

## 3. Run once or continuously

```haskell
RouterResult results <-
  runRouterOnce defaultRunCommandOptions pagingRouter recorded raised

routerOptions = defaultWorkerOptions
  & #rejectedCommandPolicy .~ RejectedHalt

runRouterWorkerWith
  routerOptions
  defaultRunCommandOptions
  pagingRouter
  adapter
  decodeMessage
```

The live worker shares the process-manager policy: transient failures retry, systemic deterministic
failures halt, and an all-rejection group follows `RejectedHalt`, `RejectedDeadLetter`, or
`RejectedSkip`. The safe default is halt.

## 4. Verify order drift and collision behavior

Use one source event and two targets:

1. Dispatch target A, interrupt before B, then redeliver with `[B, A]`; A is duplicate and B is
   applied.
2. Redeliver after both complete with `[B, A]`; both are duplicates.
3. Add C on a later resolution; C is applied and A/B remain duplicates.
4. Put A's attempted ID only in another stream; A becomes
   `PMCommandFailed targetA (StoreFailed DuplicateEvent ...)`, never `PMCommandDuplicate`.

[Inspect a rejected dispatch](/docs/keiro/how-to/inspect-a-rejected-dispatch).
