title: "Choose an outbox ordering policy"
description: "Pick PerKeyHeadOfLine, PerSourceStream, StopTheLine, or BestEffort; tune maxAttempts, backoff, and stale-publishing recovery."
---

The outbox publisher claims rows under an `OrderingPolicy`, which decides how a stuck row affects
the rows behind it. This guide helps you pick one and tune the retry, dead-letter, and stale-row
recovery knobs.

<Callout type="info">
Assumes you publish through the outbox (see [Publish with the
outbox](/docs/keiro/how-to/publish-with-the-outbox)).
</Callout>

## Goal

Choose the ordering guarantee your consumers need, and tune retry behavior, while keeping the
table healthy.

## The four policies

```haskell
data OrderingPolicy = PerKeyHeadOfLine | PerSourceStream | StopTheLine | BestEffort
```

- **`PerKeyHeadOfLine`** (the default) — within a `source`, a non-terminal row with key `k` blocks
  every later row with the same key, so each aggregate's events publish **in order**; rows with a
  `Nothing` key bypass the block. One stuck aggregate cannot stall *other* aggregates. Choose this
  unless you have a specific reason not to.
- **`PerSourceStream`** — any non-terminal row blocks every later row in the source, regardless of
  key. Strict total order across keys; rare, and it means one stuck row halts the whole source.
- **`StopTheLine`** — the worker halts after the first `PublishFailed` and records the offending id
  in `haltedOn`. Use when correctness demands a human looks at every failure before more events go
  out.
- **`BestEffort`** — no head-of-line blocking; failed rows do not block anything. Opt-in only, safe
  only when published events have no per-key or causal relationship.

## Tune retries and dead-lettering

`OutboxPublishOptions` controls the retry curve and the dead-letter threshold:

```haskell
opts = defaultPublishOptions          -- batch 32, maxAttempts 10, ConstantBackoff 2, PerKeyHeadOfLine
  { maxAttempts = 8
  , backoff = ExponentialBackoff (ExponentialBackoffOptions
      { initial = 1, maxDelay = 300, multiplier = 2 })
  , orderingPolicy = PerKeyHeadOfLine
  }
```

Each failed publish bumps `attempt_count` and sets `next_attempt_at = now + nextDelay backoff
attempt`. After `maxAttempts` consecutive failures the row goes **`dead`** (terminal) — it stays in
the table for inspection and, being terminal, stops head-of-line-blocking its key. `ConstantBackoff`
is a fixed delay; `ExponentialBackoff` grows `initial * multiplier ^ (attempt-1)` capped at
`maxDelay`.

## Recovering stale publishing rows

Schedule `outboxMaintenancePass defaultMaintenanceOptions metrics` independently from
`publishClaimedOutbox`, typically less frequently than the publish pass. Rows left in `publishing`
longer than `publishingTimeout` are reclaimed automatically:

- below the attempt ceiling, the row moves back to `failed` and becomes claimable again;
- at or above the attempt ceiling, the row moves to `dead`;
- the maintenance pass records `keiro.outbox.reclaimed`, `keiro.outbox.deadlettered`, and
  `keiro.outbox.backlog` when metrics are enabled.

Choose `publishingTimeout` above your worst expected broker publish latency. A timeout that is too
short can republish a row while the first publisher is still running; a timeout that is too long
increases recovery time after a crash. Publishing remains at-least-once: a crash after the broker
ack but before `markOutboxSent` can publish the same envelope again, and consumers must still dedupe
through the inbox.

## Verify it worked

Under `PerKeyHeadOfLine`, enqueue two rows for the same key, make the first fail repeatedly, and
confirm the second does **not** publish until the first reaches `sent` or `dead`. Enqueue a row for
a *different* key and confirm it publishes immediately (other keys are not blocked).
