---
title: "Keep target commands total"
description: "Model benign late or repeated dispatches as successful no-ops so they do not require a rejection loss policy."
---

When a process manager or router dispatches a command with no matching edge, the target returns
`CommandRejected`. The live worker then applies `RejectedCommandPolicy`: halt by default, or consume
through explicit dead-letter/skip policy. Benign “already done” behavior should not need that choice.

## Add an explicit no-op transition

For every dispatched command, identify states where a late or repeated arrival is harmless. Add an
explicit same-state, no-output edge for those cases. The command then succeeds with a no-op
`CommandResult` rather than returning `CommandRejected`.

For a race between `Acknowledge` and `Escalate`, for example:

- the state-changing edge from `Triaging` emits the winning event;
- `Acknowledge` in `Acknowledged` is an explicit no-op;
- `Escalate` in `Escalated` is an explicit no-op; and
- if the losing cross-command is also harmless, model its corresponding no-op in the other terminal
  state.

Use Keiki's explicit `noEmit` intent for silent edges. The validation floor rejects dangerous
state-changing silent transitions; a same-state benign no-op is the intended case.

## Reserve rejection for a real policy decision

If the target must genuinely refuse the command, let it return `CommandRejected` and choose the
source-event disposition explicitly:

- `RejectedHalt` keeps the position for operator action;
- `RejectedDeadLetter` records the target and acknowledges; or
- `RejectedSkip` acknowledges without a row.

Never treat `CommandAmbiguous` as a benign totality case. It means multiple transitions matched and
requires a model fix.

## Verify the worker path

Test the harmless late command through the real target runner and assert:

1. the result is `Right` with no new event/global position;
2. the process-manager/router result is not `PMCommandFailed`;
3. the source delivery reaches `AckOk`; and
4. no Keiro dispatch dead letter is written.

Separately test a genuine rejection under the selected policy so its target-bearing evidence and
checkpoint behavior are intentional.

See [Run a process manager as a subscription](/docs/keiro/how-to/run-a-process-manager-as-a-subscription)
and [Inspect a rejected dispatch](/docs/keiro/how-to/inspect-a-rejected-dispatch).
