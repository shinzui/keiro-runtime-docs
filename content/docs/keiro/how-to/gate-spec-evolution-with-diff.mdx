---
title: "Gate spec evolution with diff"
description: "Classify every persisted decode, identity, and forward-policy change as additive, warning, or breaking and use the exit code as an upgrade gate."
---

`check` proves the working spec is internally valid. `diff` separately proves how it changes stored
bytes, rows, cursors, jobs, and deterministic identity relative to a committed spec.

## Run against history

```bash
keiro-dsl diff --since origin/main service.keiro
```

The file must exist at the Git ref. The command prints one regular line per change:

```text
ADDITIVE: <node> <facet> <subject>: <detail>
WARNING: <node> <facet> <subject>: <detail> [Code]
BREAKING: <node> <facet> <subject>: <detail> [Code]
```

Only `BREAKING` makes the command exit nonzero. Warnings remain visible and exit 0; review them in
CI instead of discarding output.

After the change lines, `diff` prints the **replay-impact verdict** — whether the candidate binary
can interpret already-stored logs differently. Capture it for the deploy gate:

```bash
keiro-dsl diff --since origin/main --replay-impact-out replay-impact.json service.keiro
```

A `replay-neutral` verdict needs no data access. An `affected` verdict names the conservative
event-type set to feed the [replay audit](/docs/keiro/how-to/audit-real-logs-before-a-deploy).

## Read the non-breaking advisories

Several changes are legal but carry an operational obligation the differ cannot discharge for you.
They exit 0 — you still have to act on them.

| Advisory | What it means | What you must do |
|---|---|---|
| `AggGuardTightened` | a live transition's guard changed with no replay-only twin | paste the printed twin, or prove by audit that no stored data is affected |
| `AggFoldSurfaceChanged` | the spec-visible replay surface moved, so the fold fingerprint changed | expect one full replay per snapshot-bearing stream |
| `DeprecatedEventReplayHazard` | a deprecated event has no replay-only emitter | keep an inverting edge, or audit and truncate |
| `EventRetirementInProgress` | the safe retained shape is in place | keep the eventual cleanup on the backlog |
| `RouterDecideSurfaceChanged` / `ProcessDecideSurfaceChanged` | fan-out changed | drain the subscription's redelivery window before deploying |
| `ProcessTimerPayloadChanged` | a timer payload shape changed | deploy backward-compatible firers first |
| `ContractSchemaVersionBumped` | a contract shape moved with its version | coordinate the producer/consumer rollout by hand |

`AggGuardTightened` is the one that prints working code: a paste-ready `replay-only` twin whose guard
is the removed region `old ∧ ¬new`. See
[Tighten a guard with a replay-only twin](/docs/keiro/how-to/tighten-a-guard-with-a-replay-only-twin).

## Capture old payload shapes while both specs exist

An event version bump is the one moment when both the old and the new wire shape are knowable. Emit
the goldens then — not later, when the old shape only survives in the database:

```bash
keiro-dsl diff --since origin/main --emit-goldens golden-payloads service.keiro
keiro-dsl scaffold --goldens golden-payloads service.keiro
```

The generated conformance harness exercises each golden through `decodeRaw`. Emission never
overwrites an existing file, so a hand-captured genuine production payload always wins over a
synthesized one — and a genuine payload is the stronger evidence.

## Classify the change

| Tier | Examples |
|---|---|
| additive | new command/event/node; versioned event plus contiguous upcaster; newly declared workflow patch guarding its body |
| warning | timer window; inbox retention; publisher retry/ordering policy; forward decode posture |
| breaking decode | event field/type/removal without compatible versioning; enum wire spelling; contract field/topic/discriminator; queue payload shape |
| breaking identity | ID prefix; process name/category/timer IDs; router name/key/target; read-model registry/table/subscription; queue identity/order/group/provision; workflow stable name/shape/continuation seed |

The registry is exhaustive over all node families. An operation has no stored surface of its own, so
its references are single-spec validation concerns rather than diff entries.

## Repair an event payload break

```keiro-dsl
event TransferReservationConfirmed v2 { reservationId hospitalId triageNote:Text }
  upcast from v1 = HOLE
```

Fill the pure v1-to-v2 body, re-scaffold, and run `keiro-dsl-conformance-v2`.

<Callout type="warn">
`deprecated event` alone is **not** a safe retirement. It keeps old bytes decodable, but hydration
still needs an edge that *inverts* the event. Use the two-stage `retiring` → `deprecated` +
`replay-only` protocol in [Retire an event type](/docs/keiro/how-to/retire-an-event-type).
</Callout>

## Plan identity changes explicitly

An upcaster cannot repair a renamed cursor, queue, workflow journal, router dispatch ID, or read-model
table. Preserve the old identity or supply an application/operator migration and staged rollout.
`diff` intentionally blocks these changes because the DSL cannot move durable state implicitly.

## Gate CI

```bash
keiro-dsl check service.keiro
keiro-dsl diff --since origin/main --replay-impact-out replay-impact.json service.keiro
```

Both commands must run: a compatible diff can still leave the new spec internally invalid, and a
valid new spec can still break history.

Neither command reads your database. A clean `diff` with an `affected` verdict is not a green light —
it is a *request* for the [real-log audit](/docs/keiro/how-to/audit-real-logs-before-a-deploy).
