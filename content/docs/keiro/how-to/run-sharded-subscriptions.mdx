---
title: "Run sharded subscriptions"
description: "Run an acknowledgment-aware sharded worker pool with bounded retry, atomic dead-letter advancement, and safe rebalance behavior."
---

Use `runShardedSubscriptionGroupAck` when a high-volume category needs several workers and the
handler can fail. The worker will not checkpoint past an in-flight event: the handler must return
success, request retry, or explicitly park the source event.

## 1. Apply the schema

Run Keiro's migrations before starting the pool. Sharding uses
`keiro.keiro_subscription_shards`; terminal source failures use Kiroku's `kiroku.dead_letters`.

See [Run the migrations](/docs/keiro/how-to/run-migrations).

## 2. Choose and validate the fixed shard count

Every process using one `SubscriptionName` must agree on `shardCount`. The table rejects a later
count mismatch, so choose for peak useful parallelism and change it only through an operator-led
cutover.

```haskell
import Keiro.Subscription.Shard.Worker
import Kiroku.Store.Subscription.Types
import Kiroku.Store.Types (CategoryName (..))

let candidate =
      (defaultShardedWorkerOptions (Category (CategoryName "orders")) 8)
        { leaseTtl = 30
        , renewInterval = 10
        , handlerRetryDelay = RetryDelay 1
        , retryPolicy = RetryPolicy 5
        }

opts <- either (fail . show) pure (mkShardedWorkerOptions candidate)
```

`RetryPolicy 5` means at most five **total deliveries**: the first plus up to four
redeliveries. `handlerRetryDelay` is used for a synchronous exception that escapes the handler.

## 3. Return an explicit acknowledgment

Make the durable effect idempotent on the source `eventId`, then classify failures.

```haskell
handleOrder :: ShardDelivery -> IO ShardAck
handleOrder delivery =
  case decodeOrderEvent (delivery ^. #event) of
    Left detail ->
      pure (ShardAckDeadLetter (DeadLetterInvalid detail))

    Right orderEvent -> do
      let recorded = delivery ^. #event
      result <- applyProjectionIdempotently (recorded ^. #eventId) orderEvent
      pure $ case result of
        Applied          -> ShardAckOk
        AlreadyApplied   -> ShardAckOk
        TransientFailure -> ShardAckRetry (RetryDelay 2)
        Permanent detail -> ShardAckDeadLetter (DeadLetterPoison detail)
```

The `attempt` field is zero-based and useful for logging: `0` is the first delivery, `1` the first
redelivery. Do not build a second unbounded retry loop inside this handler; return
`ShardAckRetry` and let Kiroku enforce the configured ceiling.

<Callout type="warn">
  A synchronous exception is treated as transient and retried after `handlerRetryDelay`. Classify
  known invalid or permanently unprocessable inputs as `ShardAckDeadLetter` yourself so they do not
  consume the whole retry budget. Asynchronous exceptions from shutdown or bucket shedding are
  rethrown and produce no acknowledgment.
</Callout>

## 4. Start identical pool members

Every replica runs the same call with the same name and options:

```haskell
runShardedSubscriptionGroupAck
  store
  (SubscriptionName "orders-projector")
  opts
  handleOrder
```

The pool converges over reconcile intervals. At most `N` replicas own work; surplus replicas wait.
On graceful shutdown, the worker cancels its readers and relinquishes its current leases
immediately. On a process crash, another replica can claim after `leaseTtl`.

The older `runShardedSubscriptionGroup` accepts `RecordedEvent -> IO ()`. It is a compatibility
wrapper that returns `ShardAckOk` after the handler completes. Escaping synchronous exceptions still
use the bounded retry ladder, but the wrapper cannot express an immediate domain dead letter.

## 5. Verify the acknowledgment/rebalance timeline

```mermaid
sequenceDiagram
  participant A as Worker A
  participant K as Kiroku member checkpoint
  participant B as Worker B
  A->>A: receive event E (attempt 0)
  Note over A,K: no acknowledgment; checkpoint remains before E
  Note over A: lease lost or reader cancelled mid-handler
  B->>B: claim bucket and receive E
  alt handler succeeds
    B->>K: ShardAckOk
    Note over B,K: checkpoint saves at batch tail
  else handler requests retry
    B->>B: wait, then redeliver E with attempt + 1
    Note over B,K: checkpoint still before E
  else handler parks E
    B->>K: insert dead letter + advance checkpoint atomically
  end
```

This is the safety property: an event is either acknowledged and eventually covered by the batch
checkpoint, held for another attempt, or atomically dead-lettered and advanced. Bucket shedding does
not checkpoint first. A stale worker can overlap a successor, so duplicate execution remains
possible; monotonic checkpoint writes prevent the stale worker from moving progress backward.

## 6. Observe ownership and terminal outcomes

Inspect ownership:

```sql
SELECT bucket, owner_worker_id, lease_expires_at, heartbeat_at
FROM keiro.keiro_subscription_shards
WHERE subscription_name = 'orders-projector'
ORDER BY bucket;
```

Inspect source-event dead letters by member:

```sql
SELECT consumer_group_member, global_position, event_id,
       reason_summary, attempt_count, created_at
FROM kiroku.dead_letters
WHERE subscription_name = 'orders-projector'
ORDER BY created_at DESC;
```

Wire `Keiro.Telemetry.kirokuEventBridge` into Kiroku's store `eventHandler` to increment
`keiro.subscription.deadlettered` on the terminal event. The counter is cumulative; query the table
for current durable rows.

Use [Replay subscription dead letters](/docs/keiro/how-to/replay-subscription-dead-letters) after
repairing the cause. Replay does not delete or mark the row.
