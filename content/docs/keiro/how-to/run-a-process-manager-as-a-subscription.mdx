---
title: "Run a process manager as a subscription"
description: "Configure poison, rejection, retry, and acknowledgement policy for a live process-manager worker."
---

Use `runProcessManagerWorkerWith` when operating a manager continuously. The worker drains a Shibuya
adapter, runs one deterministic reaction per delivery, and finalizes its ack handle exactly once.

## 1. Decode only owned messages

```haskell
decodeMessage :: KirokuMessage -> Maybe (RecordedEvent, OrderEvent)
decodeMessage message = do
  recorded <- messageRecordedEvent message
  event <- decodeOrderEvent (recorded ^. #payload)
  pure (recorded, event)
```

`Nothing` is poison, not a filter. It follows `PoisonPolicy` and defaults to `PoisonHalt`. Filter the
subscription upstream when unrelated event types are expected.

## 2. Choose rejection policy explicitly

```haskell
workerOptions = defaultWorkerOptions
  & #rejectedCommandPolicy .~ RejectedHalt
  & #transientRetryDelay .~ RetryDelay 5
  & #metrics ?~ keiroMetrics

runProcessManagerWorkerWith
  workerOptions
  defaultRunCommandOptions
  fulfillmentManager
  kirokuAdapter
  decodeMessage
```

Keep `RejectedHalt` unless you have an explicit reason to consume a rejected reaction:

- `RejectedHalt` preserves the source position and stops for investigation.
- `RejectedDeadLetter` writes one `keiro.keiro_dead_letters` row per rejected target, then returns
  `AckOk`. Manager history may already be committed while the target is not.
- `RejectedSkip` returns `AckOk` without a durable row. Use it only when loss is an intentional
  domain policy.

Both `CommandRejected` and `CommandAmbiguous` enter this policy branch. Ambiguity is still a code
defect; the default halt is the appropriate response unless a durable operational repair process is
already in place.

## 3. Understand the complete disposition

| Failure group | Decision | Evidence |
| --- | --- | --- |
| No failures; appends and confirmed duplicates only | `AckOk` | Target/manager events and duplicate metric. |
| Any transient store/conflict failure | `AckRetry` | `keiro.dispatch.failed`; Kiroku attempt state. |
| Any systemic deterministic failure | `AckHalt` | Error retains the failed target stream. |
| All failures are rejection-class | Selected `RejectedCommandPolicy` | Halt, Keiro dispatch rows, or metric-only skip. |
| Decode failure | Selected `PoisonPolicy` | Halt, callback, or adapter dead letter. |

On a Kiroku-backed adapter, repeated `AckRetry` is bounded by its subscription `RetryPolicy` (five
total deliveries by default on the current adapter path). Exhaustion writes the source event to
`kiroku.dead_letters` and advances the checkpoint atomically. That is distinct from a
`RejectedDeadLetter`, which records the failed *target dispatch* in Keiro and acknowledges the source
immediately.

## 4. Preserve target-scoped duplicate proof

Do not replace Keiro's duplicate handling with “matching event ID means success.” Event IDs are
globally unique. A matching ID in another stream is a collision, not proof that this target command
ran. `confirmBenignDuplicate` point-checks the intended target; an unconfirmed collision remains a
target-bearing failure and halts.

## 5. Make correlation order-insensitive

Two source streams that correlate to the same manager instance may arrive in either order, and
shards may process them concurrently. Model both orders in the manager transducer. Do not use
delivery/global order as a business sequence across streams.

## Verify the failure paths

Test at least these cases:

1. Redeliver one source event and observe `PMStateDuplicate`/`PMCommandDuplicate` with `AckOk`.
2. Inject a transient store error and observe `AckRetry` without checkpoint advancement.
3. Reject a target under `RejectedHalt` and observe the failed `targetStreamName` and no ack.
4. Switch deliberately to `RejectedDeadLetter`; observe an idempotent Keiro row and `AckOk`.
5. Put the attempted deterministic ID in another stream; confirm the worker halts rather than
   classifying it as a benign duplicate.

See [Inspect a rejected dispatch](/docs/keiro/how-to/inspect-a-rejected-dispatch) for remediation.
