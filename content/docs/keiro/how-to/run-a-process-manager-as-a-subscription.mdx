---
title: "Run a process manager as a subscription"
description: "Drive a keiro process manager continuously by wiring runProcessManagerWorker to a shibuya adapter with a decodeMessage function."
---

You have a `ProcessManager` and you want it to react to events *as they arrive*, not one at a time
in a test. This guide wires it to a live subscription with `runProcessManagerWorker`.

<Callout type="info">
Assumes you already have a working process manager (see [Your first process
manager](/docs/keiro/tutorials/your-first-process-manager)) and a shibuya `Adapter` that delivers
messages from your event source. shibuya is keiro's supervised subscription/worker substrate.
</Callout>

## Goal

Run a process manager as a long-lived worker that drains a shibuya adapter, decoding each delivered
message into the `(RecordedEvent, input)` pair the manager reacts to.

## Steps

1. **Write a `decodeMessage` for your adapter's message type.** The worker needs to turn each raw
   `msg` into the recorded source event and the decoded `input` the manager's `correlate`/`handle`
   expect. Return `Nothing` for a message that does not concern this manager (the worker treats an
   undecodable message as fatal, so only return `Nothing` when that is what you mean):

   ```haskell
   decodeMessage :: KirokuMessage -> Maybe (RecordedEvent, OrderEvent)
   decodeMessage msg = do
     recorded <- messageRecordedEvent msg            -- the RecordedEvent carried by the message
     event    <- decodeOrderEvent (recorded ^. #payload)
     pure (recorded, event)
   ```

2. **Run the worker over the adapter.** Pass your `RunCommandOptions`, the manager, the shibuya
   adapter, and `decodeMessage`:

   ```haskell
   runProcessManagerWorker
     defaultRunCommandOptions
     fulfillmentProcessManager
     kirokuAdapter
     decodeMessage
   ```

   The worker folds the adapter's stream forever: for each message it decodes, it calls
   `runProcessManagerOnce` and acks based on the outcome.

3. **Understand the ack semantics.** They are deliberately simple:

   - decode returns `Nothing` → `AckHalt (HaltFatal "… could not decode message")`;
   - `runProcessManagerOnce` returns `Left err` → `AckHalt (HaltFatal (show err))`;
   - a `Right _` reaction → `AckOk`.

   A halted message is **retried** by the substrate. Because every write the manager makes is keyed
   by a [deterministic id](/docs/keiro/reference/process-manager#deterministiccommandid), that
   forced retry is safe: already-applied appends collapse to `PMStateDuplicate` /
   `PMCommandDuplicate` rather than duplicating events. (The same idempotency discipline is
   described for the command cycle in [Make commands
   idempotent](/docs/keiro/how-to/make-commands-idempotent).)

## Verify it worked

Append a `PaymentApproved` event to an order stream the adapter feeds from, and watch the worker
react: the manager's `fulfillment-<orderId>` stream gains a `FulfillmentObserved` event and the
order stream gains an `OrderPacked` event. Reacting to the *same* source event twice leaves both
streams unchanged the second time — the duplicate is collapsed.
