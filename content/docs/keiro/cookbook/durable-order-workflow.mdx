A durable workflow expresses a multi-step process — with waits in the middle — as one function whose
checkpoints survive a crash. This recipe is a complete order-fulfillment workflow exercising all four
primitives, plus the registry the resume worker needs.

<Callout type="info">"jitsurei" (実例) is the family's word for a runnable worked example; this recipe is the shape of `jitsurei/src/Jitsurei/DurableWorkflow.hs`.</Callout>

## Problem

Run a linear process — reserve, wait, await an external callback, charge, ship — with journaled
checkpoints that survive restarts and idle waits, without scattering it across per-event handlers.

## Solution

```haskell
import Keiro.Workflow           (StepName (..), WorkflowId (..), Workflow, step)
import Keiro.Workflow.Sleep     (sleepNamed)
import Keiro.Workflow.Awakeable (AwakeableId, awakeableNamed)
import Keiro.Workflow.Child     (awaitChild, spawnChild)
import Keiro.Workflow.Resume    (WorkflowDef (..), WorkflowRegistry)
import Keiro.Workflow.Types     (WorkflowName (..))
import qualified Data.Map.Strict as Map

orderFulfillmentWorkflowName, shipOrderWorkflowName :: WorkflowName
orderFulfillmentWorkflowName = WorkflowName "order-fulfillment"
shipOrderWorkflowName        = WorkflowName "ship-order"

-- 1. The parent: five durable checkpoints in order.
orderFulfillmentWorkflow :: (Workflow :> es, Store :> es, IOE :> es) => OrderId -> Eff es Text
orderFulfillmentWorkflow orderId = do
  _reservation <- step (StepName "reserve-inventory") (reserveInventory orderId)
  sleepNamed (StepName "cooling-off") 300                  -- durable 5-minute wait (a keiro_timers row)
  (awkId, awaitPayment) <- awakeableNamed (StepName "payment-webhook")
  _ <- step (StepName "send-payment-link") (sendPaymentLink orderId awkId)
  payment <- awaitPayment                                  -- suspend until the webhook signals
  _charge <- step (StepName "charge") (chargeCard orderId payment)
  childHandle <- spawnChild shipOrderWorkflowName (shipChildId orderId) (shipOrderWorkflow orderId)
  awaitChild childHandle                                   -- the tracking number is the result

-- 2. The child: an ordinary one-step workflow with its own execution id.
shipOrderWorkflow :: (Workflow :> es, IOE :> es) => OrderId -> Eff es Text
shipOrderWorkflow orderId = step (StepName "create-shipment") (createShipment orderId)

-- 3. Child id: reusing the same child (name,id) attaches to the existing execution.
shipChildId :: OrderId -> WorkflowId
shipChildId orderId = WorkflowId (orderIdText orderId <> "-ship")

-- 4. The external signaller receives the opaque AwakeableId from sendPaymentLink.
sendPaymentLink :: OrderId -> AwakeableId -> Eff es ()
sendPaymentLink = ...

-- 5. The registry the resume worker re-invokes through — parent AND child.
appRegistry :: (Store :> es, IOE :> es) => WorkflowRegistry es
appRegistry = Map.fromList
  [ (orderFulfillmentWorkflowName, WorkflowDef (\(WorkflowId i) -> orderFulfillmentWorkflow (OrderId i)))
  , (shipOrderWorkflowName,        WorkflowDef (\(WorkflowId i) -> shipOrderWorkflow (OrderId i)))
  ]
```

## How it runs

- **First call** to `runWorkflow orderFulfillmentWorkflowName (WorkflowId (orderIdText orderId)) (orderFulfillmentWorkflow orderId)`
  reserves inventory, arms the cooling-off timer, and returns `Suspended`.
- A **timer worker** (`runWorkflowTimerWorker`) fires the sleep; the **resume worker**
  (`resumeWorkflowsOnce defaultWorkflowResumeOptions appRegistry`) replays past it and parks on the
  payment awakeable.
- The **payment webhook** calls `signalAwakeable awakeableId confirmation` with the id it received from
  `sendPaymentLink`; the next resume pass charges, spawns and drives the `ship-order` child, and the
  parent completes with the child's tracking number.

Every committed checkpoint short-circuits on replay. Make each step action idempotent for the crash
window before its journal append commits, especially external calls such as `sendPaymentLink`.

The application layer that calls `runWorkflow` or `resumeWorkflowsOnce` includes
`Error StoreError` in its effect row. Journal errors remain visible there. Snapshot writes happen
after a checkpoint commits and are advisory: with metrics enabled, a failed write increments
`keiro.snapshot.write.failures`, while the committed checkpoint remains the result and the next run
replays the journal.
