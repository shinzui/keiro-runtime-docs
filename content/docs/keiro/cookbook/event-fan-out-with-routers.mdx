---
title: "Fan out an event with a router"
description: "Resolve a changing recipient set while keeping redelivery idempotent by target identity."
---

**Problem.** One source event must become a command on a target set held in a read model. Resolution
may reorder targets between retries, and a partial attempt must not move one target's deterministic
ID onto another stream.

**Solution.** Use `Router`, whose current IDs include the resolved target stream and same-stream
occurrence.

## Define the router

```haskell
chapterRouter :: (IOE :> es, Store :> es) => Router Transaction ChapterPhi ... es
chapterRouter = Router
  { name = "chapter-transaction-router"
  , key = \transaction -> transactionIdText (transaction ^. #transactionId)
  , resolve = \transaction -> do
      queryResults <-
        traverse (runQuery Nothing areaChaptersReadModel) (transaction ^. #areas)
      chapters <- requireAllQueries queryResults
      pure
        [ PMCommand
            { target = chapterStream chapter
            , command = RecordTransaction (transaction ^. #transactionId)
            }
        | chapter <- nub chapters
        ]
  , targetEventStream = chapterEventStream
  , targetProjections = const []
  }
```

Register `areaChaptersReadModel` at startup. `requireAllQueries` is an application failure boundary:
never map a read-model error to an empty target list, or the worker can acknowledge a source event
that should have routed somewhere.

`targetProjections = const []` selects append-only target dispatch. Return qualified inline
projections when each target's event and read-model update must commit together.

## Run with conservative policy

```haskell
options = defaultWorkerOptions
  & #rejectedCommandPolicy .~ RejectedHalt

runRouterWorkerWith
  options
  defaultRunCommandOptions
  chapterRouter
  adapter
  decodeMessage
```

Choose `RejectedDeadLetter` only when acknowledging the source while retaining one durable
`keiro.keiro_dead_letters` row per rejected target is an intentional recovery policy. Use
`RejectedSkip` only for an explicitly discardable outcome.

## Verify reordering

Suppose the first attempt resolves `[chapter-a, chapter-b]`, commits A, then stops. Redeliver the
same source event with `[chapter-b, chapter-a]`:

- B derives its B-target ID and appends;
- A derives its A-target ID and becomes `PMCommandDuplicate`;
- neither ID depends on the new list position.

If a third target appears later, it receives a new dispatch. That union-of-attempts behavior is the
router contract; use a process manager if recipients must be frozen at first observation.

## Verify collision safety

Create the attempted ID in a different stream and dispatch to chapter A. The global duplicate is
not enough: `confirmBenignDuplicate` checks A, finds no matching event there, and returns
`PMCommandFailed chapterA error`. The worker halts without acknowledging. This is the difference
between target-scoped idempotency and silently losing a cross-stream command.

  both code and resolver output change together.
</Callout>
