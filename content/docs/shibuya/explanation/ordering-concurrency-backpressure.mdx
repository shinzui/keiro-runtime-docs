---
title: "Ordering, concurrency, and backpressure"
description: "How ordering guarantees constrain the runner's concurrency mode and how bounded inboxes prevent unbounded memory growth."
---

shibuya separates the ordering guarantee from the execution strategy.

`OrderingPolicy` describes the guarantee the source requires:

```haskell
data OrderingPolicy
  = StrictInOrder
  | PartitionedInOrder
  | Unordered
```

`Concurrency` describes how the runner invokes handlers:

```haskell
data Concurrency
  = Serial
  | Ahead Int
  | Async Int
```

`StrictInOrder` must use `Serial`. The source is a single ordered stream, so running handlers ahead
of one another could commit or dead-letter later messages before earlier messages are settled.
`validatePolicy` rejects `StrictInOrder` with `Ahead` or `Async`.

`PartitionedInOrder` is for backends that preserve order inside a partition but allow different
partitions to move independently. Single-message processors enforce this with partition-keyed
dispatch: equal `Envelope.partition` values run and finalize in arrival order, while distinct
partitions can use the configured concurrency.

`Unordered` means messages can be processed independently. It can run with `Serial`, `Ahead n`, or
`Async n`.

Batching processors schedule by `BatchKey`, not by `Envelope.partition`, so `runApp` rejects
`PartitionedInOrder` with `Ahead n` or `Async n` for `BatchingProcessor`.

Backpressure is independent of those policies. The internal supervised runner creates a bounded NQE
inbox of the configured size. The ingester reads from `Adapter.source` and blocks when the inbox is
full, so a fast backend stream cannot grow memory without bound while handlers are slow.
