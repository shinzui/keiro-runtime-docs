---
title: shibuya ⇄ kiroku adapter
description: Feed Kiroku subscriptions through shibuya processors while Kiroku owns durable ordering, retry, dead letters, filters, checkpoints, and consumer-group membership.
---

`shibuya-kiroku-adapter` turns a Kiroku subscription into a shibuya `Adapter`.
Kiroku still owns the event log, catch-up/live handoff, retry policy, dead
letters, durable checkpoints, filters, and consumer-group membership. shibuya
owns processor supervision, handler execution, metrics, and coordinated
shutdown.

The adapter is intentionally thin. It builds a Kiroku `subscriptionAckStream`,
converts each `RecordedEvent` into a shibuya `Envelope RecordedEvent`, and waits
for shibuya's `AckDecision` before letting the Kiroku worker advance.

## Runtime Shape

`KirokuAdapterConfig` names the subscription and target:

| Field | Default / shape | Effect |
| --- | --- | --- |
| `subscriptionName` | required | Durable Kiroku subscription identity. |
| `subscriptionTarget` | required | `AllStreams` or a `Category`. |
| `batchSize` | `100` | Batch size passed to Kiroku subscription config. |
| `bufferSize` | `256` | Subscription stream buffer. |
| `queueCapacity` | `16` | Live publisher queue capacity for all-streams subscriptions. |
| `consumerGroup` | `Nothing` | Optional consumer-group member metadata. |
| `eventTypeFilter` | all event types | Kiroku-side event type filter. |
| `selector` | accept all | Kiroku-side predicate. |

`eventTypeFilter` and `selector` run before shibuya sees an event. Filtered
events are skipped and checkpointed past by Kiroku; they are not retried,
dead-lettered, or exposed to the handler.

## Ack Mapping

Every delivered event is ack-coupled:

| shibuya decision | Kiroku result |
| --- | --- |
| `AckOk` | `Continue`; checkpoint advances past the event. |
| `AckRetry delay` | `Retry delay`; the same event is redelivered under Kiroku retry policy. |
| `AckDeadLetter reason` | `DeadLetter`; Kiroku writes the event to dead letters and advances. |
| `AckHalt` | Cancel the subscription; the in-flight event is not checkpointed and replays on restart. |

The bridge carries a zero-based `attempt` counter on the envelope. First
delivery is attempt `0`; each Kiroku retry increments it.

The consumer-group helper wraps synchronous handler exceptions as
`AckRetry (RetryDelay 1)`. Asynchronous exceptions still propagate so
cancellation and shutdown remain responsive.

## Ordering and Backpressure

Kiroku delivers each worker's events in ascending global position and blocks on
the shibuya ack before producing the next event. That means the adapter source
does not race checkpoint advancement ahead of handler completion.

For a non-group `AllStreams` adapter, the publisher-side queue inherits Kiroku's
default pause-and-resume overflow behavior. If the queue fills, live fan-out
pauses and the worker catches up again from the durable checkpoint; events are
not dropped. Category adapters and consumer-group members fetch filtered
batches from the database when publisher position advances, so the
ack-coupled handler directly throttles those fetch loops.

## Envelope Mapping

| Kiroku field | Shibuya envelope field |
| --- | --- |
| `eventId` | `messageId`. |
| `globalPosition` | `cursor`. |
| `createdAt` | `enqueuedAt`. |
| `metadata.traceparent` / `tracestate` | `traceContext`. |
| Redelivery counter | `attempt`. |
| `RecordedEvent` | `payload`. |

Envelope attributes include `messaging.system = kiroku`, the subscription name,
optional consumer-group member, event type, and global position. `headers` is
`Nothing`; Kiroku metadata remains on the `RecordedEvent` payload and selected
trace context is lifted into `traceContext`.

## Consumer Groups

`kirokuConsumerGroupProcessors` builds a size-`N` group as `N` shibuya
processors named `<subscriptionName>-member-<m>`. Each member has its own Kiroku
checkpoint under `(subscriptionName, member)`.

The adapter validates policy before opening subscriptions. Each member must run
serially, and the group is presented as `PartitionedInOrder`: events from the
same originating stream hash to the same member, preserving per-stream order,
while different streams can run on different members.

If processor creation fails part-way through a group, already-created adapters
are cleaned up before the error escapes.
