`shibuya-kafka-adapter` turns a Kafka consumer session into a shibuya
`Adapter es (Maybe ByteString)`. Kafka owns topics, partitions, offsets, record
headers, and broker delivery. shibuya owns handler execution and ack decisions.
The adapter maps those ack decisions to local offset bookkeeping and commits
stored offsets during shutdown.

## Runtime Shape

The constructor is `kafkaAdapter` from `Shibuya.Adapter.Kafka`:

```haskell
kafkaAdapter ::
  ( KafkaConsumer :> es
  , Error KafkaError :> es
  , IOE :> es
  ) =>
  KafkaAdapterConfig ->
  Eff es (Adapter es (Maybe ByteString))
```

The adapter runs inside the `KafkaConsumer` effect supplied by the caller's
consumer properties. `KafkaAdapterConfig` supplies the topic list, poll timeout,
and batch size:

| Field | Default | Effect |
| --- | --- | --- |
| `topics` | required | Topics subscribed by this adapter. |
| `pollTimeout` | `Timeout 1000` | Kafka poll timeout in milliseconds. |
| `batchSize` | `100` | Maximum records handled from a poll cycle. |

The adapter name is `kafka:<topics>`. The adapter's `shutdown` action marks the
source as stopped and commits all offsets accumulated by successful finalization.
Call `shutdown` while the `KafkaConsumer` effect is still in scope.

Use `kafkaAdapterWith` when you need caller-owned `KafkaAdapterState`, usually so
the same state can be passed to `kafkaRebalanceHandler` in the consumer
properties before the consumer is created.

## Ack Mapping

Kafka cannot make a consumed record invisible and then re-show it after a delay.
The adapter therefore maps ack decisions to offset behavior:

| shibuya decision | Kafka adapter behavior |
| --- | --- |
| `AckOk` | Store this record's offset for a later commit. |
| `AckRetry delay` | Wait for the requested delay, record a retry barrier, and seek the partition back to this record's offset. |
| `AckDeadLetter reason` | Warn to stderr, then store the offset; the adapter does not publish a DLQ record. |
| `AckHalt` | Pause the record's partition and do not store the offset. |

`AckRetry` is a local redelivery mechanism, not a durable retry topic. The
adapter seeks the partition back and drops stale records above the retry barrier
that were already buffered. It also serializes poll, seek, store, pause, and
commit operations behind one consumer lock because librdkafka consumer handles
are not safe to poll and seek/store concurrently from shibuya's ingester and
finalizer paths.

`AckDeadLetter` moves the offset forward. If your service needs a durable DLQ,
publish to a DLQ topic from the handler before returning `AckDeadLetter`, and
make that side effect idempotent.

`AckHalt` is the only decision that deliberately withholds the offset. The
partition is paused for the current session. On a new consumer session, the
uncommitted record can be delivered again.

## Envelope Mapping

The adapter preserves Kafka's partition and header identity:

| Kafka record | Shibuya envelope |
| --- | --- |
| Topic, partition, offset | `messageId = "<topic>-<partition>-<offset>"`. |
| Offset | `cursor = CursorInt offset`. |
| Partition | `partition`. |
| Timestamp | `enqueuedAt`. |
| `traceparent` / `tracestate` headers | `traceContext`. |
| Kafka headers | `headers = Just [...]`, preserving order and duplicate keys. |
| Record value | `payload = Maybe ByteString`. |

Envelope attributes include `messaging.system = kafka`, destination partition,
and message offset. `attempt` is `Nothing`; Kafka does not expose a native
delivery attempt counter to the adapter.

## Errors and Shutdown

The source polls in batches and skips non-fatal poll outcomes. Fatal Kafka
errors are thrown through `Error KafkaError`, so normal shibuya supervision can
decide whether the processor should stop or restart.

Offset commits happen through the adapter's stored-offset ledger. A crash before
commit can replay records whose handlers already ran. Treat handlers as
at-least-once and idempotent even when they return `AckOk`.

The adapter must be run with serial processing. librdkafka stores the highest
offset per partition without gap tracking, so concurrent finalization can commit
past an earlier record that requested retry or halt. Use `Serial` for Kafka
processors until a gap-tracking commit layer exists.

Ack finalizer failures are retried a small number of times. Fatal Kafka errors,
or exhausted finalizer retries, are recorded in adapter state and then surfaced
through the source stream so shibuya supervision sees a processor failure.

## Examples

The `shibuya-kafka-adapter-jitsurei` package includes runnable examples for the
main shapes:

- `BasicConsumer` for a single-topic consumer.
- `MultiTopic` for topic fan-in.
- `OffsetManagement` for commit behavior.
- `MultiPartition` for partition-aware processing.
- `FatalErrorDemo` for fatal error surfacing.
- OpenTelemetry examples for trace/header propagation.
