keiro (経路, the event-sourcing framework) persists its events and workflow state
through kiroku (記録, the append-only PostgreSQL event store). This page documents
how to wire keiro to kiroku.

Current reviewed compatibility is Keiro `0.3.0.0` with `kiroku-store 0.3.0.1`.
Existing databases must follow both projects' one-time history and schema
upgrade paths before the native migration plan runs.

## What keiro stores in kiroku

keiro writes every aggregate, process-manager, router, outbox/inbox, timer, projection, and workflow
fact through Kiroku's PostgreSQL-backed store. Kiroku owns the append contract: stream names,
`ExpectedVersion`, duplicate event ids, global positions, transactions, and subscription delivery.
Keiro layers typed domain APIs and worker run loops over that store.

| Keiro surface | Kiroku responsibility |
| --- | --- |
| `runCommand` | Reads a stream, appends emitted events with optimistic concurrency, and retries conflict-safe commands. |
| Event streams and snapshots | Store stream events plus optional keiro snapshot rows keyed by stream id/version. |
| Async projections and read models | Supply subscription checkpoints and recorded events; Keiro adds deduplication, registry fencing, consistency waits, and rebuild coordination. |
| Process managers and routers | Consume Kiroku subscriptions and dispatch deterministic target commands. |
| Inbox/outbox | Use framework tables in the `keiro` schema and Kiroku transactions to make integration-message consume/publish paths replay-safe. |
| Durable workflows | Journal workflow steps into `wf:<name>-<id>` streams and maintain workflow lifecycle tables beside them. |

## Migration ownership

Kiroku's `kirokuMigrations` component owns the event-store objects in `kiroku`.
Keiro's `keiroMigrations` component owns framework tables in `keiro` and
declares its dependency on Kiroku. Application read models remain a third
component in an application-owned schema.

Resolve all of them in one final plan—Kiroku before Keiro, then application
components—and run that plan with connection settings before opening the
Kiroku pool. A database created by the old Codd runner needs a one-time verified
history import before native `up`; a Keiro 0.1 database also needs the documented
framework-table relocation. The import changes bookkeeping, not schema DDL.

Follow [Run the Keiro migrations](/docs/keiro/how-to/run-migrations) for fresh
composition and [Upgrade a 0.1 database](/docs/keiro/how-to/upgrade-a-0-1-database)
for the relocation and cutover path.

## Stream naming and categories

Use Keiro's typed stream/category helpers rather than hand-building Kiroku stream names. The current
safe path is `category` / `entityStream` / `streamName` from the Keiro stream surface, backed by
Kiroku's `CategoryName` and `StreamName` validation. `mkWorkflowName` and `mkWorkflowId` play the same
role for workflow journals: use the validated constructors when creating new workflow identities.

For details, read [EventStream and Stream](/docs/keiro/reference/event-stream-and-stream) and
[Kiroku core types](/docs/kiroku/reference/core-types).

## Transactions and source consistency

Keiro command transactions are ordinary Kiroku transactions. The important rule is that the local
side effect you need to be atomic with an event append must happen in the same store transaction:
read-model updates can run with a command append, outbox rows can be inserted with domain events, and
transactional job enqueueing must use the inline SQL pattern described in the Keiro cookbook rather
than a separate PGMQ send.

Kiroku now exposes point lookups such as `eventExistsInStream`, stricter stream-name validation,
typed append/link failures, and pipelined multi-stream append. Keiro's current command/read-side docs
use those semantics directly.

## Read-model schemas and checkpoints

Keep the three namespaces distinct:

- Kiroku owns event-store and subscription tables in `kiroku`.
- Keiro owns registry and projection-dedup metadata in `keiro`.
- The application owns read-model tables in a declared schema such as `app_reads`.

Set each `ReadModel.schema`, qualify its create/read/write SQL with `qualifiedTableName` or
`qualifyTable`, and call `registerReadModel` explicitly at startup. `withProjectionSchema` adds the
application schema to a Kiroku connection's `extraSearchPath` while deliberately leaving Kiroku's
`schema = "kiroku"`; changing that field would also change its notification channel. Schema creation
is application migration work—`ensureProjectionSchema` is only an opt-in helper.

For async workers, Keiro reads Kiroku's `subscriptions.last_seen` to implement `PositionWait` and
`Strong`. Pair a category subscription with `StrongScope = CategoryHead category`; `EntireLog`
targets `$all` and can be unreachable for a worker whose category cursor does not advance on
unrelated traffic. During a rebuild, `AsyncFenced` must not be acknowledged or checkpointed.

## Subscriptions and workers

Kiroku owns subscription checkpoints, retry/dead-letter behavior, consumer groups, filtering, and
trace context. Keiro's process-manager/router workers consume those subscriptions and classify their
own dispatch results: transient store/command failures retry, deterministic command failures halt,
and duplicate target command ids are treated as idempotent success.

For shibuya supervision over Kiroku subscriptions, use the
[shibuya ⇄ kiroku adapter](/docs/integrations/shibuya-kiroku-adapter). For Keiro sharded category
subscriptions, use [Subscription sharding](/docs/keiro/reference/subscription-sharding).
