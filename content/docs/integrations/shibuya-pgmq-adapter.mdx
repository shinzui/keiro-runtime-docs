title: shibuya ⇄ pgmq adapter
description: Consume a pgmq queue through shibuya's supervised processors with visibility-timeout leases, retry visibility changes, DLQ routing, FIFO reads, topic helpers, and trace propagation.
---

`shibuya-pgmq-adapter` turns a [pgmq](/docs/pgmq) queue into a shibuya
`Adapter es Value`. PGMQ owns the physical queue tables, visibility timeout,
archive table, topic routing tables, FIFO group indexes, and JSONB message
payload. shibuya owns handler execution, runner policy, ack finalization,
metrics, tracing, and shutdown.

Current compatibility is `shibuya-pgmq-adapter 0.12.0.0`, the `pgmq-* 0.4`
package family, and `shibuya-core 0.8.0.1`. The adapter's public constructor is
unchanged from 0.11; the upgrade changes schema installation and fixes idle
shutdown.

<Callout type="info">
  **Building a keiro app?** You usually do not wire this adapter directly.
  [`keiro-pgmq`](/docs/integrations/keiro-with-pgmq) wraps it in a typed `Job`
  layer that derives queue names, constructs `PgmqAdapterConfig`, decodes the
  JSON payload, and maps `JobOutcome` back to shibuya ack decisions.
</Callout>

## Install the schema before starting the adapter

The adapter consumes an existing PGMQ schema; it does not install database
objects. With `pgmq-migration` 0.4, resolve `Pgmq.Migration.pgmqMigrations`,
compose it with the application's other components through `migrationPlan`, and
run that final plan through `runMigrationPlan` or a standard `pg-migrate-cli`
host.

The migration runner takes `Hasql.Connection.Settings.Settings`, not the
`Pool` stored in `PgmqAdapterEnv`. It opens its own connection so the migration
advisory lock is not tied to the runtime pool. Once the migration gate succeeds,
create the pool, reconcile queue topology, and construct the adapter.

An existing database created by `pgmq-migration` 0.3 or earlier needs a one-time
verified import from `public.schema_migrations` before native `up`. The native
runner never discovers predecessor history implicitly, and the import adopts
bookkeeping without replaying PGMQ DDL. Follow [Install or migrate the PGMQ
schema](/docs/pgmq/how-to/install-or-migrate-schema) for the fresh and
predecessor-ledger paths.

## Runtime shape

The public constructor is `pgmqAdapter` from `Shibuya.Adapter.Pgmq`:

```haskell
pgmqAdapter ::
  ( Pgmq :> es
  , Error PgmqRuntimeError :> es
  , IOE :> es
  , Tracing :> es
  ) =>
  PgmqAdapterEnv ->
  PgmqAdapterConfig ->
  Eff es (Either PgmqConfigError (Adapter es Value))
```

Build the environment with `mkPgmqAdapterEnv pool`, or set the
`onAutoDeadLetter` / `onAckFailure` callbacks yourself when operators need a
side channel for automatic DLQs and exhausted ack retries. `pgmqAdapter`
validates `PgmqAdapterConfig` before it starts polling; invalid configs return
`Left PgmqConfigError`.

The adapter name is `pgmq:<queueName>`. The source polls PGMQ, converts each
leased `Message Value` into a shibuya `Envelope Value`, and attaches an
`AckHandle` that performs the matching PGMQ operation. Messages already in
processing are redelivered by PGMQ if their visibility timeout expires without
a final ack.

Use the adapter directly when you want the raw PGMQ payload and queue semantics.
Use `keiro-pgmq` when the queue is a typed keiro background-job queue.

## Configuration

Start with `defaultConfig` and override the fields that match your queue:

```haskell
import Shibuya.Adapter.Pgmq

case (parseQueueName "reservation_work", parseQueueName "reservation_work_dlq") of
  (Right reservationWork, Right reservationWorkDlq) ->
    let cfg =
          defaultConfig reservationWork
            { batchSize = 25
            , visibilityTimeout = 60
            , maxRetries = 5
            , deadLetterConfig =
                Just (directDeadLetter reservationWorkDlq True)
            }
     in ...
  _ ->
    error "invalid queue name"
```

| Field | Default | Effect |
| --- | --- | --- |
| `queueName` | required | Physical PGMQ queue to read. |
| `visibilityTimeout` | `30` seconds | Lease duration assigned by each read. |
| `batchSize` | `1` | Maximum messages requested per poll. |
| `polling` | `StandardPolling` | Standard sleep-after-empty polling or PGMQ long polling. |
| `pollRetry` | 5 attempts, 100 ms to 5 s backoff | Retries transient read failures before surfacing `PgmqRuntimeError`. |
| `ackRetry` | 5 attempts, 100 ms to 5 s backoff | Retries transient ack, visibility, and DLQ operations before calling `onAckFailure`. |
| `deadLetterConfig` | `Nothing` | Optional direct queue or topic route for `AckDeadLetter`. |
| `haltVisibilityTimeout` | `Nothing` | Optional visibility timeout for `AckHalt`; falls back to `visibilityTimeout`. |
| `maxRetries` | `3` | Maximum PGMQ read count before the adapter auto-dead-letters. |
| `fifoConfig` | `Nothing` | Enables grouped FIFO read strategies. |
| `prefetchConfig` | `Nothing` | Adds a bounded prefetch buffer in front of the processor. |

When prefetch is enabled, size the buffer against the lease:
`bufferSize * batchSize * averageProcessingTime` should stay below
`visibilityTimeout`, or a message can become visible again before the handler
acks it. Prefetched messages that are read but not handed to a handler before
shutdown are not lost; PGMQ redelivers them after their visibility timeout.

## Graceful shutdown timeline

Call `stopAppGracefully` while the PGMQ effect and pool are still in scope. In
0.12, an idle queue follows the same finite shutdown path as a busy one:

1. shibuya invokes the adapter's `shutdown`, which sets its stop gate.
2. The next returned PGMQ poll chunk reaches that gate **before** empty chunks
   are filtered. An empty chunk therefore ends the source instead of starting
   another poll forever. A non-empty chunk that has not reached shibuya is
   released before the source ends.
3. The ingester observes end-of-stream. The processor drains its bounded inbox
   and finalizes in-flight messages.
4. `stopAppGracefully` returns `True` when every processor finishes before
   `drainTimeout`; otherwise it stops the master and returns `False`.

The current poll must return before step 2. Standard polling is therefore
bounded by its poll/sleep cadence; long polling is bounded by the configured
server wait. With prefetch enabled, batches already held upstream of the stop
gate can remain invisible for at most their visibility timeout and are then
redelivered. They are not lost.

## Ack Mapping

PGMQ's lease model maps cleanly onto shibuya's four ack decisions:

| shibuya decision | PGMQ operation |
| --- | --- |
| `AckOk` | `deleteMessage`; the leased row leaves the live queue. |
| `AckRetry delay` | `changeVisibilityTimeout` using the requested retry delay. |
| `AckDeadLetter reason` without DLQ config | `archiveMessage`; the row moves to PGMQ's archive table. |
| `AckDeadLetter reason` with DLQ config | Write a DLQ payload to a direct queue or topic route, then `deleteMessage` the original. |
| `AckHalt` | Change visibility to `haltVisibilityTimeout`, or to `visibilityTimeout` when unset, and stop the processor. |

Before a handler runs, the adapter compares PGMQ's `readCount` with
`maxRetries`. If the count is already too high, it applies
`AckDeadLetter MaxRetriesExceeded` itself and does not deliver that message to
the handler. The count is PGMQ's broker-side delivery count, so it survives
process restarts.

`AckHandle.finalize` is idempotent per delivery. If shibuya retries the same
finalization after a transient failure, the adapter re-runs only unfinished
work; once a decision succeeds, later calls are no-ops. The DLQ path writes the
DLQ message and deletes the original in one database transaction.

## Dead Letters

`DeadLetterConfig` has two target forms:

```haskell
directDeadLetter :: QueueName -> Bool -> DeadLetterConfig
topicDeadLetter  :: RoutingKey -> Bool -> DeadLetterConfig
```

`DirectQueue` writes directly to a named PGMQ queue. `TopicRoute` publishes
through the adapter's topic-routing helper so bindings decide which queues see
the dead-letter payload. With `includeMetadata = True`, the DLQ payload includes
the original message id, original enqueue and read timestamps, read count,
headers, payload, and the shibuya dead-letter reason.

Trace propagation is deliberate. The DLQ write uses the current consumer span as
the outgoing trace context, while the original upstream `traceparent` and
`tracestate` are preserved under `x-shibuya-upstream-traceparent` and
`x-shibuya-upstream-tracestate` headers.

## FIFO and Topic Helpers

When `fifoConfig` is present, the adapter reads grouped messages instead of the
plain PGMQ queue stream. `ThroughputOptimized` lets busy groups dominate.
`RoundRobin` rotates across groups for fairer processing. In both cases, the
message payload remains raw JSON `Value`; FIFO only changes how the adapter
chooses leased rows.

The package also exposes helpers for topic bindings:

```haskell
bindQueueTopics
unbindQueueTopics
listQueueTopicBindings
testTopicRouting
```

These helpers are adapter-side conveniences over pgmq-hs topic routing. They do
not change shibuya's processor contract; they make it easier to route normal
messages or DLQ messages through PGMQ topics.

## Envelope Mapping

The adapter preserves the PGMQ cursor and lease context:

| PGMQ source | Shibuya envelope |
| --- | --- |
| `MessageId` | `messageId` as text. |
| PGMQ message id integer | `cursor = CursorInt ...`. |
| `x-pgmq-group` header | `partition`. |
| `enqueuedAt` | `enqueuedAt`. |
| `traceparent` / `tracestate` headers | `traceContext`. |
| `readCount - 1` | zero-based `attempt`. |
| JSON body | `payload`. |

`headers` is intentionally `Nothing`. PGMQ stores headers as JSONB, which is
useful for routing and metadata but is not an ordered broker header stream.
Trace context and the FIFO group header are lifted into dedicated envelope
fields; the raw payload stays as the handler's `Value`.

## Operational Notes

- The adapter is at-least-once. A handler must be idempotent because crashes,
  retry visibility changes, and lease expiry can redeliver the same message.
- `AckRetry` changes broker visibility; it does not allocate an in-process retry
  buffer.
- `AckHalt` is a stop signal, not a pause. Tune `haltVisibilityTimeout` when the
  default queue visibility timeout is too short for operator intervention.
- Standard polling is simpler for steady workloads. Long polling reduces idle
  query churn when the queue is mostly empty.
- PGMQ queue provisioning, archive reads, purges, metrics, and schema migration
  are documented in the [pgmq reference](/docs/pgmq/reference/hasql-api), the
  [queue lifecycle guide](/docs/pgmq/how-to/create-send-read-finish), and the
  [startup configuration guide](/docs/pgmq/how-to/configure-queues-at-startup).
